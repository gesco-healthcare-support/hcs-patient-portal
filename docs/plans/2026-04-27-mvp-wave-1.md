# Wave 1 plan -- minimum to make the MVP demo end-to-end visible

## Context

Wave 0 landed the foundation (`feat/mvp-wave-0`, 2 commits ahead of main:
6b699da + 459451b on top of pre-Wave-0 base 324c510). Wave 1 turns the foundation
into a visible UI demo. Adrian's demo target:

> Booker logs in -> submits appointment request -> office reviews queue ->
> approve / reject / send-back-for-info -> documents upload -> notifications fire.

Cut criterion (carries forward from Wave 0): keep ONLY caps where the app
cannot demo without them, OR a downstream W1/W2/W3 cap has a hard `Blocked by`
edge per `docs/implementation-research/dependencies.md`. Anything pure security
hardening, pure policy, pure docs, pure test coverage, or pure UX polish goes
on the ledger (`docs/plans/deferred-from-mvp.md`).

## Goal

Land 5 capability briefs on a single feature branch (`feat/mvp-wave-1` off
`main`). After Wave 1 lands, an authenticated booker can submit an appointment;
the practice admin can review, approve / reject / send-back; documents can be
uploaded; and the 4 transition emails fire. No PR per cap. No tests (all test
debt deferred to ledger under `new-qual-01-critical-path-test-coverage`). No
security hardening (still on Wave 0 ledger).

## Approach

- **Single branch, no PR ceremony.** `feat/mvp-wave-1` already exists off
  `main` (parent 324c510); the deferred-from-mvp.md ledger commit (9a0ca53) is
  its first commit. Linear sequence of checkpoint commits as work progresses.
- **OLD as reference for VALUES + ALGORITHMS only.** Per Wave 0 rule. NEW
  uses ABP 10.0.2 / .NET 10 / Riok.Mapperly / OpenIddict / Hangfire stack.
- **Sequencing.** Housekeeping (W1-0) first to sweep Wave 0 carry-overs;
  state-machine (W1-1) next as the spine; templates (W1-2) feeds notifications
  for state-machine; documents (W1-3) is the longest cap; search-listview
  (W1-4) is the smallest finishing touch and pulls forward from W2 per the
  Wave 0 plan's preview.
- **Deferred-ledger discipline.** After EACH cap, if anything was cut, append
  to `docs/plans/deferred-from-mvp.md` under `## From Wave 1`. Non-negotiable.
- **Constraints honoured throughout** (per Wave 0): ASCII only,
  `DOTNET_ENVIRONMENT=Development` for every dotnet command, never `ng serve`,
  never edit `angular/src/app/proxy/`, never edit `appsettings.secrets.json`,
  service start order via docker compose.

**MVP-active transition subset.** Stateless library wires ALL 14 states + all

- `POST /api/app/appointments/{id}/approve` (Pending|AwaitingMoreInfo -> Approved)

- `POST /api/app/appointments/{id}/send-back` (Pending -> AwaitingMoreInfo;

Cancel / reschedule transitions ARE configured in the Stateless graph but
NOT exposed -- Wave 3 adds the endpoints. This keeps the machine "complete"
so Wave 3 is purely API + Angular work.

**Slot-status sync** (per T11 sync table):
| Appointment transition | Slot transition |
| --- | --- |
| (none) -> Pending (booker submits) | Available -> Reserved |
| Pending -> AwaitingMoreInfo | (no change; stays Reserved) |
| AwaitingMoreInfo -> Pending (auto) | (no change; stays Reserved) |
| Pending -> Approved | Reserved -> Booked |
| Pending -> Rejected | Reserved -> Available |

## Risk / rollback

- Single branch isolated from `main`. Rollback: `git checkout main && git
  branch -D feat/mvp-wave-1` discards everything; no migration runs against
  anything but the docker LocalDB / SQL Server container.
- Stateless graph + slot cascade: blast radius medium-high. The `Appointment`
  aggregate's `AppointmentStatus` setter MUST narrow to `internal` to force
  callers through the manager. Existing callers were audited in the brief
  (only `CreateAsync` constructor path); confirm during execution that no
  Mapperly target depends on the public setter.
- Document upload + DB BLOB: blast radius medium. Wrap upload in a single
  `[UnitOfWork]` so blob save + entity insert succeed-or-fail together (no
  orphaned blobs).
- Stack volume: 5 caps in Wave 1 is more than W0's 8 small caps in lines of
  code (W1-3 alone is several hundred lines). If execution stretches past
  ~10 calendar days at part-time pace, evaluate cutting W1-3's Angular polish
  or W1-4 entirely (W1-4 is 0.5d so leave; W1-3's Angular is the bigger lever).
- Tracked debt (NOT lost): the deferred-from-mvp.md ledger captures every cut.
