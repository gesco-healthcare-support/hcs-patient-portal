# BUG-003 — Register endpoint returns 403 for duplicate email

## Severity
medium

## Status
**FIXED in PR #197** — error code now mapped to HTTP 400 via `AbpExceptionHttpStatusCodeOptions`.

## Affected role
Patient (and all external-user registration)

## Steps to reproduce
Same as [[BUG-001]] — POST duplicate email to `/api/public/external-signup/register`.

## Expected (REST + OpenAPI conventions)
Duplicate-email is a validation failure → 400 Bad Request or 409 Conflict (422 Unprocessable Entity also defensible). 403 Forbidden semantically means *"you do not have permission to perform this action"* — appropriate for a permission gate, not a data-validation error. Returning 403 misleads client error handlers (which often special-case 403 for re-auth flows).

## Actual (pre-fix)
HTTP 403 Forbidden, body `{"error":{"code":null,"message":"Email address is already used: ..."}}`.

## Evidence
- Network: POST `/api/public/external-signup/register` → 403 Forbidden
- `_abperrorformat: true` header set
- Correlation ID `68469a17cc0a4032aa9c84cd456cd769`

## Root cause
`UserFriendlyException` defaults to HTTP 403 in ABP 10.0.2.

## Recommended fix (applied in PR #197)
Map `CaseEvaluationDomainErrorCodes.ExternalSignupDuplicateEmail` to `HttpStatusCode.BadRequest` via:
```csharp
Configure<AbpExceptionHttpStatusCodeOptions>(options =>
{
    options.Map(
        CaseEvaluationDomainErrorCodes.ExternalSignupDuplicateEmail,
        HttpStatusCode.BadRequest);
});
```
Applied in both `CaseEvaluationHttpApiHostModule.cs` and `CaseEvaluationAuthServerModule.cs`.
