## 1. ID & Summary

OLD had TWO distinct document reminders driven off `SchedulerDomain`:
ReminderTypes id 2 (`AppointmentPackageDocumentPending`) and id 5
(`AppointmentJointDeclarationDocumentUpload`). Each had its own stored
proc, its own external trigger (cadence), and id 5 was a dedicated track
because the Joint Declaration Form (JDF) is the gating document for AME
appointments (its absence auto-cancels the appointment).

NEW collapses both into one job. `PackageDocumentReminderJob` selects
ALL outstanding non-ad-hoc docs (Pending/Rejected) on approved
appointments inside one cutoff and publishes a `PackageDocumentReminderEto`
per row carrying an `IsJointDeclaration` bool. A JDF-not-uploaded row
therefore rides the same daily 08:30 PT package-doc pass; there is no
JDF reminder with an independent cadence. The user-visible "you still
owe the JDF" reminder DOES still fire (good), but: (a) it cannot be
timed independently of package docs, and (b) `IsJointDeclaration` is
threaded all the way to the dispatcher yet NO template content branches
on it -- so today a JDF row renders byte-identical to a package-doc row.

Class confirmed: Partial behavior. Disposition `include` -> the CSV asks
to "split a dedicated JDF reminder with own cadence". This is a
buildable, M-sized split, not a pure 1:1 port (OLD cadence is unknown).

`PackageDocumentReminderJob.cs:119-122` to avoid duplicate emails -- this
is the single most likely regression. No DB schema change required

## 4. Expected vs Actual

Expected (OLD parity intent): JDF-not-uploaded produces a reminder on a
JDF-specific schedule, recognizable as a JDF reminder, distinct from the
generic package-doc reminder track (OLD had two ReminderTypes, two procs,
two triggers).

Actual (NEW): JDF-not-uploaded produces a reminder, but only as a
side-effect of the generic `PackageDocumentReminderJob` daily pass at the
shared `PackageDocumentReminderDays` cutoff. The `IsJointDeclaration`
flag is carried through the ETO and into template variables/contextTag
but no downstream template content branches on it, so the rendered email
is identical to a package-doc reminder. No independent JDF cadence knob
is wired to a reminder (the existing `JointDeclarationUploadCutoffDays`
setting feeds the auto-cancel job instead).

Delta vs the parity doc's description: parity doc says "uses one
template-selection branch instead of two distinct proc-driven sends" --
verified there is in fact NO selection branch on the JDF flag at all
(same template both ways). Also un-noted by the parity doc: OLD's JDF
send used the 3-arg no-CC overload while package used 4-arg with CC.

## 7. Candidate Solutions

### Option A -- Dedicated JDF reminder job + JDF cadence setting, exclude JDF from the package job [effort: M] [risk: low-med]

One-line: add `JointDeclarationUploadReminderJob` (own cron + own
`JointDeclarationReminderDays`/reuse `JointDeclarationUploadCutoffDays`
setting), give it a JDF-specific ETO+handler (or branch the existing
handler), and add `&& !d.IsJointDeclaration` to the package job so JDF
rows no longer ride the package pass.
- Source: OLD code -- `SchedulerDomain.cs:204-232` (id 5 distinct
  reminder) and `ReminderTypes.cs:15`; cadence is a design choice since
  OLD cadence is not in source.
- Tradeoffs: highest fidelity to OLD's two-track model; gives the clinic
  an independent JDF schedule (the CSV's explicit ask). More surface
  (4-7 files). MUST exclude JDF from the package job or recipients get
  two emails (the key regression).
- Risk: low-med -- mostly additive; the one mutation to the existing job
  (the `!IsJointDeclaration` filter) is small but load-bearing.
- Effort: M.

### Option B -- Branch the existing handler on `IsJointDeclaration` to a JDF template, keep one job/cadence [effort: S] [risk: low]

One-line: keep `PackageDocumentReminderJob` as the single pass but make
`PackageDocumentReminderEmailHandler` dispatch a distinct JDF template
(new `NotificationTemplateConsts.Codes.JointDeclarationUploadReminder`
+ subject + en.json) when `eventData.IsJointDeclaration` is true.
- Source: OLD code -- JDF reminder existed as a distinct user-facing
  message; this restores a recognizable JDF email without a second cron.
- Tradeoffs: cheap (the flag is already plumbed; only the dispatch line
  + a template are new); recipients can tell a JDF reminder apart.
  Does NOT satisfy the CSV's "own cadence" -- still one shared schedule.
- Risk: low -- additive, no change to row selection, no duplicate-send
  risk.
- Effort: S.

### Option C -- Accept as-is (per parity doc "Acceptable as-is") [effort: none-S] [risk: none]

One-line: leave folded; document that JDF reminders ride the package
pass. Revisit only if clinic wants JDF-specific timing.
- Source: parity doc lines 163-164.
- Tradeoffs: zero work; but contradicts the CSV disposition `include`
  (full), so not the chosen path -- listed for completeness. If the
  clinic confirms one shared schedule is fine, Option B is the minimal
  honoring of "include".
- Risk: none (no change).
- Effort: none.

Recommended path for the eventual fix: Option B as the parity-honoring
minimum (distinct JDF template, one cadence), with Option A reserved for
when/if the clinic confirms it wants JDF on an independent schedule. The
CSV note "own cadence" points at A, but A's value hinges on Open
Question #1 (was OLD's JDF cadence actually different from package?),
which is unanswerable from source.

## 8. Root-Cause Hypothesis

Confidence: HIGH. Root cause is a deliberate Phase 14b scope decision,
documented in source: `PackageDocumentReminderJob.cs:30-36` and
`PackageDocumentReminderEmailHandler.cs:23-43` record that NEW
consolidated OLD's separate document reminders into one daily pass with
one shared `UploadPendingDocuments` template, carrying `IsJointDeclaration`
as a flag for FUTURE differentiation that was never wired downstream.
This was a design choice (consolidate, defer the split) rather than a
bug or oversight -- OLD itself already shared the email template across
both reminders and OLD's true cadence is absent from source, so a
faithful 1:1 port was impossible and a single configurable cutoff was
chosen instead. The gap is therefore "intentional partial port awaiting
the JDF-specific split", with the flag pre-plumbed to make that split
cheap (Option B).

## 9. Open Questions

1. Did OLD actually run the JDF reminder (id 5) on a DIFFERENT cadence
   than the package reminder (id 2)? Unknowable from source -- the
   host-side trigger config and the `spm.*` proc bodies are not in the
   repo (confirmed: not in OLD docs either). This determines whether
   Option A (own cadence) adds real value over Option B (own template,
   shared cadence). Needs a stakeholder who remembers the OLD schedule.
   (Shared with G-05-03 Open Question #1.)
2. Should a dedicated JDF reminder use its own template/subject, or is
   the shared "Please Upload Pending Documents" subject acceptable? OLD
   used the SAME `UploadPendingDocuments` template for both (verified
   `SchedulerDomain.cs:145` and `:228`), so strict parity arguably keeps
   one template -- in which case the "fold" is closer to faithful than
   the parity doc implies, and only the cadence is the real deviation.
3. OLD's JDF send used the 3-arg no-CC overload (`:229`) while package
   used 4-arg with CC (`:146`). NEW applies the tenant CC appender to
   BOTH (`PackageDocumentReminderEmailHandler.cs:116`). Is CC'ing the
   JDF reminder a deviation worth preserving, or is the unified CC
   behavior acceptable? Low impact; flag for the fix author.
4. If the split is built, the package job MUST add `&& !d.IsJointDeclaration`
   (`PackageDocumentReminderJob.cs:119-122`) to prevent duplicate emails.
   Confirm no other consumer depends on the package job still emitting
   JDF rows before adding that exclusion.
