## 1. ID & Summary

OLD's Template Management gave IT Admin full CRUD over notification templates:
`POST` (Add, with a duplicate `(TemplateCode + TemplateTypeId + Active)` guard),
`PUT`, `PATCH`, and a soft `DELETE` (sets `StatusId = Status.Delete`). NEW's
`NotificationTemplatesAppService` deliberately exposes only read + `Update`
(edit Subject / BodyEmail / BodySms / IsActive). There is no Create, no Delete,
no `NotificationTemplateCreateDto`, and no `.Create` / `.Delete` permission.
NEW substitutes "disable via `IsActive`" for OLD's soft-delete.

The CSV disposition is `include`: re-add Create + Delete to the notification-
template admin surface as part of the G-07-01 admin-UI build. This is a Partial
gap (Update is present and at parity; Create/Delete are the missing slice).

Important nuance that reshapes the fix: OLD's `Template.TemplateCode` is an
**int FK to a fixed 16-value `TemplateCode` enum** (values 1-18, file
`PatientAppointment.Models\Enums\TemplateCode.cs`). OLD could NOT invent
arbitrary new codes from the UI -- a "create" could only attach a row to one of
those 16 enum codes (plus a `TemplateTypeId`, Email vs SMS). So OLD's Create
really meant "instantiate the row for an enum code that does not yet have one",
NOT "define a brand-new event type". NEW already seeds a row for every code in
the canonical 59-code set per tenant, so the OLD use-case ("a code has no row
yet") cannot arise in NEW by construction.

### Upstream callers

- The notification-template **write** surface (Update; and the would-be
  Create/Delete) has NO production caller today -- there is no Angular admin
  component (confirmed above). The only callers of the read surface are tests and
  (eventually) the unbuilt admin UI.
- The **read-by-code** path (`FindByCodeAsync` /
  `FindWithNavigationPropertiesByCodeAsync`) is the heavily-used path: the
  `Application\Notifications\Handlers\*` set (~15 handler classes:
  BookingSubmissionEmailHandler, StatusChangeEmailHandler, ChangeRequest*
  handlers, Document* handlers, DueDate* handlers, PendingDailyDigest,
  PatientPacket, AccessorInvited, ClinicalStaffCancellation, JdfAutoCancelled,
  etc.) resolve templates by `NotificationTemplateConsts.Codes.*`. These are the
  consumers a hard-delete would break.

### Downstream consumers

- `INotificationDispatcher` / `NotificationTemplateRenderer` -> handlers render
  bodies fetched by code. A deleted (hard) or wrongly-typed template surfaces as
  `NotificationTemplateNotFound` (`CaseEvaluationDomainErrorCodes`) at send time,
  or silently falls back to a stub body (per `FindByCodeAsync` doc-comment:
  "callers should fall back to a hardcoded body and log the gap").
- The seed contributor is a downstream owner of the row set: it asserts the full
  59-code set exists per tenant. A user-created code is invisible to it.

### Blast radius

A faithful "add Create + Delete" fix touches roughly these layers (Partial-scope,
because Update infrastructure already exists):

1. `Application.Contracts`: new `NotificationTemplateCreateDto` (or reuse the
   create-shape), add `CreateAsync` + `DeleteAsync` to
   `INotificationTemplatesAppService`, add `.Create` + `.Delete` permission
   constants in `CaseEvaluationPermissions.cs` AND register them in
   `CaseEvaluationPermissionDefinitionProvider.cs`.
2. `Application`: implement `CreateAsync` (with OLD's duplicate
   `(code,type,active)` guard) + `DeleteAsync` (soft via `IsActive=false` or ABP
   `IsDeleted`) in `NotificationTemplatesAppService`.
3. `HttpApi`: add `[HttpPost]` + `[HttpDelete("{id}")]` to the controller.
4. `Angular`: regenerate proxy (`abp generate-proxy`) + build the admin
   create/delete UI -- BUT this is the G-07-01 admin-UI work that does not exist
   yet; this item cannot ship a usable Create/Delete without G-07-01.
5. Seed-contributor interaction: a created code outside `Codes.All` is never
   re-seeded and never resolved by any handler -- so the new code is inert unless
   handler wiring is also added (which is a code change, defeating the runtime-
   create premise).

Regression risk: MEDIUM. Adding endpoints is additive and low-risk to existing
flows. The real risk is semantic: enabling Create invites IT Admin to make
unwired codes (dead rows), and enabling hard-Delete can break ~15 handlers via
`FindByCodeAsync`. A soft-delete-only Delete (toggle `IsActive` / ABP soft
`IsDeleted`) avoids the handler-break risk and exactly matches OLD's actual
soft-delete (`StatusId = Delete`, never hard). The seeder re-creating rows means
even an `IsActive=false` row may be re-activated on the next seed run if the body
is resource-backed (seeder overwrites Subject+Body but does NOT touch IsActive
for resource-backed rows -- it only sets IsActive on INSERT, line 128 -- so an
admin's IsActive=false survives re-seed; this should be verified by test).

## 4. Expected vs Actual

- Expected (OLD parity): IT Admin can Add a template row for a `(code,type)`
  pair not yet present (duplicate-blocked), and soft-Delete a row
  (`StatusId = Delete`). All via the Template Management UI.
- Actual (NEW): IT Admin can only Update (Subject/BodyEmail/BodySms/IsActive).
  No Create endpoint, no Delete endpoint, no Create/Delete permission, no Create
  DTO, and no admin UI of any kind (the Update endpoint itself is unconsumed).
  Disable is `IsActive=false` instead of soft-delete.
- Behavioral delta in practice: NEW pre-seeds a row for ALL 59 codes per tenant,
  so OLD's Add use-case ("a needed code has no row yet") never occurs in NEW.
  OLD's Delete use-case maps cleanly to NEW's `IsActive=false`. The genuinely
  lost capability -- admin-defined NEW codes -- did not exist in OLD either
  (`TemplateCode` was a fixed 16-value enum). So the user-visible parity loss is
  near-zero; the gap is mostly about endpoint/permission surface symmetry.

## 7. Candidate Solutions

### Option A -- Add CreateAsync + soft-DeleteAsync (parity-faithful, code-only backend) [effort: M] [risk: med]

- One-line: Add `CreateAsync` (OLD's `(code,type,active)` duplicate guard) and
  `DeleteAsync` (soft: `IsActive=false`, mirroring OLD's `StatusId = Delete`
  which was never a hard delete), plus `.Create`/`.Delete` permissions, a
  `NotificationTemplateCreateDto`, and controller POST/DELETE. Defer the UI to
  G-07-01.
- Source: OLD code -- `TemplateDomain.AddValidation/Add/Delete`
  (`TemplateDomain.cs:40-93`), `TemplatesController` POST/DELETE
  (`TemplatesController.cs:37-77`). Soft-delete chosen because OLD's `Delete`
  sets `StatusId = Status.Delete` (line 89), not a row removal.
- Tradeoffs: Restores endpoint symmetry and lets a future deployment add a code
  row. But a created code is INERT until a handler is coded to resolve it
  (`FindByCodeAsync`), and the seeder never re-creates it -- so Create has no
  runtime effect without a paired code change. Risk: invites dead rows.
- Risk: medium. Hard-delete is explicitly avoided to protect ~15 handlers; soft
  delete is safe and matches OLD.

### Option B -- Keep edit-only; formalize the exclusion (do-not-port the Create/Delete) [effort: S] [risk: low]

- One-line: Treat Create/Delete as a deliberate, OLD-bug-adjacent simplification:
  OLD's `TemplateCode` enum was fixed (16 values), NEW pre-seeds all 59 codes, so
  Create/Delete add no user-visible capability. Document the decision; do not add
  endpoints. (This contradicts the current CSV `include` and would need Adrian's
  sign-off.)
- Source: OLD code -- fixed `TemplateCode` enum
  (`PatientAppointment.Models\Enums\TemplateCode.cs`) + NEW seed contributor
  (`NotificationTemplateDataSeedContributor.cs:98-130`).
- Tradeoffs: Lowest cost, zero dead-row risk, matches the existing in-code
  rationale. Loses literal endpoint parity and the (theoretical) ability to
  retire a code. Requires reversing the CSV disposition.
- Risk: low. Only risk is mis-judging an unknown deployment that hand-added a
  code at runtime (open question Q2).

### Option C -- Delete (soft) only; no Create [effort: S] [risk: low]

- One-line: Add only `DeleteAsync` as `IsActive=false` (the one OLD capability
  that maps cleanly and is safe), and skip Create entirely since NEW already
  seeds every code.
- Source: OLD `TemplateDomain.Delete` (soft, `:86-93`).
- Tradeoffs: Captures OLD's real Delete behavior with minimal surface and no
  dead-row risk; Update already exposes `IsActive` so this is nearly a
  convenience alias. Drops Create (which had no real OLD parity value).
- Risk: low. Note the seeder may re-toggle IsActive on resource-backed rows --
  verify (see blast radius).

Recommended: Option A only if the G-07-01 UI plan genuinely needs both buttons
and Adrian wants literal endpoint parity; otherwise Option B (formal exclude) is
the most defensible given the fixed-enum + full-seed reality. Flag for Adrian
per the CLAUDE.md "ambiguous -> replicate-and-flag" policy and the parity doc's
own "flag whether any deployment created custom template codes at runtime".

## 8. Root-Cause Hypothesis

HIGH confidence. The Create/Delete omission is an INTENTIONAL design decision
made during the Phase 4 notification-template unification (documented verbatim in
the AppService and interface XML-docs and in the parity doc's "Keep in NEW?"
note), not an accidental regression. Rationale chain, all verified in code:
(1) NEW collapsed OLD's two template mechanisms (16 DB-managed int-enum codes +
43 on-disk HTML codes) into one `NotificationTemplates` table keyed by a stable
string code; (2) the canonical code set (`Codes.All`, 59 entries) is seeded per
tenant by `NotificationTemplateDataSeedContributor`, so every code a handler
needs always has a row -- removing OLD's Add use-case; (3) handlers resolve by
fixed code via `FindByCodeAsync`, so a hard-Delete would break them and an
admin-created code would be inert -- making Create/Delete unsafe-or-pointless;
(4) "disable" is reframed as `IsActive=false`, which subsumes OLD's soft-delete.
The "gap" is therefore a surface-symmetry gap, not a behavioral parity loss --
which is exactly why the CSV note folds it into the G-07-01 UI build rather than
treating it as a standalone defect.

## 9. Open Questions

1. Does the G-07-01 admin-UI plan actually require visible Create + Delete
   buttons, or is an edit-only template editor (with an IsActive toggle)
   acceptable? This determines Option A vs Option B/C. (The UI does not exist
   yet, so the answer is a forward design choice, not a parity fact.)
2. Did any OLD deployment hand-add a `Templates` row at runtime for a code
   outside the 16-value enum? Source inspection says OLD's consuming code only
   resolved the fixed enum, so this is almost certainly "no" -- but it is the
   only scenario where Create would have carried real data. (Parity doc's own
   open flag.)
3. If Delete is added, confirm the seeder's IsActive handling: the contributor
   sets `IsActive` only on INSERT (line 128) and overwrites only Subject+Body for
   resource-backed rows -- so an admin `IsActive=false` should survive re-seed.
   Needs an integration test to pin this before relying on it as the "delete"
   mechanism.
4. Should Delete be soft (`IsActive=false` / ABP `IsDeleted`) only, with hard-
   delete forbidden to protect `FindByCodeAsync` handlers? (Recommended yes;
   matches OLD's soft `StatusId = Delete`.)
