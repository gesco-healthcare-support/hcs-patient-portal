## 1. ID & Summary

At external Patient registration, NEW's `RegisterAsync` creates a separate `Patient`
demographic row seeded with HARDCODED placeholders -- `genderId: Gender.Male`,
`dateOfBirth: DateTime.MinValue`, `phoneNumberTypeId: PhoneNumberType.Home` -- because the
minimal register form collects only name + email + password. OLD had a single `Users` table
where `GenderId` and `DateOfBirth` were nullable columns left NULL until the user filled them
in later; OLD never fabricated a gender or DOB at registration, and OLD had no
PhoneNumberType column at all.

The gap has two halves, both confirmed in current code:

1. **Backend fabrication (HIPAA-adjacent).** A freshly registered NEW patient carries a real
   row that reads "Male, 0001-01-01, Home phone" until they book. Any read path that surfaces
   the Patient row before booking corrects it shows fabricated demographics on a real person.
   This is a backend stub, not just a UI default -- the fabricated value is persisted to the
   DB.
2. **Prefill rule.** The CSV requires the appointment form to prefill only fields actually
   collected at registration for a NEW user, and to prefill richer demographics only for a
   RETURNING user from a prior patient profile/appointment. The booking form
   (`appointment-add.component.ts`) currently prefills `genderId` directly from the Patient
   row (so it would surface the fabricated `Male`); it already partially guards the DOB
   prefill (suppresses years < 1900 and an "equals today" sentinel).

The OLD register form is the ground-truth contract for "what registration collects": role,
first/last name (or firm name for attorneys), email, password, confirm-password -- nothing
else.

### Blast radius

- **Backend fix (preferred), minimal:** change the 1 literal triple at
  `ExternalSignupAppService.cs:580-582`. If the chosen representation is "leave unset", that
  requires making `Patient.GenderId` / `Patient.DateOfBirth` nullable, which cascades:
  - `Patient.cs` (2 properties + constructor), `PatientManager.cs` (CreateAsync + UpdateAsync
    + FindOrCreateAsync signatures, 3 method sigs), all 3 CreateAsync callers, the
    `PatientCreateDto`/`PatientUpdateDto`/`PatientDto` contracts, the Mapperly mapper, the EF
    migration (column nullability), and the Angular forms/validators that assume non-null
    gender/DOB. That is ~6 files + 1 migration + Angular = **L** if nullability is chosen.
  - If instead the placeholder is changed to a less-real value WITHOUT nullability (e.g. add
    `Gender.Unspecified = 0` and keep DOB null-suppressed at every read), the entity stays
    non-nullable; touches `Gender.cs` + the 2 fabricating call sites + every read surface's
    "is this a placeholder?" guard. That is **M**.
- **Frontend prefill rule:** add a gender placeholder guard mirroring the existing DOB guard
  in `appointment-add.component.ts` (1 file, ~3 sites). The DOB guard already satisfies its
  half.
- **Regression risk:** MEDIUM. Making columns nullable risks NREs in any consumer that reads
  `patient.DateOfBirth.Value` or formats gender without a null check; the booking form
  validators currently require DOB on submit (line 336 `Validators.required`), so a null DOB
  from a stub is fine pre-submit but must still be required at booking time. The 30-case
  length-validation theory test and the booking dedup (3-of-6, which uses `DateOfBirth.Date`
  in `PatientManager.FindOrCreateAsync:169`) must keep working -- a null DOB would break
  `dateOfBirth.Date`, so the dedup path needs a null guard if nullability is chosen.

## 4. Expected vs Actual

- **Expected (OLD parity):** Registration persists only what the register form collects (name
  + email + password + firm name for attorneys). Gender and DOB are left UNSET (OLD: NULL
  columns) until the user supplies them at booking or profile edit. No fabricated demographic
  ever exists on a real patient. The appointment form prefills only registration-collected
  fields for a brand-new user, and prefills full demographics only for a returning user from
  their stored profile / prior appointment.
- **Actual (NEW):** Registration eagerly creates a Patient row with `Gender.Male`,
  `DateTime.MinValue`, `PhoneNumberType.Home`. The values persist. The DOB is suppressed at
  the booking-form prefill surface only; gender is not suppressed anywhere; profile/admin/
  email/report read paths surface the fabricated values verbatim until a booking overwrites
  them.

## 7. Candidate Solutions

### Option A -- Stop fabricating; defer demographic capture, keep columns non-null via a neutral sentinel [effort: M] [risk: med]

- One-line desc: Add `Gender.Unspecified = 0` (and treat `DateTime.MinValue` as the canonical
  "unset DOB" sentinel), have `RegisterAsync` pass `Gender.Unspecified` + `DateTime.MinValue`
  + drop the phone-type fabrication to a neutral default, and add read-side guards (mirror the
  existing DOB year<1900 guard) so no fabricated "Male"/date is ever displayed; require
  gender + DOB only at booking/profile-save time.
- Source: OLD code (User.cs nullable Gender/DOB left unset at `UserDomain.Add`); OLD register
  form (user-add.component.html collects neither field). NEW already has the DOB-suppression
  pattern at `appointment-add.component.ts:1560-1573` to extend to gender.
- Tradeoffs: Keeps the entity non-nullable (no EF migration of nullability, no NRE cascade in
  consumers that read `.DateOfBirth` / `GenderId`). The cost is every read surface must learn
  the sentinel ("Unspecified" / MinValue) and render it as blank -- easy to miss one surface,
  which is the residual risk.
- Risk: med (must find every read surface; missing one re-leaks a placeholder). The dedup
  path (`FindOrCreateAsync` uses `dateOfBirth.Date`) keeps working because MinValue.Date is
  valid.
- Effort: M.

### Option B -- Make GenderId / DateOfBirth nullable on the Patient entity (truest OLD parity) [effort: L] [risk: med]

- One-line desc: Change `Patient.GenderId` -> `Gender?`, `Patient.DateOfBirth` -> `DateTime?`
  (and `PhoneNumberTypeId` -> `PhoneNumberType?`), have `RegisterAsync` pass `null` for all
  three, and require them at booking/profile-save. This mirrors OLD's nullable columns
  exactly: an unbooked patient genuinely has no gender/DOB.
- Source: OLD code -- `User.cs:44,134` `Nullable<System.DateTime> DateOfBirth` /
  `Nullable<Gender> GenderId`. This is the literal OLD schema shape.
- Tradeoffs: Most faithful to OLD; "unset" is representable in the type system, so no sentinel
  to leak. But it cascades through entity + constructor + `PatientManager` (3 signatures) + 3
  CreateAsync callers + 3 DTOs + Mapperly mapper + an EF migration (column nullability) + the
  30-case length-validation theory test + the 3-of-6 dedup (`dateOfBirth.Date` needs a null
  guard) + Angular forms that assume non-null gender/DOB.
- Risk: med-to-high (NRE surface on every consumer reading `.Value`; the booking dedup and
  the booking-form `Validators.required` on DOB must be re-checked).
- Effort: L.

### Option C -- Do not persist a Patient row at registration at all; create it lazily at first booking [effort: M] [risk: med]

- One-line desc: Drop the `_patientManager.CreateAsync` call from `RegisterAsync` entirely;
  let the existing booking `FindOrCreateAsync` create the Patient row (with real demographics)
  on first booking. The IdentityUser still exists; only the demographic row is deferred.
- Source: OLD architecture -- OLD never had a separate demographic row; the booking flow
  (`IsPatientRegistered` / get-or-create) is already the NEW onboarding path
  (Patients/CLAUDE.md Business Rule 1).
- Tradeoffs: Eliminates the fabricated row at the source -- nothing to leak, no sentinel, no
  nullability change. But any read path that assumes a Patient row exists for a logged-in
  patient before they book (`GetMyProfileAsync` throws `EntityNotFoundException` when no
  Patient maps to the user -- Patients/CLAUDE.md Business Rule 3) would now throw for a
  registered-but-never-booked patient. The profile page must handle "no patient row yet"
  gracefully. The `ExternalUsersDataSeedContributor` (seed) and the auto-link flow
  (`AutoLinkAppointmentsForUserAsync`) may also assume the row exists.
- Risk: med (changes the "row exists after registration" invariant several consumers rely on;
  needs an audit of `GetMyProfileAsync` + app-init profile check).
- Effort: M.

Recommendation for the later fix: Option A is the lowest-blast-radius way to stop the
HIPAA-adjacent fabrication while preserving the existing non-null contract and the working
booking/dedup paths, and it reuses the DOB-guard pattern already in the booking form. Option C
is architecturally cleanest (closest to OLD's no-stub model) but trades the fabrication risk
for a "missing row" invariant change that touches the profile read path. Decide A vs C with
Adrian per Open question #3. Either way, also add the missing gender guard to the booking-form
prefill (the prefill-rule half) and align the seed contributor.

## 8. Root-Cause Hypothesis

Confidence: high. The new stack split identity (ABP `IdentityUser`) from a dedicated `Patient`
demographic aggregate, where OLD had one `Users` row carrying both. The minimal register form
(Adrian, 2026-04-30) deliberately collects only name + email + password, but the `Patient`
entity declares `GenderId` and `DateOfBirth` as non-nullable value types with no
null/unspecified member in the enums. To satisfy the non-nullable constructor while creating
the row eagerly at registration, the author passed real-looking literals (`Gender.Male`,
`DateTime.MinValue`, `PhoneNumberType.Home`) as placeholders, intending the booking form to
overwrite them. The fabrication is therefore a side effect of (a) eager Patient-row creation
at registration plus (b) a non-nullable demographic model with no "unset" representation. OLD
avoided this entirely because its columns were nullable and it never created an intermediate
row. The DOB-prefill guard in the booking form is evidence the team already noticed the leak
on one surface (and that the placeholder convention changed from `UtcNow.Date` to
`MinValue` -- the guard defends both).
