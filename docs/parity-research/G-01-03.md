## 1. ID & Summary
G-01-03: On the NEW stack the server correctly computes whether a booked
patient already existed (email fast-path + 3-of-6 dedup) and exposes it as
`PatientWithNavigationPropertiesDto.IsExisting`, but the Angular booking
component never copies that value into the create payload. The
`AppointmentCreateDto.IsPatientAlreadyExist` field therefore always defaults
to `false`, so every NEW appointment row records "new patient" regardless of
the actual dedup result. The full server-side plumbing exists; only the one
client-side assignment is missing.

### Blast radius
A fix is localized: it is a single client-side assignment in one Angular
component (`appointment-add.component.ts onSubmit`), echoing the already-fetched
`currentPatientProfile.patient` profile's `isExisting` into
`payload.isPatientAlreadyExist`. No backend, DTO, migration, or proxy change is
required (the field, server persistence, and dedup computation all already
exist). Regression risk is low and confined to the booking submit path; the
main subtlety is making sure `isExisting` is populated for BOTH the
external-non-patient branch (where `getOrCreatePatientForAppointment` is called
at submit, line 1028) and the case where a patient was pre-resolved earlier
(e.g. via `loadPatientByEmail`), so the assignment must read a single
authoritative source. Tests in `AppointmentsAppServiceTests.cs:395-432` already
pin the server behavior (true persists, omitted defaults false); a new client/
integration test would assert the wiring.

## 4. Expected vs Actual
- **Expected (OLD):** Each booking stamps `IsPatientAlreadyExist = true` when
  the 3-of-6 dedup resolves to an existing patient, else `false`. Staff triage,
  the "merged patient" approval card, and downstream returning-patient handling
  rely on the true value.
- **Actual (NEW):** The server computes the correct existence result and even
  returns it to the client (`isExisting`), but the booking form omits the field
  from the create payload, so the persisted `IsPatientAlreadyExist` is always
  `false`. Any feature keying off "returning patient" sees every appointment as
  a brand-new patient. The dependent approval override (G-02-08) can never fire.

## 5. Repro Status
not-reproduced-static-confirmed -- A true runtime repro would require submitting
a booking for a patient that matches an existing record (creating/mutating data),
which the read-only mandate forbids. Static confirmation is unambiguous:
`appointment-add.component.ts:1040-1076` constructs the `AppointmentCreateDto`
with no `isPatientAlreadyExist` key (grep across the component returns zero
assignments to that field), the DTO defaults to `false`
(`AppointmentCreateDto.cs:52`), and the server simply copies the input
(`AppointmentsAppService.cs:804`). Therefore the persisted value is necessarily
`false` for the booking path. The DTO's own doc-comment documents the missing
client wiring.

## 7. Candidate Solutions

### Option A -- Echo the resolved profile's isExisting into the payload [effort: S] [risk: low]
- Description: In `appointment-add.component.ts onSubmit`, set
  `payload.isPatientAlreadyExist` from the authoritative resolved patient
  profile's `patient.isExisting` (the value already held in
  `currentPatientProfile` after `getOrCreatePatientForAppointment` /
  `loadPatientByEmail`), defaulting to `false` only when no profile was
  resolved. Mirrors OLD exactly with no backend change.
- Source: OLD code: `P:/PatientPortalOld/PatientAppointment.Domain/AppointmentRequestModule/AppointmentDomain.cs:203-218`
  (the canonical true/false stamp); NEW server contract already in place
  (`PatientWithNavigationPropertiesDto.cs:21-32`, `AppointmentsAppService.cs:804`).
- Tradeoffs: Smallest, lowest-risk change; honors the existing DTO contract and
  doc-comment. Must ensure `isExisting` is sourced consistently for both the
  external-non-patient submit branch and a pre-resolved patient (e.g. staff who
  selected an existing `patientId`). For the latter, `isExisting` is genuinely
  true (the patient pre-existed), so reading the resolved profile is correct.
  This is the recommended option.

### Option B -- Server recomputes existence at create time (ignore client flag) [effort: M] [risk: med]
- Description: Have `CreateAppointmentInternalAsync` itself re-run the
  dedup/existence check (or reuse the `GetOrCreate` result) and set
  `IsPatientAlreadyExist` server-side, treating the client field as advisory.
- Source: OLD code: `AppointmentDomain.cs:203-218` (existence determined
  server-side in OLD, not by the client) -- arguably closer to OLD's actual
  architecture, where the domain layer computed it.
- Tradeoffs: More robust (cannot be wrong even if a future client forgets), but
  duplicates dedup work already done by `GetOrCreatePatientForAppointmentBookingAsync`
  and is a larger change touching the server create path + tests. The current
  NEW design deliberately split the dedup into the patient get-or-create call
  and made the appointment create trust the client flag (per the DTO comment);
  Option B would partially revert that split. Higher blast radius than A.

## 8. Root-Cause Hypothesis
The dedup result is correctly computed and surfaced server-side, but the final
client-side wiring step was never completed: the booking form's `onSubmit`
builds the `AppointmentCreateDto` without reading the `isExisting` value it
already has on `currentPatientProfile`. The DTO doc-comment explicitly flags
this as deferred ("callers that haven't been updated yet"), so this is an
incomplete-implementation / missed-wiring gap rather than a logic error.
Confidence: high.

## 9. Open Questions
- For staff bookings that select a pre-existing `patientId` directly (not the
  external-non-patient `getOrCreate` branch), confirm the resolved profile's
  `isExisting` is reliably `true` so Option A reads the right source in that
  path -- otherwise the assignment needs a small branch. Resolvable during
  implementation by tracing how `currentPatientProfile` is populated in the
  staff path; does not block choosing Option A.
