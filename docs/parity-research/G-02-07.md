## 1. ID & Summary

NEW adds a domain invariant that OLD never had: an appointment in `Pending`
cannot transition to `Approved` unless it has at least one
`AppointmentInjuryDetail` (Claim Information) row. OLD's approval path
(`AppointmentDomain.UpdateValidation`) only blocked already-terminal /
already-approved states; it never counted injury rows, so OLD would happily
approve an appointment with zero claim/injury data.

The CSV disposition is `include` with the note "Keep NEW hardening (block
approve when zero injuries); already shipped (BUG-043)." This is a
deliberate, already-implemented, already-tested hardening -- NOT a missing
feature to build. The only open work is Adrian's explicit confirmation that
the stricter behavior is desired and not a parity regression. Static
confirmation: the gate exists, is wired end-to-end (domain -> exception
translator -> localized message -> tested), and OLD has no equivalent.

This is the inverse of most parity gaps: OLD is the looser/permissive side
and NEW is the stricter side. "Parity" here means consciously deviating from
OLD (allowing a clearly-undesirable state) in favor of a guard.

## 7. Candidate Solutions

Disposition is `include` = keep the NEW hardening. The "solution" is to
RATIFY and document, not to write code. One option, plus a guarding sub-task.

### Option A -- Keep as-is; ratify the deviation and close the parity flag [effort: S] [risk: low]
- One-line desc: Confirm with Adrian that the gate is intended hardening,
  then mark the parity-doc open question (line 146) resolved. No production
  code change.
- Source: OLD code `AppointmentDomain.cs:312-344` (proves OLD had no gate) +
  NEW `AppointmentManager.cs:226-242` (proves the gate is live + correct) +
  CSV note "already shipped (BUG-043)."
- Tradeoffs: Accepts a known, deliberate divergence from OLD ground truth in
  exchange for data integrity (no approved appointment without claim data).
  The cost is that any legacy OLD workflow relying on approving an empty
  appointment is no longer possible -- the audit found no such workflow, and
  injuries are the legally-significant payload of an IME appointment, so this
  is the right tradeoff.
- Risk: low. Behavior already in production since 2026-05-27 with passing
  tests; ratification adds no risk.
- Effort: S (a confirmation + a doc-status flip).

### Option A-guard -- Close the create-as-Approved fast-path edge (only if it exists and matters) [effort: S] [risk: low]
- One-line desc: The manager comment (lines 233-235) states the gate covers
  only `Pending -> Approved`; the "create-as-Approved internal fast-path
  attaches injuries after creation and is out of scope." If an internal
  create path can land an appointment directly in `Approved` with zero
  injuries, the gate is bypassable. Verify whether such a path exists and
  whether it needs the same count check.
- Source: OLD `AppointmentDomain.Add` + NEW `AppointmentManager.CreateAsync`
  (referenced by the in-code comment); needs a follow-up read to confirm a
  create-as-Approved path is reachable.
- Tradeoffs: Closing it makes the invariant total; leaving it matches the
  documented T8 scope decision. Defer unless the fast-path is actually wired.
- Risk: low.
- Effort: S.

## 8. Root-Cause Hypothesis

Confidence: high. The "gap" is not a defect but an intentional, documented
defense-in-depth invariant (BUG-043 / Wave-3 task T8, dated 2026-05-27)
added to the NEW domain service to prevent approving an IME appointment with
no Claim Information. OLD genuinely lacked this guard
(`AppointmentDomain.UpdateValidation` checks only idempotency), so the NEW
behavior is strictly stricter. The root cause of the "deviation" is a
conscious product-quality decision, fully implemented and tested, that the
parity audit correctly flagged for Adrian's sign-off because it departs from
OLD ground truth. The disposition (`include` = keep) ratifies it. No code is
broken; the only "fix" is governance ratification (Option A).

## 9. Open Questions

1. Does an internal "create-as-Approved" fast-path exist in
   `AppointmentManager.CreateAsync` that can land an appointment in
   `Approved` with zero injury details, bypassing the gate? The in-code
   comment (lines 233-235) explicitly declares this out of scope for T8. If
   the path is reachable, Option A-guard should be evaluated; if it is not
   wired (create always lands `Pending`), the invariant is already total.
2. Confirm (Adrian) that blocking zero-injury approval is desired hardening
   and not a parity regression -- this is the literal open question the
   parity doc raises (line 146) and the gate for closing this item.
3. Relationship to G-01-01 / G-01-02 (booking-time injury-detail creation):
   if the booking client does not yet persist injury details at create time
   (those items track that), then a freshly-booked appointment may have zero
   injuries and be un-approvable until injuries are added via the
   claim-information section. The gate and the booking-injury work are
   complementary; verify the end-to-end flow (book -> add claim info ->
   approve) is unblocked once G-01-01/02 land. This is a sequencing
   dependency, not a defect in this item.
