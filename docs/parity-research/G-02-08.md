## 1. ID & Summary

At booking, both OLD and NEW run a fuzzy dedup ("3-of-6 fields" match on
last name / SSN / email / phone / DOB / claim number). When it fires, the
new appointment is **linked to the existing Patient row** and the
`IsPatientAlreadyExist` flag is set true; the freshly typed-in patient
data is discarded. This is a destructive merge: two genuinely-different
people who happen to share 3 of those fields get collapsed onto one
Patient record.

The parity doc (G-02-08) frames the gap as: NEW records a "patient-match
override" decision at approval but does not perform the downstream
patient-row split, AND the Angular approve modal hardcodes
`overridePatientMatch: false` so the override is unreachable.

Both halves of that NEW-side description are verified true. **However, a
material drift was found on the OLD side:** OLD has **no override/split
control at all**. OLD's approval screen only DISPLAYS a read-only
"Patient record has been merged with existing patient" card; it has no
button, toggle, or payload field to un-merge. `IsPatientAlreadyExist`
is written exactly once in OLD (at `Add`) and is never flipped back
anywhere in the codebase. So the feature the parity doc describes
("OLD's approve screen could split them") does **not exist in OLD**. The
NEW `OverridePatientMatch` field + `ShouldOverridePatientMatch` helper +
`PatientMatchOverridden` ETO field are a NEW-invented scaffold for an
un-merge capability that was never in OLD. This reframes the item from a
"port OLD behavior" task into a "decide whether to build a brand-new
un-merge feature" task.

### Blast radius

A real fix (un-merge / split) touches:
1. Domain -- a new `PatientManager.SplitFromMatchAsync` (or similar) that
   creates a fresh `Patient` row from the appointment's original typed-in
   data and relinks `Appointment.PatientId`. PROBLEM: NEW (like OLD)
   discards the typed-in patient at booking (`PatientId` already points
   at the matched existing row; the typed-in `PatientCreateDto` is gone).
   So the original data needed to recreate the split-off patient is **not
   persisted** -- a split would need a new source of that data (re-typed
   by staff at approval, or a booking-time snapshot we do not currently
   keep). This is the load-bearing constraint.
2. Application -- `AppointmentsAppService.Approval.cs` to call the new
   manager method instead of the no-op flag flip.
3. Contracts -- `ApproveAppointmentInput` likely needs the split-off
   patient's fields (or a patient DTO), not just a bool, if staff re-enter
   the correct identity.
4. Angular -- add the override control + the re-entry form to the approve
   modal; stop hardcoding false.
5. EF Core -- no schema change for the bool itself (exists), but a
   booking-time snapshot of typed-in patient data (if chosen) is a new
   column/table.

Layers touched: Domain, Application, Contracts, Angular (+ EF Core if a
snapshot is needed). ~5-7 files. Regression risk: MEDIUM -- it introduces
a new Patient-creation path during approval (PHI write), and a relink can
strand the now-orphaned reference if other appointments shared the merged
PatientId. Cross-tenant safety must be respected (Phase 1 single-tenant).

## 4. Expected vs Actual

- Parity-doc expectation: staff at approval can un-merge a wrongly-matched
  patient (split into a new Patient row + relink), as OLD allegedly did.
- OLD ground-truth (corrected): OLD provided NO such capability. The merge
  at booking is final; the approval screen only displays it as
  information. So strict OLD parity = do nothing (the NEW read-side
  display equivalent is the only OLD-faithful behavior).
- NEW actual: the override is plumbed (`OverridePatientMatch` field,
  `ShouldOverridePatientMatch`, `PatientMatchOverridden` ETO) but the only
  effect is flipping `IsPatientAlreadyExist` to false; no new Patient row,
  no relink. The Angular UI hardcodes false, so the path is unreachable.
  Net: dead scaffolding on both ends.

The disposition is `include`, which means Adrian wants the un-merge
capability as a deliberate improvement over OLD (OLD's destructive merge
with no escape hatch is arguably a latent data-integrity bug). This is a
NEW feature, not a port -- the candidate solutions reflect that.

## 5. Repro Status

`not-reproduced-static-confirmed`. A runtime repro would require creating
two near-duplicate patients, booking through the dedup, then approving
with `overridePatientMatch=true` -- all data mutation, and the UI cannot
even send true. Per guardrails (no mutation, no flow submission), runtime
repro is not performed. The gap is fully confirmed by static reading:
(a) `confirm()` sends `overridePatientMatch: false` hardcoded
(modal ts:149); (b) even if true, the AppService only flips the bool
(Approval.cs:122) with an explicit "split deferred" comment; (c) no
handler consumes `PatientMatchOverridden`.

## 7. Candidate Solutions

### Option A -- Strict OLD parity: drop the scaffold, keep display only [effort: S] [risk: low]
- One-line: Remove the unreachable override plumbing; surface a read-only
  "merged with existing patient" indicator on the NEW approve screen to
  match OLD's information card.
- Source: OLD code -- `appointment-view.component.html:17-121` (display
  only); `AppointmentDomain.cs` has no un-merge.
- Tradeoffs: Faithful to OLD; zero new PHI-write paths. But it leaves
  OLD's destructive-merge data-integrity weakness in place and contradicts
  the `include` disposition (Adrian wants the split). Likely NOT what the
  disposition asks for, but it is the honest "what OLD actually did"
  baseline and worth stating.
- Risk: low. Effort: S.

### Option B -- Full un-merge with staff re-entry of correct identity [effort: L] [risk: med]
- One-line: Wire the Angular override control; when staff toggle it, show
  a patient-identity sub-form; on approve, create a NEW `Patient` from the
  re-entered data via a domain `PatientManager` method and relink
  `Appointment.PatientId`, then flip `IsPatientAlreadyExist=false`.
- Source: NEW-invented (no OLD equivalent). Framework crux: ABP domain
  service + repository `InsertAsync` + multi-tenant filter discipline
  (per MEMORY: cross-tenant reads need `IDataFilter.Disable<IMultiTenant>`;
  a Patient insert must stay inside the current tenant).
- Tradeoffs: Delivers the disposition's intent. Solves the "typed-in data
  was discarded at booking" constraint by having staff re-key the correct
  identity at approval (no schema change needed). Adds a PHI-write path on
  approval -- needs audit + permission gating
  (`CaseEvaluationPermissions.Appointments.Approve` already required).
- Risk: med (new Patient creation + relink during approval; orphaned-row
  handling if the merged PatientId is shared). Effort: L.

### Option C -- Booking-time snapshot + auto-split [effort: XL] [risk: med-high]
- One-line: Persist the typed-in patient data at booking (new snapshot
  column/table) even when the dedup merges; on override at approval,
  auto-recreate the split-off Patient from the snapshot -- no staff
  re-entry.
- Source: NEW-invented. Requires EF Core migration for the snapshot.
- Tradeoffs: Best UX (one-click un-merge, no re-typing) and most faithful
  to the idea of "the booking data was right, the merge was wrong". But
  it is the largest change: schema migration, snapshot write on every
  booking (storage of duplicated PHI -- HIPAA review needed), and the
  recreate logic. Over-engineered for a single-office Phase 1.
- Risk: med-high (PHI duplication, migration, more surface). Effort: XL.

Recommendation for the eventual builder: Option B. It satisfies the
`include` disposition (delivers a real un-merge) without OLD's
data-integrity weakness, avoids a schema migration, and keeps the PHI
write inside the already-permission-gated approval path. Option A is the
strict-parity fallback if Adrian decides the un-merge is out of Phase-1
scope after seeing that OLD never had it. Confirm the framing with Adrian
first, because the parity doc mis-describes OLD.

## 8. Root-Cause Hypothesis

Root cause (high confidence): the NEW `OverridePatientMatch` plumbing was
built in Phase 12 as a deliberate placeholder for a future Session-A
"manager rewrite" that never landed (the XML docs at
`AppointmentApprovalValidator.cs:89-99` and `Approval.cs:116-121` say so
verbatim). The split was scoped out, the UI was wired to send a constant
false to keep the path dormant, and no consumer was added for the ETO
flag. So the partial state is intentional-but-unfinished scaffolding, not
a regression.

Secondary finding (high confidence): the parity doc's premise that OLD
"could split them" is incorrect. OLD's approval screen is display-only;
OLD never un-merged. The work is therefore a net-new feature (justifying
the `include` disposition as an improvement), not a behavioral port. The
G-01-03 dependency (dedup flag populate) is already satisfied in NEW, so
this item is unblocked on its input.

## 9. Open Questions

- Does Adrian want the un-merge as a real Phase-1 feature (Option B) given
  that OLD never had it, or is strict OLD parity (Option A, display-only)
  acceptable? The `include` disposition implies B, but the OLD drift
  should be surfaced before building.
- If Option B: must staff re-enter the correct patient identity, or is a
  booking-time snapshot (Option C) wanted? Re-entry avoids a migration and
  duplicated-PHI storage.
- Orphaned-reference handling: if multiple appointments were merged onto
  the same existing PatientId, splitting one must not strand or corrupt
  the others. Define the relink scope (this appointment only) explicitly.
- Multi-tenant: confirm the new Patient insert stays in the current tenant
  (single-tenant Phase 1 makes this moot now, but the code must not assume
  it).
- Should the dead `PatientMatchOverridden` ETO field be removed if Option
  A is chosen, to avoid future confusion?
