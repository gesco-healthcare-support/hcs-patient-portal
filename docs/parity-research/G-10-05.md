## 1. ID & Summary

OLD persisted per-field change-log rows in `spm.AppointmentChangeLogs`, each
carrying a nullable `IsMailSent` flag. The flag is an idempotency latch: when
staff edit a confirmed appointment, OLD writes one change-log row per changed
field with `IsMailSent = false`, then a single email job collects every
`IsMailSent == false` row for that appointment, sends ONE "here is what
changed" diff-table email to stakeholders, and flips all included rows to
`IsMailSent = true`. A subsequent edit therefore only ever emails the NEW
diffs, never re-sends ones already announced.

NEW replaces field-level diff history with ABP's built-in entity-change
tracking (`AbpEntityChanges` / `AbpEntityPropertyChanges`). That covers the
diff DATA, but ABP audit rows are immutable and have no `IsMailSent` column.
NEW also has no `AppointmentChangeLog` entity and -- critically -- no
change-notification email at all yet (the consumer that would need the latch,
G-02-03 / G-04-03, is itself Missing). So today there is no double-send to
prevent; the gap is a forward-looking one: when the change-email is built, it
will need its own "already announced" bookkeeping or it will re-notify on
every edit.

This is a Partial column-gap whose practical weight is entirely conditional on
the G-02-03 / G-04-03 email being implemented. It is effectively a sub-task of
that email rather than independent data work.

## 4. Expected vs Actual

- Expected (OLD parity): editing a confirmed appointment emails stakeholders a
  one-time diff of changed fields; repeated edits never re-announce
  already-emailed diffs. A persistent per-diff "sent" marker enforces this.
- Actual (NEW): no change-notification email exists at all; no diff-capture on
  update; no sent-marker. ABP audit captures the diff data but is immutable and
  carries no send-state. The dedup mechanism is therefore both unbuilt AND
  homeless.

## 7. Candidate Solutions

### Option A -- Event-per-edit dispatch, NO persistent latch [effort: M] [risk: low]
Build the G-02-03/G-04-03 change-email so it fires exactly once per edit
transaction, driven by the appointment-update event (mirroring how
`StatusChangeEmailHandler` reacts to `AppointmentStatusChangedEto`). The diff
is computed for THAT edit only (compare incoming DTO vs current entity, or read
the just-written ABP `EntityPropertyChanges` for that change set) and emailed
in one dispatch. Because each edit emits one event and one email, there is no
batch to re-scan and nothing to double-send -- the OLD `IsMailSent` latch
becomes unnecessary by construction.
- Source: OLD code `AppointmentChangeLogDomain.cs:334-378` (semantics to
  match); NEW pattern `StatusChangeEmailHandler.cs` + `NotificationDispatcher.cs`.
- Tradeoffs: cleanest, no schema change, fits NEW's event-driven model. Minor
  behavioral deviation from OLD's "accumulate then batch" -- but the
  USER-VISIBLE outcome (one email per edit listing that edit's changes, no
  duplicates) is equivalent or better. If OLD intentionally batched multiple
  rapid edits into one email, that nuance is lost (see Open Questions).
- Risk: low. Hangfire delivers at-least-once, so a job retry could theoretically
  resend; if exactly-once delivery matters, pair with Option C.

### Option B -- Port the latch faithfully (new AppointmentChangeLog entity) [effort: L] [risk: med]
Recreate `AppointmentChangeLog` as a NEW EF Core entity with an `IsMailSent`
(or `MailSentAt` timestamp) column, write one row per changed field on update,
and have the send job scan `IsMailSent == false`, send, then flip. This is a
1:1 OLD port.
- Source: OLD `AppointmentChangeLog.cs` + `AppointmentChangeLogDomain.cs`.
- Tradeoffs: maximally OLD-faithful and preserves the batch-across-edits
  semantics. But it DUPLICATES the diff data ABP audit already captures
  (parity doc explicitly calls the diff "Equivalent" via `AbpEntityChanges`),
  reintroducing a redundant table the NEW design deliberately dropped. Higher
  surface (entity + migration + repo + UoW logic).
- Risk: med -- a second source of truth for change history can drift from the
  audit log; more code to maintain for a feature whose value is "no duplicate
  email," which Option A achieves without the table.

### Option C -- Lightweight sent-ledger keyed by content hash [effort: M] [risk: low]
If the change-email must be exactly-once across job retries, add a minimal
dedup record (e.g. a small `AppointmentChangeNotification` row or a distributed
de-dup key) keyed by appointment + a hash of the diff set, checked before
dispatch. This is the "equivalent marker" the CSV note allows ("IsMailSent or
equivalent"). Pairs with Option A to harden against Hangfire at-least-once.
- Source: OLD intent (`IsMailSent` as send-ledger); ABP `IBackgroundJobManager`
  at-least-once semantics motivate the guard. (ABP background-job delivery
  guarantees -- verify against ABP docs at implementation time if exactly-once
  is required; not load-bearing for confirming this gap.)
- Tradeoffs: smallest persistent footprint that still survives retries; does
  not resurrect the full change-log table. Slightly more design than Option A.
- Risk: low.

Recommended: Option A as the default (matches NEW's event model, zero schema
churn, equivalent user outcome), escalating to Option C only if exactly-once
delivery across Hangfire retries is a stated requirement. Option B only if
Adrian wants strict OLD batch-across-edits fidelity. This item should be
scheduled WITH G-02-03/G-04-03, not standalone.

## 8. Root-Cause Hypothesis

High confidence. Root cause: NEW intentionally dropped OLD's per-field
`AppointmentChangeLog` table in favor of ABP's immutable entity-change audit
log (a sound modernization). The audit log captures the diff data but, being
append-only, cannot host a mutable per-row "email already sent" flag. Because
the change-notification email that the flag guarded (G-02-03/G-04-03) was also
deferred as Missing, the latch was never given a replacement home. The flag is
not "lost data" -- it is a behavioral dedup mechanism orphaned by two
independent decisions (drop the table; defer the email). It resurfaces as a
real concern only when the deferred email is built, at which point the
event-driven NEW architecture can satisfy the dedup intent without resurrecting
the column (Option A).

## 9. Open Questions

1. Did OLD's batch-across-edits behavior matter, or was each
   `Appointment.Update` effectively one edit per email anyway? OLD scans ALL
   `IsMailSent == false` rows for the appointment, so two edits before any send
   would batch -- but the send runs at the end of every update (line 569), so
   in practice each update sends its own diffs. If batching across updates was
   never observable, Option A is strictly equivalent. (Needs OLD runtime/UX
   confirmation; cannot be settled statically.)
2. Is the change-email PHI body redacted as CSV row 12 requires? The dedup
   correctness and the redaction are coupled; both belong to the same
   G-02-03/G-04-03 work item.
3. Does NEW require exactly-once delivery (Option C) or is at-least-once with a
   one-email-per-edit model (Option A) acceptable? Depends on Hangfire retry
   policy and stakeholder tolerance for a rare duplicate on job retry.
4. Out of scope here but related: `IsInternalUserUpdate` (the OTHER OLD
   change-log flag) likewise has no NEW home; if the change-email needs to
   distinguish staff vs external edits, that flag's replacement should be
   resolved alongside.
