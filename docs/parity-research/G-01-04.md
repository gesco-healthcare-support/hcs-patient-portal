> **DECISION (finalized 2026-06-01) [Gate 3]:** Effort downgraded L -> S. OLD evidence (SoCal Overview PDF p.5; the RoleAppointmentType seed was DB-managed, never committed) confirms the ONLY role x type rules that ever shipped are: AME/AME-REVAL require an attorney (PatientAttorney or DefenseAttorney); PQME/PQME-REVAL allow all 4 external roles -- which NEW already matches behaviorally. SCOPE: keep current behavior; replace the fragile AME substring match with a small per-type "allowed roles" data flag on AppointmentType. DEFER the full admin-configurable matrix + UI until a genuinely new rule is requested. See SUMMARY.md Gate 3.

## 1. ID & Summary
OLD enforced a data-driven `RoleAppointmentType` join table (RoleId x AppointmentTypeId) at
booking time: a row had to exist for the booker's role + the chosen appointment type, or the
booking was rejected with `AppointmentCanNotBook`. The OLD UI also filtered the appointment-type
picker by the booker's role. NEW replaced the entire table with one hardcoded rule: external
callers cannot book an AME / AME-REVAL type unless they are Applicant or Defense Attorney. NEW's
type-lookup is unfiltered by role, and any non-AME type is bookable by any role. The configurable
matrix and the role-filtered picker are gone; the fix is to reintroduce an admin-configurable
role x type table plus admin UI so future rules need no code change.

### Blast radius
A faithful, configurable fix is a vertical slice across ~6 layers: a new `RoleAppointmentType` join
entity (Domain) + EF config + migration + tenant/host DbContext registration (note AppointmentType is
host-scoped, Role is ABP Identity-scoped -- the join's multi-tenancy side needs a deliberate decision);
contracts + DTOs + an admin AppService/controller + permissions; a seeded default matrix (the 4-row SoCal
matrix); a server-side check in `CreateAppointmentInternalAsync` replacing the hardcoded AME branch (or

UI; and the Angular booking picker passing the caller role. Regression risk is moderate: the booking
gate is on the critical create path and is exercised by existing AME tests, so the AME outcome must be
preserved exactly while generalizing. A minimal alternative (data flag on AppointmentType) touches far
fewer files but does not satisfy the CSV's "configurable matrix + admin UI" requirement.

## 4. Expected vs Actual
- EXPECTED (OLD): A `RoleAppointmentType` join table is consulted at booking time; if no row maps the
  booker's role to the chosen appointment type, the booking is rejected (`AppointmentCanNotBook`). The
  type picker is pre-filtered to only the types the booker's role may request. Admins can reconfigure
  the matrix by editing table rows -- no code change. Seeded matrix: PQME = 4 roles, AME = 2 attorney
  roles, PQME-REVAL = 4 roles, AME-REVAL = 2 attorney roles (per G-06-01 / SoCal Overview p.5).
- ACTUAL (NEW): Only one rule exists -- external non-attorney callers are blocked from AME/AME-REVAL
  (substring "AME"). Every other role x type combination is allowed. The type dropdown shows all types
  to all roles. Reconfiguration requires a code change. A future appointment type whose name contains
  "AME" would be wrongly attorney-gated; an AME-style type without "AME" in its name would be wrongly
  open to all.

## 7. Candidate Solutions

### Option A -- Port a configurable RoleAppointmentType join entity + admin UI [effort: L] [risk: med]
- Description: Recreate OLD's `RoleAppointmentType` (RoleId x AppointmentTypeId) as a NEW Domain entity
  with EF config, migration, seeded default 4-row matrix, an admin AppService/controller + permissions,
  an Angular admin matrix screen, a server-side booking check that replaces the hardcoded AME branch, a
  role-filtered `GetAppointmentTypeLookupAsync` variant, and the Angular booking picker passing the caller
  role. This is exactly what the CSV note asks for ("admin-configurable role x type table + admin UI so
  future rules need no code change").
- Source: OLD code: `AppointmentDomain.cs:637-643,688-691` (booking check) + `RoleAppointmentType.cs:12-45`
  (entity) + `appointment-add.component.ts:681-697` (role-filtered picker). For the EF "configurable lookup
  + seed" pattern on the NEW stack: ABP "Data Seeding" docs, https://abp.io/docs/latest/framework/infrastructure/data-seeding
  (accessed 2026-06-01; credibility: official ABP docs) and EF Core relationships/many-to-many,
  https://learn.microsoft.com/en-us/ef/core/modeling/relationships (accessed 2026-06-01; credibility: official Microsoft docs).
- Tradeoffs: Full parity + future-proof; but it is a 6-layer slice and forces a multi-tenancy decision
  (AppointmentType is host-scoped, Identity roles are host-scoped, but the matrix could be tenant-specific
  if different offices want different rules -- OLD was single-tenant so this is a new question). Must
  preserve the existing AME outcome and AME tests exactly. Role identity in NEW is by ABP role name
  (string), not OLD's int RoleId -- the join must key on the NEW role concept.

### Option B -- Data flag on AppointmentType (e.g. RequiresAttorney / AllowedRoles) [effort: M] [risk: low]
- Description: Add a configurable field (or small allowed-roles set) to the AppointmentType entity + admin
  CRUD, and check it at booking instead of the substring "AME" heuristic. Removes the name-based fragility
  (G-06-01's noted divergence) and is admin-editable per type.
- Source: OLD code: `AppointmentDomain.cs:637-643` (the rule it must reproduce). Per-entity flag is a
  simpler shape than the full join; EF Core modeling docs as above (accessed 2026-06-01; official Microsoft docs).
- Tradeoffs: Far less code and lower regression risk, and fixes the "odd name mis-gated" problem; but it
  is per-type, not a true role x type matrix, so it cannot express arbitrary combinations (e.g. "Patient
  may book PQME but not OTHER") the way OLD's table could. Does NOT fully satisfy the CSV's explicit
  "role x type table" requirement -- acceptable only if Adrian confirms the per-type flag covers every
  rule that ever shipped.

### Option C -- Keep hardcoded AME rule, defer (status quo) [effort: S] [risk: low]
- Description: Do nothing; rely on the fact that for the four default types NEW's outcome already matches
  OLD's seeded matrix (G-06-01 line 79-84).
- Source: OLD code: G-06-01 matrix vs NEW substring rule equivalence (06-auth-rbac.md:79-84).
- Tradeoffs: Zero effort; but the CSV decision is "include", so this is a non-starter unless the disposition
  changes. Listed only to bound the trade space and because G-06-01's own "Keep in NEW?" note calls the
  status quo "Acceptable for Phase 1."

## 8. Root-Cause Hypothesis
The OLD `RoleAppointmentType` join table was deliberately not ported during the booking build; the
single role x type rule that mattered in the seeded production matrix (AME/AME-REVAL require an attorney)
was reimplemented as a hardcoded substring + role allow-list (OBS-23, 2026-05-21). All other matrix rows
(notably the PQME/PQME-REVAL = 4-role rows) happen to be no-ops because NEW allows everything by default,
so the collapsed rule is behaviorally equivalent for the 4 default types but not configurable and fragile
on type name. Confidence: high -- the in-code comments at `AppointmentsAppService.cs:672-677` and
`BookingFlowRoles.cs:113-115` explicitly state the join was replaced by a hardcoded allow-list, and grep
confirms no join entity exists.

## 9. Open Questions
- Did OLD's `RoleAppointmentType` table ever hold rules beyond the SoCal 4-row matrix? No INSERT seed
  data exists in the OLD repo (`generated-schema.sql` is DDL only), so the actual production rows cannot
  be confirmed from code. G-06-01 cites "SoCal Overview p.5" for the canonical matrix; if that is the
  complete set, a per-type flag (Option B) would suffice. (Parity doc itself flags this, 01-booking.md:142.)
- Should the role x type matrix be host-scoped (one matrix for all offices) or tenant-scoped (per office)?
  OLD was single-tenant so the question did not arise; NEW must decide. This drives the entity's IMultiTenant
  side and the migration shape.
- Does NEW key the matrix on ABP role name (string) or a stable role id? OLD used int RoleId; NEW roles are
  ABP Identity roles -- the join's role key must be chosen deliberately so renames do not break rules.
