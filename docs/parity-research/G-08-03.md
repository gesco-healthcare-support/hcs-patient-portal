## 1. ID & Summary

OLD let report viewers (Clinic Staff, Supervisor, IT Admin) click "Export
to PDF" on the Appointment Request Report grid and receive a print-ready
rendering of the same ten-column table. NEW has no report screen, no
export endpoint, and no PDF render of any tabular data. The capability is
entirely absent.

This item is a strict subordinate of G-08-01 (the report itself). It
cannot ship before G-08-01 lands a query + grid, because the PDF is just
a second rendering of that same filtered result set. Per the root mission
(OLD print/DOCX -> NEW immutable PDF), NEW should emit a real PDF via
QuestPDF rather than OLD's open-HTML-in-a-new-window-and-print trick.

Material correction to the CSV planning hint: the note says "Render via
QuestPDF (already used for packets)". QuestPDF is *registered* in NEW
(`CaseEvaluationDomainModule.cs:83`, license = Community) but is **not
actually used to render anything**. The packet pipeline renders DOCX
templates and converts them to PDF via a Gotenberg sidecar
(`GotenbergDocxToPdfConverter.cs`, `DocxTemplateRenderer.cs`). There is
zero existing `Document.Create(...)` / `.GeneratePdf()` QuestPDF render
code to copy. The implementer writes the first QuestPDF document in the
codebase. The library and license registration are in place, so this is a
"first use of an installed dependency", not "add a dependency".

## 4. Expected vs Actual

Expected (parity outcome, per parity doc + root mission): an authenticated
report viewer applies filters on the Appointment Request Report, clicks
"Export to PDF", and downloads an immutable PDF containing the full
filtered result set laid out as a print-friendly table (the same columns
as the grid), with SSN and DOB masked per the HIPAA decision (CSV note:
"mask SSN/DOB"). The PDF is generated server-side via QuestPDF and streamed
as a file response; access is gated by the `Reports` permission.

Actual: there is no report grid, no export button, no export endpoint, no
`Reports` permission, and no QuestPDF render code. A NEW user cannot
produce any PDF of any report data. Gap fully confirmed.

Behavioral deltas the implementer must decide (these are *improvements
over OLD*, flagged in parity doc Open Questions 3 + 5):
- OLD shipped full SSN/DOB on both grid and PDF (unmasked). NEW should
  mask -- HIPAA decision belongs to Adrian (Open Question 3), but the CSV
  disposition already directs "mask SSN/DOB".
- OLD's client handshake serialized only the rows the rx-table had
  (pagination behavior ambiguous). NEW should render the FULL filtered set
  server-side (Open Question 5), which is what users expect from an
  "export" and avoids OLD's static-cache/temp-file pattern entirely.

## 7. Candidate Solutions

### Option A -- Server-side QuestPDF table render off the G-08-01 query [effort: M] [risk: low]

One-line: After G-08-01 lands the report query + DTO, add a
`Reports.ExportPdf` AppService method that runs the *full filtered* query
(no paging), feeds rows into a new QuestPDF `IDocument` that mirrors OLD's
grid columns, and returns the bytes as an HTTP file download; mask SSN/DOB
in the row projection before rendering.

- Source: OLD code `CSVExportController.cs:53-61` + `GeneratePDF:157-241`
  (what to render -- the ten-column table, green header, title) and the
  grid column list `...search.component.html:113-140` (column order +
  labels). Library capability: QuestPDF `Table` + repeating `table.Header`
  + `.GeneratePdf()` (context7 `/questpdf/questpdf`, accessed 2026-06-01,
  official maintainer docs -- HIGH credibility). Wiring shape:
  `DashboardAppService` + `DashboardController` as the AppService+manual-
  controller template; ABP conventions per root CLAUDE.md.
- Tradeoffs: This is the mission-correct outcome (immutable PDF, full
  result set, server-rendered). It deliberately does NOT reproduce OLD's
  HTML-print mechanism, the GUID-key handshake, or the static cache (all
  flagged as OLD bugs / non-load-bearing). Slightly more work than echoing
  HTML, but avoids re-importing OLD's PHI-bypass and concurrency bugs.
- Risk: low. Additive; QuestPDF already registered; no existing path
  changed. Main risk is column-width/overflow tuning for ten columns on
  one page (QuestPDF handles this with relative columns + landscape
  `PageSizes.A4.Landscape()`; the repeating `table.Header` keeps the
  header on every page). SSN/DOB masking must be applied in C# (the
  Angular `ssn-mask.pipe` does not reach the backend).
- Effort: M -- one DTO, one permission node (shared w/ G-08-01), one
  AppService method, one controller action, one QuestPDF document class,
  one Angular button + download call. Bounded; no new dependency.

### Option B -- HTML-to-PDF via the existing Gotenberg sidecar [effort: M] [risk: med]

One-line: Reuse the packet pipeline's Gotenberg converter: build OLD's
HTML table string (port `GeneratePDF`'s markup near-verbatim) and POST it
to Gotenberg's Chromium HTML route to get a PDF.

- Source: NEW `GotenbergDocxToPdfConverter.cs` (existing sidecar) + OLD
  `GeneratePDF:157-241` (the HTML to reuse). Gotenberg HTML route docs
  (https://gotenberg.dev/docs/routes, accessed 2026-06-01, official --
  HIGH credibility; note the sidecar's current route is
  `/forms/libreoffice/convert`, so the Chromium HTML route would be a new
  endpoint integration).
- Tradeoffs: Reuses OLD's exact visual markup (closest pixel parity to
  OLD's print view) and an existing sidecar. But it pulls an external
  network hop into a read-only report export, depends on the Gotenberg
  service being up for a non-packet feature, and re-introduces HTML string
  building (the brittle `dobCount==4` column-index date hack lives here --
  do not port it; format by field). Heavier operational coupling than
  Option A.
- Risk: med -- adds a runtime dependency (Gotenberg up) to report export;
  Chromium HTML route is not yet wired (only the LibreOffice route is).
- Effort: M, but with higher operational risk than A and no QuestPDF
  reuse for G-08-04.

Recommended: Option A. It matches the root mission (true immutable PDF via
QuestPDF), is self-contained (no extra service hop), and shares the
QuestPDF-first-render + masking groundwork with G-08-04, which is also
include. Option B's only edge is verbatim HTML parity, which the parity
doc explicitly says is not load-bearing ("only the outcome -- a PDF of the
report -- matters").

## 8. Root-Cause Hypothesis

Confidence: high. Root cause is simple absence: the entire reporting/export
area (G-08-01..04) was never built in NEW. There is no report query, grid,
permission, route, or export endpoint, and no QuestPDF render has ever been
written (QuestPDF is registered but unused; the only PDF path in NEW is
DOCX+Gotenberg for packets). G-08-03 is not a regression or a deviation --
it is a not-yet-ported capability whose prerequisite (G-08-01) is also
unported. The fix is additive and gated entirely on G-08-01 landing first.
