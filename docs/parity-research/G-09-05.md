> **DECISION (finalized 2026-06-01) [Gate 5]:** Not a simple revert. Provide BOTH windows: default the Approved/Rejected counters to "This week" with an "All-time" toggle/switcher. Effort stays S (a query-param + UI toggle). See SUMMARY.md Gate 5.

## 1. ID & Summary
The NEW tenant dashboard scopes its "Approved" and "Rejected" counters
to a 7-day window (counts where the relevant date `>= lastMondayUtc`),
whereas the OLD dashboard's "Approved Appointment" / "Rejected
Appointment" cards were unwindowed status totals (all-time running
counts). A staffer reading "Approved: 4" in NEW sees this week's
approvals, not the cumulative approved backlog OLD showed -- same card,
different number, different meaning.

Disposition (CSV): include / full. The CSV note: "Revert counters to
all-time totals per OLD. NOTE: OLD proc not in source (all-time
inferred); confirm intended window." The fix is a small backend change
(drop the two date predicates) plus a label/field rename cascade. The
one caveat is that OLD's all-time semantic is INFERRED, not directly
read, because the source-of-truth stored proc body is not in the repo
(see Open Questions + Root-Cause Hypothesis).

## 4. Expected vs Actual
- Expected (parity goal): the two cards show all-time status totals --
  total appointments currently in Approved status, and total in Rejected
  status -- matching OLD's "Approved Appointment" / "Rejected
  Appointment" cumulative counts.
- Actual (NEW): the cards show only this-week counts (rows whose approve
  date / last-modified date `>= lastMondayUtc`), so the numbers are a
  strict subset of the OLD totals and reset weekly. Label text "This
  Week" is honest about the new behavior but is itself a parity
  deviation (OLD labels had no period qualifier).

## 5. Repro Status
not-reproduced-static-confirmed. The gap is fully visible in source:
NEW's two `CountAsync` predicates include `>= lastMondayUtc`
(`DashboardAppService.cs:85-93`) and the UI labels say "This Week"
(`tenant-dashboard.component.html:31,43`); OLD's cards carry no period
qualifier and its proc takes no date parameter. A runtime repro would
require seeding Approved/Rejected appointments across week boundaries
and comparing the rendered count to a manual all-time total -- that is
state mutation (creating appointment rows), which the read-only mandate
forbids. Marked not-reproduced rather than infeasible because the static
evidence is conclusive: the windowing predicate is literally in the
query, no runtime observation is needed to confirm the behavior.

## 7. Candidate Solutions

### Option A -- Revert both counters to all-time status totals (port OLD) [effort: S] [risk: low]
- One-line desc: Drop the `>= lastMondayUtc` (and `AppointmentApproveDate
  != null`) predicates so both counts become pure status totals
  (`AppointmentStatus == Approved` / `== Rejected`), then rename the DTO
  fields + UI labels to OLD's unwindowed wording.
- Source: OLD code -- `dashboard.component.ts:57-58` (raw unfiltered
  proc fields), `dashboard.component.html:31-32,44-45` (no period label),
  `DashboardController.cs:47-53` + `_local/stub-procs.sql:293-299`
  (single `@UserTypeId` param, no date input). This is the disposition
  the CSV asks for ("Revert counters to all-time totals per OLD").
- Tradeoffs: matches OLD intent and the click-through (which already
  lands on an undated status list, so card count == list size again,
  restoring the OLD invariant). Loses the "recent throughput" signal NEW
  introduced -- acceptable, since parity is the goal and that signal was
  never an OLD behavior.
- Risk: LOW. No tests, no shared labels, host dashboard untouched. Only
  visible change is larger numbers + relabeled cards.
- Effort: S (delete 2-4 predicate lines; rename 2 DTO fields; relabel 2
  cards; regenerate proxy if proxied).

### Option B -- Keep the window but make it explicit + add an all-time card [effort: M] [risk: med]
- One-line desc: Leave "This Week" counts as-is and ADD two separate
  all-time "Approved" / "Rejected" cards so both signals are visible.
- Source: NEW design intent (recent throughput) + OLD all-time totals.
- Tradeoffs: preserves the NEW throughput view while restoring OLD
  numbers, but adds two cards to an already 13-card board and diverges
  further from OLD's 12-card layout (works against G-09-01/06 parity).
- Risk: MED -- layout drift, more surface to maintain, contradicts the
  "match OLD layout" directive in CLAUDE.md.
- Effort: M.
- Recommendation: not preferred. Only consider if the owner explicitly
  wants the weekly-throughput signal retained (see Open Questions).

Recommended: Option A, gated on the Open-Question confirmation that OLD
was all-time (the CSV itself flags "confirm intended window").

## 8. Root-Cause Hypothesis
The NEW dashboard is a deliberate redesign, not a port (per the parity
doc's "Key structural finding 1"). When the 13-card board was built
(W2-6, per DTO/component comments), the author re-interpreted the two
status counters as "recent throughput" metrics and added a Monday-00:00
UTC window plus "This Week" labels, rather than reproducing OLD's
cumulative status totals. There was no proc to copy from (only a
`WHERE 1=0` stub in the repo), so the windowing was a fresh design
choice unconstrained by OLD source.

Confidence: MEDIUM. The NEW windowing is HIGH-confidence (the predicate
is literally in the query). The OLD all-time semantic is MEDIUM-
confidence because the proc body is absent; it is inferred from three
converging signals -- (1) card labels with no period qualifier, (2)
click-through to an undated status list, (3) the proc's only parameter
being `@UserTypeId` with no date/window input. All three point to
all-time, but none is the proc body itself. This is exactly the caveat
the CSV note and parity-doc Open Question 1 raise.
