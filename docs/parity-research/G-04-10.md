## 1. ID & Summary

When an email fails to send, neither OLD nor NEW retries; the message is
lost. The CSV disposition is `include` because the fix INTENTIONALLY
EXCEEDS OLD parity: add retry + dead-letter on send failure. The
parity doc itself classifies this as an "Intent deviation (parity-matched,
both swallow)" -- i.e. NEW currently matches OLD's swallow behavior, and
the decision is to deliberately do better than OLD.

Mechanism of the gap in NEW: sends route through the Hangfire job
`SendAppointmentEmailJob`. Both send paths (`SendPlainAsync`,
`SendWithAttachmentAsync`) wrap `IEmailSender.SendAsync` in a
`try { ... } catch (Exception ex) { Logger.LogWarning(...) }`. Because
the catch swallows the exception, NO exception leaks out of
`ExecuteAsync`, so Hangfire records the job as **Succeeded** and its
built-in `AutomaticRetry` (default 10 attempts) + Failed-state
dead-letter never engage. The swallow is documented as a deliberate
MVP measure while ACS placeholder credentials are in use, to be removed
when real creds land.

The fix is small and low-risk: stop swallowing (remove or narrow the
try/catch) so Hangfire's retry pipeline activates, and optionally tune
the retry/dead-letter policy explicitly via `[AutomaticRetry]`.

### Downstream consumers

- Hangfire SQL storage (job state tables) -- a propagating exception
  changes recorded state from Succeeded to Retrying/Failed.
- `/hangfire` dashboard -- failed jobs surface for manual retry.
- `IPacketAttachmentProvider.NotifySendCompletedAsync` (attachment path
  only) -- currently called in `finally` with `success=false` on send
  failure to drive the AttyCE prune-on-success retention rule. If the
  fix lets the exception propagate, the `finally` STILL runs before the
  throw, so the callback is preserved; but a retry would re-invoke the
  whole job, calling `NotifySendCompletedAsync` again. Idempotency of
  that callback must be checked (see Open Questions) -- this is the one
  non-trivial interaction.

### Blast radius

- Files a minimal fix touches: **1** (`SendAppointmentEmailJob.cs`).
  Optional global-policy tuning adds 1 more (`CaseEvaluationHttpApiHostModule.cs`
  `ConfigureHangfire`).
- Layers: Domain (the job) +/- HttpApi.Host (Hangfire config). No
  Contracts/DTO/proxy/Angular change.
- Regression risk: **low-to-medium**. Removing the swallow means:
  (a) while ACS placeholder creds are still in use, EVERY send will now
  fail -> retry 10x -> Failed, generating Hangfire churn + log noise
  (this is exactly why the swallow was added as an MVP guard); (b) the
  attachment path's `NotifySendCompletedAsync(false)` will fire once per
  failed attempt (up to 11x) unless idempotent. The mitigation is to
  gate the behavior on whether real creds are configured, or to keep a
  narrow catch that re-throws only transient SMTP exceptions.

## 4. Expected vs Actual

- **Expected (target, exceeds OLD):** A failed email send is retried by
  Hangfire (default 10 attempts with backoff). If all attempts fail, the
  job lands in Hangfire's Failed state (dead-letter), retained
  indefinitely and visible/retriable at `/hangfire`. No silent loss.
- **Actual (NEW today):** `SendAppointmentEmailJob` catches the SMTP
  exception, logs a warning, and returns. Hangfire sees no exception ->
  marks the job Succeeded -> never retries, never dead-letters. The only
  trace is a single `LogWarning` line. (Strictly better than OLD, which
  logs nothing -- but still a silent functional loss.)
- **OLD:** Catches, builds a log object, never persists it (commented
  out). Silent loss, no retry, no record at all. `async void` also means
  no caller can observe the failure.

## 5. Repro Status

`not-reproduced-static-confirmed`. The gap is a code-path certainty,
provable statically:

1. `IEmailSender.SendAsync` throwing is caught by the `catch (Exception)`
   in both `SendPlainAsync` (line 100) and `SendWithAttachmentAsync`
   (line 152); the methods return normally.
2. No exception escapes `ExecuteAsync`.
3. Hangfire's `AutomaticRetry` only triggers on an exception thrown out
   of the job method (Hangfire docs, confirmed below). No exception =>
   job state Succeeded => no retry, no Failed/dead-letter entry.

A true runtime repro (force an SMTP failure and observe the Hangfire job
table stay Succeeded) would require enqueuing a real job against a
running stack with broken SMTP creds -- i.e. data/state mutation and a
running Hangfire server. Per guardrails that is out of scope for
read-only research; the static chain is conclusive, so runtime repro is
unnecessary rather than infeasible.

## 7. Candidate Solutions

This is NOT a pure port -- OLD has no behavior worth replicating here
(OLD also swallows). The target deliberately exceeds OLD using a
framework capability (Hangfire/ABP background jobs), so the crux is the
Hangfire retry model. External citations are warranted.

### Option A -- Remove the swallow so Hangfire's default retry + Failed-state dead-letter activates [effort: S] [risk: low-medium]

- **Desc:** Delete the `try/catch` around `_emailSender.SendAsync` in
  both `SendPlainAsync` and `SendWithAttachmentAsync` (keep an
  `Information` log on success). The exception then propagates out of
  `ExecuteAsync`; ABP's `AbpBackgroundJobsHangfireModule` already applies
  Hangfire's global `AutomaticRetryAttribute` (default 10 attempts), and
  exhausted jobs move to the Failed state, retained indefinitely at
  `/hangfire`. Zero new infra -- this is exactly what the job's own XML
  doc (lines 27-29) says to do "when real ACS credentials land."
- **Source:** OLD code N/A (no port). Hangfire docs "Dealing with
  Exceptions" -- "If retry attempts exceed their maximum, the job will be
  moved to the Failed state ... you will be able to retry it manually"
  (https://docs.hangfire.io/en/latest/background-processing/dealing-with-exceptions.html,
  accessed 2026-06-01, OFFICIAL). The default-10-attempts and
  Failed-state-as-dead-letter behavior also confirmed by
  `AutomaticRetryAttribute` API docs
  (https://api.hangfire.io/html/T_Hangfire_AutomaticRetryAttribute.htm,
  accessed 2026-06-01, OFFICIAL).
- **Tradeoffs:** Cleanest, smallest, matches the documented intent.
  BUT while ACS placeholder creds are still in use, every send fails ->
  retries 10x -> Failed, flooding the dashboard and logs. Must be paired
  with real creds OR gated (Option C). Attachment path: the `finally`
  `NotifySendCompletedAsync(false)` fires on each failed attempt -- needs
  idempotency check (Open Q1).
- **Risk:** low-medium (the cred-timing coupling is the only sharp edge).

### Option B -- Explicit `[AutomaticRetry]` policy on the job + tuned dead-letter action [effort: S] [risk: low]

- **Desc:** Same as A, plus decorate `SendAppointmentEmailJob` with an
  explicit `[AutomaticRetry(Attempts = 5, OnAttemptsExceeded = AttemptsExceededAction.Fail)]`
  (or set it globally in `ConfigureHangfire` via
  `GlobalJobFilters.Filters.Add(...)`). `Fail` keeps the job in the
  Failed state as a true dead-letter for manual inspection/retry (vs
  `Delete` which discards). Makes the retry count + dead-letter
  disposition explicit and reviewable rather than relying on the global
  default of 10.
- **Source:** Hangfire `AutomaticRetryAttribute` API docs (URL above,
  OFFICIAL) -- `Attempts`, `OnAttemptsExceeded`, `AttemptsExceededAction`
  members. Hangfire "Dealing with Exceptions" for global-filter syntax
  (URL above, OFFICIAL).
- **Tradeoffs:** Slightly more code than A but documents the SLA in
  code; `Fail` matches the CSV's "dead-letter" intent precisely. Same
  cred-timing caveat as A.
- **Risk:** low. CAVEAT: community reports of method-level `Attempts`
  being ignored in favor of a global filter on some Hangfire versions
  (GitHub issue #1273) -- verify the configured form actually takes
  effect on the version in use; prefer the global-filter form if unsure.

### Option C -- Conditional propagation gated on real-creds flag (transitional) [effort: M] [risk: low]

- **Desc:** Keep a catch that distinguishes "ACS placeholder creds not
  configured" (swallow + warn, today's behavior) from a genuine
  transient SMTP failure (re-throw so Hangfire retries). Gate on a
  setting (e.g. an `Email:RealCredentialsConfigured` flag or detecting
  the placeholder ACS connection string). When real creds land, the flag
  flips and propagation/retry turns on without a code change.
- **Source:** ABP settings pattern (the codebase already uses
  `CaseEvaluationSettingDefinitionProvider`); Hangfire retry model as
  above. No OLD source (no port).
- **Tradeoffs:** Avoids the dashboard/log flood during the placeholder-
  creds window AND delivers retry the moment creds are real -- best
  operational story. More code than A/B and one more setting to manage;
  arguably over-engineered if real creds are imminent.
- **Risk:** low.

**Recommendation:** Option B (remove swallow + explicit
`[AutomaticRetry(..., OnAttemptsExceeded = Fail)]`) once ACS creds are
real; Option C if there is a meaningful window where placeholder creds
remain. Resolve Open Q1 (NotifySendCompletedAsync idempotency) before
shipping either, since retries re-invoke the attachment path.

## 8. Root-Cause Hypothesis

The NEW job intentionally swallows SMTP exceptions
(`SendAppointmentEmailJob.cs` catch blocks at lines 100-107 and 152-158)
as an MVP guard while ACS placeholder credentials are configured, so
that an appointment request still completes even though delivery cannot
happen. The swallow has the side effect of hiding the exception from
Hangfire, which therefore marks the job Succeeded and never engages its
built-in `AutomaticRetry` (10 attempts) or Failed-state dead-letter. The
"gap" is thus not a missing capability -- the durable queue + retry +
dead-letter all already exist via `AbpBackgroundJobsHangfireModule` and
Hangfire's global default -- but a deliberately placed try/catch that
short-circuits them. Removing/narrowing the catch (the job's own doc
prescribes exactly this once creds are real) restores retry + dead-letter
with essentially no new infrastructure. **Confidence: high** -- the code
path, the Hangfire default behavior, and the documented intent all
corroborate.

## 9. Open Questions

1. **Idempotency of `IPacketAttachmentProvider.NotifySendCompletedAsync`
   under retry.** With Option A/B, a failing attachment send re-runs the
   whole job up to 11 times; the `finally` block calls
   `NotifySendCompletedAsync(packetRef.PacketId, success=false)` each
   time. Is that callback safe to call repeatedly (esp. the AttyCE prune
   path on a later success after earlier failures)? Needs a look at the
   provider impl before enabling retry on the attachment path.
2. **Are ACS real credentials imminent?** Determines Option A/B (just
   remove the swallow) vs Option C (gate it). The whole swallow exists
   only as a placeholder-creds workaround.
3. **Desired retry count + dead-letter disposition.** Default 10 attempts
   may be excessive for transactional email; confirm whether 5 + `Fail`
   (keep for manual retry) or `Delete` (discard) matches the operational
   intent the CSV calls "dead-letter."
4. **Cross-ref G-04-06 (approval-path CC).** The approval email path is
   already special-cased for CC (`clinicStaffEmail` global folded in
   except on approval). Confirm the retry change does not interact with
   that path differently -- it should not, since CC is resolved before
   enqueue, but worth a glance.
5. **Hangfire version-specific attribute behavior.** Confirm
   `[AutomaticRetry]` (method-level) is honored on the ABP-bundled
   Hangfire version, or use the global-filter form (GitHub issue #1273
   reports method-level being ignored on some versions).
