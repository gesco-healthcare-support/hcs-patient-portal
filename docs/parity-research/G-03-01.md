## 1. ID & Summary

OLD had a tenant-level master list of document categories, `AppointmentDocumentType`
(`spm.AppointmentDocumentTypes`: `AppointmentDocumentTypeId`, `DocumentTypeName`,
`StatusId`), with full CRUD (admin screen) and a picker dropdown on the ad-hoc upload
modal, plus an "Other" free-text overflow (`OtherDocumentTypeName`). NEW dropped the
entity entirely: every document is classified only by a free-text `DocumentName` string.
Staff lose the controlled vocabulary and type-based filtering/reporting.

Disposition is `include` (build it). This item is the canonical entry; G-07-07
(admin-master-data view) and G-10-09 (data-model view) describe the same missing entity
from different audit angles and are explicit duplicates per the decision log.

## 4. Expected vs Actual

Expected (parity with OLD): an admin-managed list of document categories; a dropdown
picker at ad-hoc upload time populated from active types, with an "Other" option that
reveals a free-text "specify type" field; uploaded ad-hoc docs carry a nullable
`AppointmentDocumentTypeId` + `OtherDocumentTypeName`. Staff can filter/report by type.

Actual (NEW): no type concept at all. Ad-hoc uploads carry only a free-text
`DocumentName`. No admin screen, no picker, no type-based filtering. Package-queued docs
get their name copied from the master `Document`, so they are still labelled, but the
controlled vocabulary and "Other" overflow are gone.

CSV scope to satisfy: "Add doc-type entity + CRUD + picker on upload (+ Other free-text)."
This also closes the residual gap in G-03-03 (ad-hoc rows lack type metadata) and the
duplicate views G-07-07 / G-10-09. Note G-10-10 (`DocumentPackageId`/`UserType` columns)
is a related but separate column gap, not part of this item.

## 7. Candidate Solutions

### Option A -- Faithful lookup port modeled on AppointmentStatuses [effort: L] [risk: low]

Port `AppointmentDocumentType` as a tenant-scoped ABP aggregate following the existing
`AppointmentStatuses` feature template (the closest in-repo analog: `Name` + status +
full CRUD across all 6 layers + an Angular CRUD screen). Add a nullable
`AppointmentDocumentTypeId` + `OtherDocumentTypeName` to `AppointmentDocument`, surface a
picker in the ad-hoc upload UI, and seed default types including a stable "Other" row.

- Source: OLD code -- `AppointmentDocumentTypeDomain.cs:15-110`,
  `AppointmentDocumentType.cs:14-48`, `AppointmentDocumentTypesController.cs:17-73`,
  `appointment-new-document-add.component.{html,ts}` (picker + "Other"); NEW template --
  `src/.../{Domain,Application,Application.Contracts,EntityFrameworkCore,HttpApi}/AppointmentStatuses/`
  and `angular/src/app/appointment-status/`.
- Tradeoffs: highest fidelity and matches the CSV scope exactly; cost is breadth (~6
  layers + migration + Angular). Apply the bug policy: OLD did NO validation
  (`CommonValidation` empty, `DeleteValidation` commented out) -- NEW should add name
  uniqueness in the manager and an in-use check before delete (sanctioned OLD-bug fix,
  matching the precedent set for `DocumentDomain.DeleteValidation` -> `DocumentInUse`
  noted in the audit's E9). Replace OLD's hardcoded "Other == id 4" with a stable
  marker column (e.g. `AllowsFreeText`/`IsOther`) so the free-text overflow does not
  depend on a magic primary-key value.
- Risk: low. Net-new feature; migration is additive (nullable FK); ad-hoc DTO change is
  backward-compatible when the type is optional.

### Option B -- Minimal: type column on AppointmentDocument only, no admin CRUD [effort: M] [risk: low]

Add `AppointmentDocumentTypeId` + `OtherDocumentTypeName` plus a small seeded enum/lookup
exposed read-only to the picker, but skip the admin CRUD screen (types are seed-managed,
not user-editable).

- Source: OLD entity fields (`AppointmentNewDocument.cs:85-91`) + the picker UI only.
- Tradeoffs: delivers the upload-time categorization and "Other" overflow at lower cost,
  but FAILS the CSV scope ("+ CRUD") -- OLD let IT/clinic staff curate the list at
  runtime. Acceptable only if Adrian decides admin curation is not needed; otherwise it
  is a partial port that re-opens the gap.
- Risk: low, but scope-mismatch risk with the recorded decision.

### Option C -- Defer (free-text only) [effort: none-if-deferred] [risk: low]

Keep free-text `DocumentName`; do nothing now. NOT aligned with the `include` decision --
listed only for completeness because the audit's own "Keep in NEW?" line floated
deferral. Choose only if the include decision is revisited.

Recommended: Option A (matches the include decision + CSV scope; B is the fallback if
admin curation turns out to be unneeded).

## 8. Root-Cause Hypothesis

High confidence. The root cause is a deliberate MVP scope cut, self-documented in
`AppointmentDocument.cs:15-16` ("Cuts deferred to ledger: AppointmentDocumentType lookup
(free-text DocumentName at MVP)"). The new stack unified OLD's two document tables into
one `AppointmentDocument` (+ `IsAdHoc`) and, in doing so, dropped the doc-type lookup and
its `OtherDocumentTypeName` overflow rather than porting them. The gap is intentional
deferral, not an accidental omission -- which is why it is `Missing` + `include` (build
the deferred piece), not a bug to fix in place.
