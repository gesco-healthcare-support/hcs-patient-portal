## 1. ID & Summary
For a "cumulative trauma" injury (a date-range exposure window rather than a single-day
event), OLD validated the From/To pair before letting the injury row into the form: From
must be < To, To must be <= today, and From must not equal To. NEW carries the
`injuryCumulative` toggle and `injuryToDateOfInjury` field through the entire stack
(Angular modal -> POST `/api/app/appointment-injury-details` -> AppService -> domain
manager -> entity) but applies no validator to the To-date and never compares From vs To
vs today. An inverted, future, or zero-width cumulative range is accepted and persisted
silently. This is the cumulative-range sibling of G-01-01 (single-injury date <= today /
>= DOB), which covers the non-cumulative branch of the same OLD function.

### Blast radius
A faithful port touches one or two layers. Minimum (parity-with-OLD): add the three-rule
cumulative check (From < To, To <= today, From != To) where it can fire before persist.
OLD put it client-only, so the lowest-risk parity-true fix is a single client method on
`appointment-add-claim-information.component.ts` (mirroring `checkDateValidationForInjury`),
called from `saveInjuryModal` (~424). That is one file, additive, near-zero regression risk
(it only adds rejection of currently-invalid input; no existing valid path changes). A
defense-in-depth server guard (recommended, since the OLD client-only design is a known
weakness and NEW already moved sibling checks server-side) adds a check in
`AppointmentInjuryDetailManager.CreateAsync`/`UpdateAsync` -- one domain file, gated on
`isCumulativeInjury && toDateOfInjury.HasValue`, so non-cumulative rows (To == null) are
untouched. Tests that assert current accept-anything behavior, if any exist, would need
updating; a grep found a spec for `saveInjuryModal` happy-paths only. Coupling to G-01-01:
both implement halves of the same OLD function, so they should be designed/fixed together
to avoid a half-validated injury form.

## 4. Expected vs Actual
- Expected (OLD): when the cumulative toggle is on and both dates are entered, the form
  rejects (toast + no row added) if From > To, if To is in the future, or if From == To.
  A cumulative injury with only one date entered is not blocked by this rule (returns false).
- Actual (NEW): the cumulative toggle and To-date are collected and saved with no
  validation. From > To, To > today, and From == To are all accepted and persisted to
  `AppointmentInjuryDetail.ToDateOfInjury`, then surfaced verbatim in the view and the
  downstream report duration math.

## 7. Candidate Solutions

### Option A -- Client validator mirroring OLD, called from saveInjuryModal [effort: S] [risk: low]
- Description: Add a `checkCumulativeInjuryDates()` (or extend `saveInjuryModal`) on
  `appointment-add-claim-information.component.ts` that, when `injuryCumulative` is true and
  both dates present, rejects From > To, To > today, From == To with the three OLD messages
  set as `injuryModalError`; block save when it fires.
- Source: OLD code: `appointment.domain.ts:1031-1052` (authoritative -- this IS the rule to port).
- Tradeoffs: Exact parity with OLD's placement and messages, smallest blast radius. But, like
  OLD, it is bypassable by a direct API call (the To-date endpoint stays unguarded). Given
  NEW already moved sibling validations server-side, client-only is a regression in rigor.

### Option B -- Server guard in AppointmentInjuryDetailManager (defense in depth) [effort: M] [risk: low]
- Description: In `AppointmentInjuryDetailManager.CreateAsync`/`UpdateAsync`, after the
  duplicate check, add: if `isCumulativeInjury && toDateOfInjury.HasValue`, throw
  `UserFriendlyException` for `dateOfInjury > toDateOfInjury`, `toDateOfInjury.Date > Today`,
  or `dateOfInjury.Date == toDateOfInjury.Date`. Pair with Option A for UX (inline toast vs
  400 round-trip). Localize the messages via `L[...]` per project convention.
- Source: OLD code: `appointment.domain.ts:1031-1052` for the rule; ABP UserFriendlyException
  + Volo.Abp.FluentValidation pattern already in use at
  `AppointmentCreateDtoValidator.cs:8-13`. Reference: ABP domain-service validation guidance,
  https://abp.io/docs/latest/framework/architecture/domain-driven-design/domain-services
  (accessed 2026-06-01) -- credibility: official ABP docs.
- Tradeoffs: Enforces the invariant regardless of client; consistent with how G-01-01's
  sibling checks were ported server-side. Slightly more code and a localization-key addition;
  must guard on `HasValue` so single-date cumulative rows (OLD does not block them) and all
  non-cumulative rows stay unaffected.

### Option C -- FluentValidation rule on the DTO [effort: M] [risk: med]
- Description: Create the `AppointmentInjuryDetailCreateDtoValidator` (and Update) the
  existing comment already references, with cross-field rules using `When(x => x.IsCumulativeInjury)`.
- Source: OLD code: `appointment.domain.ts:1031-1052`; FluentValidation cross-property /
  conditional rules, https://docs.fluentvalidation.net/en/latest/conditions.html (accessed
  2026-06-01) -- credibility: official FluentValidation docs.
- Tradeoffs: Produces the uniform `validationErrors[]` 400 shape and satisfies the dangling
  comment. But the project convention (per layer CLAUDE.md) is that business invariants live
  in the domain service, not the DTO validator -- so Option B is the more idiomatic server
  layer. Use C only if a uniform 400 payload is specifically wanted; otherwise prefer B.

Recommended: B + A together (server invariant for correctness, client check for UX),
designed jointly with G-01-01 since they are two halves of the same OLD validator.

## 8. Root-Cause Hypothesis
During the NEW injury-detail port, only the duplicate-tuple rule and field length/required
constraints were carried over; the date-relationship logic in OLD's
`checkDateValidationForInjury` (both its cumulative branch = G-01-02 and its single-date
branch = G-01-01) was never ported. The cumulative controls were wired for data capture and
persistence (`injuryToDateOfInjury`, `ToDateOfInjury` column, view binding) but the
accompanying validation was dropped. The stale comment in `AppointmentCreateDtoValidator`
referencing a non-existent `AppointmentInjuryDetailCreateDtoValidator` suggests injury-level
validation was planned and never built. -- confidence: high.
