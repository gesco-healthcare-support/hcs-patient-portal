## 1. ID & Summary

When an internal user edits a confirmed appointment's intake-form data, OLD
collects every field-level change-log row not yet emailed (`IsMailSent==false`),
builds an HTML diff table (`FieldName / OldValue / NewValue`), and emails it to all
appointment stakeholders via the `AppointmentChangeLogs` template, then flips
`IsMailSent=true` so it never double-sends. A second template
(`AppointmentRescheduleRequestByAdmin`) fires ONCE if any changed field is an
Appointment Date or Appointment Time. NEW's update path (`AppointmentManager.UpdateAsync`)
is silent: it mutates scalar fields and saves, with no diff, no recipient fan-out,
no email, and no `IsMailSent` dedup state anywhere.

The gap is confirmed by static code reading on both sides. This item is the
notification-side companion to the change-log capture gaps G-02-01/02 and the
`IsMailSent` dedup gap G-10-05; the email-send half is independently tracked as
G-04-03 in area 04.

PHI note (from CSV): the diff table puts raw field values (claim numbers, DOB,
addresses, body parts) into the email body. Any NEW implementation MUST redact or
gate PHI per `.claude/rules/hipaa-data.md` -- OLD sent values in cleartext, which
is NOT a behavior to port verbatim.

### Blast radius

A faithful, PHI-safe port touches roughly 6-9 files across 4 layers; medium-high
regression risk because it adds a side effect to the hot edit path and introduces
new dedup state.

- Domain.Shared: add a `NotificationKind.IntakeFormChanged` value (+ keep the
  existing `AppointmentChangeLogs` template code; add localization for subject/body).
- Domain: a new event (`AppointmentIntakeFormChangedEto` or reuse a generic
  changed-eto) and the decision of WHERE the diff is computed (manager vs handler).
- Application: a new `IntakeFormChangedEmailHandler` subscribing to the event,
  computing the field diff (from `AbpEntityChanges` or a before/after snapshot),
  rendering the redacted table, resolving recipients via the existing resolver, and
  enqueuing `SendAppointmentEmailJob`.
- Persistence/dedup (G-10-05): a new "already-notified" marker -- either a small
  `AppointmentChangeNotification` table, a per-`AbpEntityChange`-batch dedup key, or
  a job-idempotency token. ABP audit rows are immutable, so the OLD in-row
  `IsMailSent` flip cannot be ported literally.
- DbMigrator: a migration if a dedup table/column is added.
- Tests: handler unit tests + a dedup test.

Regression risk drivers:
1. Side effect on every appointment edit -> must not throw/block the save (OLD ran
   it post-commit; NEW should use a queued handler off an event, not inline).
2. The OLD "fire admin-reschedule email once if Date/Time changed" branch is a
   second template with its own gate -- easy to miss.
3. PHI in body -- a literal port leaks field values; redaction is a design
   requirement, not an option.
4. Diff scope: OLD logged child-entity changes too (injury/body-part/claim,
   G-02-02); ABP `[Audited]` on the root only covers root scalars unless child
   entities are also `[Audited]` AND the diff query spans them.

## 4. Expected vs Actual

- Expected (OLD ground truth): after `Appointment.Update`, stakeholders receive an
  HTML diff email of changed intake fields (deduped so each change emails once via
  `IsMailSent`), plus a one-shot reschedule-by-admin email if a Date/Time field
  changed.
- Actual (NEW): `AppointmentManager.UpdateAsync` saves scalar changes and returns.
  No diff is computed for notification, no email is sent, no recipient fan-out runs,
  and no `IsMailSent`/equivalent dedup state exists. The two relevant template codes
  are defined but unwired. Net: appointment edits are silent.

## 7. Candidate Solutions

### Option A -- Event-driven change-mail with dedup table [effort: L] [risk: med]

One-line: publish an `AppointmentIntakeFormChangedEto` from `AppointmentsAppService.UpdateAsync`
(post-save), handle it in a new `IntakeFormChangedEmailHandler` that diffs
before/after, renders a REDACTED table, resolves recipients via the existing
`AppointmentRecipientResolver` (new `NotificationKind.IntakeFormChanged`), enqueues
`SendAppointmentEmailJob`, and records a dedup row in a small
`AppointmentChangeNotification` table (the NEW home for `IsMailSent`).

- Source: OLD code (`AppointmentChangeLogDomain.SendEmailForIntakeFormChanges` 334-378,
  `SendEmailForChangeLog` 267-311) for behavior; NEW pipeline
  (`AppointmentRecipientResolver`, `SendAppointmentEmailJob`, handler family) for wiring.
  ABP audit infra optional for diff data (`AbpEntityChanges`).
- Tradeoffs: cleanest fit with the existing notification architecture; queued so it
  cannot break the edit save (OLD ran post-commit, NEW improves by going async).
  Cost: new migration + dedup table. Resolves G-10-05's "no IsMailSent home" by
  giving the flag a purpose-built table instead of trying to mutate immutable audit
  rows.
- Risk: med. Must capture the before-state (snapshot the loaded entity, or read
  `AbpEntityChanges` for the request) and must include the Date/Time -> second
  template branch. PHI redaction is the load-bearing design decision.
- Effort: L.

### Option B -- Diff off ABP audit log, dedup via job idempotency key [effort: L] [risk: med-high]

One-line: drive the email entirely from `AbpEntityChanges`/`AbpEntityPropertyChanges`
(the NEW substitute for the OLD diff table), keyed on the audit-change id, and use a
Hangfire idempotency token (or a `processed-change-id` marker) instead of a dedup table.

- Source: ABP audit-logging docs (entity history) -- framework capability is the
  crux here, so verify the audit-change query surface before committing:
  https://abp.io/docs/latest/framework/infrastructure/audit-logging (official ABP
  docs; accessed 2026-06-01; credibility: official).
- Tradeoffs: reuses the diff data already captured by `[Audited]` (no second diff
  computation), so it stays Equivalent with G-02-02's audit approach. But it
  inherits G-02-02's limitation: root-entity scalars only unless child entities are
  also `[Audited]` AND the query joins them -- so injury/body-part/claim edits would
  NOT appear unless that gap is closed first.
- Risk: med-high. Querying audit changes for a specific request/transaction is
  fiddlier than an explicit before/after snapshot; harder to scope to "this edit."
  Idempotency without a durable marker is brittle on job retries.
- Effort: L.

### Option C -- Defer until the change-log feature (G-02-01/02) lands, then bundle [effort: M to revisit] [risk: low]

One-line: treat G-02-03 as blocked on the change-log capture work; implement the
email as a rider on whichever change-log feature delivers child-entity diff capture
and a queryable diff source.

- Source: parity doc cross-refs (this item depends on G-02-01/02 + G-10-05; G-04-03
  is the same gap from the email side); INDEX line 126 already groups them.
- Tradeoffs: avoids building a diff source twice (the change-mail and the audit
  viewer want the same field-level diff, ideally spanning child entities). Lowest
  rework risk. Cost: the notification stays silent until the bundle ships.
- Risk: low. Pure sequencing decision.
- Effort: M when revisited (the email itself is small once a diff source + recipient
  resolver exist; the resolver already exists).

Recommendation hint for the fixer: Option A is the most self-contained and matches
the existing handler/resolver/job architecture; pair it with the dedup-table half of
G-10-05. If child-entity diffs (G-02-02) are in-scope for the same slice, lean toward
Option B's audit-driven diff only after child entities are confirmed `[Audited]`.

## 8. Root-Cause Hypothesis

High confidence. The NEW app replaced OLD's bespoke `spm.AppointmentChangeLogs`
table + hand-rolled diff/email orchestration with ABP `[Audited]` entity-change
tracking, which captures the diff DATA but carries none of the OLD side effects: no
email on edit, no recipient fan-out, no `IsMailSent` dedup (audit rows are
immutable). The template codes (`AppointmentChangeLogs`,
`AppointmentRescheduleRequestByAdmin`) were scaffolded into
`NotificationTemplateConsts` during the email-template port but never given a
handler, an `Eto`, or a `NotificationKind` -- so the wiring is a known, deliberate
stub (consistent with G-04-03's "seeded but unwired" finding and the project's
Phase-1 email scope that activated only verification + requested + approved/rejected
emails). The gap is a not-yet-implemented feature, not a regression.

## 9. Open Questions

- PHI redaction policy: which intake fields may appear in an email body at all, and
  to which recipient roles? OLD sent raw values to all stakeholders; that is the one
  OLD behavior that must NOT be ported verbatim. Needs Adrian's call before build.
- Dedup home (G-10-05): new `AppointmentChangeNotification` table vs job idempotency
  token vs a column elsewhere. Affects whether a migration is needed.
- Diff scope: does the change-mail need child-entity edits (injury/body-part/claim,
  G-02-02), or root-scalar only for Phase 1? Determines whether this is blocked on
  G-02-02 and whether child entities must be `[Audited]`.
- Date/Time second email: should the NEW app keep OLD's separate
  `AppointmentRescheduleRequestByAdmin` one-shot email when a Date/Time field
  changes, or fold it into the single change-summary email? OLD sent two distinct
  emails.
- Recipient set: OLD's stakeholder JSON (`GetNotificaionEmailAndPhoneNumers`) vs the
  NEW `AppointmentRecipientResolver` -- confirm the resolver's role set matches OLD's
  intended audience for change notifications (it was built for status-change emails).
