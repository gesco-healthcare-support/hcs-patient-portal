## 1. ID & Summary
OLD's forgot-password flow returned three DISTINCT messages (not-verified / inactive /
not-found), which leak account state (account enumeration). NEW deliberately collapses all
outcomes to one generic "if the email matches a registered account, a link is on its way."
This is logged as an Intent deviation, but it is an intentional SECURITY improvement, not a
defect. Deferred pending Adrian's explicit confirmation that we will NOT revert to OLD's
three messages (reverting would re-introduce the enumeration leak, PHI-adjacent on a medical
portal). A middle option exists: keep the generic public message but log the specific reason
server-side.

## 3. Affected Code (brief: where the gap is + gating dependency)
- OLD distinct messages: UserAuthenticationDomain.cs:157-179 -- StatusId==Delete/null ->
  UserNotExist; !IsVerified -> "We have sent a verification link..."; StatusId==InActive ->
  "Your account is not activated"; else (user null) -> UserNotExist.
- NEW gate (throws distinct exceptions): PasswordResetGate.cs:49-65 -- null user -> silent
  return; !EmailConfirmed -> BusinessException(EmailNotConfirmedForPasswordReset);
  !IsActive -> BusinessException(UserInactiveForPasswordReset).
- NEW collapse point: ForgotPassword.cshtml.cs:63-94 -- OnPostAsync wraps
  SendPasswordResetCodeAsync in try/catch(Exception), swallows ALL exceptions with a
  LogWarning (lines 79-90), then always sets RequestSubmitted=true and renders the same
  generic success page. The distinct BusinessException codes never reach the user.
- Server-side reason today: SendPasswordResetCodeAsync (ExternalAccountAppService.cs:82-144)
  does NOT catch the gate's BusinessException (its own catch at :137 only covers dispatch
  failures); the exception propagates to the page catch-all and is logged as a generic
  "threw" warning carrying the email key (ForgotPassword.cshtml.cs:86-89). The specific
  DomainErrorCode is therefore NOT explicitly logged today -- relevant to the middle option.
- Gating dependency: none technical. The "dependency" is a human decision (B-06-05 / Adrian
  sign-off), not an upstream feature.

## 4. Expected vs Actual
- OLD (Expected-per-strict-parity): user told exactly why reset failed (unverified vs
  inactive vs not-found).

- Intended NEW target: KEEP the generic message (security). The "gap" is only a gap vs literal
  OLD parity; per CLAUDE.md bug-policy this is an OLD security bug deliberately not ported.

## 7. Candidate Solution (high-level; deep sourcing deferred)
Three options, decision pending:
- (A) No change -- keep OWASP-generic message. Zero code. Recommended default; matches the
  intentional security posture and the documented OLD-bug-fix (B-06-05).
- (B) Middle option (CSV note) -- keep generic public message, but log the SPECIFIC gate
  reason server-side. Concretely: in PasswordResetGate or SendPasswordResetCodeAsync, emit a
  structured LogInformation/LogWarning naming the gate outcome (email-not-confirmed vs
  inactive) before/instead of letting it surface as a bare "threw" at the page. Small,
  additive, no UX change. Effort S.
- (C) Revert to OLD three messages -- REJECTED unless Adrian explicitly accepts the
  enumeration leak. Re-introduces B-06-05 (PHI-adjacent). Not recommended.
Deep external sourcing (e.g., OWASP forgot-password cheat-sheet citation) deferred until a
decision is made; option (A) needs no sourcing, option (B) is a logging tweak.

## 8. Root-Cause Hypothesis
Not a code defect -- a deliberate divergence. NEW's authors fixed OLD's account-enumeration
leak (B-06-05) by funneling all forgot-password outcomes through a single generic response.
The gate still computes the distinct reason (so the logic exists) but the Razor page
intentionally hides it. The only open question is whether to ALSO log the reason (option B),
which today is only partially captured (generic "threw" warning, no explicit reason code).
