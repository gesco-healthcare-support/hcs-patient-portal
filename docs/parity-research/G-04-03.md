## 1. ID & Summary

When internal staff edit a confirmed appointment, OLD collects every
field-level change-log row that has not yet been mailed (`IsMailSent ==
false`), renders an HTML diff table of `FieldName / OldValue / NewValue`,
emails it to the appointment stakeholders via the
`Appointment-Change-Logs.html` template, and then flips those rows to
`IsMailSent = true`. A second, single email
(`Appointment-Reschedule-Request-Changed-By-Admin.html`) fires once per
update if a Date/Time field changed. NEW sends nothing on appointment
edit: no diff is built, no change-summary email is dispatched, and no
`IsMailSent` flag exists anywhere in the NEW schema.

This is the email leg of G-02-03 (the diff + dedup mechanism), which in
turn sits on top of G-02-02 / G-02-01 (a change-log store that captures
field-level diffs including child entities). The NEW template codes
`AppointmentChangeLogs`, `AppointmentRescheduleRequestByAdmin`, and
`PatientAppointmentRescheduleReqAdmin` are all seeded as per-tenant
NotificationTemplate rows, but no handler or appservice dispatches them.

Gap is confirmed present in current code (static confirmation).

### Blast radius

A faithful fix spans 3-4 layers and is gated on an upstream feature:

1. Domain.Shared -- a new `*Eto` (e.g. `AppointmentFieldsChangedEto`)
   carrying the diff payload + confirmation number. (1 new file.)
2. Domain / Application -- compute the field diff on Update. This is the
   blocker: there is no `AppointmentChangeLog` store, no `IsMailSent`
   dedup, and child-entity diffs (injury / body-part / claim) are not
   captured. Either (a) build the change-log store (G-02-01/02/03 work),
   or (b) derive the diff from ABP audit-log entity changes for the
   root entity only (loses child-entity fidelity). Publish the new ETO
   from `AppointmentsAppService.UpdateAsync`. (1-2 files touched + the
   change-log feature itself.)
3. Application/Notifications/Handlers -- 1 new handler subscribing to
   the ETO, dispatching `AppointmentChangeLogs` (+ the date/time
   `AppointmentRescheduleRequestByAdmin` leg) via the dispatcher.
   (1-2 new files.)
4. NotificationTemplate body -- confirm/author the
   `AppointmentChangeLogs` + `AppointmentRescheduleRequestByAdmin`
   HTML bodies in `EmailBodyResources` + subjects in `EmailSubjects.cs`.

Regression risk: LOW for existing flows in isolation (purely additive --
a new handler + a new ETO, nothing rewires existing sends). The REAL
risk and cost are in the diff source: without the G-02-03 change-log
store + `IsMailSent` dedup, the email either cannot reproduce
child-entity diffs (G-02-02) or risks re-sending the same diff on every
update (no dedup). PHI sensitivity is high: the body contains a table of
exact old/new values for intake fields (claim numbers, injury details,
patient demographics) -- recipients are the stakeholder distribution
list, so the body must be redacted in logs and the recipient set must be
correct (see HIPAA note in Open Questions).

## 7. Candidate Solutions

### Option A -- Wire on top of the G-02-03 change-log store (full OLD parity) [effort: L] [risk: medium]
- One-line: build/finish the NEW change-log store (G-02-01/02/03), add
  an `IsMailSent` flag, then add an ETO + handler that mails the unmailed
  diff and flips the flag -- a faithful port.
- Source: OLD code `AppointmentChangeLogDomain.cs:267-378`,
  `AppointmentDomain.cs:569`.
- Tradeoffs: highest fidelity (captures child-entity diffs, exactly-once
  dedup, date/time second email). But it is gated on G-02-03 landing
  first, so effort is dominated by that dependency. The CSV note
  ("build with change-diff + IsMailSent dedup") points squarely at this
  option.
- Risk: medium -- new persistence + dedup state; PHI in the body.
- Effort: L (the email handler itself is S; the change-log store is the
  weight, owned by G-02-03).

### Option B -- Derive the diff from ABP audit-log entity changes (no new store) [effort: M] [risk: medium]
- One-line: on Update, read the ABP audit-log entity changes for the
  Appointment root (the same source the per-appointment viewer already
  uses) since the prior version, format them into the
  `##AppointmentChangeLogs##` table, and dispatch the email; gate
  re-sends with a lightweight per-appointment "last mailed change time"
  marker instead of a per-row `IsMailSent`.
- Source: existing NEW pattern
  (`appointment-change-logs.component.ts:39` proves root-scalar audit
  capture); ABP audit logging
  (https://abp.io/docs/latest/framework/infrastructure/audit-logging --
  official, accessed 2026-06-01; credibility: official docs).
- Tradeoffs: avoids building a new change-log store; reuses ABP's audit
  trail. BUT loses child-entity field diffs (injury / body-part /
  claim) which OLD included (G-02-02), and the dedup is coarser
  (timestamp marker vs per-row flag) so it can miss or duplicate edge
  cases. Audit logs are async/queued in ABP, so "diff since last mail"
  timing is fragile.
- Risk: medium -- partial-fidelity diff; timing coupling to the audit
  pipeline.
- Effort: M.

### Option C -- Defer until G-02-03 lands; keep rows seeded [effort: S] [risk: low]
- One-line: leave the three template rows seeded-but-unwired and
  implement nothing now; revisit when the change-log feature (G-02-03)
  is scheduled, then take Option A.
- Source: matches the source doc's "Keep in NEW? Yes, pair with the
  appointment change-log / audit feature when it lands" (04 G-04-03)
  and the CSV framing ("Email leg of G-02-03 (already included)").
- Tradeoffs: zero immediate work, but the parity gap persists
  (stakeholders not notified of edits). Correct sequencing given the
  hard dependency on G-02-03.
- Risk: low.
- Effort: S (no code; tracking only).

## 8. Root-Cause Hypothesis

The change-log email was never wired because its data source -- a
field-level change-log store with an `IsMailSent` dedup flag -- was
never ported. NEW replaced OLD's bespoke `AppointmentChangeLog` table +
reflection-based `ChangeLogs` differ with ABP's built-in audit-log
entity-change tracking (per-appointment viewer only), which captures
root-entity scalar diffs but offers no per-row "mailed" flag and no
child-entity diff surface. The template rows were dutifully seeded
(`Codes.All`) during the email-system port, but with no diff+dedup
mechanism upstream there was nothing to dispatch them from, so the codes
landed in the "seeded, unwired" bucket. This is a sequencing gap, not a
defect: the email leg (G-04-03) is correctly blocked on the change-log
mechanism (G-02-03), which is itself blocked on the change-log store
(G-02-01/02). Confidence: high -- the absence of dispatch references,
the absence of an `IsMailSent` flag, and the silent NEW Update path are
all directly observable.
