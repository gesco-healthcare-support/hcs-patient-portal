## 1. ID & Summary

When an external user submitted a Contact-Us / "Need Question?" query in
the OLD app, the query email was routed by a single branch:

- **If** a confirmation number was supplied AND an **Approved** appointment
  matched that `AppointmentId`, the email went to that appointment's
  `PrimaryResponsibleUserId` (the owning internal staffer), with a subject
  line built from patient name + claim + ADJ.
- **Else** the email fanned out to **all IT-Admins** (`Roles.ItAdmin`).

The NEW stack has no `UserQuery` feature at all -- no entity, no submit
endpoint, no email handler -- so this routing logic does not exist. G-09-03
is the routing half of the Submit-Query trio: G-09-02 (entity + endpoint +
navbar Help button + modal) and G-04-02 (the confirmation-number capture on
the booking/home surface that feeds the optional `AppointmentId`). G-09-03
cannot be built standalone; it rides on top of G-09-02's submit AppService.

The good news: every primitive the routing needs already exists in NEW and
is exercised by a sibling handler (`StatusChangeEmailHandler`). The fix is a
small, well-precedented port, gated only on G-09-02 landing first.

## 3. Affected Code

### OLD (verified)

`P:/PatientPortalOld/PatientAppointment.Domain/UserQueryModule/UserQueryDomain.cs`
-- the routing logic spans lines 53-107 (doc cited 77-105; the
appointment lookup that drives the branch starts at line 62, so the true
range is 62-105). Verified verbatim:

- **L55-56** -- stamps `CreatedById = UserClaim.UserId`,
  `CreatedDate = DateTime.UtcNow`.
- **L62** -- looks up the matching appointment:
  `vAppointment` where `AppointmentId == userQuery.AppointmentId`
  **AND** `AppointmentStatusId == (int)AppointmentStatusType.Approved`,
  `.FirstOrDefault()`.
- **L64-73** -- when an Approved appointment matched, builds the subject
  fragment: `"Patient: <First> <Last>"` + `" - Claim: <ClaimNumber>"`
  + `" - ADJ: <WcabAdj>"`, wrapped in parentheses
  (`patientDetailsEmailSubject`).
- **L74-75** -- `RegisterNew<UserQuery>` + `Commit()` (the row is persisted
  BEFORE any email send -- relevant to B-09-02 below).
- **L77-90 (responsible-user leg)** -- `if (vAppointment != null && `
  `!string.IsNullOrEmpty(userQuery.RequestConfirmationNumber))`: looks up
  `vInternalUserEmail` where `UserId == vAppointment.PrimaryResponsibleUserId`,
  reads `.EmailId`, renders the `EmailTemplate.UserQuery` body, and sends one
  email with subject `"Patient Appointment Portal - <subject> - User query"`.
- **L91-105 (IT-Admin fan-out)** -- `else`: queries all `vInternalUserEmail`
  where `RoleId == (int)Roles.ItAdmin`, concatenates their `EmailId`s with
  `;` separators, renders the same template, and sends one email to the
  semicolon-joined recipient list.

Note the branch condition: it requires BOTH `vAppointment != null` AND a
non-empty `RequestConfirmationNumber`. The Approved-appointment lookup at
L62 is keyed on `userQuery.AppointmentId` (a `[NotMapped]` field, see
below), but the routing decision additionally re-checks the confirmation
number string. So a query with a matching AppointmentId but a blank
confirmation number still falls through to the IT-Admin fan-out.

## 4. Expected vs Actual

**Expected (OLD parity):** On query submit, if a confirmation number is
present and resolves to an **Approved** appointment, email that
appointment's `PrimaryResponsibleUserId` (the owning internal staffer) with
a patient/claim/ADJ subject line; otherwise email all IT-Admins. Render the
body from the `UserQuery` (NEW: `UserQuery`/`SubmitQuery`) template.

**Actual (NEW):** No UserQuery feature exists; queries cannot be submitted
and nothing routes. The `UserQuery`/`SubmitQuery` template codes are seeded
but have no handler. A regression vs OLD intent the moment Contact-Us is
added without the routing.

## 7. Candidate Solutions

### Option A -- Port routing into a `UserQuerySubmitted` local-event handler [effort: M (incremental over G-09-02: S)] [risk: low]

One-line: Mirror `StatusChangeEmailHandler` -- a new
`UserQuerySubmittedEmailHandler : ILocalEventHandler<UserQuerySubmittedEto>`
that runs the OLD branch using the two existing reuse patterns.

Routing logic (verbatim port of UserQueryDomain.cs:62-105, B-09-02-guarded):
1. If `AppointmentId` provided, load the appointment and check
   `Status == Approved` (OLD L62). Build the patient/claim/ADJ subject
   fragment from the resolved patient + injury details (OLD L64-73).
2. **If** appointment matched AND confirmation number non-empty (OLD L77):
   resolve `PrimaryResponsibleUserId` -> `IdentityUser.Email` using the
   guarded pattern at StatusChangeEmailHandler.cs:295-317. If the user/email
   is missing, **fall back to the IT-Admin fan-out** (fixing B-09-02 rather
   than throwing). Dispatch one email via `INotificationDispatcher` with the
   `UserQuery`/`SubmitQuery` template and the patient/claim/ADJ subject.
3. **Else** (OLD L91-105): fan out to every user in role `"IT Admin"` via
   `_userManager.GetUsersInRoleAsync("IT Admin")` (the
   `ResolveNoShowInternalRecipientsAsync` pattern at L547-569, role name
   swapped). Dispatch the same template.

- **Source:** OLD code `UserQueryDomain.cs:62-105`; NEW precedent
  `StatusChangeEmailHandler.cs:295-317` (responsible-user leg) + `:547-569`
  (role fan-out). No external dependency.
- **Tradeoffs:** Decouples the email side-effect from the persist (matches
  OLD's commit-then-mail order, and matches every other NEW notification
  handler) -- if the template is missing the submit still succeeds (the
  handler swallows `NotificationTemplateNotFound` exactly as
  StatusChangeEmailHandler.cs:219-234 does). Cleaner than inlining.
- **Risk:** Low. All primitives proven in production by the sibling handler.
- **Deviations to flag:** (a) Fix B-09-02 -- never throw on a missing
  responsible-user email; fall back to IT-Admins. (b) The subject line
  contains patient PHI (name/claim/ADJ); OLD put it in the email subject
  verbatim -- preserve for parity but confirm the per-recipient PHI scope is
  acceptable (the responsible-user leg is internal-only, the IT-Admin leg is
  internal-only, so PHI stays internal -- matches OLD).

### Option B -- Inline routing directly in the submit AppService [effort: M] [risk: low-med]

One-line: Put the branch inside `UserQueriesAppService.CreateAsync` after
the entity insert, synchronously, before returning.

- **Source:** same OLD logic.
- **Tradeoffs:** Simpler call graph (no Eto), but couples the SMTP fan-out to
  the request path -- a template-missing or SMTP hiccup would surface to the
  external user even though the query already persisted. Diverges from the
  established NEW notification pattern (every other email side-effect is an
  event handler).
- **Risk:** Low-med -- behavioral parity is fine, but it breaks the
  architectural consistency the codebase has standardized on, and re-creates
  the OLD coupling that caused B-09-02 to surface to the user.

Recommendation: **Option A.** It matches both OLD behavior and the NEW
codebase's own notification convention, reuses two already-tested code
paths, and naturally fixes B-09-02. Effort is M for the trio overall; the
G-09-03-specific increment is S once G-09-02's entity + submit endpoint
exist.

## 8. Root-Cause Hypothesis

The routing is absent because the **entire Submit-Query feature was never
ported** (G-09-02), not because the routing was specifically dropped.
Confidence: **high**. Evidence: (a) zero UserQuery/Note artifacts in `src/`
and `angular/`; (b) the `top-header-navbar` has a vestigial
`showHelp`/`helpClick` API but renders no button and nothing binds it
(per source doc L194-196); (c) the notification template codes
`SubmitQuery`/`UserQuery` are pre-seeded with an explicit "unwired until
post-parity" note (NotificationTemplateConsts.cs:56-61). This is a clean
"not-yet-built feature," not a behavioral deviation -- so G-09-03 is purely
a forward port, gated on G-09-02. The OLD logic is fully understood and
every NEW primitive it needs already exists and is exercised by a sibling.

## 9. Open Questions

1. **Confirmation-number vs AppointmentId as the routing key.** OLD's L62
   lookup keys on the NotMapped `AppointmentId`, but the responsible-user
   branch (L77) additionally requires a non-empty `RequestConfirmationNumber`
   string. In NEW the natural key is the confirmation number (the user-facing
   token); the submit DTO should resolve confirmation number -> appointment
   GUID + AppointmentId. Confirm with G-09-02's DTO design which field the
   external user actually supplies (likely just the confirmation number).
2. **Approved-only constraint.** OLD only routes to the responsible user when
   the appointment is **Approved** (L62). A query referencing a Pending /
   Rejected / Cancelled appointment falls through to IT-Admins. Confirm this
   is intended in NEW (it is a deliberate-looking filter, so port verbatim).
3. **IT-Admin fan-out recipient set.** OLD fans out to `Roles.ItAdmin` only
   (RoleId == 1). NEW role name is `"IT Admin"`. Confirm there is at least
   one IT-Admin per tenant (single-tenant Phase 1: the demo office) so the
   fallback is non-empty; if zero, the handler should log and no-op (matching
   the NoShow handler's empty-set behavior at StatusChangeEmailHandler.cs:470-476).
4. **Subject-line PHI.** OLD embeds patient name + claim + ADJ in the email
   subject. Both legs are internal-only recipients, so PHI stays internal
   (parity-safe), but confirm the redaction policy permits PHI in subjects
   for internal staff mail before shipping.
