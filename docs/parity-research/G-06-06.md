## 1. ID & Summary

OLD forced a single active session per user: on every successful login, the
auth domain deleted ALL existing `ApplicationUserToken` rows for that user
before inserting the freshly issued one. Because the OLD JWT middleware
validated each request against the stored token row (DB-backed validation),
deleting the prior rows immediately invalidated any session on another
device -- logging in on device B silently logged out device A.

NEW (ABP / OpenIddict) issues a standard authorization-code + PKCE access
token (and rotating refresh token) per login with NO prior-token revocation.
A user can hold multiple concurrent valid sessions across devices/browsers.

The gap is real and statically confirmed. Note the disposition tension: a
PRIOR decision (documented as an "Acceptable deviation", locked 2026-05-01)
treated multi-session as intentional. G-06-06's `include` disposition with
the CSV note "reverses prior accepted-deviation (multi-session)" means the
team now wants single-session RE-ADDED. This reversal is the load-bearing
fact for whoever implements it -- it is a policy decision, not a defect fix.

## 3. Affected Code

### OLD (verified)

`PatientAppointment.Domain/Core/UserAuthenticationDomain.cs` (the parity doc
cited it without the `Core/` segment -- minor path drift; behavior matches
exactly):

- Lines 83-90 -- the single-session mechanism. Verbatim:
  ```csharp
  ApplicationUserToken applicationUserToken = JwtTokenProvider.WriteToken(user);
  var applicationUserTokens = LoginUow.Repository<ApplicationUserToken>()
      .FindBy(x => x.UserId == user.UserId).ToList();
  foreach (var userToken in applicationUserTokens)
  {
      LoginUow.RegisterDeleted(userToken);
  }
  LoginUow.RegisterNew(applicationUserToken);
  LoginUow.Commit();
  ```
  Confirmed: writes the new token, finds ALL token rows by `UserId`, deletes
  every one, inserts the new one, commits in a single UoW. Net result: exactly
  one `ApplicationUserToken` row per user after any successful login.

`PatientAppointment.Infrastructure/Security/JwtTokenProvider.cs` -- the
enforcement half (why deleting the row matters):

- Lines 62-94 `Validate(HttpContext)`: looks up the request's bearer token in
  the DB (`Repository<ApplicationUserToken>().SingleOrDefault(x => x.JwtToken
  == jwtToken)`); if the row is gone, `applicationToken` is null and the method
  returns `false` (line 93). So OLD tokens were DB-backed / reference-style:
  deleting the row was an immediate, server-enforced session kill.
- Lines 27-30 `LogOut()` is an empty no-op: OLD "logout" was effectively the
  next login overwriting the row plus the client discarding its localStorage
  token. There was no explicit server-side revoke-on-logout path.

Conclusion: NEW has no single-session enforcement. Gap confirmed by static
code inspection.

### Upstream callers

- The login surface is the AuthServer Razor authorize/token endpoint chain
  (`/connect/authorize` -> `/connect/token`), driven by the SPA's
  `@abp/ng.oauth` AuthorizationCode+PKCE flow. There is no app-layer
  "PostLogin" service in NEW equivalent to OLD's `UserAuthenticationDomain`;
  the OpenIddict server module is the single integration point. A fix lives in
  `CaseEvaluationAuthServerModule.PreConfigureServices` (event handler) or a
  custom token controller -- one project, the AuthServer.

### Downstream consumers

- The HttpApi.Host validates bearer tokens via the same OpenIddict local-server
  config (`CaseEvaluationHttpApiHostModule`). For a revoked prior token to be
  rejected IMMEDIATELY at the API (matching OLD's reference-token behavior),
  token-entry validation must be active on the validation side. ABP's default
  non-degraded OpenIddict setup stores tokens in the `OpenIddictTokens` table
  and validates against it; `UseLocalServer()` inherits the server's
  token-entry validation. This must be VERIFIED for this stack (open question)
  -- if self-contained JWTs without entry-validation are in play, a revoked
  access token would still pass until its short lifetime expires.
- Refresh-token rotation already caps the practical window of a stale access
  token (line 136-149 docblock). Revoking only the refresh-token entry on a new
  login would kill the OLD session at its next access-token refresh, not
  instantly -- a softer-than-OLD variant.

### Blast radius

- Files touched by a fix: 1 primary (`CaseEvaluationAuthServerModule.cs`),
  plus possibly 1 config touch on the HttpApi.Host validation if entry
  validation must be explicitly enabled. No DB schema change (OpenIddict
  already persists token entries). No Angular change. No proxy regeneration.
- Layers: AuthServer (token issuance) + potentially HttpApi.Host (validation).
- Regression risk: MEDIUM. A `HandleTokenRequestContext` handler that revokes
  prior tokens runs inside the token pipeline; a bug there can break ALL
  logins. Revoking the wrong subject's tokens, or revoking on refresh (not
  just login), could log users out unexpectedly. The refresh-token reuse-leeway
  customization (lines 147-150) shows the team already edits this pipeline, so
  the integration point is established but sensitive.
- Reverses a documented accepted-deviation -> requires sign-off, not just code.

## 4. Expected vs Actual

- Expected (OLD parity / G-06-06 target): a successful login invalidates the
  user's other active session(s); a second device silently terminates the
  first. One effective active session per user.
- Actual (NEW): each login mints an independent token set; prior sessions stay
  valid until their own token expiry / refresh. Multiple concurrent sessions
  per user are allowed.

## 7. Candidate Solutions

### Option A -- HandleTokenRequestContext handler revokes prior user tokens on login [effort: M] [risk: M]

One-line: Add an OpenIddict server event handler in
`CaseEvaluationAuthServerModule.PreConfigureServices` that, on a successful
authorization-code token request, resolves `IOpenIddictTokenManager`, lists
prior tokens for the subject (`FindBySubjectAsync`) and revokes all but the
new one (`TryRevokeAsync`), reproducing OLD's "delete all prior rows" exactly.

- Source: OpenIddict core docs -- custom server event handlers
  (`AddEventHandler<OpenIddictServerEvents.HandleTokenRequestContext>`) and
  `IOpenIddictTokenManager` revocation API.
  https://documentation.openiddict.com (Context7 mirror
  https://context7.com/openiddict/openiddict-core/llms.txt), accessed
  2026-06-01. Credibility: official library docs (highest). OLD parity source:
  `UserAuthenticationDomain.cs:83-90`.
- Tradeoffs: Closest behavioral match to OLD. Immediate cross-device logout
  ONLY if the API validation side does token-entry validation against the
  store (default in ABP non-degraded mode -- VERIFY). If not, the killed access
  token survives until expiry (mitigated by short lifetime + the 2s
  reuse-leeway already set). Must scope revocation to the SAME client so an
  unrelated client's tokens are not nuked. Must NOT fire on refresh-token grant
  (only on fresh authorization-code login) or normal refresh would self-revoke.
- Risk: M -- runs inside the token pipeline; a fault breaks all logins.

### Option B -- Revoke only the prior refresh-token entry on login [effort: M] [risk: M-low]

One-line: Same hook point, but revoke only the user's prior REFRESH tokens, so
the old session dies at its next silent refresh rather than instantly.

- Source: same OpenIddict event-handler + token-manager APIs as Option A.
  OLD parity source: `UserAuthenticationDomain.cs:83-90` (behavioral intent).
- Tradeoffs: Lower blast radius (does not depend on access-token entry
  validation; refresh tokens are always store-checked during rotation). Plays
  naturally with the existing rotation + 2s reuse-leeway. NOT exact OLD parity:
  the old session lingers up to one access-token lifetime before its refresh
  fails. Acceptable if the team's intent is "one durable session" rather than
  "instant kill".
- Risk: M-low -- narrower surface; still pipeline-resident.

### Option C -- Document-and-decline (keep multi-session, record the reversal explicitly) [effort: S] [risk: low]

One-line: Treat G-06-06 as a policy question first: confirm with Adrian whether
single-session is a deliberate requirement before any code; if not, re-affirm
the 2026-05-01 accepted-deviation and update the parity log.

- Source: `docs/design/external-user-login-design.md` lines 176-194 (existing
  accepted-deviation) + CSV note flagging the reversal. OLD parity source:
  `UserAuthenticationDomain.cs:83-90`.
- Tradeoffs: The disposition is `include`, so this is NOT the default outcome
  -- but the CSV explicitly flags that this overturns a locked decision, and
  the parity doc itself says "Likely drop ... Flag for Adrian". Cheapest path
  if the reversal was provisional. Spend zero engineering risk until the policy
  is confirmed.
- Risk: low. The only cost is a clarification turn.

Recommended ordering: confirm the policy reversal (Option C as a gate), then
implement Option A for exact OLD parity (or Option B if "instant kill across
devices" is not required).

## 8. Root-Cause Hypothesis

Confidence: HIGH. The gap is not a defect but an architectural consequence:
OLD used a custom single-row reference-token store (`ApplicationUserToken`,
one row overwritten per user per login, DB-validated per request), so
single-session fell out for free. NEW uses ABP/OpenIddict standard OAuth2
(authorization-code + PKCE + rotating refresh, self-contained or DB-entry
tokens) which by design supports concurrent sessions and performs no
revoke-on-login. No NEW code ever ported OLD's "delete all prior tokens"
step because the team consciously accepted multi-session as a deviation
(external-user-login-design.md row 2, locked 2026-05-01). G-06-06 reverses
that accepted deviation. Re-adding single-session is a deliberate add of an
OpenIddict token-revocation hook, not the repair of a regression.

## 9. Open Questions

- Is the policy reversal final? G-06-06 `include` contradicts the locked
  2026-05-01 "multi-session is intentional" decision. Needs Adrian's explicit
  confirmation before code (this is the gate, per Option C).
- Does this stack's OpenIddict validation actually do token-entry validation
  against the `OpenIddictTokens` store (so a revoked access token is rejected
  immediately at the API), or are self-contained JWTs used (revoked token
  survives until expiry)? Determines whether Option A gives OLD's instant kill
  or only a delayed one. Verify in the resolved OpenIddict server options /
  ABP defaults for 10.0.2.
- Exact-parity semantics: OLD killed the prior session on the NEXT request to
  the API. Do we need instant cross-device logout, or is "old session dies at
  next refresh" (Option B) acceptable as the parity bar?
- Scope: should revocation be per-client (CaseEvaluation_App only) or all
  clients for the subject? OLD had a single web client, so the distinction did
  not exist; NEW has Swagger + App clients.
- Interaction with impersonation (AbpAccountPublicWebImpersonationModule,
  AuthServerModule lines 47, 368-373): impersonation mints tokens too -- the
  handler must not revoke an admin's own session when starting impersonation.
- Relationship to G-06-07 (lockout / rate-limit): same login surface; a
  combined auth-hardening slice may be more efficient than two separate edits.
