## 1. ID & Summary

G-10-02 is the **storage side** of the OLD "Submit-Query / Contact-Us"
feature. OLD has a `spm.UserQueries` table (entity `UserQuery`) holding
a logged-in user's free-text question (`Message`, max 500 chars) plus a
`UserId` FK and full audit columns. NEW has no `UserQuery` entity, no
`DbSet`, no table, no AppService and no UI.

Per the CSV note this row is a **duplicate of the included behavioral
items G-09-02 (Submit-Query modal + endpoint) and G-09-03 (email
routing)**. G-10-02 only adds the data-model contract: the exact column
shape the persisted row must have. There is no independent gap to fix
beyond what G-09-02/03 already require -- when those land, the table
lands with them. This record exists so the column contract is recorded
with verified evidence.

## 4. Expected vs Actual

- **Expected (OLD parity):** a persisted store for an external user's
  free-text question -- `Message` (<= 500 chars), the submitting user,
  and created/modified audit. One row per submitted query.
- **Actual (NEW):** no entity, no table, no DbSet. A user's question
  cannot be persisted; the feature is entirely absent (matching the
  behavioral gaps G-09-02/03).

Mapping note for the port: OLD's separate `UserId` (FK to `dbo.Users`)
and `CreatedById` are effectively the same person (the logged-in
submitter -- `Add()` sets `CreatedById = UserClaim.UserId` and the row's
`UserId` comes from the posted DTO bound to the current user). In NEW,
ABP's `FullAuditedAggregateRoot<Guid>` supplies `CreatorId`,
`CreationTime`, `LastModifierId`, `LastModificationTime`, `IsDeleted`
automatically -- so `CreatedById`/`CreatedDate`/`ModifiedById`/
`ModifiedDate` collapse into ABP audit, and OLD's explicit `UserId` FK
is redundant (the creator IS the user). NEW gains a soft-delete column
OLD lacked; that is an acceptable framework difference, not a parity
break.

## 7. Candidate Solutions

### Option A -- Port UserQuery as an ABP FullAuditedAggregateRoot, bundled with G-09-02/03 [effort: S] [risk: low]

- **Desc:** Create `Domain/UserQueries/UserQuery.cs` as
  `FullAuditedAggregateRoot<Guid>, IMultiTenant` with `Message` (max 500
  via a `UserQueryConsts.MessageMaxLength` const). Add `DbSet` + entity
  config to BOTH `CaseEvaluationTenantDbContext` and
  `CaseEvaluationDbContext` (the project registers tenant entities in
  both per the existing pattern), `b.ToTable(prefix + "UserQueries",
  schema)`, `b.Property(x => x.Message).IsRequired().HasMaxLength(500)`,
  generate one EF migration. Do NOT add a redundant `UserId` FK to
  identity -- ABP `CreatorId` already captures the submitter (OLD's
  `UserId` and `CreatedById` were the same person). Optionally carry a
  nullable `AppointmentId` column IF the routing (G-09-03) needs the
  confirmation link persisted rather than transient -- but OLD kept it
  transient (`[NotMapped]`), so default to NOT persisting it.
- **Source:** OLD code:
  `PatientAppointment.DbEntities/Models/UserQuery.cs:12-55`,
  `_local/generated-schema.sql:389-399`,
  `PatientAppointment.Domain/UserQueryModule/UserQueryDomain.cs:53-107`.
  ABP audit-base pattern matches the existing NEW entities (e.g.
  `Appointment`, `Invitation`) which use ABP audit columns rather than
  manual `CreatedById`/`CreatedDate`.
- **Tradeoffs:** Gains an `IsDeleted` soft-delete column OLD did not
  have (ABP default) and drops OLD's explicit `UserId` FK as redundant.
  Pro: matches every other NEW entity, zero new patterns. Con: schema is
  not a byte-for-byte OLD copy -- acceptable per the project's
  "logical/behavioral parity, NOT code/framework parity" directive.
- **Risk:** Low. Additive table, no impact on existing queries. Must be
  `IMultiTenant` so it lands in the tenant DB (Phase-1 single tenant).

This is the only sensible option for a pure port; no competing approach
adds value. The decision of whether to also persist `AppointmentId`
belongs to the G-09-03 routing implementation, not this storage record.

## 8. Root-Cause Hypothesis

The new app was scaffolded around the appointment-booking core; the
low-value Contact-Us channel (described in the audit as "lightweight
in-app help", "likely low priority") was never scaffolded. The entity's
absence is simply un-ported scope, not a deviation or bug. Confidence:
HIGH -- the OLD entity/table/domain/controller all verified present, the
NEW tree verified to contain zero UserQuery code in any layer, and the
CSV/audit explicitly frame this as the storage half of the G-09-02/03
behavioral feature.

## 9. Open Questions

- Does the port need the **read/queue side** (OLD `Get()` /
  `GET api/UserQueries`)? G-09-02/03 only describe submit + email
  fan-out, implying no staff inbox screen. If "staff read the queue"
  (10-data-model.md:192) is in scope, an AppService list method + UI is
  additional work beyond this entity. Default assumption: submit-only.
- Persist `AppointmentId`/confirmation number as a column, or keep it
  transient like OLD? OLD kept it `[NotMapped]` (request-only). Resolve
  inside G-09-03 routing design.
- Soft-delete: ABP adds `IsDeleted` by default; OLD had none. Accept the
  extra column (recommended, matches all NEW entities) or use a
  non-soft-delete base. Recommend accepting it.
