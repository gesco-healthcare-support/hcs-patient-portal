## 1. ID & Summary
Internal-staff per-appointment Notes: a threaded comment feature (`Note` entity
with `ParentNoteId` grouping + `EditNoteId`/`IsLatest` soft-delete edit-chain),
exposed via a "Notes" modal on the internal appointment-detail queue. OLD built
the backend (`NotesController` + `NoteDomain`) and modal UI fully, BUT the only
launcher link was commented out, so it was unreachable in the shipped OLD app.
NEW has no Note entity, AppService, DTO, controller, migration, or UI. Deferred:
latent-in-OLD + large surface; revive only post-launch if owner wants it.

## 4. Expected vs Actual
- Expected (if revived to OLD parity-of-capability): internal staff can attach
  threaded, author-editable notes to a specific appointment; edits create a new
  row and soft-delete the prior, preserving an edit-chain + original created
  date; only the author may edit (OLD gate `parentNoteId == createdById`).
- Actual (NEW): feature does not exist at all. Because OLD shipped it disabled,
  there is NO live end-user behavior to match -- only the latent capability and
  any historical Note rows in the OLD production DB.

## 7. Candidate Solution (high-level; deep sourcing deferred)
Clean ABP CRUD slice (build with G-10-01), do NOT auto-port the latent OLD code:
- Domain: `Note` entity under `Domain/Notes/` (Comments, AppointmentId FK,
  ParentNoteId, EditNoteId, IsLatest, audited Created/Modified via ABP
  `FullAuditedEntity` -- replaces OLD's manual CreatedById/StatusId soft-delete).
- App: `NoteAppService` (`[RemoteService(IsEnabled = false)]`) + DTOs
  (`NoteCreateDto`/`NoteUpdateDto`/`NoteDto`) + Mapperly mapper.
- EF: migration adding the Notes table (this IS G-10-01).
- Permissions: internal-only (`Appointments.Notes.*` or a new node), NOT granted
  to external roles.
- UI: a notes panel/modal on the NEW appointment-view page.
- Reuse ABP soft-delete (`ISoftDelete`) + full-audit instead of OLD's manual
  `StatusId=Delete`/`IsLatest` pattern; model the edit-chain as a new row
  referencing the prior (`EditNoteId`) only if owner wants visible edit history,
  else a simple editable note. Guard the lookup OLD missed (B-09-01).
Deep external sourcing (ABP audit/soft-delete specifics) deferred until the
upstream un-defers and owner confirms scope.

## 8. Root-Cause Hypothesis
No bug -- this is a deliberate scope deferral. OLD built the feature but disabled
its only entry point before release, so it was never live; NEW simply never
ported it. It is large (entity + threading + edit-chain + internal-only UI +
migration) and low-value (zero live OLD usage), so it was correctly deferred
until after production launch, to be built together with its storage half
(G-10-01) only if the owner decides to revive it.
