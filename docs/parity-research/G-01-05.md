## 1. ID & Summary
OLD's booking validation (`CommonValidation` -> `AddValidation`) collects the
patient email plus every applicant-attorney and defense-attorney email into one
list and rejects the booking with "Please insert different email-id" if any
value repeats across those stakeholders. NEW's `CreateAppointmentInternalAsync`
performs no such cross-party check -- the patient, applicant attorney, and
defense attorney can all share one email on a single appointment. The colliding
emails then flow into the notification fan-out, where the recipient resolver
dedups by email first-wins, so one party silently gets the other's
notifications (or is dropped) under a single role tag -- a wrong-recipient PHI
exposure.

### Blast radius
A fix is a single additive validator (one pure helper + one call site in
`CreateAppointmentInternalAsync`), plus one localization key and one error code.
It touches the Application layer only (validator + AppService), Domain.Shared
(error-code constant + en.json key), and optionally a mirror client guard in
`appointment-add.component.ts`. No schema change, no migration, no proxy
regeneration (no DTO change). Regression risk is low and well-bounded: the only
behavior change is rejecting bookings that today succeed with colliding emails;
existing tests assert happy-path bookings with distinct emails and would be
unaffected. The one judgement call -- whether to include the claim-examiner
email in the set -- is a behavior choice, not a structural risk (see Open
Questions).

## 4. Expected vs Actual
- EXPECTED (OLD): On booking, if the patient email equals any applicant-attorney
  or defense-attorney email (or two attorneys share one), the request is
  rejected with "Please insert different email-id" before any row is written.
  Adjuster/claim-examiner email is excluded from the set (commented out in OLD).
- ACTUAL (NEW): No cross-party check. All parties may share one email; the
  booking succeeds. The colliding email then collapses in the notification
  resolver's first-wins dedup, so one stakeholder receives another's
  notifications under a single role tag, or is dropped -- a wrong-recipient PHI
  exposure.

## 7. Candidate Solutions
### Option A -- Port the OLD set-based uniqueness check into AppointmentBookingValidators [effort: S] [risk: low]
- Description: Add an `internal static bool HasDuplicateStakeholderEmail(params string?[] emails)`
  helper to `AppointmentBookingValidators` that trims + lower-cases non-empty
  values and returns true when distinct-count < count; call it early in
  `CreateAppointmentInternalAsync` and throw a `BusinessException` with a new
  localized error code when it returns true. Mirror OLD's party set
  (patient + AA + DA; exclude claim-examiner per OLD's commented-out adjuster
  line) unless Adrian decides to include CE.
- Source: OLD code: `AppointmentDomain.cs:700-729` (the verbatim
  set-build-and-distinct logic) + `AppointmentsController.cs:33-39` (hard-block
  semantics). Matches the existing extract-pure-validator pattern already used
  for lead-time/max-time/dedup in the same file.
- Tradeoffs: Smallest, OLD-faithful change; unit-testable without DB. Decision
  needed on whether CE email joins the set (OLD excluded it; NEW added a
  ClaimExaminerEmail column it did not have at booking, so this is a genuine
  scope choice, not a pure port).

### Option B -- Enforce via ABP FluentValidation on AppointmentCreateDto [effort: M] [risk: med]
- Description: Add an `AbstractValidator<AppointmentCreateDto>` cross-field rule
  that compares the 3-4 scalar email fields.
- Source: ABP validation infrastructure --
  https://abp.io/docs/latest/framework/fundamentals/validation (accessed
  2026-06-01; credibility: official ABP docs). ABP auto-discovers
  `IObjectValidationContributor` / FluentValidation validators registered in the
  module.
- Tradeoffs: Idiomatic ABP and keeps the rule declarative, but splits booking
  business rules across two places (DTO validator + AppService/Manager), which
  the project convention discourages ("business rules belong in the domain /
  validators consumed by the manager", per layer CLAUDE.md). Also fires only on
  the create DTO; reval/re-submit reuse the same DTO so it would still cover
  them, but the rule is less discoverable than co-locating with the other
  booking validators. Higher risk of convention drift for marginal benefit.

## 8. Root-Cause Hypothesis
The OLD per-appointment stakeholder email-uniqueness validator was simply not
carried over when the booking flow was re-implemented. The other `CommonValidation`
sub-checks (lead-time, max-time, per-injury duplicate, patient 3-of-6 dedup,
accessor-email dedup) were each extracted and ported individually during
Phase 11; this one block was missed -- plausibly because it sits between the
body-part dead-commented check and the patient-dedup method and reads as part of
the same region. The NEW structural shift from AA/DA collections to single
scalar email columns also made it easy to overlook, since "uniqueness across a
list" no longer has an obvious list to iterate. Confidence: high (validator
provably absent; OLD gate provably present and binding; downstream PHI impact
provable from the resolver's first-wins dedup).
