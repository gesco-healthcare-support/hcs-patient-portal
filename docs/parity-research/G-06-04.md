## 1. ID & Summary

OLD spelled out the login gate in domain code: in `PostLogin`, an unverified
user is blocked (and re-sent a verification link), and an inactive user
(`StatusId == InActive` OR `IsActive == false`) is blocked with `UserInactivated`.
NEW has no login domain code; both gates are framework-provided by ABP
(`AbpSignInManager`) / OpenIddict. The CSV disposition is **include** with the
narrow scope: "Add integration test pinning inactive+unverified login rejection
(behavior equivalent today but implicit/unguarded)." So this item is NOT a
behavior fix; it is a **test-coverage / regression-pin** task plus an accuracy
correction to how the doc describes the wiring.

Key correction surfaced by this research (raises confidence on the disposition):
the verified-only gate is NOT a bare framework default. It is **explicitly
configured** in project code at
`CaseEvaluationSettingDefinitionProvider.cs:99-104`, which overrides
`Abp.Identity.SignIn.RequireConfirmedEmail` default to `"true"` with an
OLD-parity citation comment. The active-only gate IS framework-implicit
(no project code; `AbpSignInManager.PasswordSignInAsync` rejects `IsActive=false`).
Neither gate has an automated test today.

### NEW (verified)

- `src/.../AuthServer/Pages/Account/Login.cshtml.cs:45,118-145` -- VERIFIED.
  `LoginModel : OpenIddictSupportedLoginModel`. `OnPostAsync` calls
  `base.OnPostAsync(...)` (the ABP Pro model does the credential + gate
  checks), then at lines 138-145 intercepts the Pro module's
  `/Account/ConfirmUser` redirect (the unverified-email branch) and reshapes it
  into a generic banner with a Resend link. There is NO explicit
  `IsActive` / `EmailConfirmed` / `PasswordSignInAsync` / `CanSignInAsync`
  call in this file -- the gates are entirely in `base`.
- `src/.../Domain/Settings/CaseEvaluationSettingDefinitionProvider.cs:86-104` --
  VERIFIED, and this REFINES the doc. The verified-only gate is explicit:
  lines 99-104 override `IdentitySettingNames.SignIn.RequireConfirmedEmail`
  default to `"true"`, with a comment citing OLD parity
  (`UserAuthenticationDomain.cs:143`, dated 2026-05-06). So the verified gate
  is NOT a bare default -- it is a deliberate, OLD-cited setting override.
  Lockout (lines 115-127) is likewise explicitly tuned (10 attempts / 3600s),
  which is the wiring behind G-06-07.
- ABP framework behavior (external, see Section 6 sources):
  - Active-only gate: `AbpSignInManager.PasswordSignInAsync` rejects users with
    `IsActive == false` (ABP-added since v5.0). This is the only gate that is
    truly framework-implicit with zero project code.
  - `IdentityUser` constructor sets `IsActive = true`, so a freshly registered
    user (`ExternalSignupAppService.cs:517-526`) is active and can log in once
    verified -- matching OLD (where new users were active but unverified).
  - Verified-only enforcement runs in `SignInManager.CanSignInAsync` /
    `PreSignInCheck` only when `SignIn.RequireConfirmedEmail == true` -- which
    is exactly the override above.
- Registration leaves `EmailConfirmed` at its column default `false`
  (`Migrations/...Initial.cs:262,517` -- `EmailConfirmed ... defaultValue: false`;
  `ExternalSignupAppService.RegisterAsync` does NOT call `SetEmailConfirmed`).
  So new external users are unverified until they click the link -- the gate
  has real subjects.

### Upstream callers

- `Login.cshtml` form -> `LoginModel.OnPostAsync` -> `base.OnPostAsync`
  (`OpenIddictSupportedLoginModel`) -> `AbpSignInManager.PasswordSignInAsync`.
  The login entry point is the AuthServer Razor page only; the Angular SPA has
  no login form (memory `project_authserver-ui-not-spa.md`).
- The token endpoint (OpenIddict password/authorization-code grant) is the
  programmatic path. Both ultimately funnel through `AbpSignInManager`, so the
  gates apply identically to the Razor UI and to a direct token request.

### Downstream consumers

- On rejection, no token is issued; the SPA never receives an access token, so
  `postLoginRedirectGuard` and all `[Authorize]` AppServices are never reached.
  Nothing downstream needs to re-check active/verified -- the gate is the single
  choke point, which is why a regression there would be silent and wide.

### Blast radius

- A regression-pin (the recommended fix) is **additive and tiny**: one new test
  file under `test/.../Application.Tests/` (or `AuthServer.Tests` if introduced).
  Zero production-code change. Zero layers touched in production.
- The risk being guarded is config drift: a single edit to
  `CaseEvaluationSettingDefinitionProvider.cs:103` (flip `"true"` -> `"false"`,
  exactly what happened 2026-05-05 per the comment) or a runtime DB setting
  override of `Abp.Identity.SignIn.RequireConfirmedEmail` would silently drop the
  verified gate with no failing test. The active gate could only regress via an
  ABP major-version change or a custom `SignInManager` override (none exists).
- Test-harness constraint (material to effort): the existing auth tests
  (`ExternalAccountAppServiceUnitTests.cs:12-26`) document a **pre-existing
  Phase 4 license-checker test-host crash** that prevents the full ABP
  integration harness from booting, so they use pure unit tests + reflection
  instead. A true end-to-end "POST credentials -> assert no token" integration
  test inherits that blocker. See Section 7 for options that avoid it.

## 4. Expected vs Actual

- Expected (OLD parity): an unverified user cannot log in (and is nudged to
  verify); an inactive user cannot log in. ACTUAL in NEW: identical outcome --
  unverified is blocked (Pro redirect intercepted into a generic banner +
  Resend), inactive is blocked by `AbpSignInManager`. Behavior matches today.
- Expected (per CSV disposition): an automated test pins both rejections so the
  implicit/config-driven gate cannot silently regress. ACTUAL: no such test
  exists (grep of `test/` for `IsActive|PasswordSignInAsync|CanSignInAsync|
  SignInResult|NotAllowed` returns only seed contributors and unrelated
  AppService tests). This is the actual gap.
- Doc-accuracy delta: the doc says both gates "rely on ABP framework defaults"
  and the gate is "invisible to a code reader." Half-true. The active gate is
  framework-implicit. The verified gate is an explicit, OLD-cited setting-default
  override in project code -- visible, but easy to flip (and was flipped once).

## 5. Repro Status

`not-reproduced-static-confirmed`. A true runtime repro ("register a user, leave
unverified, POST credentials, assert no token") requires creating and mutating
user state (a registration + a login attempt), which violates the read-only
guardrail. Static confirmation is conclusive: (a) OLD gate verified in source;
(b) NEW relies on `AbpSignInManager` (active) + the
`RequireConfirmedEmail="true"` override (verified), both verified in source and
against official ABP/Microsoft behavior; (c) absence of any pinning test
verified by grep. The "gap" here is missing test coverage, not a behavioral
defect, so there is nothing broken to reproduce.

## 7. Candidate Solutions

### Option A -- Settings-level unit test pinning the configured defaults [effort: S] [risk: low]

One pure unit test that constructs `CaseEvaluationSettingDefinitionProvider`,
runs `Define`, and asserts the resolved default for
`IdentitySettingNames.SignIn.RequireConfirmedEmail == "true"` (and, as a bonus,
the lockout defaults for G-06-07). This directly pins the explicit verified-gate
wiring and the one line most likely to drift (the `"true"` at line 103). It does
NOT exercise `AbpSignInManager`, so it does not cover the active gate end to end.
- Source: OLD code `UserAuthenticationDomain.cs:55,60,143`;
  `CaseEvaluationSettingDefinitionProvider.cs:99-104`.
- Tradeoffs: cheapest, no harness blocker, follows the existing pure-unit-test
  pattern (`ExternalAccountAppServiceUnitTests`). But it pins configuration, not
  the actual sign-in rejection -- a future ABP change to how the setting is
  consumed would pass this test while the gate breaks.
- Risk: low. No production change.

### Option B -- SignInManager integration test asserting NotAllowed/IsActive rejection [effort: M] [risk: med]

Resolve `AbpSignInManager` (or `IdentityUserManager` + `SignInManager`) from the
ABP test container, seed two users (one `EmailConfirmed=false`, one
`IsActive=false`), and assert `PasswordSignInAsync` / `CheckPasswordSignInAsync`
returns `NotAllowed` for each, plus a success case for an active+verified user.
This is the literal "inactive+unverified login rejection" the CSV asks for and
covers BOTH gates at the framework boundary.
- Source: OLD `UserAuthenticationDomain.cs:55,60`; ABP `AbpSignInManager`
  (Section 6 GitHub source); test conventions in `test/CLAUDE.md`.
- Tradeoffs: highest fidelity to the disposition. BUT it depends on the ABP
  integration harness booting, which `ExternalAccountAppServiceUnitTests.cs:12-26`
  documents as blocked by the Phase 4 license-checker test-host crash. Either the
  crash must be resolved first, or `SignInManager` must be constructed with
  mocked `IdentityOptions` (`SignIn.RequireConfirmedEmail=true`) + a fake user
  store -- which reintroduces some of Option A's "tests config, not wiring"
  caveat.
- Risk: med -- harness dependency + the protected `IsActive`/`EmailConfirmed`
  setters (the existing tests already work around these via reflection).

### Option C -- HTTP-level token-endpoint test (POST credentials -> expect no token) [effort: L] [risk: med]

Drive the real OpenIddict token endpoint (or the Login Razor POST) for an
unverified and an inactive user, assert no token / a NotAllowed response, and
assert the unverified case surfaces the generic banner + Resend (also covering
the `Login.cshtml.cs` interception). True black-box parity.
- Source: OLD `UserAuthenticationDomain.cs:50-145`; NEW `Login.cshtml.cs:138-145`.
- Tradeoffs: closest to user-observable OLD behavior and the only option that
  also pins the `LoginModel` redirect-interception. But it needs the AuthServer
  host booted in-test (no such test project exists today), DataProtection/Redis
  for tokens, and inherits the same harness blocker as B at a larger scope.
- Risk: med-high relative to value for a pin; better suited to an e2e suite than
  the unit/integration projects that exist now.

Recommendation: ship **Option A** now (pins the one line that already regressed
once, zero blockers), and queue **Option B** behind the Phase 4 test-host-crash
fix so the active gate is also pinned at the `AbpSignInManager` boundary.

## 8. Root-Cause Hypothesis

The "gap" is a documentation/scope artifact, not a behavioral defect. During the
custom-JWT -> OpenIddict/ABP port, OLD's explicit two-signal login gate
(`IsVerified` + `StatusId/IsActive`) was correctly replaced by framework
mechanisms: the active gate by `AbpSignInManager` (implicit, no code) and the
verified gate by an explicit `RequireConfirmedEmail="true"` setting-default
override (with an OLD-parity citation comment). Both produce OLD-equivalent
outcomes today. The genuine deficiency the CSV flags is the **absence of any
automated test** locking these in, combined with the verified gate being a
single flippable line that already toggled false once (2026-05-05) before being
restored. Confidence: high -- every leg (OLD source, NEW setting override, ABP
framework semantics, missing-test grep) is independently verified.
