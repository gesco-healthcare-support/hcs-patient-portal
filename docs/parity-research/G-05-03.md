## 1. ID & Summary

Two NEW document-reminder jobs each fire ONCE per appointment instead of
on an escalating T-minus countdown:

- `PackageDocumentReminderJob` -- one fire when the appointment's `DueDate`
  reaches the single configurable `Documents.PackageDocumentReminderDays`
  cutoff (default 7).
- `DueDateDocumentIncompleteJob` -- one fire at a single hardcoded
  `ReminderDaysBeforeDueDate = 7` window.

OLD reminders 2 (`AppointmentPackageDocumentPendingNotification`) and 4
(`AppointmentDueDateDocumentApproachingNotification`) were each ONE-SHOT
methods that ran once per inbound `POST /api/Scheduler/postscheduler`.
The repeat cadence (e.g. T-7 / T-3 / T-1) was decided by an external
host-side trigger that is NOT in the OLD repo, and the `spm.*` selection
procs are empty stubs. So OLD's actual cadence is unknown from source.

Disposition is `include` because the CSV directs building a
**staged-reminder framework / scaffolding** (configurable T-minus
windows), with Adrian to confirm the exact cadence later. The right
jobs already run the right selection logic; only the repeat frequency
is missing. This is a Partial gap (frequency), not a Missing feature.

## 4. Expected vs Actual

- Expected (per CSV disposition `include` + note): a staged-reminder
  framework with configurable T-minus windows, so a stakeholder with an
  outstanding document is nudged multiple times as the deadline approaches
  (the escalating-pressure behavior OLD's external trigger plausibly
  produced). Exact window set TBD by Adrian.
- Actual: each job fires exactly one reminder per appointment -- the
  package job at the single `PackageDocumentReminderDays` cutoff, the
  incomplete-doc job at a single hardcoded T-7. A stakeholder gets one
  email and no follow-up nudge.
- Caveat: "Expected" cannot be pinned to an OLD source value, because OLD
  source has no cadence. The expected behavior is the framework's
  existence + configurability, not a specific window set.

## 7. Candidate Solutions

### Option A -- Promote each job's window to a configurable T-minus array [effort: M] [risk: low-med]

- Desc: Replace the single cutoff/window in each job with a
  `int[]`-shaped configurable window set, mirroring
  `DueDateApproachingJob.ReminderDaysBeforeDueDate = { 14, 7, 3 }` and its
  `Contains(daysUntil)` check (`DueDateApproachingJob.cs:37, :104`). For
  `DueDateDocumentIncompleteJob` this is a near-mechanical copy (it already
  selects by `DueDate.Date == today.AddDays(window)`; loop the window
  array). For `PackageDocumentReminderJob`, change the
  `PackageDocumentReminderDays` setting from a scalar to a
  comma-separated list (or add a sibling list setting) and fire when
  `daysUntilDue` is in the set.
- Source: OLD code establishes only that staging existed externally
  (`SchedulerDomain.cs:119, :176` are one-shot; cadence not in source).
  The authoritative in-repo precedent is the NEW
  `DueDateApproachingJob` staged pattern (OLD code: cadence absent; NEW
  pattern: `DueDateApproachingJob.cs:37,104`). ABP settings support
  string-typed values that callers parse, so storing a CSV window list in
  one `SettingDefinition` is idiomatic (ABP `ISettingProvider.GetAsync`
  returns the raw string; the job parses) -- confirmed by the existing
  `GetAsync<int>` usage at `PackageDocumentReminderJob.cs:91`.
- Tradeoffs: smallest change that satisfies "configurable T-minus
  windows"; keeps daily cron unchanged (no Hangfire change). Per-tenant
  override comes free via ABP settings. The cost is choosing the default
  window set, which the CSV explicitly defers to Adrian.
- Risk: over-notification if a window set is shipped before Adrian
  confirms. Mitigate by defaulting to the current single window (no
  behavior change until configured) and gating multi-window behind the
  setting.
- Effort: M (two jobs + one setting reshape + unit tests for the window
  selection; ~half a day).

### Option B -- Build a shared staged-cadence helper, jobs consume it [effort: M-L] [risk: low]

- Desc: Extract a small `ReminderCadence` value object / static helper
  (parse a window list from a setting, expose
  `ShouldFire(daysUntilDue)`), and have all date-driven reminder jobs
  (`DueDateApproachingJob`, `DueDateDocumentIncompleteJob`,
  `PackageDocumentReminderJob`, and future ones) consume it. This is the
  literal "framework / scaffolding" the CSV note asks for.
- Source: in-repo pattern consolidation; precedent is
  `JointDeclarationCutoff` (`src/.../AppointmentDocuments/JointDeclarationCutoff.cs`),
  a pure static helper the jobs already share for the cutoff test --
  proves the codebase favors extracting reminder-window logic into a
  unit-testable helper.
- Tradeoffs: more reusable and unit-testable (matches the
  `JointDeclarationCutoff` convention the parity doc praises), but a
  larger diff than Option A and touches three jobs instead of two. Better
  long-term fit for "framework"; slightly more up-front cost.
- Risk: low (pure helper, fully unit-testable, no I/O). Main risk is
  scope creep refactoring `DueDateApproachingJob` that is already correct.
- Effort: M-L.

### Recommendation

Option A as the minimal parity fix, structured so the window logic is a
small shared helper (the Option B seam) if a third job needs it. Ship the
default window set equal to today's single window so there is NO behavior
change until Adrian sets the cadence, satisfying "build the scaffolding,
confirm cadence later". No handler/ETO/migration changes; add unit tests
(none exist today).

## 8. Root-Cause Hypothesis

Confidence HIGH. The single-fire behavior is a deliberate, documented
Phase 14b / Phase 7 scope decision, not an accident: the
`PackageDocumentReminderJob` XML-doc comment (`:31-36`) explicitly states
OLD had multi-cadence (T-7/T-3/T-1) and NEW intentionally ships one
reminder "at the configured cutoff" as an MVP, deferring multi-cadence to
"a post-parity enhancement". `DueDateDocumentIncompleteJob` hardcodes a
single T-7 window for the same reason. The root cause of the GAP (vs. the
decision) is that OLD's real cadence is unrecoverable from source (the
external trigger config and the `spm.*` procs are not in the repo), so
the team could not port a specific cadence and chose a safe single fire.
The fix is therefore not "match OLD" (impossible) but "build the
configurable framework so the cadence can be set once Adrian/stakeholders
confirm it" -- exactly the CSV disposition.

## 9. Open Questions

1. What window set should the default be? (CSV: "Adrian to confirm exact
   cadence soon.") `DueDateApproachingJob` uses 14/7/3; OLD's real cadence
   is unknown. Until confirmed, defaulting to the current single window
   keeps behavior unchanged.
2. Should `PackageDocumentReminderDays` (currently a scalar default "7")
   become a list, or should a new list-typed setting be added alongside it
   to avoid breaking any consumer that reads the scalar? (Only the job
   reads it -- grep showed no other consumer -- so reshaping in place is
   safe, but a new setting is lower-risk for migration.)
3. Interaction with G-05-INTENT-01 (SMS dropped): if a staged cadence is
   added before Twilio lands, the extra nudges are email-only. Confirm
   that is acceptable, or defer staging until SMS returns so multi-channel
   escalation lands together.
4. Relationship to G-05-02 (JDF reminder folded into package-doc job): if
   the package-doc job gains staged windows, the folded-in JDF rows
   inherit the same cadence. Confirm a JDF-specific cadence is not needed
   separately (G-05-02 already flags this).
5. Idempotency at scale: re-publishing the ETO on each window day is the
   intended behavior, but if a future requirement adds "don't re-send if
   already sent today", that guard does not exist yet -- out of scope here,
   noted for the build phase.
