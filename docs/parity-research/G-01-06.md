> **DECISION (finalized 2026-06-01) [Gate 1 - accessor]:** Restore OLD email-based create-or-invite. The add-accessor UX takes a free-typed name + email + role (NOT a pick-existing-user dropdown). Wire the already-built but DEAD AppointmentAccessorManager.CreateOrLinkAsync as the live path -- this makes the role-conflict check (this item) AND the invite email (G-10-08) reachable. Accessors are an ADDITIONAL share set that MAY overlap the named AA/DA/CE. SEPARATELY VERIFY: NEW must grant named AA/DA/CE appointment access via their own identity (independent of accessor rows), matching OLD (whose access-scoping stored proc is not in source -- stubs only). See SUMMARY.md Gate 1.

## 1. ID & Summary
OLD rejected adding an appointment accessor when the accessor's chosen role-type
differed from the role of an already-registered (active + verified) user matched by
email ("already registered in our system with different user type"). NEW DID port this
rule -- it lives complete in `AppointmentAccessorManager.CreateOrLinkAsync` +
`AppointmentAccessorRules.ResolveOutcome`, with the verbatim message and unit tests --
but that method is dead code: no production caller invokes it. The live accessor-add
path (Angular booking form -> `AppointmentAccessorsAppService.CreateAsync` ->
`AppointmentAccessorManager.CreateAsync`) does no role-conflict check. There is also a
model mismatch: NEW selects accessors from a lookup of EXISTING users (each carrying its
real role), so the OLD "free-chosen accessor role vs registered role" conflict is not
even expressible on the current Add path -- the accessor row has only `AccessTypeId`
(View/Edit), no separately chosen role-type.

### Blast radius
A faithful fix is medium. The simplest wiring (route the live Add through the existing
`CreateOrLinkAsync`) touches ~3-4 files: the AppService (accept email + role instead of, or
in addition to, `IdentityUserId`), the CreateDto/contract, the Angular submit payload, and
possibly the proxy regen. But it collides with two model facts: (1) NEW selects accessors
from existing users, so there is no free-typed email or chosen role-type to validate; and
(2) the auto-create/invite half of `CreateOrLinkAsync` belongs to the external-signup/auth
area (G-06/G-10-08) and overlaps the deferred accessor-account-provisioning work. Regression
risk is low for read-access (MVP does not key off accessor rows) but non-trivial for the
booking-submit flow if the payload contract changes. Recommend confirming the intended NEW
accessor-add UX before wiring, rather than mechanically porting.

## 4. Expected vs Actual
- EXPECTED (OLD): adding an accessor whose email maps to an active+verified registered user
  with a role different from the accessor's chosen role-type is rejected with the
  "different user type" message, on both the booking-time path and the standalone
  add/update path.
- ACTUAL (NEW): the live Add path (`AppointmentAccessorsAppService.CreateAsync`) performs no
  role check and accepts any selected existing user + View/Edit. The ported check
  (`CreateOrLinkAsync`) exists but is never invoked. Additionally, because NEW picks from
  existing users (role already known and fixed), the literal OLD conflict ("chosen role-type
  != registered role") cannot arise on the current path -- the accessor's role IS the
  selected user's role.

## 7. Candidate Solutions
### Option A -- Wire the live Add path through the ported CreateOrLinkAsync (full OLD parity) [effort: L] [risk: med]
- Description: change the accessor-add contract to accept email + requested role (or resolve
  them server-side from the selected lookup user) and route `AppointmentAccessorsAppService
  .CreateAsync` through `CreateOrLinkAsync`, activating the RoleMismatch check + invite flow.
- Source: OLD code: `AppointmentDomain.cs:185-195` and `AppointmentAccessorDomain.cs:129-140`;
  NEW already-ported logic in `AppointmentAccessorManager.cs:81-167` and
  `AppointmentAccessorRules.cs:80-130`.
- Tradeoffs: achieves literal OLD parity AND lights up accessor invites; but it changes the
  booking payload, depends on the external-signup/auth account-provisioning work
  (G-06/G-10-08), and conflicts with the current "pick an existing user" UX. Larger surface.
### Option B -- Lightweight role-consistency guard on the existing path [effort: M] [risk: low]
- Description: in `AppointmentAccessorsAppService.CreateAsync`, look up the selected
  `IdentityUser`'s roles and reject if the user holds a recognized external role that does
  not match what the UI intends for that accessor slot -- reusing
  `AppointmentAccessorRules.HasConflictingExternalRole`. Adapts the OLD rule to the
  pick-existing-user model without a contract overhaul.
- Source: OLD code: `AppointmentAccessorDomain.cs:135-138`; NEW reusable rule
  `AppointmentAccessorRules.cs:80-103`.
- Tradeoffs: small, low-risk, reuses tested rules; but it only re-expresses OLD intent (it
  cannot reproduce the exact "chosen role != registered role" check because NEW has no
  separately chosen role-type). Needs a product decision on what "conflict" means here.
### Option C -- Document-and-defer as not-applicable-as-written [effort: S] [risk: low]
- Description: record that the conflict is structurally unreachable on the NEW
  pick-existing-user path and that the ported logic is reserved for the future
  email-based accessor-invite flow; keep `CreateOrLinkAsync` but flag it
  `// PARITY-FLAG` as not-yet-wired pending G-06/G-10-08.
- Source: OLD code: `AppointmentDomain.cs:185-195`; NEW `AppointmentAccessors/CLAUDE.md`
  MVP-intent (accessor entity dormant at MVP).
- Tradeoffs: cheapest and honest about the model gap; but leaves an OLD-visible guardrail
  absent if a future change reintroduces email-based accessor entry.

## 8. Root-Cause Hypothesis
The check was ported as a pure, well-tested unit (`AppointmentAccessorRules` +
`AppointmentAccessorManager.CreateOrLinkAsync`, Phase 11i 2026-05-04) intended to back an
email-based create-or-invite accessor flow, but the live booking form was built on a
different model -- selecting accessors from a lookup of existing users by `IdentityUserId`
+ View/Edit. The AppService Create path was never switched over to `CreateOrLinkAsync`, so
the guard is dead code. Net: the rule exists but is unreachable, and the current Add model
has no chosen role-type field for the original conflict to fire against. Confidence: high
(AppService code, dead-code grep, DTO shape, and the Angular payload all corroborate).

- Disposition vs MVP-intent: the Domain `AppointmentAccessors/CLAUDE.md` says accessor rows
