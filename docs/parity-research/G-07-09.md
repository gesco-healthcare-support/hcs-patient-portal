> **DECISION (finalized 2026-06-01) [Gate 1 - accessor]:** Part of the accessor restore (G-01-06). Keep AccessType View/Edit on the accessor row; enforce it server-side on the live path once accessor invite is wired (OLD enforced View/Edit client-side only). See SUMMARY.md Gate 1.

## 1. ID & Summary

The parity doc (G-07-09) classes the OLD `AccessType` (View=23 / Edit=24)
enum-driven per-field/role permission model as **Missing** in NEW, and frames
NEW's analogue as `AppointmentTypeFieldConfig` (Hidden/ReadOnly/DefaultValue per
field per appointment type).

**This framing is materially inaccurate and the gap is largely already closed.**
The CSV note ("Model + enforce per-accessor View vs Edit access on appointment
share") points at the correct mechanism, which is NOT `AppointmentTypeFieldConfig`.
In BOTH OLD and NEW, `AccessType` lives on the **`AppointmentAccessor`** row (the
"share this appointment with another user" / authorized-user feature), not on a
role x field matrix. NEW already:

- stores `AccessTypeId` on `AppointmentAccessor` (View=23/Edit=24, verbatim enum);
- lets the booker pick View vs Edit per authorized user in the Angular add flow;
- persists it on submit;
- **enforces** it: `AppointmentAccessRules.CanEdit` requires `AccessType.Edit` for
  the accessor pathway (View-only accessors are denied edit), and this is wired
  into `AppointmentChangeRequestsAppService.EnsureCanEditAsync`.

The real residual delta vs OLD is narrow: the **core** `AppointmentsAppService.
UpdateAsync` does not call `CanEdit` (so View-vs-Edit is only enforced on the
change-request write path, not the main appointment update), plus the documented
`AppointmentAccessorsAppService` permission gap and absence of an admin/manage UI
(those belong to G-07-01 / area 06, not here). `AppointmentTypeFieldConfig` is a
separate, unrelated capability (per-field-per-type form config) and is NOT the
View/Edit reframing the parity doc claims.

### NEW (the actual residual gaps)

1. **Core appointment Update is not edit-gated by AccessType.**
   `Application/Appointments/AppointmentsAppService.cs` `UpdateAsync` carries only
   `[Authorize]` (per `Domain/Appointments/CLAUDE.md` Business Rule 8 + the
   skipped tests) and calls `EnsureCanReadAsync` on the Get paths but NOT
   `CanEdit` on Update. So a View-only accessor blocked from change-requests is
   not blocked from the main `PUT /api/app/appointments/{id}` (subject to the
   existing `[Authorize]`-only gap). The View/Edit distinction is therefore
   enforced on the change-request path and the document path's read gate, but not
   uniformly on every mutation. This is the legitimate "enforce" half of the CSV
   note that remains open.
2. **`AppointmentAccessorsAppService` permission + edit-gate gap.** Per
   `Domain/AppointmentAccessors/CLAUDE.md` Known Gotcha #2, mutation endpoints use
   bare `[Authorize]`; any authenticated user can create/edit/delete accessor
   grants (incl. self-granting Edit). Tracked by a skipped test.
3. **No admin/manage UI for accessor grants** beyond the booking-time add flow
   (no edit/revoke screen) -- this is G-07-01 scope, not a distinct G-07-09 gap.

## 4. Expected vs Actual

- **Expected (OLD ground truth):** per-accessor View/Edit choice stored on the
  share row; enforcement is client-side (`mode` query param renders the edit page
  read-only for View accessors). No server-side write gate on AccessType.
- **Actual (NEW):** per-accessor View/Edit is stored, settable, persisted, and is
  enforced server-side on the change-request write path (stricter than OLD) and
  surfaced in the read gate. The model is present, not "absent." The only thing
  missing relative to a fully-uniform model is an AccessType edit-gate on the core
  appointment Update -- which OLD also lacked server-side. So vs strict OLD
  parity, NEW is at or beyond parity; the parity doc's "Missing" class is wrong.

## 7. Candidate Solutions

### Option A -- Document parity-met; gate core Update only if product wants server-enforced Edit [effort: S] [risk: low]
The View/Edit model is present and enforced where OLD enforced (and more). Action:
(1) correct the parity doc's classification from "Missing" to "Equivalent/met with
minor enforcement delta"; (2) decide whether the core appointment Update should
also call `AppointmentAccessRules.CanEdit`. Since OLD did NOT server-side-gate
edits (SPA `mode` flag only), strict parity does NOT require it -- so the safe
parity answer is "no server gate needed; optionally add as a security-plus."
Source: OLD code `AppointmentAccessorDomain.cs:69-127` (no AccessType write gate),
`appointment-edit.component.ts:140-143` (client gate). One well-justified port.

### Option B -- Centralize an edit gate (security-plus, beyond OLD) [effort: S] [risk: medium]
Add `AppointmentReadAccessGuard.EnsureCanEditAsync` mirroring the read gate, call
it from `AppointmentsAppService.UpdateAsync` and any other accessor-reachable
mutation, reusing the existing `CanEdit` 7-pathway predicate. This makes View-only
truly read-only server-side. Risk: could 403 external parties who currently can
PUT (regression vs current behavior, though closer to intent). Defer until the
broader Create/Update permission gap (Appointments Business Rule 8) is addressed
so the two are designed together. Source: existing `EnsureCanEditAsync` pattern
in `AppointmentChangeRequestsAppService.cs:129-157`.

## 8. Root-Cause Hypothesis

The parity doc author saw the bare `AccessType.cs` enum, did not trace it to
`AppointmentAccessor` (the share/authorized-user feature, audited in area
01-booking and area 10-data-model, not area 07), and pattern-matched it to the
nearest area-07 config artifact (`AppointmentTypeFieldConfig`). That conflation
produced a false "Missing" classification: the two mechanisms are unrelated
(per-accessor share grant vs per-field-per-type form config). The genuine,
narrow open item is that View/Edit is enforced on the change-request write path
but not on the core appointment Update -- and even that is not a strict OLD-parity
requirement because OLD enforced View/Edit only in the SPA. Confidence: high
(every claim is code-verified; the enum-to-entity wiring is unambiguous).
