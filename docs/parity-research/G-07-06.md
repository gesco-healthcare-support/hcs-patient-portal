## 1. ID & Summary
OLD `CustomField` entity carries `AvailableTypeId` (int FK to the `AvailableType`
enum, whose only value is `Appointment = 11`). It was meant to scope a custom
field to a host context ("where the field lives"), but only `Appointment` was
ever defined or used. NEW's `CustomField` drops the column. The sibling
`FieldLength` (int?, the meaningful column) is preserved in NEW. Listed as
"Partial" only because a column is absent; behavior is equivalent.

## 4. Expected vs Actual (brief)
- Expected (strict schema parity): NEW `CustomField` would carry `AvailableTypeId`.
- Actual: NEW omits it. Since OLD's value is a compile-time constant (`11`) on
  every row, written by the client without user choice and read by no business
  logic, the omission has zero functional/behavioral impact. NEW is
  behaviorally equivalent to OLD.

## 5. Repro Status
n/a-excluded. There is no user-visible behavior to reproduce: the field is a
constant in OLD with no read path. Static code confirmation (enum cardinality +
zero domain/controller reads + hardcoded client assignment) is sufficient.

## 6. Why Excluded + Revisit-If
Excluded because the column is vestigial in OLD -- a single-valued enum, always
set to the same constant, never branched on. Re-adding it to NEW would add a
dead column and a meaningless lookup with no parity benefit. The parity doc
explicitly says "Do not re-add `AvailableTypeId` -- vestigial."

Revisit IF: a future feature reintroduces multiple host contexts for custom
fields (e.g. custom fields attached to entities other than Appointment). At that
point the scoping concept would become live and a discriminator column (or a
relation) would be warranted -- but that would be net-new design, not OLD parity.

## 7. Candidate Solution
None -- excluded.
