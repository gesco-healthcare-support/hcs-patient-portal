## 1. ID & Summary
The "max custom intake fields" rule. OLD capped the TOTAL number of active
custom fields at a global 10, enforced only at exact-equals (`== 10`) and only
on the add path. NEW caps active fields at 10 PER AppointmentType, counts
correctly, uses `>= 10`, and enforces on create plus the relevant update paths.
NEW is the deliberate OLD-bug-fix / spec-intent ("10 per type") correction.
Disposition: exclude (keep NEW, do not revert).

## 3. Affected Code (spot-checked)
- OLD code: P:/PatientPortalOld/PatientAppointment.Domain/CustomFieldModule/CustomFieldDomain.cs
  - `:38` -- counts ALL active rows globally; predicate
    `customField.StatusId == (int)Status.Active` compares the INPUT to itself
    (a constant, not a per-row filter), so it never narrows the set.
  - `:40` -- `if (customFieldCount == 10)` -- exact-equals, not `>=`. A count of

  - Enforced on create (`:101`) and on update when becoming active or moving

  - Class XML-doc (`:9-33`) already documents both corrections (global->per-type,
    `==`->`>=`) citing CustomFieldDomain.cs:38 and :40.

## 4. Expected vs Actual (brief)
Expected (NEW, intended): at most 10 active custom fields per appointment type,
counted correctly, blocked robustly on add and the relevant edit paths.
Actual gap vs OLD: OLD allowed at most ~10 GLOBALLY (buggy, leakable past 10 via
the exact-equals boundary, and never enforced on update). The two systems
differ by design; NEW is the corrected behavior.

## 6. Why Excluded + Revisit-If
Excluded because the OLD behavior is itself a bug (off-by-one global cap plus an
unenforced update path), and NEW intentionally implements the spec ("10 per
type"). Reverting would re-introduce the bug. The bug/deviation policy says
clear OLD bugs are fixed silently and must not be inherited.

Revisit IF either holds:
- The product owner confirms the intended business rule is a GLOBAL cap (across
  all appointment types), not per-type -- then NEW's per-type scoping would be
  wrong (but the count/boundary fix still stands).
- The spec line cited in the NEW XML-doc ("10 per type", spec line 543) cannot
  be located or is contradicted by an authoritative requirement.
