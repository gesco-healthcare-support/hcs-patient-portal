## 1. ID & Summary

OLD attached "extra" documents to an appointment through a dedicated
`AppointmentNewDocument` table + controller + Angular `appointment-new-documents`
component. NEW unifies that ad-hoc path into the single `AppointmentDocument`
entity via an `IsAdHoc` flag (an explicitly sanctioned unification per the
project mission and audit row E2).

The unification is accepted as equivalent. The ONLY residual behavioral gap is
type metadata: OLD's ad-hoc upload let the user pick a document type from a
master-managed dropdown (`AppointmentDocumentTypeId`) with an "Other" free-text
fallback (`OtherDocumentTypeName`); NEW classifies every ad-hoc document by a
single free-text `DocumentName` only. That residual is the same thing as G-03-01
(the `AppointmentDocumentType` master library), so G-03-03 is effectively a
duplicate of G-03-01 scoped to the ad-hoc upload surface. Fixing G-03-01 (build
the doc-type master + picker + FK) closes G-03-03 with no separate work.

Every behavioral claim in the parity doc's "### G-03-03" section was verified
against OLD and NEW source; all hold. No drift found.

## 4. Expected vs Actual

- Expected (OLD parity): when attaching an ad-hoc document, the user picks a
  document type from a master-managed dropdown and may type a custom type via an
  "Other" fallback; the row stores `AppointmentDocumentTypeId` +
  `OtherDocumentTypeName` so staff can later filter/report by type.
- Actual (NEW): the user supplies only a free-text `DocumentName`. There is no
  type classification and no master vocabulary. Everything else about the ad-hoc
  flow (no upload gate, internal->Accepted / external->Uploaded,
  `ResponsibleUserId` stamp on internal accept) matches OLD.

Net: the unified `IsAdHoc` path is behaviorally equivalent EXCEPT for the absent
type metadata. That single residual = G-03-01.

## 7. Candidate Solutions

### Option A -- Defer; close under G-03-01 (no separate G-03-03 work) [effort: none] [risk: none]
- Desc: Treat G-03-03 as a duplicate of G-03-01. Build the doc-type master ONCE
  (under G-03-01's own parity audit doc), and the ad-hoc upload form's type
  picker is one of G-03-01's deliverables. No G-03-03-specific code.
- Source: OLD code (shared `AppointmentDocumentType` FK on both doc tables,
  `AppointmentNewDocument.cs:85-91` + `AppointmentDocument.cs:100-105`); CSV
  rows 20/53/78 all converge on G-03-01.
- Tradeoffs: cleanest -- avoids double-building. Requires G-03-01 to actually be
  scheduled; until then the ad-hoc form stays free-text-only.
- Risk: none (no code change here).
- Effort: none (the work is accounted under G-03-01 = M-L).

### Option B -- Build the type picker as part of the G-03-01 slice (recommended sequencing) [effort: M] [risk: low]
- Desc: When G-03-01 lands the `AppointmentDocumentType` master + CRUD, in the
  same PR add `AppointmentDocumentTypeId` (Guid?) + `OtherDocumentTypeName`
  (string?, max 100 to match OLD) to `AppointmentDocument`, thread them through
  `UploadStreamAsync` + `AppointmentDocumentDto`, and add the "Document Type"
  `<select>` + "Other" free-text fallback to
  `appointment-documents.component.ts/html` (mirroring OLD's
  `appointment-new-document-add.component.html:14-24`, including the "Other"
  conditional). Use a Guid FK (NEW convention) rather than OLD's hardcoded
  `== 4` "Other" id check -- match the OLD by a `DocumentTypeName` semantic, not
  a magic int.
- Source: OLD `appointment-new-document-add.component.html:14-24`;
  `AppointmentNewDocument.cs:66-91`; `AppointmentDocumentType.cs`.
- Tradeoffs: delivers true OLD parity for the ad-hoc surface. Slightly more than
  the master CRUD alone, but it is the natural consumer of that CRUD.
- Risk: low -- additive nullable columns, additive form control; no existing
  flow reads document type. Avoid OLD's magic-int "Other" coupling (a small
  cleanup, not a behavior change).
- Effort: M (incremental on top of G-03-01's master CRUD; this slice alone is S).

(Only two options offered: per the CSV note this is a duplicate that folds into
G-03-01, so the "fix" is really a sequencing decision, not a design fork.)

## 8. Root-Cause Hypothesis

Confidence: HIGH. The new app was deliberately built at MVP with a single
free-text `DocumentName` and the `AppointmentDocumentType` lookup explicitly
deferred (documented in `AppointmentDocument.cs:16`). The two-table -> one-entity
unification (E2) was sanctioned and is behaviorally equivalent; it did not, by
itself, drop the type metadata. The missing type fields are a separate,
intentional MVP cut == G-03-01. G-03-03 is therefore not an independent defect
but the ad-hoc-facing symptom of the deferred G-03-01 master library. No code
bug, no drift; the parity doc's classification ("Partial; residual = type
metadata = G-03-01") is accurate.
