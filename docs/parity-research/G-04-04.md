## 1. ID & Summary

OLD emails a "your password has been successfully changed" security-receipt
(template `Password-Changed.html`) after BOTH password-change paths:
the in-app authenticated change AND the forgot-password reset flow.

NEW has the template fully prepared (`PasswordChange` code seeded with a
real HTML body + subject) and the **post-reset path is already wired**
(`ExternalAccountAppService.ResetPasswordAsync` dispatches it). The gap
that REMAINS is the **in-app authenticated change-password path**: ABP's
`/Account/Manage` -> `IProfileAppService.ChangePasswordAsync` fires no
`PasswordChange` notification, so a logged-in user who changes their own
password gets no receipt.

IMPORTANT DRIFT: the source parity doc's NEW block ("no send site
dispatches it") is STALE. It predates Phase 1.C (2026-05-08), which wired
the post-reset half. The accurate gap is narrower than the CSV/doc state:
~half of OLD's behavior is already at parity. Effort downgraded to S.

### Blast radius

Small. A fix adds ONE new dispatch site for the in-app path. Touch count:
- 1 new file (an override of ABP `ProfileAppService`, OR a guarded

## 7. Candidate Solutions

OLD-code-first: OLD simply called the template after the in-app change
committed. NEW must hook ABP's equivalent in-app change seam. The crux is
a framework capability (where to hook ABP's authenticated change-password),
so two framework-validated options are presented.

### Option A -- Override ProfileAppService.ChangePasswordAsync [effort: S] [risk: low]

- Desc: subclass ABP Pro's `ProfileAppService`, override
  `ChangePasswordAsync`, call `base.ChangePasswordAsync(input)`, then on
  success dispatch `NotificationTemplateConsts.Codes.PasswordChange` via
  `INotificationDispatcher` for `CurrentUser` -- mirroring the existing
  `ExternalAccountAppService.ResetPasswordAsync` block. Register with
  `context.Services.Replace(...)` in the module.
- Source: OLD code (`UserDomain.cs:188`/`:344-345` -- send-after-commit
  pattern); ABP support #9127 (official) confirms `ChangePasswordAsync`
  override is the supported in-app seam and `Services.Replace` the
  registration; https://abp.io/support/questions/9127 (accessed 2026-06-01,
  HIGH credibility).
- Tradeoffs: precise (fires only on actual password change); reuses the
  proven dispatch pipeline; one new file. Must register on the host that
  actually serves `/Account/Manage` (AuthServer) -- support #9127's caveat
  about the override appearing not to run when the page is on a different
  app. Verify whether ABP Pro member obfuscation makes the override
  fragile (the team chose NOT to override `AccountAppService` for the reset
  path for exactly this reason -- `IExternalAccountAppService.cs` rationale).
  If override proves fragile, fall back to Option B.
- Risk: low. Cannot affect existing emails; only adds the in-app trigger.

### Option B -- ILocalEventHandler<EntityUpdatedEventData<IdentityUser>> [effort: M] [risk: med]

- Desc: subscribe to the auto-published user-updated event and dispatch
  `PasswordChange` when the password actually changed.
- Source: ABP Local Event Bus docs (official) --
  https://docs.abp.io/en/abp/latest/Local-Event-Bus (accessed 2026-06-01,
  HIGH). Confirms `EntityUpdatedEventData<IdentityUser>` is auto-published.
- Tradeoffs: avoids overriding an obfuscated Pro service. BUT the event
  fires on EVERY user update (profile edit, role change, email-confirm,
  lockout, even the post-reset path -> double-send risk). Detecting "the
  password specifically changed" is hard: `PasswordHash` is not reliably
  diffable in `EntityUpdatedEventData` (no before/after), and
  `LastPasswordChangeTime` updates may not be granular enough to gate on.
  High false-positive / double-fire risk; needs careful guarding.
- Risk: med (double-emails / emails on non-password edits if the gate is
  imperfect).

### Option C -- Defer the in-app half [effort: none] [risk: low]

- Desc: accept post-reset-only parity for Phase 1; wire the in-app receipt
  when/if an in-app change-password surface is finalized in the SPA/Manage
  page UX.
- Source: source doc open question #3 ("acceptable to defer, or is a
  security-receipt email required for HIPAA hygiene before go-live?").
- Tradeoffs: zero work now; leaves a minor security-hygiene gap for the
  authenticated change path. Decision was `include`, so Option A is
  preferred unless Adrian re-scopes.
- Risk: low (the missing email is a hygiene nicety, not a functional break).

Recommended: Option A. It is the OLD send-after-commit pattern translated
to ABP's supported in-app seam, reuses the already-live dispatch pipeline,
and is the lowest-ambiguity fix. Validate the `Services.Replace` override
runs on the AuthServer host (support #9127 caveat) during the build phase;
fall back to Option B only if obfuscation blocks the override.

## 8. Root-Cause Hypothesis

Phase 1.C (2026-05-08) wired the `PasswordChange` template into the
forgot-password RESET flow (`ExternalAccountAppService.ResetPasswordAsync`)
because that flow has a NEW custom AppService the team already owned. The
IN-APP authenticated change-password flow, by contrast, is served by stock
ABP `/Account/Manage` -> `ProfileAppService.ChangePasswordAsync`, which the
team never extended (they deliberately avoided overriding ABP Pro account
services where possible -- see the `IExternalAccountAppService` rationale).
Since ABP 10.0.2 publishes no dedicated password-changed event, the in-app
receipt was simply never hooked. Confidence HIGH: the dispatch site exists
for one path and is provably absent for the other, and the framework gap
(no PasswordChanged event) is corroborated by ABP docs.
