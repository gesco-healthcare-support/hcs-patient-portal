> **DECISION (finalized 2026-06-01) [Gate 1 - accessor]:** Add nullable name/email (+ role) columns to the accessor row as the pending-invite seed, so an invited-but-not-yet-registered accessor displays before their IdentityUser resolves. Storage side of the email-based create-or-invite restore (G-01-06). See SUMMARY.md Gate 1.

## 1. ID & Summary

OLD `spm.AppointmentAccessors` stored `FirstName`, `LastName`, `EmailId`,
`RoleId` and `AccessTypeId` **directly on the accessor row**. The row was the
record of "share this case with this name+email as this role", written at
booking/add time BEFORE any user account necessarily existed. A separate step
(`CreateAccountOfAppointmentAccessors`) read those columns to auto-provision a
`User` and email them.

NEW `AppointmentAccessor` (`src/.../Domain/AppointmentAccessors/AppointmentAccessor.cs`)
keeps only `AccessTypeId`, a required `IdentityUserId` FK, and `AppointmentId`.
The three name/email columns and `RoleId` are gone. An accessor in NEW can
therefore only exist if it points at an already-created `IdentityUser`; there is
no place on the row to hold a pending-invite name+email for a not-yet-resolved
user.

Important nuance found during research: the *invite/provision behavior* is
partially rebuilt elsewhere. `AppointmentAccessorManager.CreateOrLinkAsync`
(Phase 11i, 2026-05-04) accepts `email`, `firstName`, `lastName`, `roleName`,
looks up or creates an `IdentityUser`, and publishes `AppointmentAccessorInvitedEto`
(handled by `AccessorInvitedEmailHandler`). So NEW resolves the email to a real
user FIRST, then stores only the FK. The dropped columns are the **storage** of
the pending invitation on the row itself; in NEW that transient data lives only
in the method arguments and the email event, not persisted on the accessor row.

The decision (`include`) per the CSV is: allow accessor name/email (pending-invite)
on the row, not only the IdentityUser link.

## 4. Expected vs Actual

- **Expected (OLD parity per decision `include`):** an accessor row can carry a
  pending-invite name+email+role even when the target user does not yet exist;
  the access list renders those columns directly; account provisioning reads
  them.
- **Actual (NEW):** an accessor row requires a non-empty `IdentityUserId` FK;
  name/email/role are not stored and are derived only via a user join. The
  name+email *flow* exists (`CreateOrLinkAsync`) but resolves to a user up front
  and is not wired to any caller, so even that path is unreachable today.

## 7. Candidate Solutions

### Option A -- Add nullable name/email (+role) to the row; make IdentityUserId nullable [effort: M] [risk: med]

Restore OLD's self-contained-row semantics. Add `FirstName?`, `LastName?`,
`EmailId?` (and a role indicator -- a `RecognizedExternalRole` string or enum,
since OLD `RoleId` -> ABP roles) to `AppointmentAccessor`; make `IdentityUserId`
a `Guid?`. On invite, store the pending name/email and leave `IdentityUserId`
null until the invitee registers; on registration (or via `CreateOrLinkAsync`)
backfill the FK. Wire `CreateOrLinkAsync` into the AppService/booking flow so it
is actually reachable.

- Source: OLD code -- `AppointmentAccessor.cs:26-60`, `vAppointmentAccessor.cs`,
  `AppointmentAccessorDomain.cs:263-354` (provisioning reads row columns),
  `AppointmentDomain.cs:185-195,256-285` (booking-time accessor persistence +
  role-mismatch check).
- Tradeoffs: true OLD parity (pending invites survive as data, access list shows
  name/email without a user). Cost: nullable FK ripples through the
  `WithNavigationProperties` join and any `EfCoreAppointmentRepository` filter;
  migration; contradicts current MVP-intent doc.
- Risk: med (nullable FK + dormant-entity intent reversal).

### Option B -- Keep FK-only storage; rely on CreateOrLinkAsync resolving up front [effort: S] [risk: low]

Treat the gap as already mitigated: do NOT add name/email columns; instead wire
the existing `CreateOrLinkAsync` (which creates the IdentityUser immediately, so
the FK is always satisfiable) into the AppService + booking flow, and record the
deviation. The pending state then lives as a not-yet-confirmed IdentityUser
(invited, password-reset-pending) rather than as bare columns on the accessor
row -- functionally OLD's `IsAccessor=true` auto-created user.

- Source: NEW `AppointmentAccessorManager.CreateOrLinkAsync` (Phase 11i) +
  `AccessorInvitedEmailHandler`; OLD `CreateAccountOfAppointmentAccessors`
  (which ALSO immediately created the User -- so OLD never truly had a
  "user-less" accessor at the application level; the name/email columns were the
  *seed* for that immediate creation, not a long-lived pending state).
- Tradeoffs: smallest change, no migration, no nullable FK, respects the MVP
  intent doc. But it does NOT satisfy the CSV's literal ask ("name/email on the
  row") and the access-list display still depends on a user join. Behavioral
  parity is arguably preserved (OLD also auto-created the user at add time).
- Risk: low.

### Option C -- Hybrid: nullable name/email as display/seed cache, IdentityUserId stays required-after-resolution [effort: M] [risk: med]

Add nullable `FirstName`/`LastName`/`EmailId` purely as a denormalized
display/seed cache (mirrors how `Appointment` already denormalizes
`ApplicantAttorneyEmail`, see G-10-06), but keep the invariant that a *linked*
accessor still has an `IdentityUserId`. Populate the cache columns from the
invite args so the access list renders without a join and so re-sends do not
need the user record.

- Source: OLD `vAppointmentAccessor` (display columns) + NEW denormalized
  `*Email` columns on `Appointment` (G-10-06 precedent).
- Tradeoffs: matches NEW's existing denormalization pattern, avoids nullable FK
  churn if linking is still required, gives the access list its name/email back.
  Adds redundant columns that can drift from the user record.
- Risk: med (redundancy/drift; still a migration).

## 8. Root-Cause Hypothesis

`high`. The accessor entity was generated/modeled FK-first ("an accessor is a
link from an IdentityUser to an Appointment"), deliberately, because NEW's stated
MVP intent (feature CLAUDE.md, Adrian-confirmed 2026-04-24) is that access is
driven by the legal-party join and ad-hoc grants are post-MVP. OLD's name/email/
role columns existed because OLD provisioned the accessor user lazily from those
columns at add time. NEW shifted provisioning to up-front user resolution
(`CreateOrLinkAsync`), which makes the row columns redundant -- but that flow was
never wired to a caller, so the practical state is: NEW can only store an
accessor for a pre-existing user, and the name/email pending-invite affordance is
gone. The "drop" is an intentional modeling choice whose mitigation (the invite
manager) is incomplete (unwired), which is why the auditor classed it Partial and
the reviewer chose `include`.

## 9. Open Questions

1. Is the storage-side name/email truly required, or does Option B (immediate
   user creation via the already-built `CreateOrLinkAsync`) satisfy parity? OLD
   itself created the User at add time, so a long-lived "user-less accessor" may
   never have existed in OLD practice -- only the seed columns did. (Resolving
   this picks Option A/C vs B.)
2. Why is `CreateOrLinkAsync` unwired? It was built Phase 11i (2026-05-04) with a
   full invite/email pipeline but has no production caller. Was the booking-flow
   wiring deferred, or superseded by the `Invitations` feature
   (migration `20260515183211_Added_Invitations`)? Check overlap with the
   Invitation entity before adding columns.
3. The feature CLAUDE.md says "no runtime access decisions should depend on this
   entity" and "accessor must be a registered IdentityUser", which directly
   conflicts with restoring name/email + nullable FK. This needs an explicit
   product decision (Adrian) before any schema change.
4. Does `EfCoreAppointmentRepository` still join `AppointmentAccessor` for
   appointment-list filtering (feature CLAUDE.md Known Gotcha #7)? If yes, a
   nullable `IdentityUserId` could change that filter's results. Not verified
   read-only here.
5. Dependency sequencing: G-01-06 (role-type conflict check) and G-07-09
   (AccessType View/Edit enforcement) are both `include` and operate on this same
   entity; the `RoleId`/role indicator decision in this item should be made
   together with G-01-06's role-mismatch enforcement.
