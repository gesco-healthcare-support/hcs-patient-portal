## 1. ID & Summary

OLD had a "Reports" screen -- the **Appointment Request Report** -- a searchable,
paged, sortable HTML table listing every appointment request across all patients
with ten columns (Confirmation No, Appointment Type, Location Name, Appointment
Date Time, Status, Patient Name, Date Of Birth, Email, Phone Number, Social
Security Number), a free-text quick search, and an "Advanced Search" panel with
five filters (Appointment Type, Location, Appointment Status, Patient Name, date
range From/To). It was the central cross-appointment operational worklist for
internal staff (Clinic Staff, Supervisor, IT Admin), gated on `MODULES.Reports`.

NEW has **nothing** in this area: no report AppService/controller/DTO/permission,
no `report/` Angular feature, no route, no sidebar entry. The gap is real and
total. This item is the anchor for the three export siblings (G-08-02 Excel,
G-08-03 PDF, G-08-04 per-appointment demographics PDF), all of which hang off the
report grid that does not yet exist.

Good news for effort: NEW already has the exact query shape this report needs.
`AppointmentsAppService.GetListAsync(GetAppointmentsInput)` is a paged, sorted,
filtered query (filters: FilterText quick-search, date range, type, location,
status) returning `PagedResultDto<AppointmentWithNavigationPropertiesDto>`. The
report is largely a new read-only AppService/grid that reuses this filter set
plus a `CaseEvaluation.Reports` permission and PHI masking.

## 4. Expected vs Actual

**Expected (OLD parity):** A "Reports" menu item (visible only to internal staff)
opens a report screen with a quick search box + Advanced Search panel (5 filters:
type, location, status, patient name, date range). Below it, a paged/sortable
grid of all appointment requests, default sort `requestConfirmationNumber desc`,
10 columns including patient PII/PHI (name, DOB, email, phone, SSN), with the
Confirmation No linking to the appointment view. Toolbar has Export to PDF/Excel
and each row has a per-appointment demographics PDF icon.

**Actual (NEW):** No such screen, route, menu item, endpoint, DTO, or permission
exists. Internal staff have no cross-appointment tabular report -- they can only
navigate per-record through the appointments feature; there is no single grid to
scan/filter/find a patient by name/DOB across appointments, and no export source.

**Deltas baked into a faithful NEW build (per mission + doc, not gaps):**
- stored proc -> EF/LINQ paged query (expected view->query translation).
- `doctorName` filter omitted in Phase 1 (single-office; dead in OLD too).
- SSN/DOB masked by default per HIPAA (CSV note + Open Q3); full SSN only via the
  existing `Patients.RevealSsn` audited reveal endpoint.
- Export rendered server-side over the full filtered set, not OLD's
  client-CSV-of-visible-page handshake (Open Q5).

## 7. Candidate Solutions

### Option A -- New read-only Reports AppService reusing the appointment query, masked-by-default, + Angular grid [effort: L] [risk: low-med]

One-line: Add a `Reports` feature (permission + paged read-only AppService over an
EF projection + Angular advanced-search grid) that reuses the existing appointment
filter/paging infrastructure and the existing SSN-masking model.

- **Source:** OLD code (the authoritative spec for this port):
  `AppointmentRequestReportSearchController.cs:20-27`,
  `appointment-request-report-search.component.{ts:63-166,html:34-141}`,
  `appointment-request-report-view-report.ts`. NEW reuse target:
  `src/.../Application/Appointments/AppointmentsAppService.cs:104-126` (existing
  paged `GetListAsync` with FilterText + date range + type + location + status
  filters returning `PagedResultDto<...>`) and `GetAppointmentsInput.cs` (same
  filter set). SSN model: `src/.../Application/Patients/{SsnVisibility,SsnRevealAccess}.cs`
  + `Patients.RevealSsn` permission + `angular/.../shared/pipes/ssn-mask.pipe.ts`.
- **How:** define `GetAppointmentReportInput : PagedAndSortedResultRequestDto`
  (FilterText + 5 filters), an `AppointmentReportRowDto` (10 columns, SSN as
  masked last-4 by default + DOB masked per Open Q3), a `ReportsAppService`
  `[RemoteService(IsEnabled=false)] [Authorize(CaseEvaluationPermissions.Reports.Default)]`
  that runs a paged projection over Appointment + navigation props (reuse the
  appointment repo's filter signature or add a report projection), a manual
  controller `[Route("api/app/reports")]`, a Mapperly row mapper, the `Reports`
  permission (defined + registered + seeded to the 3 internal roles), and the
  Angular reports feature (route + menu entry + advanced-search form + Material
  paginator/sort grid; Confirmation No -> appointment view).
- **Tradeoffs:** Faithful to OLD's visible behavior and audience; reuses proven
  NEW patterns so low novelty. Larger than a one-file change because it spans 5
  layers + role seeding + proxy regen (hence L). Default sort must match OLD
  (`requestConfirmationNumber desc`). The filterReport "at least one filter
  required" rule and From<=To validation should be ported (client + ideally server
  guard).
- **Risk:** PHI exposure if masking/permission is wrong -- the load-bearing risk.
  Must NOT port OLD's auth-bypass export (OLD bug 1) or static export cache (OLD
  bug 2). Row-visibility scope (`@UserId`) is unknown (Open Q1) -- if the OLD proc
  scoped rows per user, NEW must decide whether internal staff see all rows
  (likely, since it is an internal worklist) vs scoped.

### Option B -- Extend the existing Appointments grid with a "report mode" instead of a separate feature [effort: M] [risk: med]

One-line: Reuse the `appointments` Angular feature + `AppointmentsAppService.GetListAsync`
directly, adding the extra report-only columns (DOB/SSN/email/phone) and a
permission flag, rather than a separate Reports surface.

- **Source:** Same NEW reuse target as Option A; no separate OLD anchor.
- **Tradeoffs:** Less code (no new feature folder). But it muddies the
  appointments feature's role model (the appointments list and the report have
  different audiences and column sets), risks leaking PHI columns to roles that
  see the appointments list but not the report, and diverges from OLD's distinct
  "Reports" screen (parity expects a separate screen + menu item + `Reports`
  permission). Harder to gate exports cleanly.
- **Risk:** Medium -- permission/column bleed between two audiences sharing one
  AppService. Rejected as the primary recommendation precisely because parity +
  PHI-gating want a dedicated, separately-permissioned surface.

### Option C -- Port literally (stored proc + client-CSV export handshake) [effort: L] [risk: high]

One-line: Recreate OLD verbatim -- a `spAppointmentRequestReport` stored proc
returning a JSON blob + the GUID-keyed CSV export handshake.

- **Source:** `AppointmentRequestReportSearchController.cs`, `CSVExportController.cs`.
- **Tradeoffs:** Explicitly rejected by the parity doc (Equivalent notes 3+4) and
  the mission (EF query replaces proc; server-side export replaces the handshake).
  The real proc body is not even in source (stub only), so a literal port is
  impossible to reproduce faithfully. Inherits OLD bugs 1 & 2 (auth bypass + static
  cache PHI leak).
- **Risk:** High -- recreates the PHI-exposure bugs and an unrecoverable proc.
  Listed only to document that it is the wrong path.

**Recommendation:** Option A.

## 8. Root-Cause Hypothesis

**Confidence: high.** Root cause is straightforward scope/sequencing, not a
regression or a deviation: the entire reporting-and-export area was never built on
the NEW stack. The parity-v2 audit (08) states up front that "the whole area is
expected to be MISSING in NEW," the summary table marks every reporting capability
"Absent," and static verification confirms no report AppService, controller, DTO,
permission, route, component, or menu entry exists. NEW built the dashboard
counters (Audit 09) but not the operational report grid. There is no competing
hypothesis worth holding: this is a clean greenfield port, made cheaper by the
fact that the underlying paged appointment query and the SSN-masking model already
exist in NEW.

## 9. Open Questions

1. **Row-visibility scope of the OLD proc.** The search controller passes
   `@UserId` into `spm.spAppointmentRequestReport`, but the real proc body is not
   in `P:\PatientPortalOld` (only a `WHERE 1=0` stub). Did OLD scope the report's
   rows per user, or was `@UserId` only for auditing while all rows showed? Since
   the report's audience is internal staff (an operational worklist), the likely
   intent is "all rows for internal roles," but this cannot be confirmed from
   source. Confirm before finalizing the query's row filter.
2. **PHI masking on the report grid + exports (HIPAA).** CSV note and Open Q3
   both flag: OLD shows full SSN (`*-search.component.html:130`) and DOB (`:127`).
   NEW should mask SSN (last-4 via existing `ssn-mask.pipe` / `SsnVisibility`) and
   decide whether DOB is masked too. Full SSN, if ever needed, should route
   through the existing audited `Patients.RevealSsn` reveal endpoint, not the bulk
   report. Owner decision required before build.
3. **Phase 1 filter set.** Confirm Phase 1 exposes OLD's five live filters
   (type, location, status, patient name, date range) and omits the dead
   `doctorName` field (single-office). Six-filter spec version is a Phase 2
   (multi-doctor) concern.
4. **Export scope.** Confirm NEW exports the full filtered result set
   server-side (correct, expected) rather than OLD's client-rendered visible-page
   CSV. This is an improvement over OLD; verify it is the desired behavior. Ties
   into G-08-02 (Excel, currently excluded) and G-08-03 (PDF, included).
5. **Dependency on G-09-01.** The dashboard counters (G-09-01) deep-link to
   `/appointments?appointmentStatus=N`. Confirm whether the report grid and the
   appointments list should share filter semantics / deep-link conventions to
   avoid two divergent "list of appointments" surfaces.
