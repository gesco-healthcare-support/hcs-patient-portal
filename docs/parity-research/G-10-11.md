## 1. ID & Summary

OLD `spm.AppointmentTypes` has a nullable `ReEvalId int` column. The parity
docs (G-07-02 and G-10-11) describe it as a "self-reference linking a base type
(PQME) to its re-eval variant (PQME RE-EVAL)". **That description is wrong.**
Reading the actual OLD code shows `ReEvalId` is NOT a self-FK to another
`AppointmentTypeId`; it holds values from a 3-value classification enum
`EvaluationTypeEnum { Normal = 0, Re = 1, Both = 2 }`
(`patientappointment-portal/src/app/const/evaluation-type.ts`). The frontend uses
it ONLY to filter the appointment-type dropdown: an Initial-eval booking
(`?type=1`) shows types where `reEvalId in {Normal, Both}`; a Re-eval booking
(`?type=2`) shows types where `reEvalId in {Re, Both}`.

The NEW app dropped the column entirely AND replaced the entire seed set (NEW
seeds QME / Panel QME / AME / Record Review / Deposition / Supplemental Medical
Report -- none of OLD's PQME/PQME-REEVAL/AME/AME-REEVAL/ALL/OTHER pairs survive).
NEW carries the `?type=1|2` re-eval flag through the booking form but, per an
in-code comment, deliberately defers the dropdown filtering as a future
enhancement. So the user-visible gap is: **on the NEW re-eval booking path the
appointment-type dropdown is not filtered to re-eval-eligible types.**

This is the storage half of G-07-02 (the admin-master-data item, included). It
is a true DUPLICATE of G-07-02 -- same column, same behavior, two doc sections.

## 7. Candidate Solutions

### Option A -- Port as a classification enum field + filter the lookup [effort: M] [risk: low-med]

Add `EvaluationType?` (enum Normal/Re/Both) to NEW `AppointmentType`, seed it on
the 6 NEW types, expose it in the DTO, and have `GetAppointmentTypeLookupAsync`
accept the `type` discriminator (or return the classification so the FE filters).
Then implement the deferred FE filter at `appointment-add.component.ts:184`.
- Source: OLD code -- `const/evaluation-type.ts:1-5` (the real enum),
  `appointment-add.component.ts:681-697` (the real filter logic). This is the
  faithful port of what `ReEvalId` actually did.
- Tradeoffs: requires a product decision on which NEW types are Re/Both (the seed
  set diverged, so no mechanical mapping exists). Additive + nullable, so safe.
- Risk: low technically; the only real risk is mis-seeding the classification.

### Option B -- Implement the column literally as a self-referencing FK (as the doc describes) [effort: M] [risk: med] -- NOT RECOMMENDED

Add `ReEvalId` as a nullable self-FK `AppointmentType -> AppointmentType` and seed
base->reval pairings, matching the CSV note ("self-ref column + seed base->reval
pairings") and the parity-doc description.
- Source: parity doc `10-data-model.md:374` / CSV `:80`.
- Tradeoffs: this matches the documents but NOT the OLD code. OLD `ReEvalId` is a
  classification flag (0/1/2), not a row pointer; no OLD backend code dereferences
  it as an id. Implementing a self-FK would reproduce a behavior OLD never had and
  would not reproduce the dropdown filter (which keys off the 0/1/2 flag).
- Risk: builds the wrong model; recommend correcting the parity doc instead.

### Option C -- Defer / drop (keep current behavior) [effort: S] [risk: low]

Leave NEW as-is (re-eval flag flows to payload; dropdown unfiltered). Justified
only if Adrian decides re-eval type-restriction is not required for Phase 1.
- Source: NEW already does this (`appointment-add.component.ts:183-186`).
- Tradeoffs: a Re-eval booking can pick an Initial-only type. Whether that matters
  is the Audit 01/02 open question the parity doc itself flags.
- Risk: low; it is the status quo. Decision belongs to Adrian.

## 8. Root-Cause Hypothesis

The column was dropped because (a) the NEW `AppointmentType` was modeled as a plain
`Name`+`Description` lookup (ABP Suite CRUD scaffold) with no awareness of the OLD
re-eval classification, and (b) the entire seed set was re-authored to a different
6-type California-IME taxonomy, so even the concept of paired base/re-eval rows
disappeared. The FE re-eval distinction (`?type=1|2`) was preserved at the flag
level but the dropdown-filtering enhancement that consumes the classification was
explicitly deferred. Confidence HIGH -- every claim is grounded in read source on
both sides, and the OLD consumption path is fully enumerated by grep.

Secondary finding (HIGH confidence): the parity docs misdescribe `ReEvalId` as a
self-referential FK that drives max-time gates. It is actually a 3-value
classification flag (`EvaluationTypeEnum`) consumed only by the FE dropdown filter;
the max-time gate keys off the type enum, not `ReEvalId`. The parity docs
(G-07-02 and G-10-11) and the CSV note ("self-ref column + seed base->reval
pairings") should be corrected before this is implemented.
