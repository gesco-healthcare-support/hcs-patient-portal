## 1. ID & Summary

The audit framed G-03-05 as: NEW emails a per-document secure link (one
`VerificationCode` GUID per package document) while OLD emailed one
appointment-wide upload link plus a "N documents still not uploaded"
remaining-count nudge. The CSV disposition is `include` with the planning
hint "Adopt OLD packet-wide upload link + remaining-count messaging (vs NEW
per-document links)."

Verification finding: **the gap as literally written is already largely
closed in NEW**, and the closure post-dates both the parity doc's "Keep in
NEW" conclusion and the CSV note. The actual emailed links and the
remaining-count nudge in NEW are appointment-scoped, matching OLD intent.
What is genuinely per-document in NEW is the *anonymous-upload mechanism*
(`PublicDocumentUploadController` + `VerificationCode`), which is a backend
plumbing detail and is NOT what gets emailed to the patient. So the
remaining work is small and verification-oriented, not a re-architecture.

Two competing readings, narrowed against evidence:

- Reading A (the CSV/audit assumption): NEW emails per-document
  `upload-by-code` links; we must replace them with one packet-wide link +
  count. -- **Refuted.** No NEW email handler or template references the
  `upload-by-code` route; no Angular page consumes it (only the
  auto-generated proxy).
- Reading B (verified state): NEW's Accepted/Rejected nudge emails already
  use an appointment-scoped link (`/appointments/view/{appointmentId}`) and
  an inline remaining-package-doc count via the `*RemainingDocs` templates;
  the reminder job uses an appointment-scoped `PortalUrl`. The per-document
  `VerificationCode` is only the auth token for the anonymous upload
  endpoint, not an emailed link. -- **Confirmed by code.**

Net: the include item resolves to (a) a documentation/CSV correction, and
(b) a small confirmation pass that the appointment-wide link + count is the
only patient-facing direction, plus deciding whether the orphaned
per-document `upload-by-code` anonymous surface should be wired to a real
emailed link / Angular page or left as dead-but-tested plumbing.

## 4. Expected vs Actual

- Expected per CSV: NEW emails a per-document link; we must switch to a
  packet-wide link + remaining-count nudge.
- Actual: NEW's Accepted/Rejected status-change emails already send an
  appointment-scoped `/appointments/view/{appointmentId}` link and an
  inline remaining-package-doc count (the `*RemainingDocs` templates); the
  standing reminder uses the portal/appointment URL. No NEW email embeds a
  per-document `upload-by-code` link. OLD's live behavior was likewise
  appointment-scoped (approval email -> `/appointment-new-documents/{id}`;
  JDF status email -> `/appointment-documents/?appointmentid=...` + count;
  package status email -> no link, count commented out).
- Therefore the user-visible parity target (packet-wide link + count) is
  MET. The remaining divergence is that NEW *also* carries a per-document
  anonymous-upload mechanism that OLD only had as dead code -- but that
  mechanism is not surfaced in any emailed link or UI, so it is not a
  patient-facing deviation today.

## 8. Root-Cause Hypothesis

Confidence: high. The audit doc was written against an earlier snapshot
where the per-document `VerificationCode` path was the visible novelty, and
it concluded "Keep in NEW (per-document)." Between that audit and now,
Phase 5 / Category 5 (2026-05-10) landed the appointment-scoped
remaining-count nudge (`*RemainingDocs` templates + the count branch in the
two email handlers), and the per-document link was never actually wired into
any email or UI. The CSV note ("adopt OLD packet-wide link + count")
therefore describes an outcome NEW already produces for the patient-facing
emails. The "intent deviation" is real only at the plumbing layer (NEW has
a per-document anonymous endpoint OLD only had as dead code), and that
plumbing is not patient-visible. So the root cause of the item is
**audit/CSV drift relative to shipped code**, not a missing behavior.
