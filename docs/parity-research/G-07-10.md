## 1. ID & Summary

The NEW `SystemParametersAppService.UpdateAsync` re-validates every integer
system-parameter field with `Check.Range(1, int.MaxValue)` on every update,
plus a `CcEmailIds` length check and optimistic concurrency. The parity doc
frames this as an "Intent deviation / OLD-bug-fix exception": it asserts OLD
"never validated on edit" and that OLD's `[Range(1, int.MaxValue)]` attributes
"only fire via model binding on insert."

Disposition is **include / keep NEW** -- the work is already built and unit
tested. The CSV note correctly calls it "Hardening; already built."

Key finding from verification: the gap is **narrower than the doc states**.
The OLD update path was NOT unvalidated. OLD registered a GLOBAL
`ModelValidation` action filter (`ConfigurationService.AddMvcOptions`,
`:72`) that short-circuits invalid `ModelState`. Because OLD's PUT binds
`[FromBody]SystemParameter` (whose properties carry `[Range(1, int.MaxValue)]`),
the framework's DataAnnotations validation populates `ModelState` on the
update path too, and the global filter would reject `0`/negative on PUT --
not only on insert. The empty `CommonValidation()` the doc cites is a
red herring: it was never where OLD's range enforcement lived. NEW's explicit
`Check.Range` is therefore best understood as **making the same guard explicit
and pipeline-independent** (so it also fires for in-process callers / tests),
not as inventing a guard OLD lacked. Either reading leaves the decision the
same: keep NEW.

## 4. Expected vs Actual

- Expected (parity intent + disposition): editing system parameters must reject
  `0`, negative, and out-of-range integer values, and over-long CcEmailIds.
- Actual (NEW): exactly that, enforced at both the DTO/DataAnnotations layer and
  the explicit AppService `Check.Range` layer. Matches expected.
- Actual (OLD, corrected): the doc's "OLD never validated on edit" is most likely
  INACCURATE. OLD's global `ModelValidation` filter + `[Range]` attributes on the
  bound entity would have rejected `0`/negative on PUT as well. So NEW and OLD are
  closer in user-visible behaviour than the doc claims; NEW is at most marginally
  stricter (explicit, pipeline-independent, plus the additive CcEmailIds length and
  concurrency-stamp checks OLD lacked).

## 8. Root-Cause Hypothesis

The deviation exists because the NEW implementation chose explicit, layered,
pipeline-independent validation (DTO `[Range]` + AppService `Check.Range` + entity
ctor `Check.Range`) instead of relying on a single framework-level ModelState
filter the way OLD did. The parity doc then mis-characterised OLD as having NO
update-time validation, because it only inspected `SystemParameterDomain`
(empty `CommonValidation`) and the controller (no inline `ModelState` check), and
did not trace the GLOBAL `ModelValidation` filter wired in `ConfigurationService`.
In reality OLD enforced `[Range(1, int.MaxValue)]` on PUT via that global filter.
So the true root cause of the "deviation" is a combination of (a) NEW being
explicitly hardened and (b) the parity audit under-reading OLD's MVC pipeline.
Net user-visible behaviour: NEW and OLD both reject `0`/negative on edit; NEW
additionally enforces a CcEmailIds max length and optimistic concurrency.
Confidence: high on the NEW guard (read + tests); high on OLD having SOME global
ModelState enforcement (filter registration + DLL symbols are unambiguous);
medium-high that it produces a clean 400 reject for `0` specifically (filter body
is compiled, not source-verified).
