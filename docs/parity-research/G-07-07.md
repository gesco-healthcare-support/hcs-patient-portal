## 1. ID & Summary

OLD had an admin-managed master lookup table `AppointmentDocumentType`
(`spm.AppointmentDocumentTypes`) with full CRUD (entity + domain +
controller + search controller + Angular list/add/edit screens). Uploaded
appointment documents referenced it by FK (`AppointmentDocumentTypeId`) and
fell back to a free-text `OtherDocumentTypeName` when the user picked the
"Other" option. NEW dropped the lookup entirely: `AppointmentDocument`
classifies uploads with a single free-text `DocumentName` string, and the
entity's own header comment lists "AppointmentDocumentType lookup (free-text
DocumentName at MVP)" as a deliberate deferred cut. The upload UI in NEW is a
plain text box -- no dropdown, no controlled vocabulary, no "Other" fallback.

This row is a DUPLICATE of G-03-01 (the canonical, primary include in
`03-documents-packets.md`) and G-10-09 (the storage-side data-model row in
`10-data-model.md`). It also closes the residual type-metadata gap noted in
G-03-03. The substantive work lives under G-03-01; this item exists so the
admin-master-data area cross-references the same fix. Recommended primary
action: implement once under G-03-01 (doc-type master entity + CRUD AppService
+ admin Angular screen + upload picker with "Other" free-text fallback), and
mark G-07-07 / G-10-09 resolved by that work.

## 4. Expected vs Actual

- Expected (OLD parity): an admin-managed master list of document categories
  (CRUD screen), and on upload a dropdown picker bound to that list with an
  "Other -> free-text" fallback; uploaded documents reference the category by
  FK (`AppointmentDocumentTypeId`) plus `OtherDocumentTypeName` when "Other".
- Actual (NEW): no master list, no CRUD, no picker. Uploaders type any string
  into a single free-text "Document Name" box; there is no controlled
  vocabulary, no per-category filtering, and no admin curation. Documented in
  the entity header as an intentional MVP cut.

## 8. Root-Cause Hypothesis

High confidence: deliberate, documented MVP scope cut, not an accidental
regression. The NEW `AppointmentDocument` entity was authored "W1-3" as a
"minimum viable appointment-document row" and explicitly enumerates
`AppointmentDocumentType lookup (free-text DocumentName at MVP)` in its
"Cuts deferred to ledger" list (`AppointmentDocument.cs:10-21`). The OLD
lookup, CRUD, and picker all verifiably exist; NEW chose free-text to ship
documents without the master-data dependency. The fix is a straightforward
additive port; the only real decision is the entity's multi-tenancy placement.

## 9. Open Questions

1. Business need (source-doc open Q #3, owner Adrian): does the IME workflow
   actually require a controlled document-category vocabulary, or is free-text
   sufficient? CSV already answers "include," but the source doc still flags
   it -- confirm before sizing.
2. Entity placement: host-shared reference data (like NEW `Location`,
   `AppointmentStatus`) vs tenant-scoped (each office curates its own
   categories)? OLD was single-tenant so the question did not arise. Per the
   source doc, NEW treats shared reference data as host-scoped; recommend
   host-shared unless per-office categories are required.
3. The OLD upload picker hardcodes `value == 4` as the "Other -> free-text"
   trigger (`appointment-document-add.component.html:16`). I could NOT confirm
   from source that seed doc-type id 4 = "Other" -- no seed INSERT for
   `AppointmentDocumentTypes` exists in the available OLD SQL (only DDL at
   `_local/generated-schema.sql:68`). Treat the magic `4` as a PARITY-FLAG
   candidate: NEW should model "Other" by a stable marker (a dedicated flag or
   a sentinel row), NOT by re-hardcoding `4`, to avoid inheriting an OLD
   seed-order assumption.
4. Should NEW keep BOTH `AppointmentDocumentTypeId` (FK) and
   `OtherDocumentTypeName` (free-text) as OLD did, or fold "Other" into the
   existing `DocumentName`? Folding is simpler but loses the OLD distinction
   between "the typed category" and "the user-given document name."
5. Status model: OLD doc-type used the tri-state `StatusId` (soft-delete via
   `StatusId = Delete`); NEW lookups use `IsActive` + ABP soft-delete
   (`FullAuditedAggregateRoot`). Use the NEW `IsActive` convention (matches
   G-07-13 disposition) rather than porting the tri-state enum.
