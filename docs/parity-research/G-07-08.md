> **DECISION (finalized 2026-06-01) [Gate 4]:** decision include -> exclude (no action), effort -> none. City stays free-text (G-10-04); Country is not added (US-only; Country table deferred via G-10-03); ApplicationTimeZone/GlobalSetting already function through ABP settings -- no new structured pickers. Net new work: nil. See SUMMARY.md Gate 4.

## 1. ID & Summary

OLD shipped a home-grown globalization / global-config layer: `Country`
(currency/date/phone/postal formats + `DefaultLanguageId`), `City` (FK to Country
+ State), `ApplicationTimeZone` (per-user TZ), and `GlobalSetting` + `LockRecord`
(record-lock duration, 2FA, auto-translation, social-auth, request-logging toggles).
NEW has none of these entities. The disposition is **include** but narrowly scoped
by the CSV note: "add structured pickers, not rip out ABP" -- i.e. add a structured
Country + City + TimeZone lookup set, while leaving the GlobalSetting/2FA/record-lock
toggles to ABP's native settings + Identity system.

Key research finding that re-frames the scope: in OLD, **almost none of this layer
was actually consumed**. `City` was never FK'd to any address (OLD `Location.City`
and `WcabOffice.City` are free-text strings -- same as NEW). `GlobalSetting`'s five
boolean toggles are persisted but **never read anywhere** in OLD source; only its
`LanguageId` is read (once, in `ApplicationUtility.cs:70`). `LockRecord` has an
**empty stub filter** (`LockRecordFilter.cs` is `class LockRecordFilter {}`).
`ApplicationTimeZone` is read on login to populate user context and is **hardcoded
to `ApplicationTimeZoneId = 1`** on every user create. So the "gap" is largely the
absence of latent/dead infrastructure, which ABP already replaces for the parts that
mattered (TZ, lockout, 2FA, default language).

### Upstream callers

For the **included scope** (structured Country + City + TZ pickers): there are no
existing NEW callers, because the entities don't exist. A future Location / WcabOffice
admin form *could* consume a City picker, but today both store free-text City and
neither OLD nor NEW wired City as an FK. So adding pickers is purely additive; no
existing caller forces the change.

### Downstream consumers

- A City picker would tie to `State` (G-10-04) -- City rows would FK to the existing
  `States` lookup. State is the only existing downstream dependency.
- A Country picker would be a new standalone lookup with no current consumer (US
  single-clinic: exactly one Country row).
- A TZ table would overlap the ABP `ReminderTimezoneId` setting; if added, it should
  be the source list feeding that setting's allowed values, not a parallel mechanism.

### Blast radius

Low. For the included scope (structured pickers) a fix touches the standard ABP CRUD
stack per new lookup, mirroring the existing `States` feature:

- Per lookup (Country, City, optionally TimeZone): Domain entity + IRepository +
  Manager + DataSeedContributor; Application.Contracts DTOs + IAppService +
  Permissions; Application impl + Mapperly mapper; EF config + migration; HttpApi
  controller; Angular feature folder + proxy regen. ~8-10 files per lookup x up to 3
  lookups, all NEW files (additive). Pattern is copy-of-`States`.
- Regression risk: minimal -- no existing entity, service, or migration is modified;
  Location/WcabOffice City stays free-text unless a separate decision (out of this
  item's scope) converts them to FKs. The only shared touchpoint is `States` (read-only
  FK target for City). The GlobalSetting/2FA/record-lock half is explicitly NOT ripped
  out of ABP, so no Identity/settings regression.

## 4. Expected vs Actual

- **Expected (OLD behavior that mattered):** a default-language lookup (used), a
  per-user timezone (used, but hardcoded to 1), and tri-format Country metadata
  (currency/date/phone/postal). The City table and the five GlobalSetting toggles and
  the LockRecord filter were **latent/dead** in OLD.
- **Actual (NEW):** Country/City/ApplicationTimeZone/GlobalSetting/LockRecord entities
  absent. The *used* capabilities are covered by ABP: timezone via `ReminderTimezoneId`
  setting, 2FA/lockout/confirmed-email via ABP Identity settings, default language via
  ABP localization, SystemParameter knobs via ABP settings. `State` lookup is the one
  geographic master ported with full CRUD.
- **Net deviation:** structured Country + City pickers do not exist as selectable
  master data (US single-clinic needs at most one Country and a short City list, both
  currently free-text or absent). This is the only genuine, user-visible delta; the
  rest is equivalent-or-better via ABP.

## 7. Candidate Solutions

### Option A -- Add Country + City structured lookups (mirror States), leave GlobalSetting/TZ/2FA to ABP [effort: M] [risk: low]

Build two new ABP CRUD lookups copying the existing `States` feature: `Country`
(name + optional format metadata if needed) and `City` (name + FK to `State`, FK to
`Country`). Seed one Country (USA) and the relevant California cities for the single
clinic. Do NOT add GlobalSetting/LockRecord/ApplicationTimeZone tables -- those map to
ABP settings + Identity (already present). Optionally surface a TZ *picklist* that
feeds the existing `ReminderTimezoneId` setting rather than a parallel table.
- Source: OLD code `Country.cs`, `City.cs` (field sets); NEW `States/*` (the proven
  pattern); `CaseEvaluationSettingDefinitionProvider.cs:81,99-127` (ABP TZ/2FA/lockout
  already covering the GlobalSetting half).
- Tradeoffs: matches the CSV directive ("add structured pickers, not rip out ABP");
  ties City to G-10-04 cleanly. Country format columns (currency/date/phone/postal)
  are almost certainly unneeded for a US clinic -- include only `CountryName` +
  `CountryCode` unless Adrian wants the formats. Risk is low (all-additive, no
  migration of existing data, City on Location/WcabOffice stays free-text unless a
  separate item converts them).
- Effort: M (two ABP CRUD slices + Angular + seed; ~8-10 files each).

### Option B -- Defer / minimal: add only what a future feature consumes [effort: S] [risk: low]

Do nothing now beyond confirming ABP coverage of the TZ/2FA/record-lock half; add the
Country/City pickers only when a concrete consumer appears (e.g. a Location/WcabOffice
admin form that wants a City dropdown, or G-10-04 work). Rationale: in BOTH OLD and NEW,
City is free-text and unused as an FK; a US single-clinic has one Country. Building
selectable Country/City master data with no consumer is speculative.
- Source: OLD consumption analysis (City/Country have zero domain consumers;
  Location/WcabOffice City is free-text in both apps).
- Tradeoffs: lowest cost; honors "Low value for US single-clinic; confirm scope" in the
  CSV note. Risk: leaves the parity doc's "include" technically unmet until the consumer
  lands -- acceptable if paired with G-10-04 and the Location/WcabOffice admin screens
  (G-07-01).
- Effort: S (decision + doc note only).

### Option C -- Add a dedicated GlobalSettings admin screen for the OLD toggles [effort: M] [risk: med] (NOT recommended)

Surface OLD's 2FA / record-lock-duration / auto-translation / social-auth /
request-logging toggles as a NEW admin screen, backed by new settings.
- Source: OLD `GlobalSetting.cs`.
- Tradeoffs: explicitly counter to the CSV directive ("not rip out ABP") and to the
  parity doc's "mostly do-not-port." All five toggles were **dead in OLD** (no
  consumers; LockRecord filter is an empty stub). 2FA + lockout are already first-class
  ABP Identity features with their own admin UI. Rebuilding them as bespoke toggles
  would duplicate ABP and risk drift. Risk med (touches Identity/settings semantics for
  no parity benefit).
- Recommendation: reject. Confirm ABP coverage answers Open Question 4 instead.

## 8. Root-Cause Hypothesis

high confidence. The "gap" is a deliberate stack swap, not a defect. OLD rolled its own
i18n + global-config tables (`Country`, `City`, `ApplicationTimeZone`, `GlobalSetting`,
`LockRecord`); the modern stack provides these as framework primitives (ABP settings,
ABP localization/language management, ABP Identity 2FA/lockout, LeptonX). Evidence that
the deviation is safe: in OLD itself, City/Country had no consumers, the five
GlobalSetting toggles were never read, and LockRecord's filter was an empty stub --
so dropping the tables loses no real behavior. The only OLD-used pieces (default
language, per-user timezone) are covered by ABP. The remaining true delta is the lack
of *structured Country/City pickers* as master data, which is low-value for a US
single-clinic and should be built (Option A) or deferred-with-consumer (Option B) per
Adrian's scope call, not by resurrecting the dead toggles (Option C).
