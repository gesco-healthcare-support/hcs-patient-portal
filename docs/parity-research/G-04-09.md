## 1. ID & Summary
Email subject lines in OLD are built as
`"Patient Appointment Portal - (" + patientName + injuryDetails + ") - <event>"`,
where the parenthesized identity suffix carries patient name, claim
number, and ADJ (WCAB) number. NEW reproduces the identity suffix
verbatim via `EmailSubjectBuilder.BuildIdentitySuffix` (exposed as the
`##EmailSubjectIdentity##` token) but shortens the fixed brand prefix
from "Patient Appointment Portal" to "Appointment Portal". The suffix
(the load-bearing context) is at parity; only the leading brand string
was trimmed. Cosmetic, intentional. Excluded.

- Expected (parity): subject carries Patient/Claim/ADJ identity suffix. MET --
  NEW reproduces the suffix string, separators, and skip-when-empty behavior.
- Deviation (intentional): brand prefix "Patient Appointment Portal" -> "Appointment Portal".
  Deliberate cosmetic shortening; not a defect.

## 6. Why Excluded + Revisit-If
Excluded because the deviation is purely the leading brand literal, which
is cosmetic and intentional per CSV note ("Keep NEW brand prefix
Appointment Portal; suffix already at parity. Cosmetic only.") The
parity-relevant content (identity suffix) is faithfully reproduced.

Revisit if:
- The business decides the full "Patient Appointment Portal" brand string
  must appear in subjects (e.g. client request / inbox-search expectations).
- The empty-name edge case (see Open Questions) is judged to matter for
  inbox display.

## 7. Candidate Solution
None -- excluded. (If revisited for the brand string: change the literal
prefix in EmailSubjects.cs constants from "Appointment Portal" back to
"Patient Appointment Portal"; trivial string edit, no logic change.)

## 9. Open Questions / Flags
- Minor edge-case asymmetry (cosmetic, not a parity gap): OLD always emits
  the literal `"Patient: "` label even when first/last are empty (it
  substitutes spaces), so an empty-name OLD suffix renders as
  `"(Patient:   - Claim: ...)"`. NEW's `BuildIdentitySuffix` SKIPS the
  "Patient:" label entirely when the name is empty, yielding
  `"(Claim: ...)"`. This is arguably a NEW improvement (cleaner) and does
  not change the brand-prefix decision, but flag it in case strict
  byte-for-byte subject parity is ever required.
- No FLAG-FOR-CONFIRMATION in the CSV note; feature is in active use
  (subjects fire on every lifecycle/document email), so no
  was-this-ever-used concern.
