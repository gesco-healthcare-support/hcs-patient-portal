## 1. ID & Summary

OLD carried a `TemplateCode` int enum (16 DB-managed `spm.Templates` rows, read via
`GetNotificationTemplateBody(int)`). Nine of those codes were the SMS-body half of the
appointment lifecycle: `AppointmentBooked`, `AppointmentApproved`, `AppointmentRejected`,
`AppointmentApprovedStakeholderEmails` (OLD spelt `...StackholderEmails`),
`AppointmentCancelledByAdmin`, `RejectedPackageDocument`, `RejectedJointDeclarationDocument`
(OLD spelt `RejectedJoinDeclarationDocument`), `AppointmentDueDate`,
`AppointmentDueDateUploadDocumentLeft`.

In OLD's live code every call site that read these codes for SMS is commented out, so they
never sent anything. NEW seeds all nine as per-tenant `NotificationTemplate` rows (via
`Codes.All`) but no handler, job, or appservice dispatches them. The user-visible EMAIL
behavior these codes overlapped with is already covered by the disk-HTML-derived codes that
ARE wired (e.g. approval emails fire via `PatientAppointmentApprovedExt` /
`PatientAppointmentApprovedInternal`, not `AppointmentApproved`).

This is a true partial: the rows exist (IT-Admin-editable, harmless) but the dispatch leg is
absent. Because in OLD these were exclusively the SMS half, "wiring" them only becomes
meaningful when the SMS channel lands -- which is G-04-01, deferred ~3 months. There is no
runtime regression today: nothing was sent in OLD, nothing should be sent in NEW until SMS
exists.

## 4. Expected vs Actual

- Expected (parity with OLD live behavior): nothing sent for these nine codes. OLD's live
  code never dispatched them (all SMS call sites commented / unreachable).
- Actual (NEW): nothing sent for these nine codes; nine inert seeded rows exist. This MATCHES
  OLD's live behavior. The "Partial" classification is because NEW carries the rows (storage
  parity with OLD's `spm.Templates`) without the dispatch leg -- but OLD's dispatch leg was
  itself dead, so there is no behavioral loss.
- The genuine future-state expectation: when SMS is implemented (G-04-01), the four
  staff-digest codes (`AppointmentDueDate`, `AppointmentDueDateUploadDocumentLeft`, and the
  digest reminders) should emit SMS to staff -- the only SMS that was ACTUALLY live in OLD
  (behind the `isSMSEnable` flag, see Open Questions). The other five appointment-lifecycle
  SMS codes were never live in OLD even with the flag on.

## 7. Candidate Solutions

### Option A -- Keep seeded, do not wire; revisit with G-04-01 [effort: S] [risk: low]

- One-liner: Accept the nine inert rows as-is (storage parity, IT-Admin editable); add the
  SMS dispatch leg only when G-04-01 (SMS provider + creds) is implemented, wiring SMS for the
  ~4 staff-digest codes that were actually live in OLD and leaving the other five dormant.
- Source: OLD code (these were dead in OLD live code -- `AppointmentDomain.cs:848-875`
  commented, `AppointmentChangeRequestDomain` SendSMS callers commented) + source doc
  directive "Keep seeded ... but do not build dispatch ... Revisit only if SMS (G-04-01) is
  implemented" + CSV note "wire when SMS lands."
- Tradeoffs: matches OLD behavior exactly today (nothing sent), keeps the door open for SMS,
  zero risk of duplicate emails. Cost: the Template Management UI shows nine editable rows
  that do nothing yet (the "confusing empty rows" concern in Open Question 5 of the doc).
- This is the decision-aligned option. The "include" disposition means "include the wiring
  WORK when SMS lands," not "wire now."

### Option B -- Wire SMS for staff-digest codes when SMS provider exists [effort: M] [risk: med]

- One-liner: After G-04-01 lands a concrete SMS sender, add an SMS leg to
  `NotificationDispatcher` (or a parallel `ISmsDispatcher`) and have the four staff-digest
  handlers emit SMS for `AppointmentDueDate` / `AppointmentDueDateUploadDocumentLeft` (and the
  digest codes) using the seeded `bodySms` rows.
- Source: OLD `SchedulerDomain.cs:105,:137,:191,:220,:249` (the ONLY live `SendSms` calls in
  OLD) + G-04-01 / CSV rows 36-37.
- Tradeoffs: restores the one SMS behavior that was genuinely live in OLD. Risk: depends
  entirely on SMS infra (provider module, creds, `isSMSEnable`-equivalent setting); mis-wiring
  to the email path would duplicate already-sent emails. Effort M because it is coupled to the
  whole SMS rollout, not isolated.
- Do NOT wire the five appointment-lifecycle codes (`AppointmentBooked`, `AppointmentApproved`,
  `AppointmentRejected`, `AppointmentApprovedStakeholderEmails`, `AppointmentCancelledByAdmin`,
  plus the two `Rejected*Document` codes) to email -- they were never live in OLD and their
  email equivalents already fire via the wired disk-HTML codes.

### Option C -- Prune the nine dead codes from the seed [effort: S] [risk: low-med]

- One-liner: Remove the nine never-sent codes from `Codes.All` so the Template Management UI
  is not cluttered with rows that do nothing.
- Source: Open Question 5 in the source doc ("prune them to avoid confusing empty editable
  rows").
- Tradeoffs: cleaner UI. But loses storage parity with OLD's `spm.Templates`, and if SMS later
  needs these codes the seed must be re-added (and existing tenants re-seeded). Contradicts the
  "include / wire when SMS lands" disposition. NOT recommended unless Adrian decides the SMS
  channel is permanently out of scope.

## 8. Root-Cause Hypothesis

Root cause (HIGH confidence): OLD ran two parallel template systems -- 43 disk-HTML files
(email) and 16 DB `TemplateCode` rows (SMS-only). The nine codes here are the SMS half of the
appointment lifecycle. The OLD team had disabled SMS almost entirely (commented call sites +
`isSMSEnable` flag), so these nine never sent in OLD's live code. NEW correctly unified both
template stores into one `NotificationTemplate` table and seeds all 59 codes for IT-Admin
editability, but -- because NEW has no SMS channel at all yet (`NotificationDispatcher` defers
the SMS leg; no `Volo.Abp.Sms` concrete provider registered) -- there is nothing to dispatch
these nine to. The "Partial" gap is therefore an artifact of (1) faithfully carrying OLD's DB
template rows for storage parity, combined with (2) the SMS channel being deferred (G-04-01).
It is not a defect; it is a dependency. The correct fix is sequenced after SMS lands, and even
then only the staff-digest subset should be wired (those were OLD's only live SMS). Evidence is
direct and code-level on both sides; no inference gaps.

## 9. Open Questions

1. Was `isSMSEnable` ever ON in OLD production? (Carried from G-04-01 Open Q1.) If it was off,
   even the staff-digest SMS was dormant, making the entire SMS-wiring effort -- and these nine
   rows -- optional rather than a parity requirement. Determines whether Option B is "restore a
   real behavior" or "build a never-used feature."
2. When SMS lands, should the SMS leg read these nine `bodySms` rows by code, or should the
   existing email handlers grow a parallel SMS emit? The former reuses the seeded rows (their
   intended purpose); the latter is simpler but ignores them. Affects whether these rows are
   ever truly "used."
3. UI hygiene (doc Open Q5): keep the nine inert rows visible in Template Management, or hide
   non-email rows until SMS exists? A display-filter (templateTypeId == SMS && SMS-disabled ->
   hidden) would satisfy both parity (rows kept) and UX (no confusing empty editable rows)
   without pruning the seed.
4. Confirm none of the nine should ever drive EMAIL: the source doc and OLD code both say their
   email equivalents are already wired via disk-HTML codes, so wiring them to email would
   duplicate. Worth an explicit sign-off before any future SMS-wiring task touches them.
