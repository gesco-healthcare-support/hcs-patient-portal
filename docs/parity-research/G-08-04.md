> `CaseEvaluation.Reports` + the per-appointment read guard, stricter than the
> read guard alone). Endpoint `api/app/appointment-demographics/{id}` streams the
> file; the appointment-view page has an internal-only Download Demographics

> Employer / per-injury Injury+Insurance+ClaimExaminer / Applicant + Defense

## 1. ID & Summary

OLD exposed a one-click, read-only "Patient Demographics" intake sheet for a
single appointment: a complete consolidated rendering of every party and
detail on that appointment (Appointment Details, Patient Details, Employer,
per-injury Injury/Insurance/Claim-Examiner blocks incl. body parts, Applicant
Attorney, Defense Attorney, Appointment Accessors with Edit/View rights, and
Custom Form fields). OLD served it as styled HTML that the SPA opened in a new
window for the browser's print-to-PDF (the iTextSharp real-PDF path was dead
code). It was reachable from the report row's PDF icon and (dead) from the
report service.

NEW has no equivalent. The only "Patient Demographics" surface in NEW is the
heading of an EDITABLE reactive-form section on the appointment-view screen
(`appointment-view.component.html:197`); there is no read-only printable
export and no PDF/print button anywhere. Per the root mission (OLD DOCX/print
-> NEW immutable PDF) the NEW build should emit a true PDF (QuestPDF), not the
open-HTML-and-print trick.

Critical good news for effort: the entire data graph the OLD sheet renders is
ALREADY assembled by NEW's `AppointmentsAppService.GetWithNavigationPropertiesAsync`,
and SSN is already masked to last-4 there. The gap is the render + endpoint +
button, not the data.

## 7. Candidate Solutions

### Option A -- QuestPDF intake-sheet rendered from the existing nav-DTO graph [effort: M] [risk: low]
One-line: add an AppService method that loads the appointment via the existing
`GetWithNavigationPropertiesAsync`, builds an 8-section PDF with a new QuestPDF
`IDocument`, and streams it from a manual controller action; add a
Print/Download button to the appointment-view screen.
Source: OLD code `CSVExportController.cs:73-106` + `GetPatientDataHtml:302-681`
(section list, labels, field order, body-parts, Edit/View rights); NEW data
graph `AppointmentWithNavigationPropertiesDto`; streaming pattern
`AppointmentPacketController.DownloadAsync:38-43`; QuestPDF community license
already set at `CaseEvaluationDomainModule.cs:83`.
QuestPDF doc reference (fluent layout / IDocument, free Community license for
Gesco's scale): https://www.questpdf.com/ (official; accessed 2026-06-01) and
the license page already cited in-repo at module line 82
(https://www.questpdf.com/license/community.html).
Tradeoffs: a true immutable PDF (mission-aligned); no Gotenberg dependency for
this path (QuestPDF renders in-process); reuses the already-masked DTO so SSN
masking is inherited for free. First QuestPDF render in the codebase, so it
establishes the pattern G-08-03 (report-table PDF) will reuse.
Risk: low -- additive, no DB change, reuses a tested read path. Main work is
faithfully reproducing OLD's section order/labels and the conditional bits
(has-interpreter, language-or-other-language, cumulative-trauma flag, per-
accessor Edit/View, custom fields).

### Option B -- DOCX template + Gotenberg (reuse the packet pipeline) [effort: M/L] [risk: med]
One-line: author a DOCX intake-sheet template, fill it via the existing
`DocxTemplateRenderer`/token resolver, convert with `GotenbergDocxToPdfConverter`.
Source: NEW packet pipeline (`AppointmentDocuments/Templates/*`,
`Pdf/GotenbergDocxToPdfConverter.cs`).
Tradeoffs: reuses an existing render+convert path; but it is built around
PACKET tokens/templates and a Hangfire job, and it adds a hard runtime
dependency on the Gotenberg sidecar for a synchronous user-facing download.
Heavier to thread a new template through, and the token model is packet-shaped,
not intake-sheet-shaped.
Risk: med -- sidecar availability on the request path; more moving parts than
Option A for the same outcome.

Recommendation: Option A. It is mission-aligned (true PDF), reuses the
already-assembled + already-SSN-masked DTO, avoids a synchronous Gotenberg
dependency, and seeds the QuestPDF pattern G-08-03 will also want.

## 8. Root-Cause Hypothesis

The whole Reporting & Export area (Audit 08) was never built in NEW -- the
audit's own summary says "NEW has built nothing in this area." The
per-appointment demographics sheet is one of four Missing-behavior items in
that untouched area. It is not a regression or a half-finished feature; it is a
not-yet-ported capability. The data plumbing landed incidentally (the
appointment-view + nav DTO + SSN masking were built for the booking/view
flows), which is why the FIX is small even though the feature is fully absent.
Confidence: high -- absence is statically provable and the parity doc + code
agree.

## 9. Open Questions

1. SSN/DOB on the printed copy (HIPAA). OLD printed full SSN and DOB
   (`CSVExportController.cs:410,409`). NEW masks SSN to last-4 on every read
   DTO and gates full SSN behind the audited `Patients.RevealSsn` endpoint
   (`GetFullSsnAsync`, Design B, PR #272). Decision needed: does the intake
   PDF show masked last-4 (inherits the DTO mask -- safest, default), or does
   it support a permissioned full-SSN render for internal roles? Recommend
   masked-by-default; full only via the same RevealSsn gate if the owner
   requires it for clinical hand-off. Cross-ref the patients-domain SSN-reveal
   audit.
2. Permission gate. G-08-01 proposes a new `CaseEvaluation.Reports` permission
   (which does NOT exist yet -- confirmed absent from
   `CaseEvaluationPermissions.cs`). But this sheet is per-appointment, so it
   could instead reuse `Appointments.Default` + the existing
   `AppointmentReadAccessGuard` (which already governs who may read a given
   appointment, including AA/DA/CE scoping). Decide: Reports permission vs
   appointment-read guard. Recommend the appointment-read guard for Phase 1
   (it already enforces per-role visibility) and add `Reports` when G-08-01
   lands the report grid.
3. Custom-field values exposure. OLD renders `CustomFieldsValues` ->
   `CustomField.FieldLabel: value` (lines 655-672). The NEW
   `AppointmentWithNavigationPropertiesDto` does NOT currently list a
   custom-field-values collection. Verify whether appointment custom-field
   values are persisted/readable in NEW and, if so, whether the nav DTO/repo
   must be extended to include them for the PDF (small add) -- or whether
   custom fields are themselves a separate, not-yet-ported feature.
4. Body-parts source. OLD uses `injuryDetail.BodyParts` (a denormalized string
   on the view) rather than the `AppointmentInjuryBodyPartDetail` rows it also
   loads (lines 545-556). NEW exposes `List<AppointmentBodyPartDto> BodyParts`
   on the injury nav DTO. Confirm the NEW list reproduces the same content
   (descriptions) the OLD string showed; cosmetic, not blocking.
5. Entry points. OLD also triggered this from the report grid PDF icon. In NEW
   Phase 1 only the appointment-view button exists (the report grid is G-08-01,
   not yet built). Confirm that single entry point is acceptable for the parity
   slice, with the report-grid icon deferred to ship alongside G-08-01.
