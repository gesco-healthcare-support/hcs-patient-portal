## 1. ID & Summary
OLD had a Twilio SMS path (`TwilioSmsService.SendSms`) gated by the `isSMSEnable`
ServerSetting. The ONLY live SMS call sites were the SchedulerDomain background
digests (pending queue, package-doc reminder, due-date approaching, doc-incomplete).
Every interactive/patient-facing SMS path in OLD was commented out. NEW delivers
no SMS at all: the dispatcher renders `BodySms` but never sends it; no concrete SMS
sender is registered. Deliberate Phase 1 scope cut, deferred ~3 months (planned by
2026-08-29). Same information still goes out as email digests, so no patient-facing
SMS is lost (patient SMS was never live in OLD).

## 4. Expected vs Actual
- Expected (OLD, when `isSMSEnable` ON): the 4-5 scheduler digests fire an SMS to
  internal staff in addition to the email. No patient/external SMS (those were
  commented out in OLD).
- Actual (NEW): zero SMS delivered anywhere. The same digest information reaches
  staff via email only. `BodySms` is rendered (and seeded as a stub) but discarded.

## 7. Candidate Solution (high-level; deep sourcing deferred)
When the provider lands, register a concrete `ISmsSender` (ABP's `Volo.Abp.Sms`
abstraction is already present) backed by Twilio or ACS-SMS, then have
`NotificationDispatcher` (or the per-feature handlers it notes at :26-28) deliver the
already-rendered `BodySms` for the 4-5 scheduler digests ONLY. Replace the seeder
stub at line 126 with the OLD-verbatim SMS bodies (the DB `TemplateCode` rows in
G-04-07 are the SMS-side bodies to port). Re-implement the OLD gate as an ABP setting
(`isSMSEnable` -> a per-tenant setting). Add phone normalization (OLD strips `-`,
prefixes `countryCode`; note OLD's `+91` country code is an OLD bug for a US app per
the bug policy -- use `+1`). Do NOT build patient/external SMS -- never live in OLD.
Deep provider/SDK sourcing (Twilio vs ACS SDK choice, ABP `ISmsSender` wiring) is
HELD until the upstream creds decision is made -- not worth sourcing now.

## 8. Root-Cause Hypothesis
Deliberate, documented Phase 1 scope cut. The notification pipeline was built
email-first; SMS delivery was intentionally deferred to the Twilio/ACS creds rollout
(master-plan 18.3) because it is a host-config + provider-integration change rather
than a per-feature port. The `BodySms` field, stub seeding, and `Volo.Abp.Sms`
abstraction were all left in place precisely so the SMS leg can be wired later with
minimal disturbance. Confidence: high (the deferral is stated verbatim in code).
