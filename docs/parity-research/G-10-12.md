> **DECISION (finalized 2026-06-01) [Gate 7]:** decision include -> exclude (verified no-action). OLD's InsuranceNumber/ClaimExaminerNumber were the address SUITE line (form label "STE", appointment-add.component.html:414-415 and 476-477), NOT policy/claim identifiers. The real claim identifiers are stored separately and preserved in NEW: AppointmentInjuryDetail.ClaimNumber (:31-33) + AppointmentWorkerCompensation.ClaimNumber (:19-21) / WcabAdj (:48-50). No PolicyNumber field ever existed in OLD. NEW's rename to Suite is parity-preserving and reduces misinterpretation risk -- keep it. Optional micro-cleanup: rename the two domain-manager params to `suite`. A real insurance policy number would be a NEW enhancement, out of parity scope. See SUMMARY.md Gate 7.

## 1. ID & Summary

OLD `AppointmentPrimaryInsurance.InsuranceNumber` (varchar 255) and
`AppointmentClaimExaminer.ClaimExaminerNumber` (varchar 255) were
renamed to `Suite` in NEW (Issue 2.3, 2026-05-12, EF migration
`20260512214140_Rename_InsuranceNumber_And_ClaimExaminerNumber_To_Suite`).
The CSV decision-log row dispositioned this `include` with a FLAG:
"Restore distinct InsuranceNumber + ClaimExaminerNumber fields ...
confirm OLD actually captured these numbers (reverses a prior resolved
rename)."

**Verification verdict: the FLAG question is answered, and the answer
argues AGAINST the include disposition.** OLD did NOT use these columns
as insurance/claim-examiner numbers in live behavior. In both the OLD
Angular intake form and the OLD CSV/print export, the column is the
address suite line (USPS "STE"), rendered between Street and City. The
"appointment number / Appt #" interpretation was explicitly commented
out in the OLD export. The NEW rename to `Suite` is therefore a faithful
relabel of the field's real meaning, not a meaning-loss. There is no
separate Suite/address-line-2 column in OLD to preserve -- OLD had only
`Street` plus this single number/suite field.

Recommendation for Adrian's review: keep the `Suite` rename; do NOT
restore distinct number fields. If retained anyway, the change is small
and low-risk (see Option B). One genuine residual sub-question: was the
ORIGINAL design intent (pre-relabel) ever exercised by a back-office
report outside the two export paths examined here? No such code found.

- Effort: S (documentation / decision only; optional trivial param

### Option B -- Honor disposition literally: re-add distinct number fields [effort: M] [risk: medium]
- One-line: add back `InsuranceNumber` + `ClaimExaminerNumber` columns

## 8. Root-Cause Hypothesis

The original OLD developers named two address-suite columns
`InsuranceNumber` / `ClaimExaminerNumber` -- almost certainly a
copy-from-a-template / misnomer (the commented-out `Appt #` export lines
show an early "number" intent that was abandoned in favor of using the
field as the address suite). The NEW team correctly diagnosed this
during Issue 2.3 (2026-05-12) and renamed to `Suite`, matching the
on-screen "STE" label and the way OLD's export actually used the value.

The G-10-12 gap exists in the parity catalogue only because the audit
classified any column-NAME change as an "intent deviation" worth logging,
and the CSV reviewer -- lacking the form/export evidence -- hedged toward
"restore the distinct fields" pending confirmation. This research
supplies the missing confirmation: OLD's live behavior is a suite/address
line, so the rename is parity-PRESERVING, not parity-breaking.
Confidence: HIGH (form labels + export concatenation + dead number-render
all agree; data preserved by RenameColumn).
