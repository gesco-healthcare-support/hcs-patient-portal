## 1. ID & Summary
OLD `dbo.Users` carried profile columns beyond auth. The audit flags the
ones without an obvious NEW home -- chiefly `SignatureAWSFilePath` (the
e-signature image stamped into generated packets). The gap title's
"non-doctor signature image" phrasing implies OLD might embed a
per-recipient or per-non-doctor signature that NEW omits. Static review
shows this is NOT the case: every OLD packet embeds the SAME signature,
keyed on `Appointment.PrimaryResponsibleUserId` (the responsible
internal staff user), and the NEW stack already replicates that exactly.
Other flagged columns map cleanly to NEW. Excluded.

## 2. Docs Reviewed
- docs/parity-v2/10-data-model.md, section `### G-10-07` (OLD/NEW blocks).
- docs/parity-review-log.csv, row 76 (disposition + note).

- `PatientAppointment.DbEntities/Models/User.cs` -- `SignatureAWSFilePath`
  (MaxLength 255) plus `FirmName`, `FirmEmail`, `IsAccessor`,
  `ApplicationTimeZoneId`, `Address/City/StateId/ZipCode`, `DateOfBirth`,

  -- signature embed sites at lines 477, 568, 656, 703, 754, 815. EVERY

- `Domain/AppointmentDocuments/Templates/PacketTokenResolver.cs:197-204`
  -- resolves `ResponsibleUserSignature` via the responsible user's blob
  keyed on `appointment.PrimaryResponsibleUserId`, stamped at the
  `##Appointments.Signature##` token. Matches OLD attribution exactly.
- `Domain/BlobContainers/UserSignaturesContainer.cs`,
  `HttpApi/Controllers/UserProfile/UserSignatureController.cs`,
  `Application.Contracts/UserProfile/{IUserSignatureAppService,UserSignatureInfoDto}.cs`

Other flagged columns (per source doc + CSV note, accepted without deep

masters; `IsAccessor` -> `AppointmentAccessor` rows; `ApplicationTimeZoneId`
-> tracked separately as G-10-OQ.

## 4. Expected vs Actual (brief)
Expected (if a gap): NEW has nowhere to pull a user signature for packet
generation, or omits a non-doctor signature OLD embedded.
Actual: NEW has end-to-end per-user signature storage AND stamps the
responsible-user signature into packets via PrimaryResponsibleUserId --
the same and only signature OLD ever embedded. No behavioral delta.

## 5. Repro Status
not-reproduced-static-confirmed. Confirmed by reading OLD embed sites and
the NEW PacketTokenResolver. A runtime repro would require uploading a
signature and generating a packet (state mutation) -- not done per
read-only guardrail; unnecessary given the static match.

## 6. Why Excluded + Revisit-If
Excluded because the signature behavior is already at parity and the
remaining profile columns have confirmed NEW homes. The gap title's
"non-doctor signature" worry is unfounded: OLD never embedded a
per-recipient/per-non-doctor signature; it always used the one
responsible-user signature, which NEW reproduces.

Revisit if any of:
- A generated OLD packet template (`.docx` under the AWS packet buckets)
  is found to contain a SECOND signature placeholder fed from a different
  user (e.g. doctor's own signature distinct from PrimaryResponsibleUser).
  Not observed in code, but the actual template files were not inspected.
- A profile column flagged here (e.g. `DateOfBirth`, `PhoneNumber` on the
  staff user) turns out to drive a NEW-missing behavior. Out of this
  row's scope; would be its own gap.

## 7. Candidate Solution
None -- excluded.

## 8. Decision Rationale + confidence
Confidence HIGH. The exclude rests on directly observed code, not just
the CSV note: NEW `PacketTokenResolver.cs:197-204` mirrors OLD
`AppointmentDocumentDomain.cs` signature attribution
(PrimaryResponsibleUserId) line-for-line in intent, and the full
UserSignature feature exists in NEW. The CSV phrasing "Doctor signature"
is a minor mislabel -- OLD attributes the signature to the responsible
internal staff user, not a Doctor entity -- but the conclusion (covered)
holds.

## 9. Open Questions / Flags
- FLAG (low risk, not blocking the exclude): the OLD packet `.docx`
  template files themselves were not opened, only the embed code. The
  code proves only ONE signature image is inserted per packet. If a
  template carried a second signature merge field wired elsewhere, that
  would be a distinct gap -- no evidence of one exists.
- Note for the log: gap title says "non-doctor signature image"; OLD has
  no such concept. The single embedded signature is the responsible
  staff user's, covered by NEW UserSignatureAppService + PacketTokenResolver.
