## 1. ID & Summary

In the OLD app, `CustomField.MultipleValues` (comma-separated picklist/radio
options) and `DefaultValue` existed in the schema but were never editable from
the admin add/edit form -- the form only rendered `fieldLabel`, `fieldTypeId`,
`fieldLength`, `isMandatory`. So Picklist (14) and Radio (17) custom fields
could be stored but never configured through the UI (latent schema).

In the NEW app the backend is COMPLETE: the entity, `CustomFieldCreateDto`,
`CustomFieldUpdateDto`, `CustomFieldDto`, the AppService Create/Update, the
manual controller, and the auto-generated Angular proxy all carry and persist
`MultipleValues` + `DefaultValue` + `AppointmentTypeId`. The booking-form
renderer already consumes `MultipleValues` to render Picklist/Tickbox/Radio
option lists. The ONLY missing piece is the admin Angular screen to enter those
values -- which is precisely the scope of G-07-01 (no admin UI for six config
areas, custom-fields among them). G-07-05 is the "do not forget the
MultipleValues/DefaultValue editor when you build that screen" line item.

This is NOT an independent gap to fix in isolation: it is a sub-requirement of
G-07-01. The CSV note says verbatim "part of G-07-01". No backend change is
required; the effort is purely adding two form controls (a textarea/repeater for
options + a default-value input) inside the custom-fields admin form that
G-07-01 will create.

## 4. Expected vs Actual

- Expected (parity-plus target): IT Admin opens the custom-fields admin screen,
  picks field type = Picklist or Radio, and enters the option list (stored as
  comma-separated `MultipleValues`) plus an optional `DefaultValue`. The booking
  form then renders the dropdown/radio with those options. This is "parity-plus"
  -- OLD never let you do this; NEW intends all 7 types to be usable.
- Actual (NEW today): no admin screen exists at all (G-07-01). Even though the
  API accepts `MultipleValues`/`DefaultValue`, an admin has no UI to enter them.
  Picklist/Radio fields can only get options via a direct DB write or data seed.
- Actual (OLD): same end state by a different route -- the add/edit form omitted
  the inputs, so options were un-configurable through the UI (latent schema).

## 7. Candidate Solutions

### Option A -- Fold the MultipleValues + DefaultValue editor into the G-07-01 custom-fields admin form [effort: S] [risk: low]

When G-07-01 builds the custom-fields add/edit Angular component, include two
extra reactive-form controls:
1. `multipleValues` -- shown only when `fieldType` is Picklist (14) or Radio
   (17) (and optionally Tickbox 15, which the renderer also supports as a
   multi-option group). Bind to a textarea or a small add/remove option
   repeater that serializes to a comma-separated string (mirroring the
   `optionsFromMultipleValues` contract the renderer already expects -- split on
   `,`, trim, drop empties; reverse it on save by joining with `,`).
2. `defaultValue` -- a free-text input, shown for all types or constrained to
   the option types.

- Source: OLD code `CustomField.cs:78-80` (schema is the option store; comma
  separated) + NEW renderer
  `appointment-add-custom-fields.component.ts:51-57` (the consumer's parsing
  contract). The proxy `create`/`update` (`custom-fields.service.ts:14-20,56-62`)
  and DTOs (`models.ts:13-14,38-39`) already carry the fields, so no backend or
  proxy regeneration is needed.
- Tradeoffs: keeps the change inside the natural G-07-01 work; zero backend
  churn. The only decision is the input shape (raw comma textarea vs. structured
  repeater). A repeater is friendlier and avoids the user typing literal commas,
  but a textarea is the smallest faithful port of the comma string. Recommend a
  repeater that serializes to the comma string -- best UX while keeping the
  storage shape identical to OLD/NEW.
- Risk: LOW. No existing code changes; the renderer already defends against
  empty/malformed option strings. Validate option count vs. `FieldLength`? Not
  required -- `FieldLength` only applies to text types per the entity comment.
- Effort: S. Two controls + a conditional `@if (fieldType is option-type)`
  visibility rule + a serialize/parse pair, all inside the G-07-01 form.

(One option is sufficient -- this is a pure UI-completion of an
already-supported backend, dependent on G-07-01. There is no architecturally
distinct alternative worth listing; the only sub-choice is textarea vs.
repeater, captured in Option A's tradeoffs.)

## 8. Root-Cause Hypothesis

Root cause (confidence: HIGH): the gap is a missing admin UI, not a missing
capability. The NEW backend (entity, DTOs, AppService Create/Update, controller)
and the auto-generated Angular proxy already model and persist
`MultipleValues`/`DefaultValue`; the booking renderer already reads them. The
only thing absent is the admin Angular form to enter them -- which is the
unbuilt custom-fields admin screen tracked by G-07-01. G-07-05 exists purely to
ensure that, when G-07-01 is implemented, the new form does NOT repeat OLD's
omission (OLD's add form lacked these inputs). It is therefore a field-set
requirement riding on G-07-01, with effort S and near-zero regression risk.

## 9. Open Questions

- Input shape: structured option repeater vs. raw comma-separated textarea for
  `MultipleValues`? (Recommendation: repeater serializing to the comma string,
  to match the renderer's existing parse contract while improving UX.) Decide
  during G-07-01 design.
- Should the `MultipleValues` editor also show for Tickbox (15)? The booking
  renderer DOES support multi-option Tickbox via `MultipleValues`
  (`appointment-add-custom-fields.component.html:98-112`), so exposing it for
  Tickbox is parity-plus-consistent. OLD never configured any of these via UI,
  so there is no OLD precedent to violate. Confirm scope with Adrian.
- `DefaultValue` semantics for option types: must the default be one of the
  entered options (validated) or free text? OLD never enforced this (the field
  was never set via UI). Lowest-risk choice: free text, no cross-validation, to
  match the un-opinionated schema.
