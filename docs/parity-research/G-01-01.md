## 1. ID & Summary
OLD rejected a booking (and the standalone injury add/edit) when any injury's
Date Of Injury was in the future (DateOfInjury.Date > today) or predated the
patient's birth (DateOfBirth.Date > DateOfInjury.Date). NEW enforces neither:
the per-injury manager only checks claim/body-part length and the
(AppointmentId, ClaimNumber, DateOfInjury) duplicate, and the Angular injury
modal only marks the date `required`. A future-dated or pre-birth injury is
accepted and persisted, corrupting the legal intake record and the downstream
PDF report.

## 4. Expected vs Actual
- EXPECTED (OLD): On booking or standalone injury add/edit, a future Date Of
  Injury is rejected ("Injury date should be less then from current date" /
  `InjuryDateLessThanToday`), and an injury date earlier than the patient's DOB
  is rejected ("Injury date must be less than to date of birth"). The whole
  submit is blocked until corrected.
- ACTUAL (NEW): Both pass through. The modal only requires the date be present;
  the server only blocks an exact (claim#, date) duplicate. A 2099 injury date
  or a date before the patient was born is accepted and persisted.

## 7. Candidate Solutions

### Option A -- Port both checks into AppointmentInjuryDetailManager (domain service) [effort: S] [risk: low]
- Description: In `CreateAsync`/`UpdateAsync`, after the duplicate guard, throw
  `UserFriendlyException` when `dateOfInjury.Date > Clock.Now.Date`, and load the
  appointment's patient to throw when `patient.DateOfBirth.Date > dateOfInjury.Date`.
  Mirrors OLD's two server enforcement points exactly.
- Source: OLD code: `AppointmentDomain.cs:645-695` and `AppointmentInjuryDetailDomain.cs:153-168`.
- Tradeoffs: Domain layer is the correct home per CLAUDE.md ("business rules
  belong in domain services"). Requires injecting a patient/appointment
  repository into the manager (currently injury-only) to resolve DOB, exactly as
  OLD's `AppointmentInjuryDetailDomain` loaded the Patient via the appointment.
  Catches every path (booking POST cascade + standalone edit) since both funnel
  through the manager. Use ABP `IClock` for testable "today".

### Option B -- Add an FluentValidation validator for the injury Create/Update DTO + a client validator [effort: S] [risk: low]
- Description: Add `AppointmentInjuryDetailCreateDtoValidator` enforcing
  `DateOfInjury <= today` (the DOB check cannot live in a stateless DTO validator
  since DOB is not on the DTO). Pair with an Angular max-date validator on
  `injuryDateOfInjury`.
- Source: ABP validation via FluentValidation -- https://abp.io/docs/latest/framework/fundamentals/validation
  (accessed 2026-06-01; credibility: official ABP docs). OLD code:
  `appointment.domain.ts:1053-1055` for the client future-date message.
- Tradeoffs: Clean for the future-date half, but a DTO validator cannot do the
  DOB comparison (no patient context), so this MUST be combined with a domain
  check for DOB -- i.e. it does not fully replace Option A. Best treated as the
  client-side complement to Option A, not a standalone server fix.

### Option C (recommended composite) -- Option A for the server (both checks) + client today-max validator [effort: S] [risk: low]
- Description: Authoritative server enforcement in the manager (Option A) plus a
  defensive Angular validator that marks `injuryDateOfInjury` invalid when
  > today, matching OLD's client toast. Defer the client DOB comparison (DOB not
  in the modal) to the server message, matching OLD (which only enforced DOB
  server-side). Implement alongside G-01-02 (cumulative From/To) since both share
  the OLD client method `checkDateValidationForInjury`.
- Source: OLD code: `AppointmentDomain.cs:647-654`, `AppointmentInjuryDetailDomain.cs:160-168`,
  `appointment.domain.ts:1031-1060`.
- Tradeoffs: Matches OLD's exact client/server split (future-date both sides,
  DOB server-only). One coherent change; minimal surface; low regression risk.

## 8. Root-Cause Hypothesis
The injury date validations were simply never ported. The NEW `AppointmentInjuryDetail`
domain was scaffolded as a CRUD entity (ABP Suite-style manager with length +
duplicate guards) without re-implementing OLD's `CommonValidation` intake rules,
and because injuries are persisted via a SEPARATE post-booking POST (not nested in
`AppointmentCreateDto`), the booking-flow `CommonValidation` parity work also
missed them -- the booking AppService never sees injury rows to validate. The
Angular modal received only `Validators.required` and never the procedural
`checkDateValidationForInjury` logic. Confidence: high.
