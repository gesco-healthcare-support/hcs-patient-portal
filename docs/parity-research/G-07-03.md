## 1. ID & Summary
Bulk doctor-availability slot generation: when a requested date/time range
collides with existing slots, OLD rejects the ENTIRE batch (all-or-nothing) and
saves nothing; NEW inserts only the non-conflicting slots and returns a report
of inserted vs skipped/conflicted slots. This is a deliberate behavioral
deviation kept for better UX. Excluded -- no revert planned. Related to G-07-04
(the overlap-detection math), which is a separate row.

## 3. Affected Code (spot-checked)
- OLD: `PatientAppointment.Domain/DoctorManagementModule/DoctorsAvailabilityDomain.cs`
  `AddValidation` (:46-99). Confirmed: a nested loop over generated slots runs
  four conflict checks per slot -- contained-in (`FromTime > x && ToTime < x`,
  :59), exact-duplicate (`==`, :65), `TimeSlotValidation` same-location overlap
  (:71), and booked/reserved (:77). On the FIRST hit it `break`s and sets
  `isAlreadyExist`/`isBookedByUser`, which appends `TimeSlotExists`/
  `TimeSlotBooked` to `ValidationMessages`. A non-empty message set makes the
  controller return BadRequest -> nothing is saved. All-or-nothing confirmed.
- NEW: `src/HealthcareSupport.CaseEvaluation.Application/DoctorAvailabilities/DoctorAvailabilitiesAppService.cs`
  `CreateRangeAsync` (:314-357). Confirmed: re-runs `GeneratePreviewAsync`
  server-side, splits flat slots into `slotsToInsert` (`!IsConflict`) and
  `conflictedSlots`, inserts only the clean ones inside one transactional UoW,
  and returns `DoctorAvailabilityCreateRangeResultDto { InsertedCount,
  SkippedConflictCount, ConflictedSlots }`. Partial-insert + report confirmed.

## 4. Expected vs Actual (brief)
- OLD (expected by strict parity): a range overlapping one occupied day fails
  entirely; admin re-picks a clean range.
- NEW (actual): same range succeeds for the clean days; collisions are skipped
  and reported. Different observable outcome -- intentional.

## 5. Repro Status
n/a-excluded. No defect to reproduce; both behaviors are statically confirmed

## 6. Why Excluded + Revisit-If
Excluded because the deviation is a UX improvement, not a defect: NEW's
partial-insert + inserted/skipped report is strictly more useful to an admin
than OLD's "fix the whole range and resubmit." CSV note: "Keep NEW
partial-insert + inserted/skipped report (better UX); do not revert to OLD
all-or-nothing."

Revisit if: (a) Adrian confirms strict OLD parity is required for slot
generation (open-questions item 2 still flags this for confirmation); or (b) a
downstream consumer/report depends on the all-or-nothing atomicity (none found);
or (c) the partial-insert UoW semantics cause a data-integrity issue (e.g.
partial commit on error -- not observed; insert loop is inside a single
transactional UoW).
