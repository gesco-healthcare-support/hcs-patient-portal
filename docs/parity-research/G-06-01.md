> **DECISION (finalized 2026-06-01) [Gate 3]:** Duplicate of G-01-04; same resolution. Effort L -> S. Harden the AME rule with a per-type allowed-roles flag; DEFER the configurable matrix + admin UI. See G-01-04 and SUMMARY.md Gate 3.

## 1. ID & Summary

OLD enforced "which roles may book which appointment types" through a data-driven
join table, `spm.RoleAppointmentTypes` (RoleId x AppointmentTypeId), checked at
booking time. NEW never ported that table. Instead NEW collapses the entire matrix
to a single hardcoded rule: external callers booking an appointment type whose name
contains the substring "AME" must hold an attorney role, otherwise booking is
rejected with `AppointmentAmeRequiresAttorneyRole`. Internal-user callers bypass.

For the four appointment types that ever shipped (PQME, AME, PQME-REVAL, AME-REVAL)
the NEW outcome is behaviorally identical to OLD's seeded matrix. The gap is that
the rule is (a) not configurable without a code change and (b) name-fragile -- a
future appointment type whose name happens to contain/omit "AME" would be mis-gated.

This is flagged in the CSV as a DUPLICATE of **G-01-04** (the same gap viewed from
the booking-authorization angle in `01-booking.md`); G-06-01 is the auth/RBAC angle.
Both carry disposition `include` with the same planned fix: build an
admin-configurable role x appointment-type table + admin UI so future rules need no
code change. The work is owned by G-01-04; G-06-01 should not be implemented
separately.

### Blast radius

A faithful fix (data-driven role x type table + admin UI, per G-01-04) touches:

- Domain.Shared: localization keys for the new admin screen + any new error message.
- Domain: new `RoleAppointmentType` (or `AppointmentTypeRole`) entity + domain
  service/repository, or alternatively a `RequiresAttorney` bool on `AppointmentType`.
- Application.Contracts: DTOs + permissions for the new admin CRUD.
- Application: rewire `CreateAppointmentInternalAsync:672-686` from the substring rule
  to a table lookup; mapper entries.
- EntityFrameworkCore: DbContext config + migration + data seed contributor for the
  canonical 4-row matrix.
- HttpApi: manual controller for the admin CRUD.
- Angular: admin role x type management screen + booking-time enforcement/type filtering.
- Tests: replace `BookingFlowRolesUnitTests` AME cases with table-driven cases.

Roughly 6-7 layers + frontend. Regression risk MEDIUM: the booking gate is a
security-relevant chokepoint; an incorrect migration/seed could either over-block
(legit attorney/internal bookings rejected) or under-block (non-attorneys booking AME).
The seed must reproduce the canonical matrix exactly to preserve current behavior.
A `RequiresAttorney` flag is a much smaller blast radius than a full RoleId x TypeId
join (no new join entity, no per-role admin grid), at the cost of not reproducing
OLD's arbitrary per-role configurability.

## 4. Expected vs Actual

- Expected (parity target per G-01-04 disposition): an admin-configurable mapping of
  role -> bookable appointment type, enforced at booking, seeded to the canonical
  matrix, so adding/retyping appointment types needs no code change.
- Actual (NEW today): one hardcoded rule -- external + type-name-contains-"AME" +
  not-attorney => reject. Behaviorally identical for the 4 shipped types; diverges only
  for new/oddly-named types. No admin configurability.

## 7. Candidate Solutions

Note: implementation is owned by **G-01-04** (this row is its duplicate). Options
below are the design choices that G-01-04 must pick between; recorded here so the
auth/RBAC angle is captured.

### Option A -- Data-driven RoleAppointmentType join + admin UI (faithful port) [effort: L] [risk: med]

One-line: port OLD's `RoleAppointmentTypes` join (RoleId x AppointmentTypeId) as a NEW
entity + admin CRUD screen, seed the canonical 4-row matrix, replace the substring rule
with a table lookup at `AppointmentsAppService.cs:679-686`.
Source: OLD code `RoleAppointmentType.cs` + `AppointmentDomain.cs:637-643,688-691`;
schema `_local/generated-schema.sql:231-238`.
Tradeoffs: highest fidelity to OLD's "admins can configure any role->type combo"; the
only option that fully closes the configurability gap the CSV asks for. Heaviest blast
radius (entity + migration + seed + admin UI + booking rewrite). Must map ABP role
names/ids to the matrix (OLD used integer RoleId; NEW uses ABP role strings -- the seed
keys on role name).
Risk: med -- security chokepoint; seed must reproduce the canonical matrix exactly.

### Option B -- `RequiresAttorney` boolean flag on AppointmentType [effort: M] [risk: low]

One-line: add a single `RequiresAttorney` bool to the NEW `AppointmentType` entity,
default-seed it true for AME/AME-REVAL, replace the name substring with the flag.
Source: parity doc G-06-01 line 90-91 ("add an explicit RequiresAttorney flag on
AppointmentType"); the actual matrix only ever distinguished attorney-only vs all.
Tradeoffs: removes the name-fragility (an "AME"-named type no longer auto-gated; an
attorney-only type without "AME" can be flagged), and is admin-editable per type via the
existing AppointmentType admin screen. Does NOT reproduce OLD's full per-role grid (it is
attorney-yes/no, not arbitrary role->type). Matches the only distinction the OLD matrix
ever actually expressed, so behavioral parity holds while staying simpler.
Risk: low -- additive column + flag check; no new entity or admin screen.

### Option C -- Keep hardcoded rule, defer to Phase 2 [effort: S] [risk: low]

One-line: leave the substring rule; add a regression test pinning the 4-type outcomes;
revisit when appointment types become tenant-configurable.
Source: parity doc G-06-01 line 89 ("Acceptable for Phase 1").
Tradeoffs: zero behavioral change today, but the CSV disposition is `include` (build the
configurable table), so this contradicts the recorded decision unless Adrian re-confirms
the duplicate-with-G-01-04 status means G-01-04 alone satisfies it. Lowest effort.
Risk: low functionally; risk is process (CSV says include).

Recommendation for the eventual G-01-04 implementer: Option B is the best
effort/fidelity trade for Phase 1 (closes the name-fragility, admin-editable, low blast
radius, matches every distinction the OLD matrix actually made). Option A only earns its
cost once a tenant genuinely needs arbitrary per-role->type rules beyond attorney-gating
(Phase 2). Decision belongs to Adrian via G-01-04.

## 8. Root-Cause Hypothesis

high confidence. During the port the OLD `RoleAppointmentTypes` join was deliberately
not carried over (the seed never shipped configurable rows -- the only matrix that ever
existed was the canonical attorney-gating one). To preserve behavior cheaply, OBS-23
(2026-05-21) introduced `BookingFlowRoles.IsAmeAppointmentType` + `IsAttorneyCaller` as a
name-substring stand-in for the one rule that table effectively encoded. The code and
error-code comments explicitly state this ("RoleAppointmentType join ... was not
ported"). Root cause is an intentional simplification, correct for Phase 1, that trades
admin-configurability and name-robustness for speed -- exactly what G-01-04 is slated to
restore.
