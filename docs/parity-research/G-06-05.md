## 1. ID & Summary

NEW carries FIVE external-user role values where OLD has exactly FOUR. OLD's
single Adjuster role (`Roles.cs:15`, value 5) was renamed to "Claim Examiner"
during the port, but the NEW enum kept BOTH names: `ClaimExaminer = 2` (added
in an early session) AND `Adjuster = 5` (re-added in Phase 8 "per OLD parity"
on 2026-05-03). `ExternalSignupAppService.ToRoleName` maps each to a DISTINCT
role string ("Adjuster" vs "Claim Examiner"), but the role SEEDER only ever
creates "Claim Examiner". The result is a naming/identity contradiction: a
registration or invite posting `UserType=Adjuster` would call
`EnsureRoleAsync("Adjuster")` and auto-create an unseeded, permission-less
"Adjuster" role -- a dead-end account that exists in OLD as the same role the
NEW seeder calls "Claim Examiner".

This is the NEW-side cleanup the CSV note describes: collapse/remove the stray
`Adjuster = 5` so only `ClaimExaminer = 2` remains. There is no OLD behavior to
port here -- OLD is the ground truth that has ONE role; NEW must shed the extra
value to match. The gap is wholly internal to NEW.

## 4. Expected vs Actual

- EXPECTED (OLD parity): exactly four external role values; the value-5
  claim-examiner party maps to ONE role name. OLD's role is literally "Adjuster"
  (Roles.cs) but the NEW codebase has already, deliberately, and consistently
  renamed it to "Claim Examiner" everywhere that matters (seeder, all
  classifiers, demo seed, locked memory). Parity here means ONE external role for
  the claim-examiner concept, not the literal OLD string.
- ACTUAL: five values; `Adjuster=5` is a vestigial second name for the same
  concept. It is wired into `ToRoleName` (producing an unseeded "Adjuster"
  string), exposed in the Angular enum options, and pinned by two unit tests, but
  recognized by ZERO downstream consumers -- a dead-end role.

## 7. Candidate Solutions

### Option A -- Remove `Adjuster = 5` from the enum entirely [effort: S] [risk: low]

One-line desc: Delete `Adjuster = 5` from `ExternalUserType`, delete the
`ExternalUserType.Adjuster => "Adjuster"` arm from `ToRoleName`, regenerate the
Angular proxy, and update/remove the two locking test assertions
(`ExternalSignupValidatorUnitTests.cs:134,243,256`).
Source: OLD code (`Roles.cs:15` = one role at value 5) + the NEW seeder's own
reconciliation comment (`ExternalUserRoleDataSeedContributor.cs:39-49`). This is
the literal CSV directive ("collapse/remove stray Adjuster=5 enum value so only
ClaimExaminer=2 exists").
Tradeoffs: Cleanest end state -- one value, one name, no dead-end path. BUT if
any persisted invitation/user row or wire payload carries `UserType=5`,
deserialization to the enum still works (it is just an undefined value), but
`ToRoleName` would then hit the default `_ => throw "Invalid user type."` arm,
turning a silent dead-end into a hard failure. For Phase-1 single-tenant with no
real Adjuster accounts seeded, this is acceptable and arguably better (fail loud).
Risk: low. No production branch depends on value 5 or string "Adjuster".

### Option B -- Keep value 5, remap `ToRoleName(Adjuster) => "Claim Examiner"` [effort: S] [risk: low]

One-line desc: Leave `Adjuster = 5` in the enum but make both Adjuster AND
ClaimExaminer map to the single seeded "Claim Examiner" role string; update the
test assertion at line 256 to expect "Claim Examiner".
Source: the parity doc's own alternative ("...or make `ToRoleName(Adjuster)`
return 'Claim Examiner'", `06-auth-rbac.md:193-194`).
Tradeoffs: Backward-compatible for any in-flight `UserType=5` payload (no
deserialization/throw concern), and no proxy/wire shape change. BUT it leaves two
enum values for one concept -- the identity contradiction persists at the type
level; the Angular dropdown still offers two synonymous options to users. Treats
the symptom, not the cause.
Risk: low. Does not fully satisfy "exactly one external role value" intent.

### Option C -- Collapse to value 2, drop value 5, keep the name "Claim Examiner" [effort: S] [risk: low]

One-line desc: Same as Option A but additionally confirm `ClaimExaminer = 2`
remains the single canonical value (no renumbering), matching the CSV note
verbatim. Effectively Option A with an explicit assertion that the surviving
value is 2.
Source: CSV note ("so only ClaimExaminer=2 exists").
Tradeoffs: Identical to Option A; called out separately only because the CSV
specifies value 2 as the survivor. The enum XML doc warns "do NOT renumber"
existing values 1-4, so keeping ClaimExaminer at 2 (its original slot) and merely
dropping the appended 5 honors that constraint.
Risk: low.

Recommendation for the eventual fix: Option A/C (they are the same net change:
drop value 5). It is the only option that satisfies the intent ("exactly one
external role value"). Option B is a fallback if wire/DB compatibility with an
existing `UserType=5` value is later found to matter.

## 8. Root-Cause Hypothesis

Mid-port naming churn left two synonyms for one role. Sequence (high confidence,
corroborated by the enum's own XML doc at lines 16-25 and the seeder comment at
lines 39-49):

1. An early session named the value-5 claim-examiner role "Claim Examiner" at
   enum position 2 -- likely borrowing OLD's `UserTypesForEmail.ClaimExaminer=5`
   email label rather than OLD's `Roles.Adjuster=5` role name, and aligning with
   the `AppointmentClaimExaminer` entity. The seeder, all role classifiers, the
   Angular role list, and the demo seed all standardized on "Claim Examiner".
2. Phase 8 (2026-05-03) re-added `Adjuster = 5` "per OLD parity" because OLD's
   `Roles.cs` literally says Adjuster -- a faithful-but-redundant port of the OLD
   string, appended at the end to "preserve wire compatibility."
3. `ToRoleName` was given arms for BOTH, producing two distinct strings, while
   the seeder was never updated to create an "Adjuster" role (because the team
   had already decided Adjuster == Claim Examiner). The contradiction is the
   collision of step 1's rename with step 2's literal re-port.

The CSV's "Intent deviation" class is exact: the data MODEL diverged from OLD's
intent (one role) even though the live BEHAVIOR matches OLD (everything uses the
single "Claim Examiner" role). It is a latent defect, not an active one.
