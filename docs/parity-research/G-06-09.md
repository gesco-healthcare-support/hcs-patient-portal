## 1. ID & Summary
G-06-09 records the anonymous/post-login root-redirect behavior. Hitting the bare site
address (`/`) routes: anonymous -> login; authenticated external user -> `/home`;
authenticated internal user -> `/dashboard`. OLD had no public landing page; everything
required login. The parity doc lists this only to document that the behavior was verified,
NOT as a real gap (it is the mirror of equivalent E-06-10). Disposition: exclude.

## 2. Docs Reviewed
- `docs/parity-v2/06-auth-rbac.md` -- section `### G-06-09` (lines 268-285) and the
  paired equivalent row `E-06-10` (line 300).
- CSV note: "VERIFIED implemented: post-login-redirect.guard.ts -> anonymous hitting /
  calls navigateToLogin (AuthServer); external->home, internal->dashboard. No action."
- Locked memory `project_root-redirect-anonymous.md` (referenced by the doc).

## 3. Affected Code (spot-checked)
- NEW guard: `angular/src/app/shared/auth/post-login-redirect.guard.ts:30-61`. Three
  outcomes: anonymous -> `auth.navigateToLogin()` + `return false` (47-50); external-only
  roles -> `return true` (HomeComponent loads at `/`) (53-55); internal/mixed ->
  `router.parseUrl('/dashboard')` (60).
- NEW wiring: `angular/src/app/app.routes.ts:29` -- `canMatch: [postLoginRedirectGuard]`
  on the `/` route (loads HomeComponent). Confirms the guard is live on bare `/`.
- OLD external/internal split: `patientappointment-portal/src/app/components/login/login/
  login.component.ts:44-69` (ngOnInit) -- if authed, `UserTypeId == ExternalUser` ->
  `navigate(['home'])` (or callBackUrl), else -> `navigate(['dashboard'])`.
- OLD discriminator source: `PatientAppointment.Domain/Core/UserAuthenticationDomain.cs:99`
  (`UserTypeId = user.UserTypeId`) -- the field login.component switches on.

## 4. Expected vs Actual (brief)
Expected (OLD): anon -> login; external -> home; internal -> dashboard. Actual (NEW):
identical. The guard runs as `CanMatchFn` before the lazy HomeComponent chunk loads, which
also removes a "flash of empty home shell" the OLD app's client-side redirect could show --
strictly better, outcome-equivalent. No behavioral divergence found.

## 5. Repro Status
n/a-excluded. There is no defect to reproduce; static confirmation shows NEW == OLD.
A runtime repro would require an authenticated session per role (state-dependent login),
which is out of scope for read-only research and unnecessary given the static match.

## 6. Why Excluded + Revisit-If
Excluded because it is not a gap: the parity doc filed it informationally and pairs it with
equivalent E-06-10; the CSV dispositioned it exclude with "No action." NEW already
replicates OLD exactly and is wired to the `/` route.
Revisit if: (a) the `/` route's `canMatch: [postLoginRedirectGuard]` is removed or the guard
logic changes; (b) a public landing page is introduced (would break the "everything requires
login" OLD invariant); (c) the external/internal role partition (`hasOnlyExternalRoles`)
diverges from OLD's `UserTypeId` split; (d) AuthServer login redirect path changes such that
`navigateToLogin()` no longer reaches the Razor login.

## 7. Candidate Solution
None -- excluded. Already implemented and verified.

## 8. Decision Rationale + confidence
Exclude is correct: HIGH confidence the NEW implementation matches OLD (both citations on
both sides resolve and agree; guard is wired on `/`). root_cause_confidence is n/a because
there is no root cause -- nothing is broken. The only residual is whether OLD users ever
relied on `callBackUrl` deep-linking after login (OLD `login.component.ts:53-55`), which the
SPA guard does not reproduce, but that is post-login deep-linking, separate from the
anonymous/role-split outcome this row covers, and AuthServer OAuth `redirect_uri` handles
return-to-app at the framework level.
