## 1. ID & Summary

When an external user (patient / attorney / claim examiner) submits a
free-text question through the OLD portal's "Help / Need Question?"
modal, OLD persists a `UserQuery` row AND emails the question to staff
(routed to the appointment's primary-responsible internal user when a
confirmation number is present, otherwise broadcast to all IT-Admins).

The NEW app has none of this. The two template codes (`UserQuery`,
`SubmitQuery`) exist only as string constants and seeded
`NotificationTemplate` rows -- and those rows carry STUB subject + body,
not OLD-verbatim content. There is no `UserQuery` entity, AppService,
domain service, controller, DTO, handler, or Angular UI anywhere. The
email is the last leg of an entirely-absent feature.

G-04-02 is the **email leg** of the Submit-Query trio. The form + entity
+ navbar Help button is G-09-02; the responsible-user-vs-IT-Admin routing
rule is G-09-03; the storage-side `UserQueries` table is G-10-02. All
four are dispositioned `include`. This item cannot be built in isolation:
there is no submit trigger to fire the email from until G-09-02/G-10-02
land.

## 4. Expected vs Actual

- **Expected (OLD parity):** Submitting a query emails staff. With a
  confirmation number for an Approved appointment, the email goes to
  that appointment's primary-responsible internal user; otherwise it
  broadcasts to all IT-Admins. Subject:
  `"Patient Appointment Portal - (Patient: <name> - Claim: <n> - ADJ:
  <wcab>) - User query"` (NEW brand-prefix shortened to "Appointment
  Portal" per G-04-09). Body: the question text rendered into the
  `UserQuery` template (`##UserQueryMessage##`).
- **Actual (NEW):** Nothing. No form, no entity, no endpoint, no email.
  The `UserQuery`/`SubmitQuery` template rows are seeded with stub
  subject + stub body and are never dispatched.

## 7. Candidate Solutions

### Option A -- Port as a local-event handler off the new Submit-Query AppService [effort: M] [risk: low]

- **One-line:** Build the `UserQuery` feature (entity + AppService + UI,
  = G-09-02/G-10-02), publish a `UserQuerySubmittedEto` on create, and
  add a `UserQuerySubmittedEmailHandler` that resolves the recipient
  (responsible-user vs IT-Admin, = G-09-03) and calls
  `INotificationDispatcher.DispatchAsync("UserQuery", recipients,
  variables, ...)`.
- **Source:** OLD code `UserQueryDomain.cs:53-107` (routing + subject) +
  `User-Query.html` (body) + NEW pattern from
  `BookingSubmissionEmailHandler.cs` and `NotificationDispatcher.cs`.
- **Tradeoffs:** Matches the established NEW handler architecture exactly
  (20 sibling handlers); the email leg reuses the dispatcher, subject
  builder, and Hangfire pipeline with no new infra. Requires the parent
  feature to exist first.
- **Risk:** Low for the email leg; the new entity/migration/UI is the
  bulk of the work and lives in the dependency items. Must guard the
  responsible-user email lookup (OLD bug B-09-02 -- fall back to the
  IT-Admin pool when the responsible user has no email).
- **Effort:** M for the trio overall; the G-04-02-specific handler is S.

### Option B -- Seed OLD-verbatim template content now, defer dispatch [effort: S] [risk: low]

- **One-line:** Add an `EmailSubjects.ByCode["UserQuery"]` entry
  (`"... - User query"` with `##EmailSubjectIdentity##`) and an embedded
  `EmailBodies/UserQuery.html` resource (port of `User-Query.html`,
  `##UserQueryMessage##`), so the seeded row carries parity-correct
  content the moment the feature is wired -- but do not build dispatch
  yet.
- **Source:** OLD `User-Query.html`; NEW
  `NotificationTemplateDataSeedContributor.cs:88-117` (resource-backed
  overwrite-on-seed behavior).
- **Tradeoffs:** Tiny, isolated, removes the stub now and de-risks the
  later handler work; but delivers no user-visible behavior on its own
  (the email still never fires without Option A's trigger).
- **Risk:** Low. Idempotent re-seed; only touches the two stub rows.
- **Effort:** S. Best used as a sub-step of Option A, not a substitute.

Recommended: Option A, sequenced with G-09-02/G-10-02, folding Option B
in as the template-content sub-step.

## 8. Root-Cause Hypothesis

The Submit-Query / Contact-Us feature was simply not yet ported. During
the Phase 1 notification work the team unified OLD's 59 template codes
into one seeded table so that handlers could resolve templates by code,
but deliberately deferred ~26 codes (including `UserQuery`/`SubmitQuery`)
to "post-parity feature phases" (`NotificationTemplateConsts.cs:56-61`).
The email is the trailing leg of a feature whose form, entity, endpoint,
and routing all also remain unbuilt. There is no defect to fix -- it is
absent-by-design pending the feature build. Confidence: **high** (grep
exhaustively confirms absence at every layer; the deferral is documented
in code comments and the audit doc).

   `CreatedById = UserClaim.UserId` and the OLD audit describes the Help
   link as gated to ExternalUser (authenticated). Was there ever a truly
   anonymous (unauthenticated) submit path? If so, the NEW endpoint needs
   an `[AllowAnonymous]` variant; if not, the feature is
   authenticated-only and simpler. (The source-doc G-04-02 text mentions
   "anonymous (:102)" but that branch is the general/no-conf# branch, not
   an unauthenticated one -- it still runs under an authenticated
   `UserClaim`.)
3. **"PrimaryResponsibleUser" mapping:** G-09-03 assumes OLD's
   `vAppointment.PrimaryResponsibleUserId` maps cleanly to the NEW
   appointment's owning internal user. Confirm the NEW appointment model
   carries an equivalent owner field before building the routing.
4. **Template-content port:** should the seeded `UserQuery` row's body
   reproduce `User-Query.html` verbatim (Option B), or adopt the newer
   NEW email styling used by the recently-built templates? The OLD body
   is minimal ("User has sent query: ##UserQueryMessage##").
