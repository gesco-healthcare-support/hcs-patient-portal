## 1. ID & Summary

OLD's appointment change log was a hand-rolled, reflection-based diff engine that
wrote one `spm.AppointmentChangeLogs` row per changed FIELD across the appointment
root AND its child rows -- injury details, injury body parts, claim examiners,
primary insurance. Each row carried `TableName` (a human-readable entity name like
"Appointment Injury Body Part Detail"), `FieldName`, `OldValue`, `NewValue`,
`ChangedById`, `ChangedDate`, `IsInternalUserUpdate`, `IsMailSent`.

NEW replaced the custom log with ABP's built-in audit-logging (`[Audited]` on the
entities; the per-appointment viewer reads ABP entity-changes). The NEW viewer
(`appointment-change-logs.component.ts:39`) hardcodes a SINGLE entity FQN filter --
`HealthcareSupport.CaseEvaluation.Appointments.Appointment` -- and calls
`AuditLogsService.getEntityChangesWithUsername({ entityId, entityTypeFullName })`.

Verified finding (corrects parity-doc uncertainty): the four child entities ARE
`[Audited]`, so ABP DOES persist their property changes. The gap is purely in
SURFACING them. ABP stores each entity's changes as a separate `EntityChange` row
keyed by that row's OWN `EntityId` + `EntityTypeFullName`; the appointment's
`entityId` never matches a child row, so the single-FQN viewer can never show them.
The gap is real and the data is recoverable -- this is an aggregation + redaction
problem, not a data-loss problem.

## 4. Expected vs Actual

- Expected (OLD parity): the change-log view for an appointment shows field-level
  edits to the appointment AND its injury details, body parts, claim examiners, and
  primary insurance, each labelled by which entity/table the field belongs to, with
  old -> new values, who, and when. New body parts appear as an add (empty old value).
- Actual (NEW): the viewer shows ONLY the Appointment root entity's own scalar
  property changes. Edits to claim number, date of injury, body parts, claim-examiner
  name, etc. are recorded by ABP but invisible in this screen because it filters on
  the single Appointment FQN + Appointment entity id.

## 7. Candidate Solutions

### Option A -- Custom backend aggregation AppService + redaction, frontend reads it [effort: L] [risk: med]
- Desc: Add `AppointmentChangeLogAppService.GetForAppointmentAsync(Guid appointmentId)`
  that (1) loads the appointment's child Guids (injury details, then their body parts /
  claim examiners / primary insurance), (2) unions ABP `EntityChange` rows for the
  Appointment + each child Guid/FQN via `IAuditLogRepository.GetEntityChangeListAsync`
  (`includeDetails:true`), (3) maps each to a flat DTO with a human-readable
  `entityLabel` (mirror OLD `TableName`), and (4) REDACTS PHI property values
  server-side (claim number, date of injury, WCAB ADJ, body part description,
  examiner name -- redact-or-mask per a fixed allowlist). Frontend calls the one
  endpoint. Source: OLD code `AppointmentChangeLogDomain.cs` (label/diff intent) +
  ABP `IAuditLogRepository.GetEntityChangeListAsync` (official docs / source).
- Tradeoffs: cleanest parity, single permission gate, server-side redaction (HIPAA-
  correct -- never ship raw PHI to the client), reusable by G-02-01's global list.
  Cost: a new AppService + contracts + controller + proxy regen, and the child-Guid
  enumeration must follow the reschedule-clone chain to be complete.
- Risk: med -- correctness of child enumeration + redaction allowlist completeness.

### Option B -- Frontend fan-out across known child FQNs [effort: M] [risk: high]
- Desc: Keep using the stock `getEntityChanges`/`getEntityChangesWithUsername` proxy;
  the component fetches the appointment's children (existing `GetByAppointmentIdAsync`
  etc.), then issues one audit query per child Guid/FQN and merges client-side.
- Tradeoffs: no new backend. But it ships RAW PHI to the browser (redaction would be
  client-side = HIPAA-weak, violates "redact at the boundary"), N+1 calls, and the
  audit query is per-entity-id so you still must enumerate every child Guid first.
- Risk: high -- PHI leak to client, chatty, and the redaction guardrail in the CSV
  note effectively rules this out. Listed for completeness; not recommended.

### Option C -- AuditLogContributor that merges child changes into the parent's record [effort: L] [risk: med]
- Desc: Register a custom `AuditLogContributor.PostContribute` that, at write time,
  re-parents child `EntityChange` property changes onto the Appointment's
  `EntityChange` (ABP-documented pattern, e.g. the `IdentityUserRole` example), so the
  EXISTING single-FQN viewer surfaces them. Source: ABP docs "Customizing Audit
  Logging" / UserRoleAuditLogContributor pattern (official, high).
- Tradeoffs: viewer stays unchanged; child edits show under the appointment. But it
  permanently rewrites how audit rows are stored (loses the per-child entity identity,
  harder to label by table, affects ALL audit consumers including the admin Audit
  Logs module), and still needs PHI redaction at the viewer boundary. Higher blast
  radius on the audit subsystem than Option A.
- Risk: med-high -- changes global audit-write behavior; not isolated to this feature.

Recommended: Option A (server-side aggregation + redaction; pairs with G-02-01).

## 8. Root-Cause Hypothesis

The NEW app swapped OLD's bespoke per-field `AppointmentChangeLogs` table for ABP's
generic audit-logging. ABP records changes per audited entity instance, keyed by that
instance's own `EntityId` + `EntityTypeFullName`, with no built-in "all changes in
this aggregate tree" query. The per-appointment viewer was implemented as the minimal
W2-4 cap -- a single-FQN, single-entity-id lookup on the Appointment root only -- so
it structurally cannot reach child rows even though those rows ARE audited. Root cause
is therefore a VIEWER/AGGREGATION limitation (and a missing PHI-redaction layer), not
missing audit data. Confidence: high (child `[Audited]` confirmed in source; ABP's
per-entity EntityChange model confirmed in official docs + ABP source; viewer's single
FQN confirmed at component line 39/52).

  child values. (CSV note explicitly flags this for deeper HIPAA planning.)

  the PRE-reschedule appointment row + its (now-orphaned) child Guids, or only the
  current row? OLD kept the same `RequestConfirmationNumber` across reschedules;
  the audit-by-Guid model splits them. Decide whether to follow `OriginalAppointmentId`.
- OLD scope precision: OLD's reflection diffed injury-detail + body-part VIEWS; whether
  claim-examiner / primary-insurance OWN scalar edits were logged depends on the
  `vAppointmentInjuryDetail` view columns (not read here). Confirm desired NEW scope
  covers all four child entities regardless of how OLD's view was shaped.
- Pairing with G-02-01: a shared aggregation query should back both the per-appointment
  viewer (this item) and the global filterable list (G-02-01). Confirm sequencing.
- `IsInternalUserUpdate` parity: OLD tagged each row internal-vs-external. ABP records
  the acting user; do internal users need a derived "internal update" flag in the NEW
  view, or is the username sufficient?
