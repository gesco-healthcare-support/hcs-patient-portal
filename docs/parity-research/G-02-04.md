> **DECISION (finalized 2026-06-01) [Gate 2]:** Confirmed -- treat as the regression it is. On reschedule-reject, RELEASE the slot this change request reserved, guarded/idempotent (only the reserved DoctorAvailabilityId, only if still Reserved), mirroring OLD AppointmentChangeRequestDomain.cs:600. "Reserved" is a transient hold, NOT a manual-close state. (The frontmatter dependency "G-01-xx-slot-rework" is not a real gap ID -- it refers to the 2026-05-15 NEW slot rework whose overloaded Reserved semantics are the context.) See SUMMARY.md Gate 2.

## 1. ID & Summary

When a supervisor rejects a reschedule request, the OLD app freed the slot the
patient had tentatively grabbed (new slot -> `Available`) and parked the source
slot back to `Booked`. The NEW app leaves the user-picked new slot stuck in
`Reserved` forever. Because the NEW booking gate treats `Reserved` as
"manually closed by doctor's-admin -> always blocks", that orphaned hold
permanently removes the slot from the bookable pool.

This is an internally inconsistent state introduced by the 2026-05-15 slot
rework: the reschedule-submit path STILL programmatically sets the new slot to
`Reserved` (`AppointmentChangeRequestManager.cs:256`), but NEITHER the approve
nor the reject path clears it. Submit reserves; nothing un-reserves. Confirmed
a real regression (bug), not intended behavior.

Severity: a single rejected reschedule silently and permanently burns one
schedulable slot per occurrence. Over time this shrinks doctor capacity with no
UI signal and no automatic recovery (only a manual doctor's-admin re-open).

### Blast radius

- A minimal fix touches 1 file / 1 layer: re-introduce a slot-release step in
  `RejectRescheduleAsync` (Application layer). It needs the
  `_doctorAvailabilityRepository` (already injected, lines 59/79/95).
- A correct fix likely touches 1-2 files: the reject path PLUS a decision about
  whether submit should set `Reserved` at all (manager:256), because the root
  conflict is that `Reserved` is now overloaded (manual-close vs transient-hold).
  If submit is changed to stop reserving, the approve path comment and the
  booking-gate semantics stay consistent without any reject-side release.
- Regression risk: MEDIUM. The danger is the overloaded `Reserved` value -- a
  naive "on reject, set new slot Available" can stomp a slot a doctor's-admin
  genuinely closed (e.g. if the user-picked slot was later manually closed, or
  the override-slot path moved the booking elsewhere). The OLD code did not have
  this ambiguity (Reserved meant only "held"). A safe fix must release only the
  slot THIS change request reserved, and only if it is still `Reserved`
  (idempotent guard). Concurrency: two pending reschedules cannot target the
  same slot (submit gate `IsSlotAvailable` rejects a non-`Available` slot at
  manager:233-238), so a released slot won't collide with another live hold.

## 4. Expected vs Actual

- Expected (OLD parity): rejecting a reschedule frees the requested slot ->
  `Available` so other patients can book it; source slot stays `Booked`;
  appointment returns to `Approved`. (OLD lines 595/600/606.)
- Actual (NEW): appointment returns to `Approved` and the request is marked
  `Rejected` (correct), BUT the requested slot stays `Reserved` and the booking
  gate then refuses it permanently (`AppointmentBookingSlotClosed`). Net:
  capacity silently shrinks by one slot per rejected reschedule, with no UI
  cue and no auto-recovery.

## 5. Repro Status

not-reproduced-static-confirmed. A true runtime repro requires submitting a
reschedule (mutating slot + appointment + change-request rows) then rejecting it
-- prohibited by the read-only guardrail. The gap is fully confirmed statically:

1. submit sets new slot `Reserved` (`AppointmentChangeRequestManager.cs:256`,
   unconditional write).
2. `RejectRescheduleAsync` (`...Approval.cs:341-403`) contains NO
   `DoctorAvailability` write and an explicit removal comment (381-385).
3. the booking gate (`AppointmentsAppService.cs:889`) blocks any `Reserved`
   slot.
4. `SlotCascadeHandler` is a no-op, so no event-driven path clears it.

The chain 1->2->3 deterministically produces an un-bookable orphaned slot.

## 7. Candidate Solutions

### Option A -- Release the reserved slot in RejectRescheduleAsync (idempotent, guarded) [effort: M] [risk: med]
- One-line desc: in `RejectRescheduleAsync`, load
  `changeRequest.NewDoctorAvailabilityId`; if its `BookingStatusId == Reserved`,
  set it back to `Available` and persist -- mirroring OLD line 600 but guarded
  so a manually-closed slot is not stomped.
- Source: OLD code `AppointmentChangeRequestDomain.cs:600`. Repository already
  injected (`_doctorAvailabilityRepository`, ctor line 79). The reject method
  already loads the source appointment; add a slot fetch by
  `changeRequest.NewDoctorAvailabilityId`.
- Tradeoffs: smallest faithful port of OLD behavior. The guard
  (`only if currently Reserved AND this is the slot this CR reserved`) resolves
  the overloaded-`Reserved` ambiguity introduced by the rework. Does not address
  the APPROVE path leaving the new slot `Reserved` (separate latent bug) -- but
  that is out of this gap's scope.
- Risk: med. Must confirm with Adrian whether `Reserved` is allowed to mean both
  "transient hold" and "manual close". If a doctor's-admin could have manually
  closed exactly the user-picked slot mid-pending, the guard cannot distinguish
  intent. Mitigation: only release if `NewDoctorAvailabilityId` is non-null and
  the request type is `Reschedule`.

### Option B -- Stop reserving on submit; rely on capacity-aware model end-to-end [effort: M] [risk: med]
- One-line desc: remove the `Reserved` write at
  `AppointmentChangeRequestManager.cs:256` so submit never tentatively closes
  the slot; let the capacity-aware booking gate + active-appointment-count be
  the single source of truth (matching what `CreateAsync` already does per
  Business Rule 4).
- Source: NEW `Appointments/CLAUDE.md` Business Rule 4 (capacity model) +
  the parity doc Open-Question note (line 144) which explicitly suggests "if
  Reserved now means manually-closed, then submit should NOT be setting it
  either."
- Tradeoffs: makes the slot-status model internally consistent (only the
  doctor's-admin ever sets `Reserved`; the RescheduleRequested appointment
  itself counts toward capacity until it terminalizes). Removes the orphaned-hold
  class of bugs entirely (no reserve -> nothing to release on reject OR approve).
  But it changes the user-visible "tentative hold" semantics: between submit and
  approval the requested slot stays bookable by others, so two patients could
  request the same slot. OLD held the slot exclusively, so this is a behavior
  DEVIATION from OLD, not strict parity.
- Risk: med. Needs a capacity check: does a `RescheduleRequested` appointment
  count toward `GetActiveCountForSlotAsync`? Per Business Rule 4 only
  `Rescheduled*`/`Cancelled*` terminals are excluded -- `RescheduleRequested`
  (12) is NOT in that list, so it WOULD count, meaning the new slot's capacity
  is consumed by the pending request anyway. Verify this before choosing B.

### Option C -- Distinguish hold vs manual-close with a dedicated flag/status [effort: L] [risk: high]
- One-line desc: stop overloading `Reserved`; add an explicit
  "held-pending-reschedule" marker (e.g. a nullable `HeldByChangeRequestId` on
  `DoctorAvailability`, or a 4th `BookingStatus` value) so release-on-reject can
  target only true holds.
- Source: design decision; no direct OLD analog (OLD never had the manual-close
  meaning).
- Tradeoffs: cleanest long-term model, removes all ambiguity, supports both
  exclusive-hold (OLD parity) AND manual-close. But it is a schema + migration +
  multi-path change (submit, approve, reject, booking gate, doctor-availability
  admin UI).
- Risk: high (migration + touches every slot-status consumer). Overkill for a
  Phase-1 single-tenant parity slice; revisit if exclusive holds are required.

Recommendation: Option A for strict OLD parity (it restores the exact OLD
behavior with a guard), pending Adrian's decision on the `Reserved` semantics.
If the team accepts the capacity-model deviation, Option B is structurally
cleaner and kills the whole bug class. Option C only if exclusive tentative
holds are a hard requirement.

## 8. Root-Cause Hypothesis

High confidence. The 2026-05-15 slot rework ("plan 3") redefined `Reserved`
from "transient hold while a change request is pending" (OLD semantics) to
"manually closed by doctor's-admin", and removed the `SlotCascadeHandler`
status->slot mapping plus the `ReleaseSlotIfReservedAsync` calls in
approve/reject. But the rework did NOT also remove the `Reserved` WRITE in
`SubmitRescheduleAsync` (manager:256). Result: the submit path keeps producing
`Reserved` slots under the OLD "hold" intent, while the release side that used
to clear them was deleted under the NEW "manual close" intent. The two halves
now disagree, leaving every rejected (and even approved) reschedule with an
orphaned `Reserved` slot that the booking gate permanently rejects. This is a
half-applied refactor, i.e. a real bug, consistent with the CSV note ("Likely
slot-rework regression").

3. The APPROVE path (`ApproveRescheduleAsync`) also leaves the new slot
   `Reserved` (it relies on a no-op handler to drive it to `Booked`). Is that a
   separate gap to file, or should it be fixed in the same slice? It shares the
   same root cause but is outside G-02-04's stated scope (reject path).
4. Override-slot interaction: when a supervisor uses
   `ApproveRescheduleInput.OverrideSlotId` (different from the user-picked slot),
   the user-picked `Reserved` slot is definitely orphaned. A release-on-reject
   fix (Option A) covers reject; the approve+override case needs separate
   handling.
