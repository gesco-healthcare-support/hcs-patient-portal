> **DECISION (finalized 2026-06-01) [Gate 4]:** decision include -> exclude (true parity). OLD's spm.City table was vestigial -- never bound to any dropdown; OLD's actual city UX was free-text, identical to NEW. Keep free-text; no picker. A validated autocomplete (US Census 2024 Gazetteer seed, public domain) remains available as a FUTURE enhancement if data quality becomes a concern, but it exceeds OLD and is out of near-term scope. See SUMMARY.md Gate 4.

## 1. ID & Summary

OLD shipped a normalized `spm.City` lookup table (`CityName`, `CityCode`,
`StateId`, `CountryId` FK, `StatusId`) and exposed it as a read-model
lookup endpoint (`citylookups` -> `vCityLookUp`). NEW has no `City`
entity; every address-bearing NEW entity stores `City` as a free-text
`string?`.

The CSV disposition is `include` ("Add City lookup/picker (part of
G-07-08, included). Country half deferred (G-10-03); City proceeds.").
That disposition is a forward-looking PRODUCT decision, not a parity
restoration -- because the central finding of this research is that
**OLD never actually used the `City` table as an address picker.**
Every OLD address-bearing entity stores `City` as a free-text
`[Column("City")] string`, every OLD address form renders City as
`<input type="text">` (only State is a `<select>`), and the
`cityLookUps` data, although fetched into the Angular lookup groups, is
**never bound to any dropdown in any OLD HTML template**. The OLD
`spm.City` table was effectively dead reference data for the address
workflow.

Therefore "adding a City lookup/picker" is a NEW enhancement that goes
*beyond* OLD behavior (which would actually be a parity regression for
the address forms, since OLD's forms were free-text). The behavior NEW
ships today (free-text City) is the OLD address-form behavior. This
research records the evidence so Adrian can confirm whether to build the
new picker (product enhancement) or keep free-text (true OLD parity).

### Blast radius

A full "City lookup + picker" build (the CSV disposition) is a
cross-layer feature touching, per the `States` template, roughly:

- **Backend (new `Cities` vertical, ~13 files):** Domain entity +
  Manager + IRepository; Domain.Shared `CityConsts`; Contracts
  (`CityDto`, `CityCreateDto`, `CityUpdateDto`, `GetCitiesInput`,
  `ICitiesAppService`); Application (`CitiesAppService` + Mapperly
  mappers `CityToCityDto`, `CityToLookupDto`); EF Core repo +
  `DbSet<City>` registration in `CaseEvaluationDbContext` (host-scoped,
  `IsHostDatabase()` guard, mirroring `States`); a migration; a
  `CityDataSeedContributor`; permissions (`CaseEvaluation.Cities`
  Default/Create/Edit/Delete) in `CaseEvaluationPermissions.cs` +

- **Frontend (admin CRUD + picker wiring):** a `cities/` ABP-Suite
  feature folder (abstract/concrete components, services, routes,
  menu under Configurations) + regenerate `proxy/`.
- **Migration of the picker into address forms:** the 8 entities and
  their **14 City-bearing HTML forms** would each swap
  `<input formControlName="city">` for the picker, and the 8
  entities/DTOs would need a decision on data shape (keep `string City`
  vs add `CityId` FK -- see Options).
- **Regression risk: MEDIUM-HIGH if a FK is introduced.** Switching
  `string City` to `CityId` FK changes 8 entities, their migrations,
  ~14 forms, list-filter inputs, Excel exports, and 8 PDF placeholder
  resolutions; it also requires a complete US-city seed (tens of
  thousands of rows) or an admin-curated subset, plus a data-migration
  story for any existing free-text values. **LOW if the picker is a
  type-ahead that still persists a `string`** (no FK; cities table is
  just suggestion source) -- then only the forms change and all
  downstream consumers keep working.

## 4. Expected vs Actual

- **Parity-strict expected (match OLD user-visible behavior):** City is
  a free-text input on every address form; a `City` reference table may
  exist in the schema but is not wired into any address picker. NEW
  already matches this exactly.
- **CSV-disposition expected (product enhancement):** City becomes a
  lookup/picker (consistent with State), backed by a `Cities` table.
- **Actual (NEW today):** free-text `string?` City on all 8 entities;
  no `Cities` table; City input next to a State picker. This is the
  parity-strict expected state, and a deviation from the CSV
  disposition (which asks for more than OLD had).

The "Intent deviation" classification is accurate at the schema level
(OLD had a normalized table, NEW does not) but the user-facing address
*behavior* is identical (free-text in both). The real gap the CSV wants
closed is a UX-consistency enhancement (City should pick like State),
not an OLD-parity restoration.

## 7. Candidate Solutions

### Option A -- Keep free-text City; close as parity-equivalent [effort: S] [risk: low]
- **Desc:** Treat NEW's free-text City as the correct parity outcome
  (it is exactly OLD's address-form behavior) and record the decision.
  No code change.
- **Source:** OLD code: `Patient.cs:33-35`, `Location.cs:23-25` (string
  City), `appointment-add.component.html:181/267/...` (`<input>` City);
  `docs/parity-v2/07-admin-master-data.md:230-232` ("City lookup was
  already unused for addresses").
- **Tradeoffs:** Maximal parity fidelity; zero blast radius. Does NOT
  satisfy the CSV's "add a picker" disposition -- so it requires Adrian
  to revise the disposition once shown this evidence.
- **Risk:** Low. The only risk is the disposition assumed OLD had a
  working picker; this option corrects that assumption.

### Option B -- Type-ahead City suggestions backed by a Cities table, still persist string [effort: M] [risk: low-medium]
- **Desc:** Build the `Cities` lookup table + admin CRUD (mirroring the
  `States` feature), and surface it as an autocomplete/type-ahead that
  suggests known city names but stores the chosen/typed value back into
  the existing `string City` columns. No FK, no schema change on the 8
  entities, downstream PDF/export/filters untouched.
- **Source:** NEW template `src/.../Domain/States/CLAUDE.md` (full
  feature scaffold: entity, Manager, repo, DTOs, AppService, mappers,
  permissions, Angular CRUD, host-scoped `IsHostDatabase()` guard, seed
  contributor). Picker UI mirrors the existing `app-lookup-select`
  used for State (`appointment-add-patient-demographics.component.html:150-155`).
- **Tradeoffs:** Delivers the UX-consistency the CSV wants (City picks
  like State) without the FK migration cost; keeps free-text tolerance
  (typing an unlisted city still works, matching OLD). Needs a seed
  source for city names (curated subset, or none initially with
  free-text fallback). Only the ~14 forms change; all consumers keep
  reading `string City`.
- **Risk:** Low-medium. Main risk is seed-data scope (how many cities,
  who curates) and ensuring the type-ahead still allows free typing so
  no address entry is blocked.

### Option C -- Full normalization: `CityId` FK on the 8 entities [effort: XL] [risk: high]
- **Desc:** Add a `Cities` table AND replace `string City` with a
  `CityId` Guid FK on all 8 address entities, with the picker writing
  the FK. Closest to OLD's *schema* intent.
- **Source:** OLD `City.cs` schema; NEW `States` FK pattern
  (`State` inbound FKs in `States/CLAUDE.md`).
- **Tradeoffs:** Strong data consistency. But it exceeds OLD behavior
  (OLD never FK'd City), demands a full US-city seed or a constrained
  curated list, a data migration for existing free-text values
  (resolve string -> FK, handle unmatched), and touches 8 entities, 8
  DTO sets, ~14 forms, list-filter inputs, Excel exports, and 8 PDF
  placeholder resolutions (`DocxTemplateRenderer.cs:417-475`).
- **Risk:** High. Largest blast radius, breaks the simple
  string-passthrough that all downstream consumers rely on, and risks
  blocking address entry if a city is not in the list. Not recommended
  for Phase 1.

## 8. Root-Cause Hypothesis

**Confidence: high.** Root cause of the schema gap: NEW deliberately
dropped OLD's `spm.City` lookup table because it was **already
vestigial in OLD** -- the table and its `vCityLookUp` endpoint existed,
but no OLD entity FK'd to it and no OLD form bound it to a dropdown;
City was always free-text on the address forms. The new stack's
ABP-Suite scaffolding generated `string City` columns matching OLD's
actual `[Column("City")] string` columns, so it faithfully reproduced
OLD's real behavior while skipping the dead reference table. The "gap"
is therefore an *intent* deviation only at the normalized-schema level;
the user-facing behavior is in parity. The CSV's `include` disposition
("add a picker") is a forward product enhancement layered on top of
parity, not a correction of a regression -- a distinction worth
surfacing to Adrian before any build.

## 9. Open Questions

1. **Disposition basis:** the CSV says "Add City lookup/picker (part of
   G-07-08)" -- but G-07-08's own parity doc concluded the City lookup
   "was already unused for addresses." Does Adrian still want the picker
   as a deliberate UX enhancement (City should behave like State), or
   does the new evidence (OLD City was free-text) move this to
   exclude/keep-free-text?
2. **If building (Option B/C): seed source.** Where do city names come
   from? A curated per-state subset (small, admin-managed like States),
   a full US-city dataset (tens of thousands of rows), or no seed +
   free-text fallback? This drives effort and the S/M/XL split.
3. **Cascade vs flat:** OLD's `vCityLookUp` was a flat list (no
   StateId), and State exposes no City collection -- so even OLD's
   data shape was not a true State-filtered cascade. If NEW wants a
   state-filtered City picker, that is net-new behavior; confirm whether
   filtering City by selected State is desired.
4. **Data shape on the 8 entities:** keep `string City` (Option B,
   downstream-safe) or introduce `CityId` FK (Option C, full
   migration)? This is the single biggest effort/risk driver and should
   be decided before scoping.
5. **Country dependency (G-10-03):** OLD `City.CountryId` pointed at the
   deferred `Countries` table. Confirm the NEW `Cities` table omits
   `CountryId` entirely (US-only, per G-10-03 deferral) so it does not
   re-introduce the deferred dependency.
