> **RESOLUTION (Group L, 2026-06-06): DEFERRED.** Adrian decided not to build
> the owner-targeted pending-document reminder. OLD's send was dead code (both
> the SMS and email sends were commented out), so there is no live behavior to
> restore -- this would be a fresh feature, not a parity restoration. Revisit
> post-parity if staff want the personal nudge.

## 1. ID & Summary

OLD `SchedulerDomain` defined reminder type 8,
`AppointmentPendingDocumentSendToResponsibleUser()`, intended to send a
pending-document nudge addressed to the appointment's internal owner (the
"ResponsibleUser") listing the request number, due date, and outstanding
document names. In OLD this path is DEAD: both the Twilio SMS send
(`SchedulerDomain.cs:352`) and the email send (`:361`) are commented out,
so OLD never actually delivered it.

NEW has no equivalent job. None of the 9 registered Hangfire recurring
jobs targets the appointment's responsible user with a pending-document
reminder. The closest job, `PackageDocumentReminderJob` ->
`PackageDocumentReminderEmailHandler`, deliberately resolves the EXTERNAL
stakeholder fan-out (attorneys, claim examiners, office mailbox) via
`IAppointmentRecipientResolver`, not the internal owner. The internal
owner field does exist in NEW (`Appointment.PrimaryResponsibleUserId`,
set on approval), so the recipient target is available -- only the
reminder job/handler is missing.

Decision is `include`: build a fresh staff-side reminder to the
responsible user. This is a NEW feature build (the OLD send was dead
code), not a literal port of a working OLD behavior. No live OLD
behavior is lost by NEW's current absence.

## 7. Candidate Solutions

### Option A -- New owner-targeted reminder job, direct email resolution [effort: M] [risk: low]

One-line: add `ResponsibleUserPendingDocReminderJob` (Domain) +
`ResponsibleUserPendingDocReminderEto` (Domain.Shared) +
`ResponsibleUserPendingDocReminderEmailHandler` (Application), register in
`ConfigureHangfireRecurringJobs`, reuse the `UploadPendingDocuments`
template; handler resolves `PrimaryResponsibleUserId` -> email directly
via `IdentityUserManager` (no recipient-resolver change).

- Source: OLD code `SchedulerDomain.cs:336-364` (intent + view-model
  fields RequestConfirmationNumber / DueDate / PendingDocList); NEW
  templates `PackageDocumentReminderJob.cs` (selection + tenant
  iteration) and `InternalStaffQueueDigestJob.cs:126-142` (resolving a
  user id -> IdentityUser email). All internal-code sources; no external
  URL needed (pure port pattern).
- Selection logic: reuse PackageDocumentReminderJob's query (approved
  appointments, outstanding `Pending`/`Rejected` non-ad-hoc docs), but
  group by appointment and address only the appointment's
  `PrimaryResponsibleUserId`. Fix the OLD `AppointmentId=1` hardcode by
  iterating all qualifying appointments (per root CLAUDE.md bug policy).
- Tradeoffs: smallest blast radius (no resolver / no enum change);
  reuses the existing `UploadPendingDocuments` template so no new
  localization. Cons: addressing is appointment-owner-specific rather
  than going through the standard recipient-resolver seam, so it is a
  slightly different addressing path than other reminders.
- Risk: low (additive; no existing job changed). Skip appointments with
  null `PrimaryResponsibleUserId` (unassigned). SMS leg stays dropped
  (consistent with G-05-INTENT-01).

### Option B -- New `NotificationKind` + recipient-resolver branch [effort: M] [risk: low-med]

One-line: same job + ETO, but add
`NotificationKind.ResponsibleUserPendingDocReminder` and a branch in
`IAppointmentRecipientResolver` that returns just the responsible user's
email, so the handler uses the standard resolver seam like every other
reminder.

- Source: NEW `NotificationKind.cs` + the recipient-resolver
  implementation; OLD intent as above.
- Tradeoffs: consistent with the established recipient-resolution
  architecture (handlers do not resolve users ad hoc). Cons: touches the
  shared resolver and the shared enum -- slightly wider blast radius; a
  resolver-branch bug could (in theory) affect resolution for other
  kinds if the switch is not isolated.
- Risk: low-med (shared resolver edit). Prefer only if Adrian wants all
  reminders to flow through one recipient seam.

### Option C -- Do not build; rely on package-doc reminder [effort: none] [risk: n/a]

One-line: treat the external package-doc reminder as sufficient and not
build an owner-specific reminder.

- Source: parity doc "Keep in NEW? Optional" (line 131-133).

## 8. Root-Cause Hypothesis

Confidence: high. Root cause is a deliberate scope omission, not a defect.
During Phase 7 (Category 7, 2026-05-10) the team ported the OLD
`SchedulerDomain` reminders that actually SENT something into Hangfire
jobs. Reminder type 8 was dead in OLD (both sends commented out at
`:352`/`:361`), so it was correctly skipped as "OLD-bug, do not port the
dead send" (parity doc OLD-bug item 3). The parity audit then re-listed it
as "Missing" rather than "OLD-bug" only because the INTENT (a personal
owner-targeted doc-chase nudge) is coherent and a plausible future ask --
so it surfaces as a gap to decide on, not a regression. The CSV
dispositioned it `include` (build fresh). The underlying NEW field
(`PrimaryResponsibleUserId`) and the job/handler/dispatch pattern all
exist; only the owner-targeted reminder job was never written.
