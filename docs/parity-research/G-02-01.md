## 1. ID & Summary

OLD had a standalone, internal-only **global change-log list page** (route `appointment-change-logs`) that listed every field-level change across ALL appointments and let staff filter by `requestConfirmationNumber`, `fieldName`, `oldValue`, `newValue`, `tableName`, `appointmentStatusName`, `changedDate`, plus a free-text `searchQuery` -- paged, sortable, default sort `RequestConfirmationNumber`, with a row click that opens the underlying appointment. Use case: "who changed what claim number last Tuesday" without knowing the appointment up front.

NEW has only a **per-appointment** drill-down (`appointment-change-logs.component.ts`, route `appointments/view/:id/change-log`). It reads the route `:id`, calls ABP's stock `AuditLogsService.getEntityChangesWithUsername` for a single `Appointment` entity, and renders property diffs. There is no global list, no cross-appointment query, and none of OLD's field-level filters. Confirmed there is NO custom global change-log AppService/controller/projection anywhere in the NEW backend.

The gap is real and confirmed statically. It is a Missing feature, not a partial deviation.

- Cross-cutting: multi-tenancy -- ABP audit `EntityChange` rows carry `TenantId`; under Phase 1 single-tenant this is moot, but a cross-tenant global list would be a Phase-1 violation. Must scope to current tenant.
- Dependency on G-02-02: OLD's global list surfaced child-entity changes (injury/body-part/claim, `tableName` filter). If G-02-02 (child-entity audit) is not also implemented, the NEW global list cannot reproduce the `tableName` filter dimension faithfully -- only `Appointment`-root rows would appear.

## 4. Expected vs Actual

- Expected (OLD parity): an internal-only page listing field-level changes across ALL appointments, filterable by confirmation#, field name, old value, new value, table/entity name, status, and modified date; paged + sortable (default `RequestConfirmationNumber`); row click opens the appointment.
- Actual (NEW): no such page exists. Staff can only view the audit history of ONE appointment they already navigated to. They cannot search the audit trail by field/value/date/confirmation# across appointments.

Key capability shortfall in NEW's data model: ABP's audit projection (`EntityChange` + `EntityPropertyChange`) stores `entityTypeFullName`, `entityId`, `changeTime`, `changeType`, and per-property `propertyName`/`originalValue`/`newValue`. It does NOT store the business `RequestConfirmationNumber` or `AppointmentStatusName`, and it does NOT support filtering by `propertyName`/`originalValue`/`newValue` (field-level filters). OLD's SP joined those business fields and filtered on field-level columns; ABP's stock surface does neither.

## 7. Candidate Solutions

### Option A -- Custom global change-log AppService + projection (OLD-faithful) [effort: L] [risk: med]

Port OLD's behavior on ABP audit data. New `GetAppointmentChangeLogsInput` (paged/sorted DTO with `fieldName`, `oldValue`, `newValue`, `tableName/entityTypeFullName`, `appointmentStatus`, `changedDateFrom/To`, `requestConfirmationNumber`, free-text) -> new `AppointmentChangeLogAppService` (`[RemoteService(IsEnabled=false)]`, manual controller per ABP convention) that queries ABP's `EntityChange`/`EntityPropertyChange` (via `IEntityChangeRepository` or a custom EF query) joined to `Appointment` for `RequestConfirmationNumber` + status, projecting one row per property change. Gate with the existing `CaseEvaluation.AppointmentChangeLogs` permission. New Angular list component (filter form + paged table + row-click -> appointment view) registered as a top-level internal route + nav-menu entry.
- Source: OLD code (`appointment-change-log-list.component.ts:39,67-103`, `appointment-search-view-model.ts`, `AppointmentChangeLogsSearchController.cs`, `AppointmentChangeLog.cs`); ABP `IEntityChangeRepository` / `AuditLogsAppService` override pattern.
- Tradeoffs: faithfully reproduces field-level filters + confirmation#/status columns + default sort. But couples directly to ABP's audit-log schema (a documented but internal-ish surface); field-level filtering over `EntityPropertyChange` rows can be expensive at scale (ABP support explicitly warns against large unpaged audit pulls -- keep it paged + indexed).
- Risk: medium. Joining audit tables to business tables is non-trivial and tenant-scoping must be explicit (Phase 1). Faithful `tableName` filter depends on G-02-02 (child-entity audit) landing first; otherwise only `Appointment`-root rows appear.

### Option B -- Reuse ABP's stock Audit Logs "Entity Changes" UI [effort: S] [risk: low]

ABP Pro's Audit Logging module ships a built-in "Audit Logs" page with an "Entity Changes" tab and `AuditLogsService.getEntityChanges(GetEntityChangesDto)` -- paged, filterable by `entityTypeFullName`, `entityChangeType`, `startDate`, `endDate`. Expose that page (scoped to the `Appointment` FQN) to internal users instead of building a custom one.
- Source: ABP Audit Logging Module (Pro) docs (official) -- https://abp.io/docs/latest/modules/audit-logging-pro (accessed 2026-06-01, credibility: official vendor docs); proxy surface verified locally in `@volo/abp.ng.audit-logging/proxy/index.d.ts` (`getEntityChanges`, `GetEntityChangesDto` lines 139-146, 176).
- Tradeoffs: near-zero build cost; immutable audit data is exactly what OLD's grid showed. BUT it does NOT meet parity: no `fieldName`/`oldValue`/`newValue` field-level filters, no `requestConfirmationNumber` filter/column (audit projection lacks business fields), no status column, default sort differs. It is a global cross-appointment list but with the wrong filter set.
- Risk: low technically, but HIGH parity-shortfall -- fails the CSV's explicit filter list (field/old/new/table/status/date/conf#). Best as an interim stopgap, not the parity target.

### Option C -- Dedicated change-log projection table fed by an audit handler [effort: XL] [risk: med]

Build a denormalized read-model table (mirroring OLD's `spm.AppointmentChangeLogs`: tableName, fieldName, old/new, changedBy, changedDate, appointmentId + denormalized confirmation#/status) populated by subscribing to ABP entity-change events (`IEntityChangeEventHandler` / audit contributor) or by a projection job. The global list queries this table directly with all OLD filters; cheap to query, exact OLD shape.
- Source: OLD `AppointmentChangeLog` table + `AppointmentChangeLogDomain` reflection-based logging (the conceptual model); ABP audit contributor extension points.
- Tradeoffs: best query ergonomics + exact OLD parity (including `tableName` for child entities, subsuming G-02-02). But it duplicates audit data, adds a write path to maintain, and is the most code. Effort XL; only justified if field-level filtering at scale proves too slow over ABP's native tables, or if G-02-02 child-entity capture is built this way anyway.
- Risk: medium. New write path + dedup against ABP's own audit log; consistency burden.

Recommended path: Option A for parity (it satisfies the CSV's filter list); fall back to B only as an interim. Decide A vs C jointly with G-02-02 -- if child-entity audit is built via a projection, C becomes the cheaper combined solution.

## 8. Root-Cause Hypothesis

high confidence. NEW deliberately replaced OLD's custom `AppointmentChangeLog` table + `spm.spAppointmentChangeLogs` SP with ABP's stock `[Audited]` + Audit Logging module (confirmed by the parity-doc "Equivalent" note and the `[Audited]` attribute). The W2-4 work item built only the **per-appointment viewer** on top of `AuditLogsService.getEntityChangesWithUsername` (single entity id) and reused the existing `CaseEvaluation.AppointmentChangeLogs` permission for that one route. The **global filterable list** was never ported because ABP's audit surface does not natively expose OLD's field-level filters or business identifiers (confirmation#/status), so it would have required a custom projection/AppService that was descoped. The gap is an unbuilt feature, not a regression -- the permission and localization scaffolding for it already exist, signalling the per-appointment slice was treated as the W2-4 deliverable and the global grid deferred.
