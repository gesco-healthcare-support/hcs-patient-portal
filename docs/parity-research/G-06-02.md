## 1. ID & Summary
OLD rendered a reduced "search-only" home for accessor users (an external user
added to someone else's appointment as a view-only viewer, not a self-registrant).
The accessor home hid the two booking buttons; only the confirmation-number search
remained. NEW shows every external role the same home, including the book buttons.
The gap is purely the missing accessor-only layout branch; booking is still
ownership/permission-gated server-side, so the security outcome is bounded.

## 6. Why Excluded + Revisit-If
Excluded because:
- Cosmetic only -- the sole behavioral difference is two extra buttons being visible.
- Server-side access scoping already limits an accessor's data visibility (S-NEW-2
  involvement filter narrows the appointment list); booking is downstream
  ownership/permission-gated, so a stray click cannot create an unauthorized booking.
- The OLD accessor template hides nothing security-relevant -- it only removes the
  two book CTAs; search behavior is identical for both variants.

Revisit if ANY of:
- The accessor flow (parity area 01 / G-01-06, G-10-08) is finalized and we want the
  home layout to match OLD pixel-for-pixel for accessor users.
- Downstream booking is found NOT to fully ownership-gate an accessor (would upgrade
  this from cosmetic to a real access-control gap -- but that would be a separate
  booking-flow defect, not this UI item).
- Product wants accessors explicitly steered away from booking (UX clarity), not just
  blocked after the fact.

## 7. Candidate Solution
None -- excluded. (If revisited: small S-effort change -- read the already-surfaced
`IsAccessor` profile flag in home.component.ts and `*ngIf`-gate the two book buttons,
mirroring OLD's `!isAccessor` pattern. No backend change; flag is already exposed.)

## 8. Decision Rationale + confidence
Confidence HIGH. Both the OLD restricted layout and the NEW unconditional layout are
confirmed in source; the only divergence is the visibility of the two book buttons for
accessor users, and downstream booking is permission-gated. The CSV "cosmetic +
server-side scoping" rationale matches the code exactly. Exclude is sound for Phase 1.
