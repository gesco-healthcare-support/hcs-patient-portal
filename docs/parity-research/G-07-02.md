## 1. ID & Summary

OLD `AppointmentType.ReEvalId` (`Nullable<int>`) is dropped in NEW. The parity
doc and CSV both describe it as a "self-referential pointer" that "pairs a base
type (PQME) with its re-eval variant (PQME RE-EVAL)". **That description is
WRONG** -- verified against OLD code. `ReEvalId` is NOT a self-FK to another
`AppointmentType` row. It is a classification flag holding an `EvaluationTypeEnum`
value (`Normal = 0`, `Re = 1`, `Both = 2`) that drives the booking-form
appointment-type dropdown filter:

- Fresh booking (`type == "1"`): show types where `reEvalId == Normal || Both`.
- Re-eval booking (`type == "2"`): show types where `reEvalId == Re || Both`.
- Location admin form: exclude types where `reEvalId == Re`.

The real downstream consumer is the OLD booking client's dropdown filter. NEW has
no equivalent: it neither models the flag NOR seeds RE-EVAL variant types, and it
resolves the only reval-aware backend rule (per-type max-time horizon) by
**name-substring matching** on `AppointmentType.Name` (`name.Contains("AME")` /
`name.Contains("PQME")`) -- exactly the "name match (not data)" brittleness the CSV
note calls out. NEW also renamed the entire seed set (QME / Panel QME / AME /
Record Review / Deposition / Supplemental Medical Report), so the substring router
already mis-classifies most seeded names.

## 4. Expected vs Actual

- Expected (OLD parity): each appointment type carries an evaluation-type
  classification; the booking form shows only Normal/Both types when starting a
  fresh request and only Re/Both types when re-evaluating; the per-type max-time
  horizon and reval routing are resolved from data, not from a display-name
  substring.
- Actual (NEW): no classification field; no reval-context dropdown filtering;
  reval endpoint takes any type id; max-time horizon resolved by brittle uppercase
  name-substring matching that already mis-classifies most NEW seed names to OTHER.

## 7. Candidate Solutions

### Option A -- Model an `EvaluationType` classification flag (faithful OLD port) [effort: M] [risk: medium]

- Desc: Add a nullable `EvaluationType` enum (Normal=0, Re=1, Both=2) to
  `AppointmentType`, migrate, seed per-row, and (a) replace `ResolveMaxTimeDaysForType`
  name-substring routing with a data-driven mapping, (b) add reval-context filtering
  to the type lookup so fresh booking shows Normal/Both and reval shows Re/Both.
- Source: OLD code: `evaluation-type.ts:1-5`, `appointment-add.component.ts:681-698`,
  `appointment-edit.component.ts:991-1003`, `location-add.component.ts:56`.
- Tradeoffs: faithfully reproduces OLD dropdown behavior and removes the brittle
  name match; matches the CSV's "resolve variant from data" intent. Does NOT
  require literal base->variant row pairing (which OLD never had).
- Risk: medium -- changes the max-time horizon that currently (mis-)applies; must
  re-seed and re-test the booking gate.

### Option B -- Nullable self-`Guid? ReEvalId` literal pairing (what the parity doc DESCRIBES) [effort: M] [risk: medium-high]

- Desc: Add `Guid? ReEvalId` self-FK pairing each base type to its RE-EVAL variant,
  plus seed RE-EVAL variant rows so the FK has targets.
- Source: the source parity doc's (inaccurate) description + CSV "self-ref column
  + seed base->reval pairings".
- Tradeoffs: matches the literal CSV instruction, but it reproduces a relationship
  that DID NOT EXIST in OLD (no self-FK in schema; ReEvalId held an enum, not a
  row id). It also forces re-introducing RE-EVAL variant rows the NEW seed set
  deliberately dropped.
- Risk: medium-high -- builds a structure on a misread of OLD; likely the wrong
  shape. Flag to Adrian before choosing this over A.

### Option C -- Document-and-defer; only fix the name-substring brittleness now [effort: S] [risk: low]

- Desc: Leave reval-context dropdown filtering out (no admin UI yet, G-07-01), but
  replace `ResolveMaxTimeDaysForType`'s name-substring match with an explicit
  per-type-id mapping table keyed off the canonical seed GUIDs.
- Source: OLD code: `AppointmentDomain.cs:127-155` (the real backend routing is by
  type identity, not name); NEW `CaseEvaluationSeedIds.AppointmentTypes`.
- Tradeoffs: removes the immediate correctness bug (4/6 names mis-route to OTHER)
  cheaply; defers the full classification + dropdown filter until the admin/booking
  UI work lands.
- Risk: low; but it's a partial fix -- the reval-context dropdown filter remains a
  later task tied to G-07-01 / G-01-07.

## 8. Root-Cause Hypothesis

High confidence. Two compounding root causes:

1. The NEW rebuild simplified `AppointmentType` to a plain Name+Description lookup
   and re-named/re-scoped the seed set, dropping the `ReEvalId` column without
   carrying its real semantics (an evaluation-type classification driving the
   booking dropdown).
2. To preserve OLD's per-type max-time rule without the classification, NEW
   substituted **name-substring matching** on the display name -- a fragile proxy
   that breaks the moment the seed names stop containing "PQME"/"AME" (which they
   already do). The parity doc compounded this by misreading `ReEvalId` as a
   self-FK, which would lead a fixer to build the wrong structure (Option B).

The correct fix is data-driven classification (Option A), not literal row pairing.

## 9. Open Questions

1. Does NEW still need distinct RE-EVAL appointment-type rows at all, or is the
   re-eval distinction now carried entirely by the lifecycle flow
   (`CreateRevalAsync`) + an evaluation-type flag on the existing rows? (Open
   question 1 in the source doc, restated.)
2. Should the classification be an `EvaluationType` enum (Normal/Re/Both, faithful
   to OLD `ReEvalId` values) -- which I recommend -- or the literal self-FK the CSV
   note and parity doc (incorrectly) describe? This determines Option A vs B.
3. The NEW max-time router only has PQME / AME / OTHER horizons, but the NEW seed
   set has QME, Panel QME, AME, Record Review, Deposition, Supplemental -- how
   should each NEW type map to a max-time horizon (which one is "PQME-like")? This
   needs a business decision regardless of which option ships, because the current
   name match silently buckets most into OTHER.
4. Cross-ref G-01-07 (reval pre-load) and G-10-11 (data-model storage side): the
   reval-context dropdown filter should be built together with the reval pre-load
   UI entry path so "type == reval" is actually reachable in NEW.
