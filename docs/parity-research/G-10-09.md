## 1. ID & Summary

OLD has a tenant-scoped master library `spm.AppointmentDocumentTypes`
(`AppointmentDocumentTypeId`, `DocumentTypeName`, `StatusId`) that
classifies uploaded documents. Both document tables
(`AppointmentDocument`, `AppointmentNewDocument`) carry a nullable
`AppointmentDocumentTypeId` FK plus an `OtherDocumentTypeName` free-text
overflow used when the user picks "Other". The upload modal renders a
type dropdown from a status-filtered view (`vAppointmentDocumentTypeLookUp`)
and a free-text "Please Specify Document Type" box that appears only when
the selected type id == 4 (the magic "Other" id).

NEW has none of this. `AppointmentDocument` (the single unified entity)
classifies documents with a free-text `DocumentName` string only; its
own XML doc-comment (line 16) lists "AppointmentDocumentType lookup
(free-text DocumentName at MVP)" as a deliberate W1-3 cut. There is no
document-type entity, DTO, AppService, controller, permission, or seed
contributor anywhere in NEW; `OtherDocumentTypeName` is also gone.

G-10-09 is the **storage-side facet** (the lookup table + FK columns) of
one underlying gap. It is an explicit DUPLICATE of **G-03-01** (the
behavior/CRUD/picker facet, decision = include) and **G-07-07** (the
admin master-data facet, decision = include). All three resolve with one
implementation slice: port the master entity + CRUD + seed, add the FK
to `AppointmentDocument`, restore the "Other" free-text overflow, and add
the type picker to the upload UI.

## 4. Expected vs Actual

  no admin curation screen. The type field was a deliberate MVP cut.

## 7. Candidate Solutions

### Option A -- Port the full AppointmentDocumentType master, cloning the AppointmentType feature [effort: M] [risk: low-medium]

Treat the existing NEW `AppointmentType` feature as the copy template
(it is a host-scoped lookup with Name + optional field, full 7-layer
CRUD, a `DataSeedContributor`, and an Angular list/detail UI -- see
`src/.../Domain/AppointmentTypes/*` and its CLAUDE.md file map). Steps:

1. Create `AppointmentDocumentType` entity (`DocumentTypeName` max 100,
   matching OLD), Manager, repository, seed contributor, DTOs, AppService,
   controller, permissions, EF registration + migration, Angular CRUD.
2. Add nullable `AppointmentDocumentTypeId` FK + `OtherDocumentTypeName`
   (max 100) to the existing `AppointmentDocument` aggregate, its DTO,
   the upload AppService signature, and the Angular upload form (type
   `<select>` + conditional "Other" free-text box).
3. Decide multi-tenancy: OLD's table had `StatusId` and was per-install;
   for Phase 1 (one tenant), follow the `AppointmentType` precedent
   (host-scoped lookup, NO `IMultiTenant`, soft delete via ABP's
   `IDeletionAudited` instead of OLD's `StatusId = Status.Delete`).

- **Source:** OLD code (entity `AppointmentDocumentType.cs:12-48`, domain
  `AppointmentDocumentTypeDomain.cs:15-110`, controller, upload picker
  HTML). NEW template: `src/.../Domain/AppointmentTypes/` +
  `AppointmentTypeDataSeedContributor.cs`.
- **Tradeoffs:** Closest parity, reuses a proven in-repo pattern, fully
  satisfies G-03-01 + G-07-07 + G-10-09 in one slice. Cost is the 7-layer
  surface + a migration on the live document table.
- **Risk:** low-medium (additive nullable FK; signature ripple to the
  public/package upload paths).

### Option B -- Add the FK + "Other" overflow now, defer the admin CRUD UI [effort: S] [risk: low]

Add the `AppointmentDocumentType` entity + seed (canonical rows) + the
FK/`OtherDocumentTypeName` columns + the upload picker, but skip the
admin list/detail CRUD screen and permission tree until staff actually
need to curate categories. The picker reads seeded rows; "Other" still
works.

- **Source:** same OLD code as Option A.
- **Tradeoffs:** Lower effort, restores the user-facing classification
  immediately. But G-07-07 (admin master-data CRUD) is explicitly
  "include", so this only partially closes the linked items and leaves
  the admin curation gap open.
- **Risk:** low. Misses part of the agreed scope.

Recommendation: Option A. The CSV dispositioned all three linked items as
`include`, and the `AppointmentType` template makes the full slice cheap.

## 8. Root-Cause Hypothesis

`high`. The absence is an intentional, documented MVP scope cut, not an
oversight or bug: `AppointmentDocument.cs:16` explicitly enumerates
"AppointmentDocumentType lookup (free-text DocumentName at MVP)" among
its deferred cuts. The W1-3 minimum-viable document entity shipped with
free-text `DocumentName` to avoid the master-data + FK + UI surface; the
parity review (G-03-01/G-07-07/G-10-09) later decided to restore it. Root
cause = deliberate deferral now reversed by the include decision.

## 9. Open Questions

1. **Canonical seed rows + the "Other" id.** OLD's picker hardcodes the
   "Other" type as **id 4** (template `value == 4`), implying a fixed,
   ordered set of seed rows (likely ~3 real types + "Other" at position
   4). Those rows lived in the live AWS DB, not in repo SQL (no INSERT
   seed found). Adrian must supply the canonical `DocumentTypeName` list
   and confirm "Other" semantics. NEW uses GUID PKs, so the magic-4
   coupling must be replaced by matching on the seeded "Other" GUID/name,
   not a numeric id.
2. **Multi-tenancy / status model.** OLD used a per-install table with a
   nullable `StatusId` soft-delete enum. Confirm the NEW lookup should be
   host-scoped (like `AppointmentType`) for Phase 1 vs tenant-scoped, and
   that ABP soft-delete replaces OLD's `StatusId = Status.Delete`.
3. **Scope of the picker.** OLD surfaced the type picker on both the
   ad-hoc upload and the package-doc upload. Confirm NEW should show it on
   every upload path (ad-hoc, package-queue, public upload-by-code) or
   only ad-hoc -- and whether package-queued docs auto-inherit a type
   from the master `Document.Name`.
4. **Filtering/reporting.** G-03-01 cites "filter/group uploaded docs by
   type." Confirm whether a type filter on the document grid is in scope
   for this slice or a follow-up.
