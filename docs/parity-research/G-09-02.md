## 1. ID & Summary

OLD let an external user (Patient / Applicant Attorney / Defense Attorney /
Claim Examiner) open a "Help" link in the top bar that popped a "Need
Question?" modal, type a free-text question (<= 500 chars), optionally carry
a read-only confirmation number, and submit. Submit wrote a `UserQuery` row
and emailed staff (smart-routed -- see G-09-03). NEW has none of it: no Help
button rendered, no `UserQuery` entity, no submit endpoint, no admin list, no
email trigger. This is a clean MISSING feature to port. It is a multi-layer
slice (entity + AppService + controller + migration + Angular modal + navbar
wiring + email dispatch) but every piece maps to an existing NEW pattern, and
the two hardest dependencies (email template + appointment fields) already
exist in NEW.

## 4. Expected vs Actual

  Approved appointment matched) else broadcast to all IT-Admins -> success

  endpoint, no email. The navbar exposes a vestigial `showHelp`/`helpClick`
  API that nothing binds. External users have zero in-app contact channel.

## 7. Candidate Solutions

### Option A -- Full ABP port of UserQuery + Help modal + dispatcher email [effort: L] [risk: low]

One-line: Add a `UserQuery` aggregate, a create-only AppService + manual
controller, an Angular "Need Question?" modal launched from a wired-up navbar
Help button, and route the email through `INotificationDispatcher`
(template code `SubmitQuery`/`UserQuery`, already seeded).

Source: OLD code -- `UserQueryDomain.cs:53-107`, `UserQuery.cs:12-55`,
`user-query-add.component.{ts,html}`, `top-bar.component.{ts:47-58,html:77-80}`.
NEW infra: `INotificationDispatcher.cs`, `Appointment.cs:33,90`,
`NotificationTemplateConsts.cs:86,102`.

Mapping notes (port, not 1:1 framework copy):
- Entity goes in `Domain/UserQueries/UserQuery.cs` as an ABP aggregate. OLD
  `UserId`/`CreatedById` int FKs -> ABP audited-entity `CreatorId` (Guid) +
  a `UserId` Guid; `Message` keep `MaxLength(500)` + required. `AppointmentId`
  + `RequestConfirmationNumber` map to optional fields (OLD had them
  `[NotMapped]` view-model carriers; NEW can persist them or keep transient
  on the DTO). Drop OLD's `Query` (serialized search VM) -- it was unused
  dead state.
- Email routing (this is G-09-03): if `RequestConfirmationNumber` present AND
  an `Approved` appointment matches, dispatch to that appointment's
  `PrimaryResponsibleUserId`; else dispatch to all IT-Admin users. Use
  `INotificationDispatcher.DispatchAsync` with the seeded `UserQuery`/`SubmitQuery`
  template. GUARD the responsible-user email lookup and fall back to the
  IT-Admin pool (do NOT replicate OLD B-09-02 NRE; root CLAUDE.md bug policy
  -- clear bug, fix silently).
- Business rule lives in a domain service (`UserQueryManager`) per CLAUDE.md
  ("never skip the domain service on create flows").
- Frontend: build the modal natively in Angular 20 (Material/Lepton),
  applying OLD copy verbatim ("Need Question?", the inquiry blurb, "Please
  write down your question below.", Send/Close). Wire the navbar Help `<a>`
  + `(helpClick)` binding in `home.component.html` and ensure it shows for
  the same external roles OLD did (resolve Open Question 1 first).

Tradeoffs: Largest single item in this area because it spans every backend
layer + frontend. But each piece is a well-trodden ABP/Angular pattern and
the two riskiest dependencies (email template, appointment fields) already
exist. Pair-ship with G-09-03 (routing) and the email leg G-04-02; entity
tracked as G-10-02. Effort L (not XL) precisely because no new infra is
needed.

Risk: low. Additive table + endpoint; only shared edit is navbar markup with
one consumer.

## 8. Root-Cause Hypothesis

The feature was never ported. The NEW navbar component was scaffolded with a
forward-looking `showHelp`/`helpClick`/`onHelpClick` API (so the wiring point
exists) but no consumer binds it and no backend was built, because Submit-Query
sits in the "post-parity" bucket: `NotificationTemplateConsts.cs:60-61`
explicitly lists `SubmitQuery` among codes "seeded but unwired until
post-parity feature phases." So the absence is intentional sequencing, not an
accidental drop -- the email template was seeded ahead of the feature, and the
navbar API stub was left as a deliberate hook. Confidence: HIGH (grep/Glob
prove total backend absence; the seed comment names this feature as deferred;
the navbar stub + appointment fields confirm the intended wiring points).
