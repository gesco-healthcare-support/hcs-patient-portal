## 1. ID & Summary

OLD applies the cancel-time window gate (`AppointmentCancelTime`) ONLY at change-request
submit time. At supervisor approval, OLD does NOT re-check the window and does NOT
auto-derive the outcome: the supervisor manually picks `CancelledNoBill` or `CancelledLate`
from a dropdown. NEW ports this faithfully -- the window gate runs at submit
(`AppointmentChangeRequestManager.SubmitCancellationAsync`) and the supervisor's manual
outcome choice runs at approval (`ApproveCancellationAsync` +
`ChangeRequestApprovalValidator.EnsureCancellationOutcome`).

**There is NO functional parity gap today.** The "Partial" class is narrow: it flags two
forward-looking concerns the CSV note raises as *enhancements*, not OLD parity:
1. Optionally re-evaluate the window at approval and/or default NoBill-vs-Late from it
   (an improvement OLD never had).
2. Verify the timezone anchor (`DateTime.Today` = machine-local) once multi-tenant lands.

Both are correctly characterized in the parity doc as "no functional gap today; flagged for
timezone verification only." The single concrete code finding worth acting on is the raw
`DateTime.Today` anchor (manager line 131), which is machine-local and tenant-blind.

## 7. Candidate Solutions

### Option A -- Confirm parity, harden timezone anchor only [effort: S] [risk: low]
Treat G-02-06 as "already at parity" for the manual-outcome behavior (no action needed
there), and make the ONE defensible code change: replace `DateTime.Today` at
`AppointmentChangeRequestManager.cs:131` with an injected ABP `IClock` so the date anchor is
centralized and testable, deferring true per-tenant timezone resolution to Phase 2.
- Source: OLD code `AppointmentChangeRequestDomain.cs:87` (the gate this mirrors); ABP
  `IClock` precedent at `InvitationManager.cs:41`. ABP IClock docs:
  https://abp.io/docs/latest/framework/infrastructure/timing (official, accessed
  2026-06-01) -- `Clock.Now` respects `AbpClockOptions.Kind`; UTC mode normalizes
  `DateTime` kinds, which is the multi-tenant-safe foundation.
- Tradeoffs: minimal change, keeps strict OLD parity, adds testability. Does NOT fully solve
  multi-tenant timezone (no per-tenant zone field exists) but removes the machine-local
  coupling and gives a single seam to extend.
- Risk: LOW (1 file, behavior identical under default local-clock config).
- Effort: S.

### Option B -- Re-evaluate window at approval AND auto-default NoBill vs Late [effort: M] [risk: medium]
Build the enhancement the CSV note floats: at `ApproveCancellationAsync`, recompute
`IsWithinNoCancelWindow` against the slot date and pre-select / default the outcome to
`CancelledLate` when inside the window, `CancelledNoBill` when outside (supervisor can still
override).
- Source: NONE in OLD -- OLD never re-checks or auto-derives (verified `Update` 262-291).
  This is a NET-NEW behavior, i.e. a deliberate deviation from parity.
- Tradeoffs: reduces supervisor error (mis-billing a late cancel as NoBill) but BREAKS the
  "strict parity" directive; must be explicitly approved by Adrian as hardening, like
  G-02-07's injury-detail gate. Also requires the Angular approval UI to exist first (it does
  not -- see Open Questions).
- Risk: MEDIUM -- changes user-visible billing behavior; needs sign-off + the
  PARITY-FLAG/parity-flags.md process per branch policy.
- Effort: M.

### Option C -- No-op: document parity, defer timezone to Phase 2 multi-tenant work [effort: S] [risk: low]
Record that the behavior is at parity and fold the timezone-anchor fix into the Phase 2
multi-tenant epic (where a per-tenant timezone field would be designed holistically across
ALL date math, not just this one site).
- Source: parity doc's own conclusion ("No functional gap today... Flagged for timezone
  verification only").
- Tradeoffs: zero risk now; but leaves `DateTime.Today` machine-local coupling in place,
  which a future Phase-2 dev must remember to fix here AND at every other `DateTime.Today`/
  `.Now` site (booking lead-time, etc.).
- Risk: LOW.
- Effort: S (doc-only).

**Recommendation:** Option A. It is the smallest change that (a) preserves strict OLD parity
for the manual outcome, (b) removes the one real code smell (machine-local anchor) flagged by
the CSV, and (c) lays an `IClock` seam for Phase 2 without prematurely building a per-tenant
timezone system. Option B should be raised with Adrian as a separate, explicitly-approved
hardening item (parity deviation), not bundled here.

## 8. Root-Cause Hypothesis

There is no defect to root-cause for the manual-outcome behavior -- NEW correctly replicates
OLD (window gate at submit, manual NoBill/Late at approval, no auto-derive). The "Partial"
classification is a forward-looking annotation, not a present bug. The single legitimate
code observation is that the cancel-window date math anchors on `DateTime.Today`
(machine-local), which is correct for a single-tenant, single-timezone deployment (OLD's
exact behavior) but becomes ambiguous once tenants span timezones, because the server's local
midnight may not match the tenant's local midnight, shifting the strict `< cancelTimeDays`
boundary by a day at the edge. Confidence HIGH: confirmed by direct reads of both OLD and
NEW source and by the absence of any per-tenant timezone field.
