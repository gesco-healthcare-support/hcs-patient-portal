## 1. ID & Summary
OLD had a one-click "Export to Excel" button on the Appointment Request Report
toolbar that downloaded the report rows (header + data) as a real `.xlsx`
spreadsheet. NEW has no such capability and no xlsx writer at all. Excluded
because it rides on G-08-01 (the report grid itself), which does not yet exist;
revisit only if a billing / offline-analysis need surfaces.

## 2. Docs Reviewed
- docs/parity-v2/08-reporting-export.md -- section "### G-08-02" (OLD/NEW blocks,
  inventory row, Equivalent note 1: ClosedXML -> any xlsx writer / CSV).
- docs/parity-review-log.csv -- row 61 (decision=exclude, the revisit note).

- Actual (NEW): no report grid (G-08-01) and no export path -> no spreadsheet

## 6. Why Excluded + Revisit-If
Excluded per CSV row 61: the xlsx export is a child capability of G-08-01 (the
Appointment Request Report grid). Until that report exists there is nothing to
export, so building xlsx now is premature. The library choice is explicitly NOT
load-bearing (parity doc Equivalent note 1: ClosedXML / EPPlus / even CSV all
satisfy the outcome "user downloads the report as a spreadsheet").

Revisit if:
- Billing / management / external hand-off needs the appointment list as a
  spreadsheet for offline sort/filter/pivot, OR
- A downstream process expects an `.xlsx`, OR
- G-08-01 ships and Excel export is the natural next slice (parity doc says it
  "ships with G-08-01").

Before building, resolve the HIPAA decision (CSV note + parity doc Open Q 3):
the report includes full SSN and DOB; the exported file must mask SSN (and
possibly DOB). Also do NOT port the two OLD bugs in this controller -- the
auth/authz bypass (`AllowedApis.cs:22,36`) and the process-wide static export
cache + named temp files (`CSVExportController.cs:31-32`). Render server-side
per-request, streamed, behind a `CaseEvaluation.Reports` permission.

## 7. Candidate Solution
None -- excluded. (If revisited: server-side xlsx/CSV render from the G-08-01
paged query, PHI-masked, behind authentication + Reports permission.)

## 8. Decision Rationale + confidence
Confidence: high. The exclude rationale is fully supported by static evidence:
the OLD path is real (verified ClosedXML xlsx write + stream), NEW is genuinely
empty (verified no xlsx lib / endpoint), and the dependency on G-08-01 is
structurally correct (no grid -> nothing to export). Effort tagged M for the
work-if-revisited (xlsx/CSV writer + PHI masking + auth, ~one slice once the
report exists). This is a deferral of timing, not a disagreement with keeping
the capability eventually (the parity doc marks it "Keep in NEW -- Yes").
