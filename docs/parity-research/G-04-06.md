## 1. ID & Summary

OLD CC'd a global clinic-staff mailbox (`ServerSetting.clinicStaffEmail`) on
every Approved status email. The parity-v2 doc (and the CSV row) record this
as "not reproduced" in NEW and recommend wiring `CcRecipientAppender` into the
approval dispatch.

KEY FINDING: the gap is ALREADY CLOSED in the current code. The Approved path
in `StatusChangeEmailHandler` calls `_ccAppender.AppendAsync(...)` on BOTH the
stakeholder leg (`:268`) and the responsible-user leg (`:322`). Git blame dates
the CC wiring to commit `da459ed4` (2026-05-12). The parity-v2 doc was written
later (commit `3c82ff2`) but still quotes a since-removed XML summary comment
(`:46-47`) that said "not yet reproduced." This is parity-doc drift: the
recommended fix had already shipped when the gap was catalogued.

One residual nuance worth a flag (NOT specific to this item): NEW does not emit
a true SMTP `CC:` header. The dispatcher fans out one separate email per
recipient (each appended CC address becomes its own `To:`). The clinic-staff
mailbox still receives the approval email, so the user-visible parity intent
(clinic staff are notified of approvals) is satisfied; only the delivery
mechanism (separate To vs. shared-message CC header) differs.

  This is really a NotificationDispatcher-wide concern, not a G-04-06 fix.
