## 1. ID & Summary

OLD lets an internal user cancel an already-Approved appointment in ONE step
from the appointment detail/edit screen: a "Cancel Appointment" button opens
the view-page modal pre-set to `CancelledNoBill`, the user types a cancellation
reason, submits, and the appointment is written straight to `CancelledNoBill`
(plus an Accepted `AppointmentChangeRequest` audit row is auto-created and the
slot is freed). NO patient/attorney change-request is required.

NEW has no such path. Cancellation is reachable ONLY through the two-step
external change-request lifecycle (`RequestCancellationAsync` submit ->
supervisor `ApproveCancellationAsync`). The state machine has no direct
`Approved -> CancelledNoBill` edge, no AppService method writes a terminal
cancel status outside change-request approval, and the NEW frontend exposes no
cancellation UI of any kind (not even the external submit form). Gap confirmed
statically; classified Missing; disposition include.

### Upstream callers

- View page is the OLD trigger surface; in NEW the equivalent surface is
  `appointment-view.component` (the only place office actions are dispatched).
  A direct-cancel button would be added to its action toolbar (currently
  Approve/Reject only).
- `AppointmentApprovalController` (`api/app/appointment-approvals`,
  `{id}/approve`, `{id}/reject`) is the natural HTTP home for a `{id}/cancel`
  sibling endpoint; it delegates to `IAppointmentApprovalAppService`.

### Downstream consumers

- `AppointmentStatusChangedEto` (published by `ApproveCancellationAsync`
  146-154) drives the slot cascade (the five terminal cancel/reschedule
  statuses free capacity without a status flip per Appointments CLAUDE.md
  business-rule 4). A direct-cancel must publish the same Eto so the slot
  frees consistently.
- `Notifications.Events.AppointmentChangeRequestApprovedEto` (157-167) feeds
  the cancellation email handler. A direct cancel would need an equivalent
  notification fan-out (or to reuse this Eto) to match OLD's `SendEmail`
  status mail -- subject to Phase 1 email scope (verification +
  requested + approved/rejected are the only active types; a direct-cancel
  mail may be out of Phase 1 scope and can be gated like the others).
- Audit: OLD wrote an Accepted `AppointmentChangeRequest` audit row. NEW's
  equivalent audit is ABP `[Audited]` on `Appointment` + the change-request
  table; a direct cancel that does NOT create a change-request row would
  still be audited on the Appointment status property, so the OLD audit-row
  side effect is not strictly required for parity (the change-request row in
  OLD was a workaround to reuse the cancel pipeline; NEW audits natively).

### Blast radius

A fix touches ~5-7 files across 4 layers (medium):

1. Domain: add an `Approved -> CancelledNoBill` (and optionally `-> CancelledLate`)
   permit + a `DirectCancel` trigger in `AppointmentTransitionTrigger` +
   `AppointmentManager` (state machine + a public `DirectCancelAsync` that

5. Permissions: reuse `Appointments.Edit` or add a dedicated cancel permission.
6. Angular: add a `'cancel'` action to `TransitionAction`, surface it in

Regression risk: LOW-MEDIUM. The new transition is additive (a new edge +
trigger); existing Approve/Reject/RequestCancellation paths are untouched.
The main risk is the slot-cascade Eto and capacity model: the direct cancel
must emit `AppointmentStatusChangedEto` with the right from/to so a Booked
slot frees exactly as the change-request path does -- skipping it would orphan
capacity. Email scope must be respected (gate any new mail like the other
Phase-1-deferred handlers).

## 4. Expected vs Actual

- Expected (OLD parity): An internal user viewing an Approved appointment can
  click "Cancel Appointment", enter a cancellation reason, and the appointment
  becomes a terminal cancelled status in ONE action; the slot frees. No
  patient/attorney request is needed.
- Actual (NEW): The Approved view page offers no cancel control. The only way
  to reach `CancelledNoBill`/`CancelledLate` is for a Cancel change request to
  be submitted (no UI exists for that either) and then supervisor-approved.
  Internal staff cannot directly cancel.

## 7. Candidate Solutions

### Option A -- Direct-cancel transition + approval-service method + view-page modal (port) [effort: M] [risk: low-med]

One-line: Add an additive `Approved -> CancelledNoBill/CancelledLate` edge and
a `DirectCancelAppointmentAsync` AppService method (mirroring the existing
Approve/Reject surface), plus a cancel modal on the view page.

- Source: OLD code `AppointmentDomain.cs:532-611` (slot free + audit) and the
  trigger `appointment-edit.component.ts:881-893` + view modal HTML 136-139.
  NEW pattern to clone: `AppointmentApprovalAppService.Approval.cs` (Approve/
  Reject), `AppointmentApprovalController` `{id}/approve`, and the
  `approve-confirmation-modal`/`reject-appointment-modal` components.
- Mechanics: new `AppointmentTransitionTrigger.DirectCancel` (+ optional
  `DirectCancelLate`); `BuildMachine` permits it from `Approved`;
  `AppointmentManager.DirectCancelAsync` stamps `CancellationReason` +
  `CancelledById`, transitions, publishes `AppointmentStatusChangedEto` so the
  slot frees identically to `ApproveCancellationAsync`. Reuse G-02-06's manual
  NoBill/Late outcome choice on the input DTO + a `ChangeRequestApprovalValidator`-
  style outcome guard. Do NOT auto-create an `AppointmentChangeRequest` row
  (OLD did this only to reuse its pipeline; NEW audits the Appointment natively
  via `[Audited]`, so the row is redundant -- a justified deviation under the
  bug/deviation policy).
- Tradeoffs: cleanest parity; matches the existing approval-service shape; no
  fragile two-step kludge. Slightly more surface than a UI-only hack but it is
  the correct domain placement (business rule in the domain service per branch
  CLAUDE.md).
- Risk: must emit the slot Eto (capacity model); must respect Phase 1 email
  scope for any cancellation mail; confirm whether a direct cancel should offer
  NoBill only (OLD's direct path always wrote CancelledNoBill) or both
  NoBill/Late. OLD `cancelRequest()` hardcoded `CancelledNoBill`; the CSV note
  says "reason + NoBill/Late", so confirm with Adrian (see Open Questions).

### Option B -- Reuse the change-request pipeline server-side (submit-then-auto-approve in one call) [effort: M] [risk: med]

One-line: A new AppService method that internally creates a Cancel
`AppointmentChangeRequest` and immediately calls the existing
`ApproveCancellationAsync` logic, exposed as one endpoint + one view-page modal.

- Source: reuses `SubmitCancellationAsync` + `ApproveCancellationAsync`
  verbatim; closest to OLD's literal behavior (OLD did create an Accepted
  change-request row).
- Tradeoffs: maximal reuse of validated cancel logic and produces the same
  change-request audit row OLD had. But it threads a user-facing one-step
  action through a two-step machine, mixing the "request" and "confirm"
  semantics; the submit-side cancel-window gate (`IsWithinNoCancelWindow`)
  would fire and could block a staff cancel that OLD allowed unconditionally.
- Risk: medium -- the submit-time window gate is an unwanted side effect for a
  staff direct cancel (OLD's direct path had no such gate); would need a bypass
  flag, re-introducing branching the state machine was meant to remove.

Recommendation: Option A. It places the rule in the domain service (branch
convention), is additive/low-regression, and avoids dragging the submit-time
cancel-window gate into a staff action OLD never gated.

## 8. Root-Cause Hypothesis

The NEW build implemented cancellation as the EXTERNAL two-step change-request
lifecycle only (Phases 15-18: submit + supervisor approve), and Wave-1
appointment transitions deliberately exposed only Approve/Reject/Reschedule
(`AppointmentTransitionTrigger` doc comment: "Cancel ... triggers are
configured in the graph but not exposed as endpoints until Wave 3"). The OLD
single-step internal-staff cancel (a separate code path in `AppointmentDomain.
Update`, launched from the edit page's `cancelRequest()`) was simply never
ported as its own surface -- it fell through the cracks between the approval
service (approve/reject only) and the change-request service (external
request/approve). The state machine has the destination states but no direct
edge, and the frontend has no cancellation UI at all. Confidence: high --
every layer independently confirms the absence.

## 9. Open Questions

- Outcome set: OLD's direct internal cancel (`cancelRequest()`) always wrote
  `CancelledNoBill`. The CSV planning note says "reason + NoBill/Late". Should
  the NEW direct cancel offer both outcomes (matching G-02-06's supervisor
  choice) or stay NoBill-only to match OLD's literal direct path? Recommend
  offering both for consistency with G-02-06, but confirm with Adrian.
- Cancel-window gate: should a staff direct cancel respect the
  per-AppointmentType cancel-time window (`IsWithinNoCancelWindow`)? OLD's
  direct path did NOT gate it (the gate lived only on the external submit
  path, G-02-06). Recommend no gate on the staff direct cancel.
- Permission: reuse `Appointments.Edit`, or add a dedicated
  `Appointments.Cancel` / `AppointmentChangeRequests`-style permission? OLD
  gated by internal-user role only.
- Email: is a cancellation notification in scope for Phase 1? Per the email-
  scope memory, only verification + requested + approved/rejected are active;
  a direct-cancel mail likely defers and should be gated like the others.
- Audit row: confirm the OLD Accepted `AppointmentChangeRequest` audit row is
  intentionally dropped in NEW (native `[Audited]` covers it) rather than
  replicated.
