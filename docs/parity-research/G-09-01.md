## 1. ID & Summary

OLD's internal-staff dashboard had four "headcount" cards in its third row --
Patient, Claim Examiner (Adjuster), Applicant Attorney (PatientAttorney), and
Defense Attorney -- each showing how many external users of that role are
registered, and each clicking through to a role-filtered user list
(`/users?userRoleTypeId=N`). NEW's redesigned 13-card dashboard has NO
external-role headcount card and NO equivalent role-filtered user-list
click-through. The four counters were dropped wholesale in the redesign.

This is a clean MISSING-behavior gap. The disposition is **include**: re-add
per-role registered-user counts with click-through. The work is straightforward
but spans backend (a new count source over ABP Identity, not the appointment
repository) and frontend (cards + a click-through target). Effort M, not S,
because (a) counting users by role is NOT a built-in ABP `GetCountAsync`
overload, and (b) NEW has no `/users?userRoleTypeId=N`-style list -- it uses
ABP's stock Identity module UI, so the click-through target has to be decided.

### Upstream callers

- Backend: `DashboardAppService.GetAsync()` is the single entry point
  (`HttpApi/Controllers/Dashboards/DashboardController.cs` -> `GET /api/app/dashboard`).
  A fix adds fields to `DashboardCountersDto` and a count source inside
  `BuildAsync`; no new endpoint is required (the existing single GET carries
  the extra fields).
- Frontend: `DashboardComponent` (`angular/src/app/dashboard/dashboard.component.ts:9-10`)
  permission-switches between `HostDashboardComponent` and
  `TenantDashboardComponent`. The cards live in the tenant variant (and
  possibly host); a fix renders four more cards there. No router changes for
  the cards themselves -- only for the click-through target.

### Downstream consumers

- The Angular proxy (`angular/src/app/proxy/`) is regenerated from the DTO; new
  DTO fields propagate automatically via `abp generate-proxy` (do NOT hand-edit).
- Click-through TARGET: OLD navigated to `/users?userRoleTypeId=N`. NEW has NO
  such custom user list. NEW uses ABP's stock Identity module mounted at
  `/identity` (`app.routes.ts:52-55`, `@volo/abp.ng.identity`). The Identity Pro
  UI does not accept a simple `?userRoleTypeId=N` query param to pre-filter by
  role the way OLD's custom `/users` page did. So the click-through is the
  harder half: either (a) deep-link into `/identity/users` and accept it lands
  unfiltered, (b) deep-link with whatever filter param the Identity Pro list
  supports (needs verification -- Open Question 3), or (c) build a small
  role-filtered user-list page in NEW. There is an `external-users` feature
  folder and a `/users/invite` route (`app.routes.ts:143-151`) but NO read-only
  external-user list to deep-link to.

### Blast radius

- Files a minimal fix touches: 2 backend
  (`DashboardCountersDto.cs`, `DashboardAppService.cs`) + 1 EF (a new
  `IIdentityUserRepository`-based count or a custom count method) + 2 frontend
  (`tenant-dashboard.component.{ts,html}`; optionally `host-dashboard` too) +
  regenerated proxy + `en.json` localization keys for the four card labels.
  Roughly 3-4 layers, 5-7 hand-edited files.
- Regression risk: LOW. Additive only -- new DTO fields default to 0 for
  existing callers; new cards are independent of the existing appointment
  counts. The one cross-cutting concern is multi-tenancy: a Tenant-scoped
  dashboard must count users in the CURRENT tenant (ABP's `IMultiTenant`
  filter handles this automatically), while a Host dashboard must count
  cross-tenant via `_dataFilter.Disable<IMultiTenant>()` -- the same pattern
  `GetHostCountersAsync` already uses (`DashboardAppService.cs:59-65`). Getting
  the host/tenant scope wrong would leak or undercount across tenants, so the
  count must thread through the existing `scopedToTenant` switch.
- HIPAA note: a per-role COUNT is an aggregate with no individual identifier
  attached -- not PHI under Safe Harbor (the DTO doc already makes this
  argument for the appointment counts, `DashboardCountersDto.cs:8-11`).

## 7. Candidate Solutions

### Option A -- Port the four cards, count via ABP Identity, click-through into the stock Identity user list [effort: M] [risk: low]
- One-line: Add four role-count fields to `DashboardCountersDto`; in
  `DashboardAppService.BuildAsync`, count users per role through ABP Identity
  (threading the existing `scopedToTenant`/host-filter switch); render four
  cards that deep-link to `/identity/users`.
- Mechanics (backend): ABP's `IIdentityUserRepository.GetCountAsync(filter)`
  filters only by search string, NOT by role -- there is no built-in count-by-role
  overload (verified, ABP API docs below). So either (a) resolve each role then
  `(await _userRepository.GetListByNormalizedRoleNameAsync(role.NormalizedName)).Count`,
  or (b) add a custom repository count over the `IdentityUserRole` join table
  (`LongCountAsync(set<IdentityUserRole>().Where(ur => ur.RoleId == roleId))`)
  for efficiency at scale. The four NEW roles are "Patient", "Claim Examiner",
  "Applicant Attorney", "Defense Attorney" (role-name strings already used in
  `AppointmentsAppService.cs:161` and `BookingFlowRoles.cs`). Host scope counts
  cross-tenant under the existing `_dataFilter.Disable<IMultiTenant>()`; tenant
  scope counts within the current tenant via the default filter.
- Click-through: OLD's `/users?userRoleTypeId=N` has no NEW equivalent; deep-link
  to `/identity/users` (lands the staffer on the stock Identity user list). If
  the Identity Pro list accepts a role pre-filter param, pass it (Open Q 3);
  otherwise the card click lands unfiltered -- still better than no jump.
- Source: OLD code (`dashboard.component.{ts,html}`); ABP Identity repository
  API (URL below).
- Tradeoffs: faithful to OLD intent; additive/low-risk. The click-through is
  weaker than OLD's exact role-filtered list unless the Identity UI supports a
  role filter param. `GetListByNormalizedRoleNameAsync(...).Count` materializes
  the user list (fine for a single-office tenant; prefer the join-count for the
  cross-tenant host view).
- Risk: low. Multi-tenancy scope is the only sharp edge; reuse the existing
  host/tenant switch.

### Option B -- Port the cards + build a dedicated role-filtered external-user list page as the click-through target [effort: L] [risk: medium]
- One-line: Same backend as Option A, but instead of deep-linking into ABP's
  Identity UI, add a small read-only external-user list (filterable by role) in
  NEW so the click-through reproduces OLD's `/users?userRoleTypeId=N` exactly.
- Source: OLD `/users` list behavior; reuses the `external-users` feature folder.
- Tradeoffs: closest visual+behavioral parity to OLD (a role-filtered roster
  the staffer lands on); but it is a whole new list feature (AppService + DTO +
  permission + Angular list component + route), which overlaps the eventual
  external-user-management surface. Larger surface = more to test (sort/page,
  permission gating, tenant scoping).
- Risk: medium -- new endpoint + list UI + permission gating; more regression
  surface than Option A.

### Option C -- Defer the click-through; ship the four counts only [effort: S] [risk: low]
- One-line: Add the four counts + cards (Option A backend) but make the cards
  non-clickable for now, matching the current STUB-card pattern where placeholder
  cards are not links (`tenant-dashboard.component.html`).
- Source: OLD code for the counts; NEW's existing non-clickable card pattern.
- Tradeoffs: restores the at-a-glance census cheaply and removes the most
  visible part of the gap; defers the click-through (which is really G-09-06's
  territory) until the external-user list question is settled. Partial parity.
- Risk: low. The cosmetic risk is a card that looks clickable but is not --
  mitigate by styling it like the other non-link cards.

Recommendation: Option A is the right default -- it restores both the counts
and a (best-effort) click-through with low risk and modest effort, and reuses
the host/tenant scoping that already exists. Escalate to Option B only if the
owner wants the exact OLD role-filtered roster as the landing target.

## 8. Root-Cause Hypothesis

The NEW dashboard was a deliberate ground-up redesign (audit doc key finding
#1), reoriented around an appointment-status + multi-tenant operational view
(pending/approved/rejected, doctors, tenants, legal-deadline) rather than OLD's
mix of status counts + user-roster counts. In that redesign the four external-
role headcounts were simply not carried over: the new `DashboardCountersDto` and
`DashboardAppService` were built to count over the appointment/doctor/tenant
repositories and never wired an Identity-user count source. There is no
intentional suppression in the code -- it is an omission of scope during the
redesign, plausibly because (a) the source-of-truth `spDashboardCounters` body
was never in the repo to copy from, and (b) the equivalent "registered users by
role" data lives in ABP Identity, which the dashboard service does not touch.
Confidence: HIGH -- both code bases were read in full at the cited lines and the
absence is unambiguous; the only unverifiable detail is the OLD proc's internal
SQL (Open Q 1), which does not affect the conclusion that the four counts exist
in OLD and are absent in NEW.

## 9. Open Questions

1. Exact OLD count semantics. `spDashboardCounters` is only a `WHERE 1=0` stub
   in the repo; the production proc that computed Patient/Adjuster/
   PatientAttorney/DefenseAttorney is not available. The bindings + labels +
   click-through filters confirm "count of registered users in role N", but
   whether the proc excluded soft-deleted/inactive users, or scoped by tenant
   (OLD was single-tenant, so likely not), is inferred. Confidence MEDIUM on the
   internal filter; HIGH on the headline behavior.
2. Owner decision (audit Open Q 2): are the four role-headcount cards wanted on
   the NEW staff dashboard, or are they redundant now that NEW has an Identity
   user-management UI? The disposition is "include", so assume yes, but confirm
   placement (tenant dashboard only, or host too).
3. Click-through target. Does ABP's stock Identity Pro user list
   (`/identity/users`) support a role pre-filter via query param so a card click
   can land role-filtered (matching OLD's `/users?userRoleTypeId=N`)? If not,
   decide between deep-linking unfiltered (Option A) vs building a dedicated
   role-filtered list (Option B). Needs verification against the installed
   `@volo/abp.ng.identity` version.
4. Role mapping. OLD used `Adjuster` (enum 5) labelled "Claim Examiner"; NEW has
   both a `ClaimExaminer` role and the locked-memory position that "Claim
   Examiner is metadata not a role". Confirm which NEW role name the
   Claim-Examiner card should count ("Claim Examiner" string is the one used in
   `AppointmentsAppService.cs:161` / `BookingFlowRoles.cs:101`).
