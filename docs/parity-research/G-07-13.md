## 1. ID & Summary

OLD modeled three master-data entities (`AppointmentType`, `Location`,
`WcabOffice`) with a tri-state `StatusId` enum (`Status { Active=1, InActive=2,
Delete=3 }`). NEW collapsed that to a simple `IsActive` bool plus ABP
`FullAuditedAggregateRoot.IsDeleted` soft-delete.

The collapse is correct and parity-equivalent for `Location` and `WcabOffice`:
both NEW entities carry `IsActive`, both expose it in their create/edit modal,
and the OLD `Status` enum had no states beyond Active/InActive (the only third
value, `Delete=3`, is a tombstone now covered by ABP soft-delete).

The single real gap is `AppointmentType`: in NEW it has **no active toggle at
all** -- `FullAuditedEntity<Guid>` with only `Name` + `Description`. OLD's
`AppointmentType.StatusId` could be set to InActive, so OLD could deactivate an
appointment type; NEW cannot. The decision is to **restore an `IsActive` toggle
on `AppointmentType`** to match the `Location`/`WcabOffice` pattern (keeping ABP
soft-delete for the OLD `Delete` state).

Important nuance found during verification: the OLD `AppointmentType` add/edit
Angular pages were **never built** -- both `appointment-type-add.component.html`
and `appointment-type-edit.component.html` are stub `<h1>AppointmentType</h1>`
pages. So in practice OLD had no UI to toggle `AppointmentType.StatusId` either;
it defaulted server-side and could only be changed at the DB/seed level. The gap
is therefore "schema + intent" (the deactivate capability existed in the data
model and in the booking lookup view) rather than "OLD had a working screen we
dropped." This makes the fix parity-PLUS for the UI, parity-restore for the
schema.

## 4. Expected vs Actual

Expected (OLD-parity intent): an appointment type can be deactivated so it stops
appearing as a selectable option, without hard-deleting it (which OLD's
`RegisterDeleted` did, and which FKs from `Appointment`/`DoctorAvailability`
would block anyway). OLD encoded this via `AppointmentType.StatusId = InActive`,
and the booking lookup view (`vAppointmentTypeLookUps`) carried `StatusId`.

Actual (NEW): `AppointmentType` has no active concept. Every existing type is
permanently "on"; the only way to remove one is ABP soft-delete (`IsDeleted`),
which is all-or-nothing and not exposed as a user-facing "deactivate." Location
and WcabOffice already have the `IsActive` toggle; AppointmentType is the lone
holdout, producing an inconsistency within the same master-data cluster.

## 5. Repro Status

`not-reproduced-static-confirmed`. The gap is a missing field/column, fully

## 7. Candidate Solutions

### Option A -- Add `IsActive` to AppointmentType, mirroring Location/WcabOffice [effort: M] [risk: low-medium]

One-line: add a non-nullable `IsActive` bool to the `AppointmentType` entity +
DTOs + manager + a column migration (default `true`) + an Angular checkbox, with
the lookup active-filter deferred behind the booking-parity decision.

- Source: OLD code -- `AppointmentType.cs:40-42` (`StatusId` exists) +
  `Location.cs:72-74` / `Location` NEW pattern as the canonical NEW shape to
  copy. This is a pure port-by-analogy: Location/WcabOffice already prove the
  exact `IsActive` pattern in this codebase.
- Steps: (1) entity prop + constructor param; (2) `LocationCreateDto`-style
  `IsActive { get; set; } = true;` on Create DTO, `IsActive` on Update + read
  DTO; (3) thread through `AppointmentTypeManager` + `AppointmentTypesAppService`;
  (4) EF migration `Added_AppointmentType_IsActive` with `defaultValue: true`;
  (5) Angular checkbox in the detail modal + form control; (6)
  `abp generate-proxy`.
- Tradeoffs: matches the existing intra-cluster pattern exactly (low cognitive
  load, consistent admin UX). The only judgment call is whether booking lookups
  should filter on `IsActive`; recommend adding the column + admin toggle now and
  the lookup filter as a follow-up gated by Open Question 1.
- Risk: migration must default existing rows to `true` or they disappear from
  any future active-filtered lookup. Non-nullable bool with default is the safe
  shape (matches how Location was added).
- Approach flag (per RPE): `tdd` for the manager/AppService active-state logic
  if a lookup filter is added; `code` for the pure column/DTO/UI plumbing.

### Option B -- Reuse ABP soft-delete (`IsDeleted`) as the deactivate mechanism, no new column [effort: S] [risk: medium-high]

One-line: treat ABP's `IsDeleted` (already on `FullAuditedEntity`) as
"inactive," exposing a soft-delete/restore toggle instead of a separate
`IsActive`.

- Source: ABP `FullAuditedEntity` already provides `IsDeleted` + the automatic
  `ISoftDelete` query filter (ABP docs, https://abp.io/docs/latest/framework/architecture/domain-driven-design/entities
  and data-filtering docs; official, accessed 2026-06-01).
- Tradeoffs: zero schema change. BUT it conflates two distinct OLD states:
  `InActive` (still exists, just hidden from new bookings) vs `Delete`
  (tombstoned). OLD modeled them separately; ABP's soft-delete filter hides
  `IsDeleted` rows from ALL queries by default, which would also break
  historical reads of appointments that reference a "deactivated" type unless
  the filter is disabled per query. Mismatched semantics -> higher risk.
- Risk: medium-high -- diverges from the Location/WcabOffice pattern, introduces
  global query-filter behavior, and loses the active/deleted distinction OLD
  had. Not recommended; documented to show it was considered and rejected.

Recommendation: **Option A.** It is the OLD-code-first, intra-codebase-consistent
choice and is what the CSV note prescribes ("Keep ABP soft-delete; restore
active/inactive toggle"). Option B is rejected for conflating InActive with
Delete.

## 8. Root-Cause Hypothesis

Confidence: HIGH. When the master-data cluster was scaffolded on ABP, the OLD
tri-state `Status` enum was deliberately and correctly simplified to `IsActive`
(active/inactive) + ABP soft-delete (the `Delete` tombstone). Location and
WcabOffice received the `IsActive` bool; AppointmentType did not -- almost
certainly because OLD's AppointmentType admin screen was a non-functional stub
(`<h1>` only), so the scaffolder (or its human reviewer) saw no working
status control to replicate and generated AppointmentType as a bare
Name+Description lookup. The deactivate capability lived only in the schema
(`AppointmentType.StatusId`) and the booking lookup view
(`vAppointmentTypeLookUps.StatusId`), both of which are easy to miss when
porting from the (empty) UI rather than the data model. Result: an unintended
intra-cluster inconsistency, not a considered decision to drop the capability.
