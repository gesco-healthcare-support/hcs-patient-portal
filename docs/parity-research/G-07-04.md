## 1. ID & Summary

OLD slot-generation conflict detection runs FOUR discrete predicates per generated
slot (containment, exact-duplicate, range-overlap, booked/reserved-match) in TWO
separate passes with DIFFERENT location scoping:

- A BLOCKING pass (`AddValidation`) -- all 4 checks scoped to `LocationId`; any hit
  rejects the entire batch (this all-or-nothing policy is G-07-03's concern).
- A DISPLAY-ONLY pass inside `GenerateDoctorsAvailabilityByDays` -- the SAME 4 checks
  but with the `LocationId` filter REMOVED on 3 of them, producing the non-blocking
  advisory string "TimeSlot Already Exist in the System for different location" on the
  returned preview view-model.

NEW collapses all of this into ONE half-open interval test
(`x.FromTime < slot.ToTime && x.ToTime > slot.FromTime`), always scoped to
`LocationId` + date, classifying only Reserved-overlap vs other-overlap for the message.

Net behavioral delta (CSV-flagged crux): NEW DROPS OLD's cross-location advisory --
the warning that the same doctor is double-booked across two different locations on the
same date/time. The CSV note flags this as the priority item ("esp. cross-location
overlap warning ... relevant under one-doctor-per-tenant"). The single-interval math
itself is a net improvement (catches partial-edge overlaps OLD's `==` and contained-in
checks miss), so the gap is specifically the missing cross-location advisory, not the
overlap math.

Verdict from the parity doc: keep NEW's interval test; flag the dropped cross-location
advisory for Adrian (likely an OLD artifact, not a real requirement). This research
confirms that disposition and adds the architectural reason the advisory may be moot.

## 4. Expected vs Actual

Expected (OLD ground truth):
- Same-location overlap on generate -> hard `BadRequest`, whole batch rejected
  (G-07-03 owns this), via 4 location-scoped discrete checks.
- A passing batch that nonetheless overlaps the SAME doctor's slots at a DIFFERENT
  location on the same date/time -> non-blocking advisory string shown on the preview
  ("TimeSlot Already Exist in the System for different location").

Actual (NEW):
- Same-location overlap -> per-slot `IsConflict`, skipped on insert, partial success
  (G-07-03 deviation, dispositioned separately).
- One half-open interval test, same-location only. No cross-location advisory at all.
- Even the same-location advisory message NEW computes (`GenerationConflictExists/
  Reserved`) is not rendered by the current Angular component (boolean-only UI).

Edge-case math delta (not a regression, an improvement): OLD's checks 1+2
(containment + exact `==`) miss partial-edge overlaps; OLD's check 3
(`TimeSlotValidation`) is inclusive on boundaries (`>=`/`<=`) so back-to-back slots
touching at an endpoint count as overlap. NEW's half-open test
(`<` / `>`) treats touching endpoints as NON-overlapping (correct for adjacent slots)
and catches all true partial overlaps. NEW is cleaner and more correct same-location.

## 7. Candidate Solutions

### Option A -- Keep NEW interval math; add a non-blocking cross-location advisory [effort: M] [risk: low-med]

Port ONLY the OLD cross-location advisory, not the 4-check structure. In
`GeneratePreviewAsync`, after the existing same-location flagging, run a second overlap
query that drops the `LocationId` filter (same doctor/tenant, same date, half-open
overlap) and, on a hit, set an ADVISORY-ONLY field on the per-day preview DTO (new
`CrossLocationWarning` string + localization key). Crucially do NOT set `IsConflict`
(keeps G-07-03 partial-insert intact). Surface it in the Angular generate component
(which today ignores even the existing per-day message).

- Source: OLD code
  `DoctorsAvailabilityDomain.cs:408-445` (the display-only pass + its advisory strings)
  and controller `DoctorsAvailabilitiesController.cs:39` (advisory is on the preview,
  non-blocking).
- Tradeoffs: faithful to OLD's user-visible advisory under one-doctor-per-tenant;
  keeps NEW's superior same-location math. Adds one query per generate-preview over the
  date span (small).
- Risk: the OLD advisory only makes sense when one doctor maps to multiple locations.
  Under Phase 1 (tenant IS the doctor) this holds; if Phase 2 introduces multiple
  doctors per tenant, the warning needs to filter by doctor/availability owner or it
  will mis-fire (warn about a DIFFERENT doctor's slot). Mitigate by scoping the
  cross-location query to the same availability owner once a doctor dimension exists.

### Option B -- Do not port the advisory; document the deliberate drop [effort: S] [risk: low]

Treat the cross-location advisory as an OLD artifact (the parity doc's stated leaning:
"likely an OLD artifact, not a real requirement"). Keep NEW as-is. Record the decision
(close OBS-26 with rationale; add a one-line note to the parity flags). Optionally,
separately, fix the smaller latent issue that the Angular component never surfaces the
EXISTING per-day same-location message.

- Source: OLD code (advisory is non-blocking and exists only in the display pass --
  weak evidence it was load-bearing), `docs/parity-v2/07-admin-master-data.md:158-159`.
- Tradeoffs: zero backend blast radius; loses a (marginal) operator nicety. Defensible
  because OLD never BLOCKED on cross-location and the message text reads like a
  developer placeholder.
- Risk: if an office genuinely runs one doctor across multiple locations and relied on
  the warning, NEW silently allows the double-book. Low likelihood; the operator would
  still see the slots when picking each location.

### Option C -- Port OLD's 4 discrete checks verbatim (both passes) [effort: L] [risk: med]

Replicate OLD literally: 4 predicates x 2 passes (blocking location-scoped + display
cross-location).

- Source: OLD code `DoctorsAvailabilityDomain.cs:46-99` and `:400-445`.
- Tradeoffs: maximal literal parity. But it RE-INTRODUCES OLD's weaker math
  (misses partial overlaps; inclusive-boundary false positives on adjacent slots) and
  conflicts with G-07-03's already-shipped partial-insert policy. Net negative.
- Risk: regresses correctness and contradicts the disposition that NEW's interval test
  is the keeper. NOT recommended.

Recommendation: Option A if the cross-location advisory is confirmed useful under the
one-doctor-per-tenant model; Option B (with the small UI message fix) if Adrian agrees
it is an OLD artifact. Both are cheaper and safer than C.

## 8. Root-Cause Hypothesis

The NEW slot-generation rework (2026-05-15 plan series) intentionally re-designed
conflict detection around one canonical half-open interval test (the modern, correct
overlap primitive) and a partial-insert UX, superseding OLD's accreted 4-predicate /
2-pass structure. In doing so it carried forward ONLY the blocking, same-location
semantics and dropped OLD's display-only cross-location advisory -- a non-blocking
warning string that lived solely in the generate-preview view-model. The drop was a
side effect of consolidating to the single interval test scoped to `LocationId`, not a
considered decision, which is why it surfaces now as a parity gap and why OBS-26 raised
the same scoping question independently. Confidence: HIGH -- both code paths verified at
the cited lines, the advisory strings are demonstrably absent from NEW, and the
intent-deviation classification is consistent with the shipped rework plan.
