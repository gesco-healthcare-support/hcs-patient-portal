## 1. ID & Summary
On a re-evaluation (OLD `IsRevolutionForm`) or re-request (OLD `IsReRequestForm`),
the OLD app looked up the source appointment by confirmation number and re-bound the
FULL prior intake (patient, injuries, body parts, attorneys, employer, custom fields)
into the new booking form -- resetting all child PKs to 0 -- so the booker edited a
pre-populated copy. NEW has the server-side gates ported and tested
(`CreateRevalAsync`/`ReSubmitAsync` + status validation + confirmation-number
reuse-vs-fresh), but the Angular booking form has no reval entry path: it only reads
`?type=2` to flip a cosmetic heading flag, always POSTs the plain
`/api/app/appointments` create with a fresh empty form, and never looks up or copies a
source appointment's intake. The reval/re-submit endpoints are unreachable from the UI,
and even if reached they accept a fresh `AppointmentCreateDto` (no server-side
injury/attorney copy).

## 4. Expected vs Actual
- EXPECTED (OLD): Booker picks Re-evaluation, enters/looks up the prior approved (reval) or
  rejected (re-request) confirmation number, and the form loads pre-populated with the prior
  patient, injuries, body parts, attorneys, employer, and custom fields (child PKs reset to 0);
  the booker edits the copy and submits; reval gets a fresh conf#, re-request reuses the
  source conf#.
- ACTUAL (NEW): Re-evaluation only changes the page heading. The booker starts from an empty
  form and must re-enter everything. The form always submits a plain create. The reval/re-submit
  endpoints (which DO carry the correct conf# semantics) are never invoked from the UI, and they
  would not copy intake even if called.

## 7. Candidate Solutions

### Option A -- Frontend reval entry + prefill from GetByConfirmationNumber [effort: L] [risk: med]
- Description: Add a source-confirmation-number input + "Load" action to the reval/re-submit
  entry path; on load call `getByConfirmationNumber`, validate source status client-side
  (defensive), and map the returned nav-props DTO into the form, `injuryDrafts`, and the
  employer/attorney/accessor sections with all child PKs reset to 0; switch `onSubmit` to call
  `createReval`/`reSubmit` (passing the source conf#) when in reval/resubmit mode.
- Source: OLD code: `appointment-add.component.ts:298-374` + `appointment-add.component.ts:542-567`
  + `appointment.domain.ts:325-417` (the authoritative copy-then-reset-PKs logic to port);
  NEW read source `AppointmentsAppService.cs:283-293` + `AppointmentWithNavigationPropertiesDto.cs:17-38`.
- Tradeoffs: Mirrors OLD trust model (server gate is the real guard, server-validated on submit
  via `LoadReval/ResubmitSourceAsync`). Most logic is a frontend port; the heavy part is the
  DTO->form mapping and PK-reset, which must align with NEW's dedup / per-injury-duplicate /
  per-type validators. Does not harden the endpoints against a bare-DTO client.

### Option B -- Option A plus server-side intake copy in CreateAppointmentInternalAsync [effort: XL] [risk: med]
- Description: In addition to A, copy the source appointment's child intake (injuries, attorneys,
  employer, accessors, custom fields) server-side inside `CreateAppointmentInternalAsync` when a
  `sourceConfirmationNumber` is present, so the reval/resubmit endpoints produce a correct copy
  even if the client sends a sparse DTO.
- Source: OLD code: `AppointmentDomain.cs:243-275` (server reuses source data + conf# logic);
  NEW pipeline `AppointmentsAppService.cs:647+`.
- Tradeoffs: Defense-in-depth and keeps the contract honest (the endpoints "copy prior data" by
  name). But it duplicates copy logic across client+server, raises the effort to XL, and risks
  double-creating child rows if the client ALSO sends them. Needs a clear ownership decision
  (client-authoritative vs server-authoritative copy) before building.

### Option C -- Confirm reval is intentionally deferred (no build now) [effort: S] [risk: low]
- Description: Per parity doc open-question line 144 ("needs a running-app check or area-02
  confirmation that reval is intentionally deferred"), confirm with Adrian whether reval-with-prefill
  is in scope for the current parity slice; if deferred, record it and stop.
- Source: `docs/parity-v2/01-booking.md:144`.
- Tradeoffs: Cheapest, but the CSV disposition is `include`, so this is a scope-confirmation
  gate, not a solution. If "include" stands, Option A is the implementation.

## 8. Root-Cause Hypothesis
The reval/re-submit feature was deliberately split: the server-side lifecycle (status gates +
confirmation-number reuse-vs-fresh) was ported and unit-tested in Phase 11g (2026-05-04), but the
Angular booking form was shipped as an MVP that surfaces re-evaluation as a heading-only flag
(`?type=2`) without the source lookup, the DTO->form prefill mapping, or the switch to the reval
endpoints. The author's own comment (`appointment-add.component.ts:180-186`) names the
prior-appointment picker and prefill as an explicit "future enhancement," so this is deferred work,
not a defect introduced by a regression. The read-side data source for a prefill
(`GetByConfirmationNumberAsync` + the nav-props DTO) already exists, which makes the remaining work
a frontend port of OLD's `bindFormGroup` copy-then-reset-PKs logic. -- confidence: high
