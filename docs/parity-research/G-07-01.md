## 1. ID & Summary

OLD shipped six IT-Admin master-data screens (System Parameters edit, Custom
Fields list/add/edit, Notification Templates list/add/edit/delete, Documents
list/add/edit, Document Packages / Package Details, plus the field-config /
field-permission surface). NEW has the **entire backend stack** for all six --
domain entities, AppServices, manual controllers, permissions, and generated
Angular proxy clients -- but **zero Angular feature folders, components, routes,
or menu entries**. Result: an IT Admin cannot change any system parameter, add
a custom intake field, edit an email/SMS template body, or curate the
document/package catalog at runtime. The only way to change this data today is
a DB edit or a re-run of the data seed.

This is a **pure frontend build** (no backend work required for the core CRUD)
and is the single largest UI parity gap in the project. Confirmed via static
inspection: backends + proxies present, frontend absent.

### Upstream callers

There are no upstream UI callers to break -- the screens do not exist yet.
The backend AppServices ARE consumed internally by other features (the booking
flow reads custom-field configs), so adding admin UI does not change any
existing call path; it adds new ones.

### Downstream consumers

Grep across `angular/src/app` (excluding `proxy/`) for the six proxy modules
returns only the booking renderer:
- `appointments/appointment-add.component.{ts,html}`
- `appointments/sections/appointment-add-custom-fields.component.{ts,html}`

These consume `CustomFields` / `AppointmentTypeFieldConfigs` proxies to RENDER
the booking form (read-only consumers), confirming the audit-doc claim: the
only non-proxy consumer is the booking-form renderer, not an admin editor.
Building the admin screens does not touch these files.

Backend downstream consumers of the DATA the admin screens edit (so admin edits
have real effect once screens exist): booking gates / cancel / reschedule / JDF
read `SystemParameter`; the booking form reads `CustomField` +
`AppointmentTypeFieldConfig`; email/SMS handlers resolve `NotificationTemplate`
by code; packet generation reads `Document` + `PackageDetail`.

### Blast radius

- **Layers touched by a fix:** Angular ONLY (feature folders + route provider +
  localization keys + LeptonX menu). Backend, controllers, proxies, permissions
  are already in place. Localization: `Domain.Shared/Localization/CaseEvaluation/
  en.json` needs new `Menu:*` + field-label keys (keys must exist before
  referenced -- root CLAUDE.md).
- **File count (new, not modified):** ~6 feature folders. Mirroring the
  established `appointment-statuses/` shape (abstract+concrete component,
  abstract+concrete service-wrapper, route provider, base routes, detail
  component) that is roughly 8-12 files per entity -> 50-70 new files total.
  Plus edits to `route.provider.ts` (or per-feature route providers wired from
  `app.routes.ts`) and `en.json`.
- **Regression risk:** LOW for existing features (additive; no existing file's
  behavior changes except menu/route registration which is purely additive).
  MEDIUM intra-feature risk because two screens are not plain CRUD: System
  Parameters is a singleton edit form with optimistic-concurrency stamp +
  Range(1,max) validation echoed client-side; Package Details has an M:N
  document-linking sub-UI (`LinkDocuments` / `UnlinkDocument`) and a one-active-
  per-AppointmentType rule; Documents has file upload (multipart -> blob) and a
  "still linked" delete guard; Custom Fields needs the MultipleValues editor
  (G-07-05) and the 7 field types.
- **Sequencing:** independent screens, parallelizable, but System Parameters is
  the smallest (single form, no list) and the best pilot to lock the
  LeptonX/Material + design-token pattern before the M:N and upload screens.

## 7. Candidate Solutions

### Option A -- Port all six OLD screens onto the NEW stack, mirroring the established `appointment-statuses` feature pattern [effort: XL] [risk: low-existing / medium-intra-feature]

- **One-liner:** Build six Angular standalone feature folders (component +
  detail/edit component + service wrapper + route provider), wire each into the
  LeptonX menu via `route.provider.ts` with `permissionGuard` + `requiredPolicy`
  on the existing `.Default/.Create/.Edit/.Delete` permissions, calling the
  already-generated proxy clients. Apply OLD field sets + business-rule echoes;
  style with LeptonX/Material + `docs/design/_design-tokens.md` colors/fonts.
- **Source:** OLD code (authoritative for field sets + rules):
  - System Parameters form fields: OLD
    `system-parameter-edit.component.html:11-69` (+ show the two fields OLD
    commented out at `:43-50`, per audit OLD-bug-3 and CSV note -- confirm with
    Adrian).
  - Custom Fields form: OLD `custom-field-add.component.html:26-52` (+ add
    MultipleValues editor for Picklist/Radio per G-07-05; expose AppointmentType
    that OLD commented out at `:15-24`; add IsActive).
  - Templates list/edit: OLD `template-list.component.html` (+ NEW omits
    Create/Delete per G-07-11 -- edit-only + IsActive toggle).
  - Documents / Package Details: OLD `document-management/` screens.
  - NEW pattern to mirror: `angular/src/app/appointment-statuses/
    appointment-status/` (abstract+concrete component, providers, services).
- **Tradeoffs:** Largest effort item; but no backend risk and high parity value.
  Two screens exceed plain CRUD (Package Details M:N linking; Documents file
  upload), so they cannot be pure ABP-Suite generation -- they need hand-built
  sub-UI. Recommend sequencing: System Parameters (pilot, smallest) -> Custom
  Fields -> Notification Templates -> Documents -> Package Details -> Field
  Configs.
- **Risk:** Low to existing features (additive). Medium per-screen for the three
  non-trivial ones. Must respect the "never `ng serve`" build constraint and
  "never edit `proxy/`" rule.

### Option B -- ABP Suite scaffold the simple CRUD screens, hand-build only the non-trivial ones [effort: L-XL] [risk: medium]

- **One-liner:** Use ABP Suite (already evidenced by the abstract+concrete
  component pattern in `appointment-statuses`) to generate the straightforward
  list/detail screens (Custom Fields, Documents catalog, Package Details base
  CRUD, Field Configs), then hand-author System Parameters (singleton form, not
  a list) and the Package-Details document-linking sub-UI.
- **Source:** ABP Suite Angular generation
  (https://abp.io/docs/latest/suite/index, official ABP docs, accessed
  2026-06-01; credibility: official). Pattern already in repo
  (`appointment-statuses` abstract files note "generated by ABP Suite").
- **Tradeoffs:** Faster for the 3-4 plain-CRUD screens; but Suite assumes a
  standard `GetList/Get/Create/Update/Delete` AppService shape. System
  Parameters (no list, singleton Get/Update) and Notification Templates (no
  Create/Delete) deviate, so Suite output needs manual trimming. Suite-generated
  styling may not match OLD design tokens without post-editing.
- **Risk:** Medium -- regenerating Suite could clobber hand-edits; must guard
  the customized files. Net effort similar to A once the non-CRUD screens are
  factored in.

### Option C -- Single consolidated "Admin Settings" hub page with tabbed sections [effort: L] [risk: medium-parity]

- **One-liner:** One route (e.g. `/admin/settings`) with tabs for each of the
  six areas instead of six separate menu entries/routes.
- **Source:** Inference from LeptonX layout capability; no OLD precedent (OLD
  used six distinct routes/screens).
- **Tradeoffs:** Less menu clutter and arguably faster to wire one route. BUT it
  deviates from OLD's six-screen IA, which the Primary Mission says to match
  ("UI layout ... every small UI detail"). Reserve for a later UX pass, not the
  parity build.
- **Risk:** Medium parity risk -- diverges from OLD navigation; would need
  Adrian sign-off given the strict-parity directive.

**Recommendation:** Option A (pure OLD-faithful port, six screens), sequenced
with System Parameters as the pilot. It carries the lowest risk to shipped
features and best satisfies the strict-parity directive; the CSV note already
scoped it as "~6 admin CRUD/edit screens ... mostly frontend ... needs
sequencing/planning."

## 8. Root-Cause Hypothesis

**Confidence: high.** This is not a regression or a logic bug -- it is planned-
but-unbuilt scope. The backend for all six master-data areas was deliberately
built first (entities, AppServices, controllers, permissions, proxies all
present and richly documented with phase markers like "Phase 4 (2026-05-03),"
"Phase 5 (2026-05-03)"), following a backend-first sequencing. The Angular admin
screens were simply never reached -- the team prioritized the booking/approval/
email flows (which DO have UI) and the priority lookup CRUD screens
(appointment-statuses, appointment-types, locations, states, wcab-offices). The
six admin master-data screens were left for a later UI wave. Evidence: backend
artifacts exist and are complete; zero frontend artifacts exist; the only
runtime path that touches this data on the frontend is the booking-form
renderer (read-only consumption), not an editor. Hence "Missing behavior" =
missing UI layer, backend ready.
