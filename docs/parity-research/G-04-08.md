## 1. ID & Summary
OLD's mailer (`SendMail`) accepted a `mailBCC` parameter on every overload and
would add those addresses as blind-copy recipients. NEW's email path
(`SendAppointmentEmailJob` / `IEmailSender.SendAsync`, with CC via
`CcRecipientAppender`) models recipients as To + CC only -- no BCC. Disposition:
exclude. Verification shows BCC was a dead/latent capability in OLD: no live call
site ever passed a non-empty BCC, so no real behavior is lost.

  (lines 317-332).

  live call used it; no real behavior lost. Add later only if a concrete need

## 3. Affected Code (spot-checked)
OLD `SendMail` (P:/PatientPortalOld/PatientAppointment.Infrastructure/Utilities/SendMail.cs):
- BCC-add blocks confirmed at `:221-232` and `:435-446` (split on `;`, add each
  to `message.Bcc`) -- matches the parity-doc citation exactly.
- Signatures: `SendSMTPMail(mailTo, mailSubject, mailBody, mailCC="", mailBCC="", filePath="")`
  (`:166`) and `SendSMTPMailWithAttachment(... mailCC="", mailBCC="", files=null)`
  (`:388`); interface `:469-470`. `mailBCC` is the 5th positional param,
  defaulting to "".
- Call-site enumeration (all of OLD, grep of `SendMail.SendSMTP*`): ~50 live
  calls. None passes a non-empty 5th positional arg. Patterns observed:
  - 3-arg calls (To, Subject, Body) -> mailBCC defaults to "".
  - 4-arg calls pass a non-empty CC (e.g. AppointmentDomain.cs:933/966/980
    `..., emailCC`; SchedulerDomain.cs:146 `..., item.PrimaryEmailList`) -> still
    no BCC.
  - `SendSMTPMailWithAttachment(email, subject, body, "", "", attachment)`
    (AppointmentDocumentDomain.cs:491,683,731,777,791,838,851) -> CC and BCC both
    literal "".

NEW: no BCC argument in the dispatcher path (per parity doc; `CcRecipientAppender`
handles CC only). Not re-read in light tier; the gap is "feature absent", which
the OLD-side dead-capability finding makes moot.

## 4. Expected vs Actual (brief)
- Expected (OLD intent): mailer can blind-copy recipients.
- Actual behavior (OLD runtime): BCC never populated -- latent code, zero live use.
- NEW: no BCC support. Net user-visible behavior: identical (nobody was ever BCC'd).

## 5. Repro Status
n/a-excluded. There is no observable OLD behavior to reproduce -- BCC was never
exercised at runtime. A behavioral diff would require fabricating a call site that
never existed; not applicable.

## 6. Why Excluded + Revisit-If
Excluded because porting BCC would replicate dead code, not behavior. Parity is
defined as user-visible + business-rule behavior; an unused parameter is neither.
Revisit if: a concrete NEW feature needs blind-copy (e.g. compliance/archival BCC
to a monitoring mailbox, or a future per-tenant "BCC all outbound to office
inbox" requirement). At that point it is a new feature, not a parity regression.

## 7. Candidate Solution
None -- excluded. (If revisited: add an optional bcc list to the NEW dispatcher /
`IEmailSender` call alongside the existing CC handling; trivial, additive.)

## 8. Decision Rationale + confidence
Confidence: high. Static evidence is complete and unambiguous: the BCC-add code
exists in OLD (`SendMail.cs:221-232,:435-446`) but the full set of ~50 live call
sites was enumerated and none passes a non-empty BCC. The exclude decision does
not merely rest on the CSV note -- it is independently confirmed by source.
