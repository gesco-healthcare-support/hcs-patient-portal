## 1. ID & Summary

OLD's `spm.AppointmentDocuments` table carried an `int DocumentPackageId`
column (and an `int UserType` column). NEW's `AppointmentDocument` entity
has neither. The CSV asks to "re-add package-item linkage so each uploaded
doc maps to the specific required package item it satisfies (vs looser
queued-row)."

The research finding **contradicts the framing of the gap as a real loss**:
in the OLD app, `AppointmentDocument.DocumentPackageId` was a **dead /
vestigial column**. The only code that ever wrote it is a private method
whose every call site is commented out, and that method hardcodes
`DocumentPackageId = 77` with the OLD developer's own comment
`//Not needed as changes in requirement. (Will be removed)`. No OLD UI,
controller, or live domain path ever populated it with a meaningful
package-item reference -- not for `AppointmentDocument` and not for
`AppointmentJointDeclaration`.

NEW's "looser queued-row" model (`PackageDocumentQueueHandler` queues one
`AppointmentDocument` per active `DocumentPackage`, copying the master
`Document.Name` into `DocumentName`, plus a per-row `VerificationCode`) is
in fact **strictly more functional** than OLD's live behavior, because OLD
never created those per-package rows at all (G-03-B1).

The legitimately portable kernel of this item is small: NEW currently
discards the identity of WHICH master `Document` / `DocumentPackage` each
queued row corresponds to (it keeps only the name string). Re-adding a
nullable `DocumentId` (master-template) reference on the queued row would
deterministically answer "which required doc is still outstanding" without
relying on name-string matching. That is the only defensible piece of the
"re-add the FK" request -- and it points at the master `Document`, not at
OLD's `DocumentPackageId` semantics.

### Blast radius

If the legitimate kernel is implemented (add a nullable `DocumentId`
master-template reference to the queued row):
- Layers touched: Domain (entity + manager factory), Application (queue
  handler), EntityFrameworkCore (mapping + 1 migration adding a nullable
  column). 4 files + 1 migration.
- DTO/proxy: optional -- only if the UI needs to surface it; if added,
  `AppointmentDocumentDto` + regenerate proxy.
- Regression risk: **low**. The new column is nullable and additive; ad-hoc
  and JDF rows leave it null. No existing query filters on it, so adding it
  cannot change current behavior. The queue handler already has `documentId`
  in scope, so no new query is needed.
- If instead one literally re-adds OLD's `DocumentPackageId` semantics
  (pointing at the `DocumentPackage` M:N row): higher modeling cost for zero
  OLD-parity benefit, since OLD never populated it.

## 4. Expected vs Actual

- **CSV expectation:** each uploaded doc maps via a hard FK to the specific
  required package item it satisfies, matching OLD.
- **OLD actual (verified):** `AppointmentDocument.DocumentPackageId` was
  never populated by live code; its only writer is dead and hardcodes `77`
  with a "will be removed" comment. The JDF FK column existed but was also
  never set by any live UI/API path. So OLD had NO working
  upload-to-package-item linkage on `AppointmentDocument`.
- **NEW actual (verified):** no `DocumentPackageId`/`UserType` columns; NEW
  links queued rows to packages by generating one row per active
  `DocumentPackage` and copying `Document.Name` + a `VerificationCode`. This
  is more capable than OLD's live behavior, but it loses the *identity* of
  the source master `Document`/`DocumentPackage` (kept only as a name
  string).
- **Net:** the "FK dropped" is real at the schema level but represents
  removal of a dead column, not a behavioral regression. The only behavioral
  improvement available is to persist the master `DocumentId` on the queued
  row for deterministic outstanding-doc resolution.

## 5. Repro Status

not-reproduced-static-confirmed. The gap is a schema/column difference and a
data-linkage design choice, fully confirmable by reading the OLD entity +
the single dead writer and the NEW entity + EF mapping + queue handler. A

NEW mapping provably lacks the column.

## 7. Candidate Solutions

### Option A -- Confirm-and-exclude the literal FK; persist master `DocumentId` on queued rows instead [effort: S] [risk: low]
- One-liner: Do NOT re-add OLD's `DocumentPackageId` (dead/hardcoded in OLD).
  Instead add a nullable `Guid? SourceDocumentId` (master-template
  reference) to `AppointmentDocument`, set it in `PackageDocumentQueueHandler`
  (it already has `documentId` in scope), so "which required doc is
  outstanding" is resolvable by id rather than by `DocumentName` string.
- Source: OLD code -- `AppointmentDocumentDomain.cs:1102-1123` (dead writer,
  hardcoded `77`), Audit 03 G-03-B1 (`03-documents-packets.md:139`); NEW
  `PackageDocumentQueueHandler.cs:148-170` (documentId in scope, discarded).
- Tradeoffs: gives deterministic linkage and survives master-`Document`
  renames (a name-string match would break on rename); nullable + additive
  so no regression. Points at the master `Document` (the meaningful target),
  not OLD's vestigial `DocumentPackage` join semantics.
- Risk: low (nullable additive column; no existing filter depends on it).
- Effort: S (entity field + manager param + handler line + nullable-column
  migration; mapping update).

### Option B -- Re-add `DocumentPackageId` verbatim to match OLD schema [effort: M] [risk: med]
- One-liner: Recreate an `int`/`Guid` column named after OLD's
  `DocumentPackageId` referencing the NEW `DocumentPackage` M:N row.
- Source: OLD `AppointmentDocument.cs:57-60` (the column as it existed).
- Tradeoffs: matches OLD schema shape literally but reproduces a column OLD
  never populated; the M:N `DocumentPackage` row is keyed
  `(PackageDetailId, DocumentId)` in NEW, so an FK to it is awkward and adds
  no behavioral value over Option A. Contradicts the "do not port dead/buggy
  OLD code" directive.
- Risk: med (modeling churn; invites future code to depend on a semantically
  empty column).
- Effort: M.

### Option C -- Do nothing; record exclude with evidence [effort: none] [risk: low]
- One-liner: Treat NEW's queued-row + `VerificationCode` + copied
  `DocumentName` as sufficient parity (it exceeds OLD's live behavior) and
  close the item.
- Source: Audit 03 row E9 + G-03-B1; NEW queue handler.
- Tradeoffs: zero work; accepts name-string matching for outstanding-doc
  resolution (fragile only if a master `Document` is renamed after rows are
  queued -- an edge case). This is defensible because OLD itself had no
  working linkage.
- Risk: low.
- Effort: none.

Recommended: **Option A** if the team wants the small robustness win (the
CSV decision is `include`); otherwise Option C is fully justified by the OLD
evidence. Option B is not recommended.

## 8. Root-Cause Hypothesis

The FK was "dropped" deliberately and correctly. The NEW design (Phase 1.6 /
Phase 12-14) consolidated OLD's three document tables into one entity and
implemented the per-package queueing that OLD only ever stubbed out (every
`AddAppointmentDocument` call site commented; `DocumentPackageId = 77`
placeholder marked "will be removed"). The NEW author dropped
`DocumentPackageId` and `UserType` because: (a) `DocumentPackageId` was dead
in OLD, and (b) `UserType` (recipient stakeholder type) is superseded by
NEW's per-recipient `AppointmentPacket` rows + `IsAdHoc`/`IsJointDeclaration`
flags + the read-access guard. The residual, legitimately-portable concern
is purely that the queued row keeps the master document's NAME but not its
ID -- a minor robustness gap, not the FK the CSV literally names.
Confidence: high (OLD writer provably dead; NEW mapping provably lacks the
column; Audit 03 already dispositioned the source behavior).
