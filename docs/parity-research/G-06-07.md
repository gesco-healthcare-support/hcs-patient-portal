## 1. ID & Summary
Brute-force protection on login. OLD had NO server-side rate-limit or account
lockout: the only "failed attempt" tracking was a cosmetic client-side counter
(`failedCount`) the server never read. NEW relies on ABP Identity's default
account lockout plus explicit per-email rate limits on the account-recovery
endpoints. NEW is therefore strictly STRICTER than OLD. Disposition: exclude
(no action) -- adding protection that OLD lacked is a security improvement,
not a parity regression to fix.

## 4. Expected vs Actual (brief)
- Expected (parity to OLD literal behavior): unlimited login attempts, no
  lockout, cosmetic client counter.
- Actual (NEW): account lockout after N failed attempts (ABP default) +
  throttled recovery emails. Divergence is intentional hardening.

## 5. Repro Status
n/a-excluded. There is no defect to reproduce -- the "gap" is that NEW does
MORE than OLD. A runtime repro of lockout would require submitting repeated
bad-credential logins (state-adjacent / mutating auth-attempt counters) and is
unnecessary; the static evidence (LockedOut page wired + zero OLD server-side
lockout) fully establishes the divergence direction.

## 6. Why Excluded + Revisit-If
Excluded because the divergence makes NEW more secure, not less faithful in any
way a user/business rule cares about: no OLD workflow depended on unlimited
attempts. Revisit IF:
- A real user-facing complaint about over-aggressive lockout appears (then tune
  `IdentityOptions.Lockout` thresholds -- still no parity restore of OLD's
  unlimited attempts).
- A business stakeholder explicitly requests OLD's permissive behavior (would
  be a deliberate security downgrade -- escalate, do not silently restore).

## 7. Candidate Solution
None -- excluded. (If lockout tuning is ever wanted, it is an ABP settings
change: `Abp.Identity.Lockout.*` / `IdentityOptions.Lockout`, not code parity.)
