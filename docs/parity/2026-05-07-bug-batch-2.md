## Issue #1 - Slot deletion 403 with booked slots

### NEW behavior (current state)
`DELETE /api/app/doctor-availabilities/by-date` -> `DoctorAvailabilitiesAppService.DeleteByDateAsync` at `src/HealthcareSupport.CaseEvaluation.Application/DoctorAvailabilities/DoctorAvailabilitiesAppService.cs:142-170`. Loads all rows for `LocationId + AvailableDate.Date`; if ANY row has `BookingStatus.Reserved` or `Booked` (`:412-415`), throws `BusinessException(DoctorAvailabilityCannotBulkDeleteWithBookedSlots)` (`:163`), surfaced as HTTP 403. Single-row `DeleteAsync` (`:102-118`) blocks even Available rows when an `Appointment` or `AppointmentChangeRequest` references them. Error string: `src/HealthcareSupport.CaseEvaluation.Domain.Shared/Localization/CaseEvaluation/en.json:225`.

### OLD behavior (ground truth)
OLD soft-deletes (sets `StatusId = Status.Delete`); does NOT hard-delete. `DoctorsAvailabilityDomain.DeleteValidation` at `P:\PatientPortalOld\PatientAppointment.Domain\DoctorManagementModule\DoctorsAvailabilityDomain.cs:141-157` blocks bulk delete only if `Booked` or `Reserved` rows exist; `Delete` at `:159-177` then soft-deletes ONLY `BookingStatus.Available` rows in the same date+location (`x => x.BookingStatusId == BookingStatus.Available`, `:163`). Effectively a partial-delete: leaves Booked/Reserved rows alone, removes the freely-deletable rest.

### Gap
NEW correctly mirrors the "block bulk delete when any slot is booked" check but loses OLD's quiet partial-delete of Available rows. From the admin UI a date with even one booked slot becomes un-prunable. Additionally OLD's flow returns the validation as HTTP 400 from `BadRequest(...)` at `DoctorsAvailabilitiesController.cs:87`, not 403; ABP's `BusinessException` is mapping to 403 here.

### Fix options
1. **Mirror OLD partial-delete** - skip Booked/Reserved rows, soft/hard-delete only Available rows for that date+location, return a `{deletedCount, skippedSlotIds}` response so the UI can render "3 of 5 deleted; 2 are booked". Blast radius: changes admin-side semantics; users may need to refresh to see remaining booked rows.
2. **Keep block, return 400 with detail** - throw `UserFriendlyException` (HTTP 400) with the conflict slot IDs so UI can show inline message + "cancel the appointment first" CTA. Blast radius: error-code rename; any caller switching on the current code must change.
3. **Add admin force-delete override** - new `force` flag gated on `DoctorAvailabilities.ForceDelete` permission; cascades through linked appointments to Cancelled state. Blast radius: large; needs cancellation emails, audit log entries, role review.

### Recommendation
Option 1: strict-parity match to OLD, smallest UX surprise. Option 3 is a future enhancement.

### Gap
NEW carries a `Role` tag OLD never had. Two failure modes:

### Fix options

### Recommendation
