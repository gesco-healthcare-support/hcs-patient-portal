## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict (root `CLAUDE.md` Stack table). `[AllowAnonymous]` is the ASP.NET Core primitive for anonymous endpoints; ABP AppServices accept it at method level even though `AbpApplicationServiceAttribute` default is `Authorize`.
- ADR-002 (manual controllers with `[RemoteService(IsEnabled = false)]`) + root `CLAUDE.md` "Controllers" subsection -- a new `AnonymousDocumentUploadAppService` must carry `[RemoteService(IsEnabled = false)]` and a hand-written `AnonymousDocumentUploadController` in `src/.../HttpApi/Controllers/AppointmentDocuments/`. The controller must add `[AllowAnonymous]` on the controller class (or on each anonymous verb) AND `[IgnoreAntiforgeryToken]` if it accepts POST of file bytes -- same envelope `ExternalSignupController` uses at line 14.
- ADR-001 (Riok.Mapperly) -- any DTO mapping in this capability lives in `CaseEvaluationApplicationMappers.cs` as a `[Mapper]` partial.
- ADR-003 (dual DbContext) + ADR-004 (doctor-per-tenant) -- the anonymous request has no tenant context. The token must resolve `AppointmentId -> TenantId` from the host-side `Appointment` lookup (with `IDataFilter.Disable<IMultiTenant>()` scoped narrowly), then `using (CurrentTenant.Change(tenantId))` wraps the save so tenant isolation holds for `AbpBlobs.TenantId` and any downstream domain-event subscribers. Pattern is identical to `ExternalSignupAppService.cs:175` with its `CurrentTenant.Change(tenantId)` block.
- ADR-005 (no `ng serve`) applies transitively; not an anonymous-specific concern.
- HIPAA -- the uploaded files ARE PHI. Token must be time-limited, bound to a single appointment + document-type + recipient email, single-use where possible, logged with uploader's email + IP, and the upload must flow into the same `IBlobContainer` consumer chain that authenticated uploads use (so the row-level tenant filter still applies). The OLD defect class of "token enforced on list but not on PUT" must not recur; every verb on the anonymous controller must re-validate the token.
- Q25 (gap-analysis README line 261: "Anonymous endpoints in OLD (DocumentUpload/Download/CsvExport/Dashboard/Scheduler): intentional or legacy bug?") -- Adrian acknowledges the OLD anon surface as a defect. Track 10 scheduler probe (line 166-170) showed it IS actually exploitable. This brief treats the capability as "replicate the magic-link UX with hardened semantics", NOT "replicate the anonymous surface verbatim".
- Q26 (line 263: `server-settings.json` committed AWS + Twilio keys) -- not a blocker for this brief (DB-BLOB recommendation avoids S3), but cross-referenced because the OLD implementation relied on those keys.
- Q15 VERBATIM (gap-analysis README line 245): "Anonymous upload via magic link (OLD `/upload-documents/:id/:type`): required for external attorneys/patients to submit documents without login? (Tracks 4, 7, 9)". This brief assumes Q15 = yes; if Adrian says no, the capability collapses to "require login" and the recommended solution is "delete this capability from scope; route external uploads through `account-self-service` + authenticated `appointment-documents`."

## Alternatives considered

- A. **ABP `[AllowAnonymous]` AppService + `api/public/appointment-documents/upload-by-token` controller, token issued via `IDataProtectionProvider.CreateProtector("AppointmentDocuments.UploadLink").ToTimeLimitedDataProtector()`, blob via `IBlobContainer<AnonymousUploadsContainer>`, single-use state in `IDistributedCache`, rate-limited via `AddTokenBucketLimiter` keyed by (token, IP)** -- `chosen`. Uses only ABP primitives + .NET shipped libraries; zero new NuGet. Token is opaque (Base64Url-encoded Protect output) and carries `{AppointmentId, DocumentTypeId, RecipientEmail, Exp}` server-signed. 24-hour TTL from issuance. Each upload validates protector + cache-backed used-flag + rate-limit partition. Flow: staff clicks "Send Upload Link" on Appointment detail -> `AppointmentDocumentsAppService` (MVP authenticated surface) issues token via protector, persists a row in `AppointmentDocument` with `Status=PendingUpload`, builds URL `https://<frontend>/upload-documents/{token}`, sends email through `IEmailSender` with the TextTemplate template. External user opens link; Angular calls `GET /api/public/appointment-documents/upload-token-info?token=...` to display appointment and doc-type context (no PHI beyond the user's own context). User selects file, POSTs to `/api/public/appointment-documents/upload-by-token` (multipart form with token + file); AppService protector.Unprotect + cache-check + rate-limit + tenant-context-switch + blob SaveAsync + audit-log entry + single-use flag.
- B. **Require login (break OLD parity)** -- `conditional rejected`. Chosen only if Q15 = no. Forces external attorneys/patients through the `account-self-service` flow (forgot-password / email-verification) for every document drop. Two operational costs: more staff hand-holding for attorney firms that don't keep accounts, and more invitation email cost (one invite + one doc-link instead of one doc-link). Reduces blast radius to zero anonymous surface. Reason this loses absent Q15 = no: a primary OLD use case is "intake staff schedules appointment for a patient represented by an external attorney firm; firm's paralegal uploads the court-order PDF without creating a portal account." Forcing a full account-creation flow for every paralegal is a meaningful UX regression.
- C. **Pre-signed S3 / Azure Blob URL issued server-side, client PUTs directly to the object store** -- `rejected for MVP`. Cleanest scale story (file never touches .NET). Requires the AWS/Azure provider (`Volo.Abp.BlobStoring.Aws` or `.Azure`), BAA signing, KMS keys, CORS, and bucket policy. All four are out-of-scope per Q17 (`solutions/blob-storage-provider.md` chose DB-BLOB for MVP). Re-opens as a post-MVP option once Q17 flips.
- D. **JWT (or signed JWE) as the magic-link token instead of `ITimeLimitedDataProtector`** -- `rejected`. JWT is a plaintext payload over an HMAC signature. The token goes in a URL, so anyone with the link can decode and see the recipient's email + appointment ID + doctor tenant ID. `ITimeLimitedDataProtector` encrypts the payload with keys managed by ASP.NET Core's Data Protection (already Redis-backed per root `CLAUDE.md` "Data protection via Redis in non-dev"), producing an opaque ciphertext. Same verification semantics; smaller disclosure surface. Also avoids importing `System.IdentityModel.Tokens.Jwt` separately.
- E. **Hand-rolled HMAC token + bespoke persisted `UploadToken` entity** -- `rejected`. Gives flexibility to revoke individual tokens via a `RevokedAt` column, but duplicates what `IDistributedCache` already does for single-use enforcement and what Data Protection already does for signed opaque encoding. Revocation is achievable in A by writing a `revoked:{tokenId}` marker into the cache -- simpler than a table.

## Recommended solution for this MVP

Add a new `AnonymousDocumentUploadAppService` under `src/HealthcareSupport.CaseEvaluation.Application/AppointmentDocuments/Anonymous/` plus a matching `AnonymousDocumentUploadController` under `src/HealthcareSupport.CaseEvaluation.HttpApi/Controllers/AppointmentDocuments/Anonymous/`. Mirror the `ExternalSignupController` envelope: `[IgnoreAntiforgeryToken] [Area("app")] [ControllerName("AnonymousDocumentUpload")] [Route("api/public/appointment-documents")]` at class level, `[AllowAnonymous]` on each of three verbs (`GET upload-token-info`, `POST upload-by-token`, `POST complete-by-token`). The AppService is `[AllowAnonymous]` per method and carries `[RemoteService(IsEnabled = false)]` at class level per ADR-002. Token issuance lives on the authenticated side of the house: the existing `appointment-documents` capability exposes `IAppointmentDocumentsAppService.IssueUploadLinkAsync(Guid appointmentId, Guid documentTypeId, string recipientEmail)`, which calls `IDataProtectionProvider.CreateProtector("AppointmentDocuments.UploadLink").ToTimeLimitedDataProtector()`, serialises `{AppointmentId, DocumentTypeId, RecipientEmail, NonceId}` (System.Text.Json), protects with 24-hour lifetime, Base64Url-encodes, composes the URL `https://<frontend>/upload-documents/{token}`, and dispatches an email via `IEmailSender` using a `TemplateDefinitionProvider` keyed `AppointmentDocumentUploadLink` (template registered alongside `email-sender-consumer` rollout). The anonymous controller's GET returns `{AppointmentConfirmationNumber, DocumentTypeName, ExpiresAt, AllowedFileTypes, MaxSizeBytes}` only -- no PHI beyond what the recipient already received in the email. POST accepts multipart form data (token as form field, file as IFormFile). The AppService: (1) Unprotect token via the same protector; reject on `CryptographicException` with 410 Gone. (2) Read cache key `anon-upload:{NonceId}:used`; reject with 409 if already marked. (3) Rate-limit check: per-token bucket (3 attempts / 10 min) + per-IP bucket (10 attempts / 10 min), `RateLimitPartition.GetTokenBucketLimiter` configured in `CaseEvaluationHttpApiHostModule.cs:ConfigureServices`. (4) Resolve `AppointmentId -> TenantId` by reading `Appointment` with `IDataFilter.Disable<IMultiTenant>()` scoped to the single query. (5) `using (CurrentTenant.Change(tenantId)) { ... }` wraps: validate file size + MIME against `AppointmentDocumentType`'s allowed set, write bytes via `IBlobContainer<AnonymousUploadsContainer>.SaveAsync(key, stream)`, create an `AppointmentDocument` row `Status = Pending`, `VerificationNonceId = NonceId`, `UploadedByEmail = recipientEmail`, `UploadedByIp = requestIp`, `UploadedAt = Clock.Now`. (6) Mark cache key used with absolute expiry = token expiry (so an expired token never collides with a replay). (7) Log via `IAuditingManager.BeginScope` with reason = "anonymous-upload" for HIPAA granularity. No schema migration beyond what `appointment-documents` introduces; `VerificationNonceId` is a `Guid?` column on `AppointmentDocument` handled in that brief. Angular: add a standalone public route `{ path: 'upload-documents/:token', loadComponent: () => import('./appointment-documents/anonymous/upload-documents.component'), canActivate: [] }` in `app.routes.ts`, outside the `authGuard` chain. The component is a single page: shows `{AppointmentConfirmationNumber, DocumentTypeName}` from the info-endpoint, file input, client-side size + extension gate, submit button that POSTs multipart form to `/api/public/appointment-documents/upload-by-token`, success screen. Proxy is hand-written (do not regenerate for `/api/public/**`; abp-generate-proxy ignores routes outside `/api/app/**`). Flag to toggle the capability off (post-MVP rollback): feature flag `AppointmentDocuments.AnonymousUpload` via ABP FeatureManagement; when off, `IssueUploadLinkAsync` throws `FeatureNotEnabledException` and the anonymous controller returns 404 on every verb via a `[FeatureGate(...)]` attribute.

## Why this solution beats the alternatives

- A honours Q15 if answered "yes" while hardening every OLD defect: token enforced on every verb (OLD only on list), single-use (OLD allowed replay), time-limited (OLD had no expiry), rate-limited (OLD had none), encrypted payload (OLD stored a bare GUID in DB). No new NuGet, uses only ABP + .NET primitives already wired.
- A does not reopen the S3 credential + BAA debate (Q17 + Q26); reuses the DB-BLOB marker class `AnonymousUploadsContainer` that `blob-storage-provider` already reserved.
- A reuses the `ExternalSignupController` envelope pattern, so the reviewer has prior art in-repo for the anonymous-surface shape. Adding a second `api/public/**` route cluster follows a pattern staff already audit.
- A keeps all OLD-level defect classes closed: token-replay (single-use cache), path-traversal (no filesystem pathing), PHI-in-URL (opaque encrypted token, not `?query=<plain-guid>`), unguarded mutations (every verb re-checks token), permission-drift (feature gate + rate limit in-middleware + token validation in-AppService).

## Risk and rollback

- Blast radius if implementation is wrong:
  - Token leakage: an attacker forwarding the link could upload on behalf of the intended recipient. Mitigated by single-use + short expiry + rate limit + audit-logged IP/email; recipient can complain if they never got to upload.
  - File-bomb: an attacker who guessed or stole a token uploads a 4 GB file. Mitigated by `MaxSizeBytes` MIME + size gate in `AnonymousDocumentUploadAppService` and by ASP.NET's `KestrelServerLimits.MaxRequestBodySize`.
  - Cross-tenant leak: the `CurrentTenant.Change` wrapper must be present on every write path. If omitted, the blob lands in a null-tenant partition. Covered by review + test (test in `new-qual-01-critical-path-test-coverage` scope: one test asserts tenant filter on `AbpBlobs` after anonymous upload).
  - Rate-limiter misconfiguration: if the partition key is based purely on IP, a shared NAT (firm office behind one public IP) could self-DoS. Primary partition MUST be token; IP partition is a secondary guard.
  - Feature-gate failure: if the flag is `enabled` but the email template isn't registered, the staff click "Send Upload Link" and it 500s. Mitigated by template-existence precondition in `IssueUploadLinkAsync`.
- Rollback: toggle `AppointmentDocuments.AnonymousUpload` feature flag to disabled via ABP Feature Management admin UI. Controller returns 404 on every verb, `IssueUploadLinkAsync` throws `FeatureNotEnabledException`. No schema migration to revert (the `VerificationNonceId` column is benign when unused). Existing `AppointmentDocument` rows uploaded via this path are `Status=Pending` and remain visible to authenticated staff.

## Open sub-questions surfaced by research

- Does Adrian want the magic link to carry the recipient's email inside the token, or only an opaque NonceId with the email looked up from an `UploadTokens` table? The recommendation uses "email inside token" for zero-DB-round-trip issuance, but if Adrian prefers revocable + re-issuable tokens, a small `UploadTokens` entity (columns: `Id`, `AppointmentId`, `DocumentTypeId`, `RecipientEmail`, `ExpiresAt`, `UsedAt`, `RevokedAt`) simplifies admin operations at the cost of one INSERT per issuance. Both variants keep the `ITimeLimitedDataProtector` wrapper.
- Should the upload flow allow N files per token ("attorney uploads packet of 5 PDFs") or strictly one file per token ("each file gets its own link")? OLD was effectively one-file-per-doc-row. The recommendation keeps single-file-per-POST but allows the same token to be used up to `MaxUploadsPerToken` times (default 1, configurable per AppointmentDocumentType).
