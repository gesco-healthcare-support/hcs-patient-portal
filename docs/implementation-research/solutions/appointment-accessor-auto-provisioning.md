# AppointmentAccessor auto-user-provisioning

## Status (scope-locked 2026-04-24 -- SCOPE EXPANDED)

Adrian's Q&A answer Q12 elevates this capability:
- **OLD scope:** specialised accessor-only `FindOrCreateAccessorUserAsync` method on `AppointmentAccessorManager`.
- **NEW scope (locked):** canonical `FindOrCreateExternalUserAsync` method used by ALL external-user creation (Patient, Claim Examiner, Applicant Attorney, Defense Attorney). This becomes the backbone of the email-invite signup flow.
- Flow: admin creates pending user via `/identity/users` Admin UI -> selects tenant + role + name + email -> system calls `FindOrCreateExternalUserAsync` -> creates IdentityUser with role, generates password-reset token, emails invite link via `IAccountEmailer.SendPasswordResetLinkAsync` -> user clicks, sets password, logs in.
- **For Patient role invite specifically:** after creating IdentityUser, call `PatientManager.FindOrCreateAsync` (patient-auto-match logic) to create/attach the Patient entity. This replaces the anonymous signup flow that `new-sec-04` just removed.
- Effort revised: **L+ (~5-6 days)**, up from L (~4 days). Additional work: tenant-scoped parameter handling in Manager, 4-role branching for Patient-vs-non-Patient entity creation, audit log of who-invited-whom.

## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict. Row-level `IMultiTenant` per ADR-004. Manual controllers + Mapperly per ADR-001/ADR-002. Dual DbContext per ADR-003.
- HIPAA: no plaintext passwords emailed; the invited accessor is by definition granted access to a patient's appointment, so auditability is required. Invite email must carry a one-time reset link, not a password.
- Tenant-scoped: each AppointmentAccessor row is `IMultiTenant` -- the user being provisioned MUST exist inside that tenant context so ABP's automatic filter matches. OLD had no tenant model; we explicitly switch `CurrentTenant`.
- ABP Identity already supplies `IdentityUserManager.CreateAsync`, `AddToRoleAsync`, `GeneratePasswordResetTokenAsync`, account lockout, audit logs. Reuse these rather than rolling our own password/user store.
- Roles already seeded by `ExternalUserRoleDataSeedContributor` (Patient/Claim Examiner/Applicant Attorney/Defense Attorney). The 4 external role names are the universe of accessor role assignments; `AccessType` (View/Edit) is orthogonal.
- `ExternalSignupAppService.GetExternalUserLookupAsync` already offers the search path for "existing user by email/name within the 3 external roles" -- the new provisioning service should extend this shape, not duplicate it.
- Depends on `email-sender-consumer` (CC-01): the worktree's `probes/service-status.md` and gap-analysis note `IEmailSender` is `NullEmailSender` in debug. Without CC-01 merged, invite emails silently drop. Fallback: return the reset link in the API response for admin-to-accessor out-of-band delivery, and flip on when CC-01 ships.
- `AppointmentAccessorsAppService` class currently carries only `[Authorize]`. Method-level `Create/Edit/Delete` permissions will be added in NEW-SEC-02 (separate capability); this brief must not regress that fix.

## Alternatives considered

- **A. Extend `AppointmentAccessorsAppService.CreateAsync` with a find-or-create path driven by email** -- chosen. Adds a new DTO variant (or nested field) carrying `{email, firstName, lastName, accessorRoleName, accessTypeId, appointmentId}`. If `FindByEmailAsync(email)` returns an existing user in the current tenant, reuse it. Otherwise, call `IdentityUserManager.CreateAsync` with a random strong password, `AddToRoleAsync` with the chosen external role, `GeneratePasswordResetTokenAsync`, then `IAccountEmailer.SendPasswordResetLinkAsync`. Finally create the `AppointmentAccessor` row with the resolved `IdentityUserId`. Single API call, ABP-native primitives, HIPAA-compliant invite.
- **B. Require admin to pre-provision users in the Users admin UI and link from the accessor UI** -- rejected. Blocks the booking workflow for every new accessor, defeats the single-action UX that OLD had, and becomes a training burden for clinics Gesco is onboarding. No material security gain over A (A uses the same ABP primitives under the hood).
- **C. Reuse `ExternalSignupAppService.RegisterAsync` as the provisioning backend** -- rejected. `RegisterAsync` is `[AllowAnonymous]` and requires a plaintext password from the caller; it also hardcodes `Gender.Male / DOB = today / PhoneNumberType.Home` in the Patient branch (NEW-SEC-04). Coupling accessor-provisioning to a defective anonymous self-service flow mixes authorization tiers and drags the SEC-04 defect into the booking path. Cleaner to keep them distinct -- A delegates to the same underlying `IdentityUserManager` but in an `[Authorize]` admin-triggered context.
- **D. Port OLD's plaintext-password email verbatim** -- rejected. HIPAA non-starter; emailing credentials to the accessor violates the zero-trust-verification and code-standards rules and materially raises breach liability. Invite-link flow is industry standard and costs no additional development once CC-01 ships.
- **E. Use `IIdentityUserAppService` (Volo built-in CRUD) for the user creation step, then link the accessor row** -- conditional, not chosen for MVP. `IIdentityUserAppService.CreateAsync` exists but requires `AbpIdentityPermissions.Users.Create` on the caller. Internal staff who add accessors during booking may not carry that permission, and we do not want to broaden their Identity admin grants just to support this workflow. `IdentityUserManager` at the domain layer avoids the permission coupling.

## Recommended solution for this MVP

Extend `AppointmentAccessorManager` with a new `FindOrCreateAccessorUserAsync(email, firstName, lastName, accessorRoleName, appointmentTenantId)` domain method that injects `IdentityUserManager`, `IdentityRoleManager`, `ICurrentTenant`, and `IAccountEmailer`. Inside, wrap the work in `using (CurrentTenant.Change(appointmentTenantId))` so the row created is `IMultiTenant`-scoped to the appointment's tenant. Call `FindByEmailAsync(email)`; if non-null, return that user's `Id`. Otherwise: generate a random strong password (`GuidGenerator.Create().ToString("N")` padded to meet ABP password policy), `new IdentityUser(GuidGenerator.Create(), userName: email, email: email, tenantId: CurrentTenant.Id) { Name = firstName, Surname = lastName }`, `await _userManager.CreateAsync(user, tempPassword)` (throw on non-success), ensure and assign the caller-specified external role via `EnsureRoleAsync` (existing pattern in `ExternalSignupAppService`), then `var token = await _userManager.GeneratePasswordResetTokenAsync(user)` and `await _accountEmailer.SendPasswordResetLinkAsync(user, token, appName: "CaseEvaluation", returnUrl: <angular /account/reset-password route>, returnUrlHash: null)`. Return `user.Id`. Callers compose: `AppointmentAccessorsAppService.CreateAsync` branches on whether the input DTO carries an `IdentityUserId` (existing path) or an email/name/role triple (new path); when the new path fires, it calls `FindOrCreateAccessorUserAsync` to resolve `IdentityUserId`, then the existing `AppointmentAccessorManager.CreateAsync` persists the accessor row. Add a new DTO `AppointmentAccessorCreateWithUserDto` or a nested optional `NewUser` sub-object on `AppointmentAccessorCreateDto` -- prefer a new endpoint `POST /api/app/appointment-accessors/with-user` so the existing shape stays backward-compatible with the generated Angular proxy. Regenerate proxy (fixes A8-04 in the same PR). On the Angular side, add `appointment-accessors.service.ts` wrapper plus a modal form wired into the appointment-add / appointment-view pages that offers a "pick existing user" option (using `GetIdentityUserLookupAsync`) or "invite new user" fields. While CC-01 is still in flight, gate `SendPasswordResetLinkAsync` behind an `IEmailSender` probe -- if the resolved sender is `NullEmailSender`, return the reset URL in the `AppointmentAccessorDto` response so staff can deliver the link out-of-band.

## Why this solution beats the alternatives

- Uses ABP Identity + Account primitives (lockout, audit logs, password policy, token signing) instead of reinventing.
- HIPAA-safe: no plaintext credentials emailed; the accessor sets their own password via reset link.
- Single API call preserves OLD's booking-flow UX while letting the accessor-add modal gracefully fall back to "pick existing user" when the email is already in Identity.
- Additive: new endpoint + new DTO keeps the existing `POST /api/app/appointment-accessors` contract intact so partial migration is possible and NEW-SEC-02 method-level authorization work can proceed on the existing endpoints in parallel.

Inventory says L (~4 days). ## Dependencies

- Blocks: `appointment-change-requests` (when an accessor is added during a change request, reuse this flow), `appointment-documents` (anonymous-upload magic-link logic depends on accessor role-scoping), `external-user-home` (the invited accessor lands on `/home` after setting password).
- Blocked by: `email-sender-consumer` (CC-01) -- without it the invite email silently drops; mitigate with the fallback return-link. `internal-role-seeds` (DB-16 / 5-G01..04) -- note this capability actually depends on `ExternalUserRoleDataSeedContributor` (already seeded), not on internal-role-seeds. `account-self-service` (5-G13/5-G14) -- the reset link terminates on the Account module's `/account/reset-password` page which must be functional.
- Blocked by open question: Q22 (`External role default permissions: add a seed contributor so the 4 external roles have baseline grants out-of-the-box, or rely on admin to assign?`). Affects whether a freshly-invited accessor can actually see anything on first login. MVP-blocking until resolved.

## Risk and rollback

- Blast radius: new endpoint only; existing `POST /api/app/appointment-accessors` contract unchanged. Angular accessor modal is additive. If the invite-email step throws, the new endpoint returns 500 and the accessor row is not created (wrap the create+invite in a single UoW). Worst case: an orphan `IdentityUser` if the accessor insert fails after the user insert commits; mitigate by issuing both writes inside one `IUnitOfWorkManager.Begin(requiresNew: true, isTransactional: true)` block (ABP default).
- Rollback: revert the feature branch; no EF migration change required (entity shape unchanged); remove the new endpoint from the controller; delete the added Angular component. Re-run `abp generate-proxy` to erase the proxy entry.

## Open sub-questions surfaced by research

- Cross-tenant accessors: one Applicant Attorney or Defense Attorney may need to access appointments across multiple doctor-tenants. ADR-004 says the doctor IS the tenant. Do we (a) create a fresh IdentityUser in each tenant (duplicate identities) or (b) keep the user host-scoped and allow cross-tenant JWT claims? Resolve with Adrian -- affects how `FindByEmailAsync` is scoped and whether `CurrentTenant.Change` is even correct.
- Invite email branding (ties to BRAND-03): until `templates-email-sms` + `email-sender-consumer` land, invite emails use Volo default templates with no clinic branding. Acceptable for MVP?
- Password policy vs temp-password: ABP's password policy (default: 6+ chars, 1 upper, 1 digit) is higher than a plain `Guid.ToString("N").Substring(0, 8)`. The generated temp password must satisfy policy or `CreateAsync` fails. Use `GuidGenerator.Create().ToString("N") + "Aa1!"` or similar guaranteed-valid pattern.
- Do accessors need `IsAccessor` on IdentityUser (OLD had a boolean column)? NEW's IdentityUser has no such flag. If we need to distinguish "system admin-provisioned accessor" from "self-registered external user" (for audit reports or revocation flows), we need an ExtraProperty or a separate AppointmentAccessorAudit entity. Defer until change-log capability is designed.
