## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict. Row-level `IMultiTenant` (ADR-004). Mapperly (ADR-001), manual controllers (ADR-002), dual DbContext (ADR-003), no ng serve (ADR-005). No suggestion may reverse these.
- HIPAA: attorney records reference IdentityUsers and appointments, both PHI-adjacent. Any split model must preserve audit trail across both sides (`FullAuditedAggregateRoot` is already in use).
- `ExternalUserRoleDataSeedContributor` has already established two distinct attorney roles at the identity layer (`Applicant Attorney`, `Defense Attorney`). The signup / authentication layer therefore treats the two as distinct actors. Any schema choice must reconcile identity-side separation with the entity-side unification that NEW currently has.
- `ExternalUserType` enum already contains `ApplicantAttorney = 3` and `DefenseAttorney = 4`, so workflow code already branches on the distinction.
- Workers-comp semantics (capability-specific): Patient attorney represents the injured worker (applicant / plaintiff); Defense attorney represents the employer's insurance carrier. Both may be present on one IME appointment. Access scoping differs: Patient attorney sees the patient's case; Defense attorney sees the carrier's discoverable items and typically cannot reach patient self-service screens.
- `UpsertApplicantAttorneyForAppointmentAsync` assumes one attorney per appointment (maxResultCount: 1 read, first-link update). Supporting Defense + Patient simultaneously requires either (a) a second discriminator column on the join OR (b) a second parallel join entity.
- Uniqueness on the join: today, `AppointmentApplicantAttorney` has no composite unique index. Either solution must add one (`AppointmentId + AttorneyType`-style unique on the join) or enforce single-link semantics in the domain service.
- Cross-project influence: `appointment-accessor-auto-provisioning` and `appointment-documents` both anchor on attorney-role scoping; their briefs already list this capability as a blocker. Whatever naming falls out here propagates to those capabilities.

Workers-comp IMEs explicitly contemplate parallel representation: injured worker's attorney vs. defense (employer / carrier) attorney; both parties have right to examiner selection and records access. Confirms the two sides are durable, distinct roles.
- ABP DDD entity discriminator guidance -- `https://docs.abp.io/en/abp/latest/Entities#defining-entities` and `https://abp.io/docs/latest/framework/architecture/domain-driven-design/entities` (accessed 2026-04-24). ABP does not take an opinion on TPH vs. split tables; both are supported EF Core patterns and both work with `FullAuditedAggregateRoot<Guid>, IMultiTenant`. Choice is driven by whether the two sides share behaviour.
- Microsoft EF Core inheritance (TPH / TPT / TPC) -- `https://learn.microsoft.com/en-us/ef/core/modeling/inheritance` (accessed 2026-04-24). TPH (single table + discriminator column) is the EF Core default and has the best read performance for lookup-heavy entities; TPT adds a join per subtype and is preferred when subtypes diverge substantially.
- ABP Commercial Identity + role-based access -- `https://abp.io/docs/en/abp/latest/Modules/Identity` (accessed 2026-04-24). Confirms `IdentityRole` is the canonical identity dimension; an entity attribute that mirrors the role name is acceptable as long as the domain enforces consistency.
- Riok.Mapperly `[Mapper]` + derived-type mapping -- `https://mapperly.riok.app/docs/configuration/derived-types/` (accessed 2026-04-24). Supports mapping source/destination pairs for sealed class hierarchies; would apply if alternative B (separate entities) is chosen.
- ABP support thread "Single table inheritance with `HasDiscriminator`" (abp.io/support question 10243, accessed 2026-04-24) -- confirms ABP apps can use EF `HasDiscriminator<T>` on an aggregate root and keep a single repository. HIGH confidence (pattern used across ABP samples).

## Alternatives considered

- **A. Keep unified `ApplicantAttorney`, add `AttorneyType` enum on the join (`AppointmentApplicantAttorney`) to distinguish Patient/Defense context per appointment.** Chosen if Q1 = no and Q2 = rename. Mechanics: add `AttorneyType` enum to `Domain.Shared/AppointmentApplicantAttorneys/` with values `Patient=1, Defense=2`; add column + EF migration; optionally rename `ApplicantAttorney` -> `PatientAttorney` for workers-comp correctness (Q2). Upsert path adds `AttorneyType` parameter; `GetAppointmentApplicantAttorneyAsync` returns a 2-element DTO shape (PatientSide + DefenseSide) instead of a single object. Unique index on `(AppointmentId, AttorneyType)`.
  - Pros: one entity to maintain, one Manager, one AppService, one controller, one Angular surface. Row-level `IMultiTenant` stays in place unchanged. Firm reuse across both sides works naturally (a single firm may represent patient on one case and defense on another). Smallest migration surface.
  - Cons: single table collects rows for both sides -- admins of the applicant-attorneys list see mixed rows unless the list page filters by role. Firm-detail metadata does not naturally diverge between the two roles today, but if workers-comp ever demands a defense-only field (e.g. carrier ID) it must live on both sides or be ExtraProperties.
- **B. Split into two entities: `PatientAttorney` + `DefenseAttorney`, each with identical firm-field schema. Add parallel join entities `AppointmentPatientAttorney` + `AppointmentDefenseAttorney`.** Chosen if Q1 = yes. Mechanics: duplicate the existing `ApplicantAttorneys` feature folder under both names; each has its own Manager, AppService, controller, Angular module, permission tree. Appointment upsert branches on side; `GetAppointmentApplicantAttorneyAsync` splits into two methods.
  - Pros: each side has its own permission tree, admin surface, and audit scope; matches OLD schema one-for-one; future defense-only columns are trivial.
  - Cons: roughly doubles the surface area (entity + Manager + 9 endpoints + mapper + proxy + Angular module + permission set x 2). Breaks the single upsert endpoint into two. Firm reuse across sides requires duplicate rows or a foreign-key to a shared Firm aggregate that doesn't exist yet. Migration is larger (two new tables, a new join, data copy).
- **C. Keep current unified model; treat Defense Attorney as a role on `AppointmentAccessor` only.** Rejected. Loses the firm-detail separation (AppointmentAccessor has `IdentityUserId` + `AccessTypeId` only; no firm fields). Defeats the purpose of tracking Defense firm info at all, which is a standard workers-comp reporting requirement.
- **D. TPH inheritance: `ApplicantAttorney` base + `PatientAttorneyEntity` + `DefenseAttorneyEntity` subclasses with EF `HasDiscriminator<string>`.** Conditional. Works cleanly with ABP and Mapperly (derived-types support), but is overkill when the subclasses have zero divergent fields today. If Q1 = yes AND Gesco later wants per-side-specific fields, re-open as a refactor.
- **E. Move the discriminator to the main `ApplicantAttorney` entity (not the join): add `AttorneyType` on `ApplicantAttorney`.** Rejected. Couples firm identity to role; the same firm (e.g. Gallagher) may represent the patient on one case and the defense on another. `AttorneyType` belongs on the appointment-scoped join, not the firm aggregate.

## Recommended solution for this MVP

Default (if Adrian answers Q1 = no and Q2 = rename): **Alternative A with a rename**. Add a `Domain.Shared/AppointmentApplicantAttorneys/AttorneyType.cs` enum (`Patient = 1`, `Defense = 2`); add a non-nullable `AttorneyType` column to `AppointmentApplicantAttorney` with a sensible default (`Patient` backfill for existing rows) via EF migration; add a composite unique index `(AppointmentId, AttorneyType)` on the join. Update `AppointmentApplicantAttorneyManager.CreateAsync/UpdateAsync` to accept and persist `attorneyType`. Update `AppointmentsAppService.UpsertApplicantAttorneyForAppointmentAsync` signature to carry `AttorneyType` (extend `ApplicantAttorneyDetailsDto` with the field) and change read path (`GetAppointmentApplicantAttorneyAsync`) to return a two-slot DTO (`PatientSide`, `DefenseSide`) pulling up to two rows. The underlying `ApplicantAttorney` firm aggregate is unchanged; optionally rename `ApplicantAttorney` to `PatientAttorney` across code + folder + route (`api/app/patient-attorneys`) in a follow-up ticket to track Q2's "rename" intent -- this is a cosmetic rename, not a behaviour change. Regenerate Angular proxy; the `applicant-attorneys` page keeps the same logic but is re-labelled. Add a second `AttorneyType` toggle on the appointment booking page to let a clerk add a Defense attorney link alongside the existing Patient link. Permission-wise, gate the Defense section behind the already-seeded `Defense Attorney` role's view permission (create a new child perm `CaseEvaluation.AppointmentApplicantAttorneys.ManageDefense` and wire it through the permission provider). Locations in NEW to change: `Domain.Shared/AppointmentApplicantAttorneys/AttorneyType.cs` (new), `Domain/AppointmentApplicantAttorneys/AppointmentApplicantAttorney.cs` (add property), `Domain/AppointmentApplicantAttorneys/AppointmentApplicantAttorneyManager.cs` (accept + persist), `Application.Contracts/AppointmentApplicantAttorneys/*Dto.cs` (surface field), `Application/AppointmentApplicantAttorneys/AppointmentApplicantAttorneysAppService.cs` (include in CRUD), `Application/Appointments/AppointmentsAppService.cs:370-463` (two-slot read + type-aware upsert), `EntityFrameworkCore/Migrations/<ts>_AddAttorneyTypeToAppointmentApplicantAttorney.cs` (migration), `angular/src/app/appointments/...` (UI toggle) plus auto-generated proxy files.

Fallback (if Adrian answers Q1 = yes): Alternative B. Add `src/.../Domain/DefenseAttorneys/` mirroring `ApplicantAttorneys/` (entity + Manager + Repository + WithNavigationProperties) and `src/.../Domain/AppointmentDefenseAttorneys/` mirroring `AppointmentApplicantAttorneys/`, plus full stack through Application.Contracts, Application, HttpApi, Angular. Firm reuse across the two sides is given up; if Gesco wants it, add a shared `AttorneyFirm` aggregate in a later iteration.

## Why this solution beats the alternatives

- Minimises migration surface -- one new enum, one new column, one new unique index, one new permission, vs B's full-feature duplication.
- Preserves Mapperly source-gen contracts (ADR-001): only the existing mappers grow a field. B requires a whole new set of mappers.
- Keeps the single `api/app/appointment-applicant-attorneys` endpoint stable (ADR-002 manual controller is edited, not duplicated). B forces a brand new controller + proxy set.
- Defense-side workflows (accessor auto-provisioning, appointment-documents) gain the `AttorneyType` discriminator without waiting for a second entity -- fewer cross-capability blockers.
- Reversible: if Q1 changes to yes later, we can extract the Defense rows into their own table via a data migration without touching OLD data (we never imported it).

## Risk and rollback

- Blast radius (Alternative A): `AppointmentApplicantAttorney` table gains one column + one unique index; rows default to `AttorneyType = Patient` so existing behaviour is preserved. Appointment upsert + read endpoints gain a parameter / shape field; Angular must be shipped together to avoid a UI crash on DTO shape change. If a prod deploy lands backend-only, the existing UI keeps working against the default `Patient` AttorneyType (backward-compatible behaviour).
- Blast radius (Alternative B): two new tables, two new permission groups, two new Angular modules, two new controllers. Higher deployment risk; requires full-stack atomic release. No silent fallback.
- Rollback (Alternative A): revert the EF migration down (`dotnet ef database update <prev>`; see `rules/dotnet-env.md`), revert the branch; the `AttorneyType` column and unique index disappear, AppService and DTO changes revert in the same diff. LocalDB data with defense rows would be lost -- acceptable at MVP since no prod data exists yet per track 10.
- Rollback (Alternative B): revert migrations for both new tables (pre-deployment if feature-flagged); the new controllers/modules simply cease to exist on the generated proxy's next run.
