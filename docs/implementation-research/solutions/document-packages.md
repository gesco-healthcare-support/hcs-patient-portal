## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, EF Core 10, Angular 20. Per root `CLAUDE.md` Stack Table and `docs/decisions/`.
- ADR-001 (Riok.Mapperly): any DTO mapping for package aggregates lives as `[Mapper]` partial classes in `CaseEvaluationApplicationMappers.cs`; no AutoMapper calls.
- ADR-002 (manual controllers): a `DocumentPackageAppService` MUST carry `[RemoteService(IsEnabled = false)]` and a hand-written controller in `src/.../HttpApi/Controllers/` (pattern as in `AppointmentTypeController.cs`).
- ADR-003 (dual DbContext): if the package entity is tenant-scoped, config lives in `CaseEvaluationDbContext` but MUST NOT be guarded by `IsHostDatabase()`. If it is host-scoped (shared-template model), it IS guarded.
- ADR-004 (doctor-per-tenant): "packages differ per clinic" in the kickoff prompt implies tenant-scoped. The package entity therefore SHOULD implement `IMultiTenant` so the automatic tenant filter keeps each doctor's template list isolated.
- HIPAA: package rows themselves contain no PHI -- they describe a TEMPLATE (cover letters + intake forms + history forms). But the actual file bodies behind each `DocumentPackageDetail` are blobs that are NOT PHI at rest (they are blank form templates for the clinic) and become PHI only when filled and attached to an appointment. The `appointment-documents` flow handles PHI attachments; package management handles template metadata. Document this boundary in the brief; do not confuse the two.
- Blob provider: per `blob-storage-provider` sibling brief recommendation, use ABP default DB-backed `IBlobContainer<T>` with a marker container class. Introduce `DocumentPackageTemplatesContainer` for the template file bodies.
- Windows short-path constraint (root `CLAUDE.md`): no impact -- new file additions under an already-short path.
- Q9 open scope (gap-analysis README line 239): "Document Packages + Package Details: required for MVP? (Tracks 1, 2, 3, 4, 9)" is the single blocker. If Adrian answers "no for MVP", this capability is post-MVP and no code is written; if "yes", the recommendation below applies.
- Schema-gap erratum from this research: the kickoff prompt's proposed shape `DocumentPackage (Id, TenantId, Name, Description) + PackageDetail (Id, PackageId, FileDescriptorId, Order)` has the roles reversed vs OLD. NEW should use the OLD semantics but with canonical ABP names: a header aggregate + a line-item child, not two parallel peers.

## Alternatives considered

- A. **Skip document packages entirely for MVP; revisit post-MVP on request.** `conditional on Q9 = no`. Zero MVP implementation cost. Matches the "dead-code in OLD's runtime" evidence: nothing breaks by not porting a feature that was already abandoned in production. Admin clinic staff currently attach form templates manually via the `appointment-documents` upload surface once that capability lands. If clinic ops want template reuse later, A is the starting point of a post-MVP feature design.
- B. **Port DocumentPackage + DocumentPackageDetail as new ABP aggregates, admin CRUD only, NO auto-attach.** `conditional on Q9 = yes`. Matches the OLD admin tab's observable surface without replicating the dead auto-attach path. Adds two tenant-scoped aggregates to `CaseEvaluationDbContext`, one `IDocumentPackageRepository`, one manager + appservice + manual controller, one Mapperly mapper, four permissions (`DocumentPackages.{Default,Create,Edit,Delete}`), and one Angular list/detail module. Effort ~M (4-5 days). Template files live in a typed blob container; each `DocumentPackageDetail` row stores a `FileDescriptorId` or a blob key. Intake staff attach templates manually at booking; no event handler is wired.
- C. **Port DocumentPackage + DocumentPackageDetail AND add an `AppointmentCreated` event handler that clones package rows into `AppointmentDocument` rows.** `conditional on Q9 = yes AND explicit auto-attach request`. Extends B with a `AppointmentCreatedEvent` publisher in `AppointmentManager.CreateAsync` (or use ABP's built-in `EntityCreatedEventData<Appointment>`) plus an `ILocalEventHandler<AppointmentCreatedEvent>` in `.Application/` that selects the tenant's `DocumentPackage` for the appointment's `AppointmentTypeId` and creates one `AppointmentDocument` row per `DocumentPackageDetail`. Effort adds +1-2 days vs B. Risk: copies template files into every appointment at booking, multiplying blob storage; also ties booking performance to package lookup. Higher blast radius on refactor. This is the pattern the kickoff prompt described but no OLD evidence supports as ever having worked.
- D. **Use ABP's `SettingManagement` with per-tenant keys `packet.patient.url`, `packet.doctor.url`, ...** `rejected`. Closest match to OLD's actual runtime (static per-role S3 URLs). Zero schema work; wires Adrian's existing `ServerSetting` mental model to `ISettingProvider`. Rejected for two reasons: (1) OLD maps keys by appointment-type, not package-template; SettingManagement would either flatten that or require encoded keys, both brittle. (2) Template metadata (display name, ordering, status) has no natural home in a setting value and is exactly the admin UX OLD's tab provided.
- E. **Use ABP File Management folders-as-packages.** `rejected`. Viable structurally: use a `DirectoryDescriptor` per `AppointmentType` and put each template file inside. Lets the admin UI be the stock File Management page. Rejected because (a) `DirectoryDescriptor` has no `AppointmentTypeId` FK or typed metadata -- the mapping is string-name convention, brittle to rename, and not query-friendly; (b) ABP docs confirm File Management lacks named-collection metadata beyond folder name, so we cannot carry `Description`/`Order`/`Status` on a package without a second table anyway -- which eliminates the simplicity argument.

## Recommended solution for this MVP

If Q9 answers `no`, recommend **Alternative A**: document as post-MVP, no code written, tracked as follow-up. Clinic staff use the `appointment-documents` capability to attach individual template files when `appointment-documents` (DB-01 / G2-08) lands.

If Q9 answers `yes`, recommend **Alternative B** (NOT C): port the admin CRUD cleanly, skip the auto-attach event handler, let intake staff trigger template selection explicitly at booking. Concrete shape:

- Domain entity `src/HealthcareSupport.CaseEvaluation.Domain/DocumentPackages/DocumentPackage.cs`: `FullAuditedAggregateRoot<Guid>, IMultiTenant`. Fields: `Name` (string, required, max 100), `Description` (string?, max 500), `AppointmentTypeId` (Guid FK to host `AppointmentType`, required), `TenantId` (Guid, set from `ICurrentTenant`), `Details` (ICollection<DocumentPackageDetail>, aggregate-owned).

Decision between `FileDescriptorId` vs `BlobName` path is sub-question 1 below.
- `DocumentPackageManager` in `src/HealthcareSupport.CaseEvaluation.Domain/DocumentPackages/`: `DomainService` with `CreateAsync` (name uniqueness-per-appointment-type-per-tenant validation), `UpdateAsync`, `AddDetailAsync`, `RemoveDetailAsync`, `SetActiveAsync`.

## Why this solution beats the alternatives

- A vs B is purely scope: if Q9=no, A is correct by design; if Q9=yes, B is correct because the admin CRUD surface is what OLD's clinic staff rely on, and NEW should provide it.
- B beats C because the "auto-attach on booking" behavior is not evidenced in OLD's runtime (the `//Not needed as changes in requirement` dead-code literal `77` at `AppointmentDocumentDomain.cs:1109` is the only auto-attach shaped thing in OLD). Adding an event handler that was never proven in production adds migration risk (blob duplication, booking latency) and a new failure mode (booking fails if package lookup fails) with no observable OLD-parity benefit. If clinic ops later confirm they want auto-attach, C is a strict extension of B and can land as a follow-up.
- B beats D (`SettingManagement` packet URLs) because it preserves the admin UX (list/add/edit packages in the UI) and the metadata (description, per-detail ordering). D collapses templates into opaque setting strings that the admin cannot manage through a dedicated UI.
- B beats E (File Management folders) because `DirectoryDescriptor`/`FileDescriptor` lack the typed FK to `AppointmentTypeId` and the per-detail `Order`/`Status` the OLD surface carried. Using File Management would require a shadow metadata table anyway, so the simplicity argument fails.

## Risk and rollback

- Blast radius if implemented poorly: tenant-scoped package rows leak across tenants if `IMultiTenant` is not implemented correctly -- but this is the same risk every tenant entity already carries, and the Initial migration's `AbpBlobs.TenantId` index pattern gives the blueprint. Package row corruption cannot break appointment booking because booking does not read packages. The only user-facing surface at risk is the admin "Document Packages" page.
- Rollback path: revert the single `AddDocumentPackages` migration (`dotnet ef database update <previous-migration> --project src/.../EntityFrameworkCore --startup-project src/.../HttpApi.Host`), delete the 2 tables, remove the feature files, redeploy prior app version. No foreign keys from other entities (outside the aggregate itself) are introduced; removal is clean.
- If the blob provider is swapped post-MVP (DB -> S3 per `blob-storage-provider` brief), template blobs follow the same offline-migration path as appointment-documents. Package row metadata is unaffected.

## Open sub-questions surfaced by research

1. **Template file storage primitive**: `FileDescriptorId` (ABP File Management's `FmFileDescriptor`) vs `BlobName` (raw `[BlobContainerName]` marker). Harmonise with the `appointment-documents` decision; whichever brief answers first sets the pattern. Default: blob-container marker (simpler, avoids wiring `FileDescriptor` record creation for admin-uploaded templates).
2. **Auto-attach on booking (Alternative C)**: not supported by OLD evidence and not requested explicitly in Q9's verbatim text. Flag to Adrian with this brief's finding and wait for a yes/no. If yes, add as a follow-up extension of B (+1-2 days).
3. **Package-header tenancy model**: the kickoff prompt said per-clinic (tenant-scoped). Confirm Adrian wants per-clinic customisation, not a shared host-wide catalogue (in which case the entity becomes host-scoped + no `IMultiTenant`, and `AppointmentType`-level uniqueness alone suffices).
4. **Versioning**: if a clinic updates a package template file, does the rev replace the blob (simple) or append a new version (audit-friendly)? OLD did not version. Default: replace-in-place for MVP, note as post-MVP upgrade.
5. **Name uniqueness scope**: OLD's `PackageDetailDomain.AddValidation` enforced one package per AppointmentType (not unique name). The recommended NEW constraint is `(TenantId, AppointmentTypeId, Name)` uniqueness, which is stricter. Confirm this matches clinic workflow (allow multiple packages per type, each with a distinct name) vs OLD's one-per-type rigidity.
