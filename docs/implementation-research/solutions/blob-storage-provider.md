## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, EF Core 10 per root `CLAUDE.md` Stack table. ABP's `Volo.Abp.BlobStoring` is the canonical provider abstraction; every provider (Database, FileSystem, AWS, Azure, MinIO) swaps behind the same `IBlobContainer` interface.
- ADR-003 (dual DbContext, host + tenant with `IsHostDatabase()` guard) -- the default `BlobStoringDatabaseModule` writes `AbpBlobs` rows into the same physical DB as `CaseEvaluationDbContext`. With ABP's "database-per-tenant" pattern and `AbpBlobs.TenantId` being part of the unique index (migration Initial line 1162-1163), blob rows are automatically isolated by tenant.
- ADR-004 (doctor-per-tenant) -- patient/appointment documents are tenant-scoped; anonymous-document-upload documents need to land with a resolved tenant id (via appointment-id-to-tenant lookup) so the multi-tenant filter keeps them from leaking.
- ADR-001 (Riok.Mapperly), ADR-002 (manual controllers with `[RemoteService(IsEnabled = false)]`), ADR-005 (no `ng serve`) apply transitively: any new `DocumentAppService` must carry the `[RemoteService(false)]` attribute and a hand-written controller in `HttpApi/Controllers/`, mappers must live in `CaseEvaluationApplicationMappers.cs` as `[Mapper]` partial classes.
- HIPAA -- appointment documents ARE PHI. Bucket ACL must be private; object-level encryption at rest is mandatory for any S3 path. The OLD `PublicReadWrite` ACL must not be replicated. For MVP-on-DB, SQL Server TDE + connection encryption carry the burden.
- Windows short-path constraint (root `CLAUDE.md` Critical Constraints) -- no impact on blob storage but relevant if future tests drop files to disk.
- Q17 open scope (gap-analysis README line 250): "Storage provider for blobs: DB BLOB (ABP default, works now) or S3 (OLD parity, needs creds)?" -- this is the eliminating constraint that narrows alternatives B and C below.
- File Management module (Pro) is already licensed and wired at every layer; choosing it vs. rolling our own `IBlobContainer` consumers is itself a sub-decision.
- DB BLOB performance ceiling -- `varbinary(max)` + `AbpBlobs.Content` is adequate for IME PHI packets (typical 1-20 MB PDFs) up to the 2 GB SQL Server per-row limit. Above that, S3 is required. Gap-analysis README line 250 does not constrain volume, so we assume single-digit MB per doc.

## Alternatives considered

- A. **DB BLOB provider via ABP's default `BlobStoringDatabaseModule`** -- `chosen`. The module is already in `[DependsOn]`, tables `AbpBlobContainers` + `AbpBlobs` already exist in the Initial migration, tenant filter applies through `AbpBlobs.TenantId`. We add marker container classes under `src/HealthcareSupport.CaseEvaluation.Domain/BlobContainers/` plus thin `IBlobContainer<T>` consumers in four appointment-related AppServices (appointment-documents, joint-declarations, document-packages, anonymous-document-upload). Reason: zero new infrastructure, zero credentials, zero per-environment config drift; works with the existing LocalDB dev setup today.
- B. **S3 via `Volo.Abp.BlobStoring.Aws` v9.1.1 with `UseCredentials = true` (IAM role)** -- `conditional post-MVP`. Same marker container classes and consumers as A; the only delta is a config section in `appsettings.json` (`BlobStoring:Default:UseAws = true` plus `BucketName`, `CreateBucketIfNotExists`) plus an `AbpBlobStoringOptions` registration in `CaseEvaluationHttpApiHostModule.cs`. Reason for deferral: requires AWS account + IAM + KMS + BAA setup, none of which are in place for MVP per Q17. Adrian flagged `server-settings.json` committed keys (Q26) as a burn-before-cutover item, so tying MVP to a dependency on rotated credentials adds scope.
- C. **Volo File Management Module (Pro) with hierarchical folders + per-tenant quotas instead of per-appointment `IBlobContainer`** -- `rejected`. The module is already wired (the 12 Swagger paths in Probe 1 prove it) and could serve as the entire storage layer, but its REST surface exposes a generic file-tree with a single shared set of directories, not appointment-scoped buckets. Reason for rejection: replicating OLD's seven logical containers (patientpacket, doctorpacket, etc.) maps more naturally to `[BlobContainerName]` markers than to File Management directories, and mixing generic file-manager uploads with appointment-document workflow complicates audit trails and permission checks. We may later use File Management for ad-hoc admin file storage, but appointment documents belong in a purpose-built container set.
- D. **Local filesystem provider (`Volo.Abp.BlobStoring.FileSystem`) with `wwwroot/Documents/`** -- `rejected`. Matches OLD's de-facto behavior (local disk under `wwwroot/Documents/<bucket>`) but is unsuitable for multi-node deployment and does not align with the HIPAA encryption-at-rest stance. Reason for rejection: plain-disk write is a step backward from the DB-BLOB baseline and adds no capability.

## Recommended solution for this MVP

Use ABP's `BlobStoringDatabaseModule` (already in `[DependsOn]`) with marker container classes consumed via `IBlobContainer<T>` in the appointment-document AppServices. Concretely: add four C# marker classes under `src/HealthcareSupport.CaseEvaluation.Domain/BlobContainers/` -- `AppointmentDocumentsContainer`, `JointDeclarationsContainer`, `DocumentPackagesContainer`, `AnonymousUploadsContainer` -- each decorated with `[BlobContainerName("appointment-documents")]`, `[BlobContainerName("joint-declarations")]`, etc. Inject `IBlobContainer<AppointmentDocumentsContainer>` into the blob-backed AppServices (shape defined in the dependent briefs) and expose upload/download through hand-written manual controllers (ADR-002) on top of Mapperly-authored DTOs (ADR-001) in the `AppointmentDocuments` feature tree. Keep `appsettings.json` untouched for MVP: the default provider resolves at runtime to DB and `AbpBlobs.Content` rows are written into the same DbContext. Post-MVP migration to S3 is a config-only swap: install `Volo.Abp.BlobStoring.Aws`, add `[DependsOn(typeof(AbpBlobStoringAwsModule))]` to the HttpApi.Host module, and register `AbpBlobStoringOptions` with the AWS-backed default provider. Because the consumer-side `IBlobContainer<T>` call-sites stay identical, no application-layer code needs to change on cutover. For the File Management module (already mounted at `/api/file-management/**`), suppress the auto-generated controllers until a decision is needed (either remove the module from `[DependsOn]` or keep the API surface dark by removing its `IHttpApiModule` wiring) -- see Open sub-questions below. No schema migration is needed: the Initial migration already has `AbpBlobContainers` + `AbpBlobs`.

## Why this solution beats the alternatives

- A eliminates the Q17 dependency for MVP: DB BLOB works today on LocalDB with no credentials, no bucket setup, no region choice. B requires Q17 to be answered before any appointment-document work can start, and requires Gesco AWS + KMS + BAA to be in place.
- A preserves a zero-friction migration path to B (same `IBlobContainer<T>` consumers; config-only switch) -- the marker-class pattern is ABP's native affordance for exactly this scenario.
- A avoids the ACL + encryption-at-rest hazards that OLD's S3 integration brought with it (`PublicReadWrite` at `AmazonBlobStorage.cs:159`). Keeping blobs in the tenant DB puts them under SQL Server TDE/row-level-security boundaries that the tenant filter already enforces, without new HIPAA-compliance surface.
- A tracks ADR-004 (doctor-per-tenant) cleanly: `AbpBlobs.TenantId` column + `IX_AbpBlobs_TenantId_ContainerId_Name` unique index map directly to the existing per-tenant isolation that the rest of the app depends on. File Management (C) and filesystem (D) would both require manual tenant-scoping enforcement on top of a generic directory tree.

## Risk and rollback

- Blast radius if DB-BLOB is wrong: PHI documents land in `AbpBlobs.Content varbinary(max)` rows inside the tenant DB. The tenant filter is automatic (ABP behavior; confirmed by `IX_AbpBlobs_TenantId_ContainerId_Name` scope), so cross-tenant leak risk is bounded by the same filter that protects every other tenant-scoped entity. Performance ceiling is high-MB files bloating SQL Server row storage -- mitigated by keeping IME PDFs at single-digit MB each.
- Rollback path: remove the four marker classes and the `[DependsOn(typeof(BlobStoringDatabaseDomainModule))]` line (it is already imported via the existing module graph, so removal is a revert of the four marker classes + their four consumers). No schema rollback is needed because `AbpBlobs` + `AbpBlobContainers` are already in the Initial migration and are harmless when empty.
- Post-MVP migration risk: switching to S3 requires a one-time blob migration (`SELECT ... FROM AbpBlobs` → S3 upload) per tenant. This is a known, tractable operation and can be done offline per tenant.

## Open sub-questions surfaced by research

- The File Management module (Pro) is wired at every layer, and its 12 HTTP endpoints are live at `/api/file-management/**` (Probe 1). Is this intentional? If yes, we should define whether appointment-documents reuses the `FmFileDescriptor` tree or stays in a purpose-built `[BlobContainerName]`-annotated container set. If no, we should suppress the module's controllers (either remove from `[DependsOn]` or null out its HTTP API module). The recommendation above assumes we use `[BlobContainerName]`-annotated containers, not File Management descriptors.
- HIPAA audit-log expectations: does every blob read/write need an `AbpAuditLog` row (audit middleware covers the HTTP request but not the in-AppService `IBlobContainer.SaveAsync` call). If yes, we add `[Audited]` attributes or explicit `IAuditingManager` calls in the four consumer AppServices. Track 06 line 130 confirms `UseAuditing()` middleware is on, but does not address the blob-level granularity.
- Anonymous-document-upload (R-09) requires writing a blob from a non-tenant-resolved request context. The magic-link token resolver must set `using (_currentTenant.Change(tenantId)) { ... }` around the `SaveAsync` call. This is a constraint for the anonymous-upload brief, not this one, but it means the blob-storage-provider decision depends on consumers knowing they must switch tenant context.
