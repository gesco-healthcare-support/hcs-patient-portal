## Constraints that narrow the solution space

- **ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict.** Standard stack. No framework blockers.
- **Row-level `IMultiTenant` (ADR-004).** `AppointmentChangeRequest` is a child of `Appointment`. `Appointment` already implements `IMultiTenant` with a nullable `Guid? TenantId`. The child entity inherits the parent's tenant implicitly via the `AppointmentId` FK but MUST also implement `IMultiTenant` itself so ABP's automatic tenant filter applies to repository queries without a manual join on `Appointment.TenantId`. Pattern matches `AppointmentEmployerDetail` (a parallel child entity at `src/.../Domain/AppointmentEmployerDetails/AppointmentEmployerDetail.cs:15`) which declares `IMultiTenant` independently.
- **Mapperly (ADR-001).** Add three partial mapper classes to `CaseEvaluationApplicationMappers.cs`: `AppointmentChangeRequestToAppointmentChangeRequestDtoMapper`, `AppointmentChangeRequestWithNavigationPropertiesToWithNavigationPropertiesDtoMapper`, and one `LookupDto<Guid>` mapper if any dropdown consumes the list. No AutoMapper. Compile-time safety.
- **Manual controllers (ADR-002).** Each new AppService declared `[RemoteService(IsEnabled = false)]`. Controller at `src/.../HttpApi/Controllers/AppointmentChangeRequests/AppointmentChangeRequestController.cs` extends `AbpController, IAppointmentChangeRequestAppService` with explicit `[Route("api/app/appointment-change-requests")]`.
- **Dual DbContext (ADR-003).** `DbSet<AppointmentChangeRequest>` added in both `CaseEvaluationDbContext` (host+tenant) and `CaseEvaluationTenantDbContext` (tenant-only). `OnModelCreating` configuration lives in both contexts outside `IsHostDatabase()`. Generates two migrations (one per context).
- **HIPAA.** `Reason` text field may inadvertently contain PHI (a patient describing their injury history to justify a reschedule). Constrain UX copy (placeholder: "Enter scheduling reason -- do not include medical details") and enforce a server-side `MaxLength(500)` ceiling matching OLD. Audit logs redact the `Reason` field via ABP's `[DisableAuditing]` attribute or per-field `AuditingDataSelector` wiring. All workflow events land in [appointment-change-log-audit](appointment-change-log-audit.md) for HIPAA trail continuity.
- **Capability-specific: tight coupling to appointment-state-machine (G2-01).** Creating a ChangeRequest must flip the parent Appointment from `Pending`/`Approved` to `RescheduleRequested (12)` or `CancellationRequested (13)`. Approval must flip to `RescheduledNoBill (7) / RescheduledLate (8) / CancelledNoBill (5) / CancelledLate (6)`. Rejection flips back to the prior status. These transitions MUST go through the state-machine primitive shipped by G2-01; this capability does not ship its own state-transition logic. Effort assumes G2-01 exposes a `TransitionToAsync(appointmentId, targetStatus, reason)` or equivalent entry point the AppService can call.

## Alternatives considered

1. **A1. Full entity + state-machine integration (chosen).** Create `AppointmentChangeRequest` aggregate root + `AppointmentChangeRequestManager` domain service + `AppointmentChangeRequestAppService` + manual controller + Mapperly mappers + Angular module with three routes. Approve/reject calls through to G2-01's state-machine primitive on the parent `Appointment`. **Tag: chosen.** Mirrors the proven NEW pattern for `Appointment`, `ApplicantAttorney`, `AppointmentEmployerDetail` -- each a tenant-scoped aggregate with a Manager + AppService + Mapperly + manual controller. Queryable for the admin queue UI (UI-04, UI-05). Supports per-field audit via `FullAuditedAggregateRoot` plus the planned [appointment-change-log-audit](appointment-change-log-audit.md) capability. Minimal architectural novelty.
2. **A2. Use `Appointment.ExtraProperties` bag to hold reschedule/cancel request fields.** ABP provides an extensible property bag on every entity (`ObjectExtensionManager.MapEfCoreProperty<T, TProperty>()`). Storing `ReScheduleReason`, `CancelReason`, `RequestStatus`, `OldDoctorAvailabilityId` as extra properties on `Appointment` avoids a new table. **Tag: rejected.** `ExtraProperties` is stored as JSON and is **not LINQ-queryable** from the admin queue UI (UI-04 wants "all appointments where `ExtraProperties.RequestStatus = Pending AND ExtraProperties.Type = Reschedule`"). You would need to load every appointment and filter client-side, or fall back to raw SQL / stored procs (against ADR direction). Also produces no audit trail for repeated requests (a patient reschedules twice -- the second request overwrites the first in the bag). ABP docs explicitly list "queryability" as a reason to prefer a normalized entity over `ExtraProperties`.
3. **A3. Piggy-back on ABP's generic audit log (`AbpAuditLogs`, `AbpEntityChanges`) to track change requests as audit entries.** Every state transition already lands in the audit log; treat a "request" as a special-tagged log entry. **Tag: rejected.** Audit logs are descriptive (they record what happened), not workflow-capable (they do not model approval/rejection decisions, document attachments, or the wait-state pending queue). The queue UI needs a list of "pending-approval" items -- audit logs have no "pending" semantic; they are append-only records of things that already happened. Confirmed by ABP audit-logging docs and the existing `AuditLogging` module wiring in `CaseEvaluationHttpApiHostModule`.
4. **A4. Replicate OLD exactly: store procedure `spAppointmentChangeRequests` + nested route `/api/appointments/{id}/change-requests`.** Minimize porting work by keeping OLD's shape. **Tag: rejected.** NEW's core ADR direction (per gap-README "Intentional architectural differences" table) replaces 40+ stored procs with LINQ. Importing a stored proc for this capability alone reopens that architectural decision. Nested route is also fine, but inconsistent with every other NEW controller which uses the flat `/api/app/{entity-plural}` root. A1 produces the same functionality using the ADR-blessed pattern.
5. **A5. Defer entirely: live without change requests until post-MVP, rely on admin manual editing of appointments.** **Tag: rejected.** All four external roles (Patient, Adjuster, Applicant Attorney, Defense Attorney) have the self-service "Rescheduled Requests" use case in OLD. Removing it means external users cannot initiate a reschedule; they must phone the clinic. That regresses from OLD's external-user workflow and breaks the "external users book and reschedule their own appointments" MVP contract (per `docs/business-domain/USER-ROLES-AND-ACTORS.md`).

## Recommended solution for this MVP

Create a new aggregate root `AppointmentChangeRequest` under `src/HealthcareSupport.CaseEvaluation.Domain/AppointmentChangeRequests/`. Fields: `Id` (Guid, inherited), `AppointmentId` (Guid, FK), `RequestType` (enum `Reschedule = 1 / Cancel = 2`), `Status` (enum `Pending = 1 / Approved = 2 / Rejected = 3`), `RequestedByUserId` (Guid, FK IdentityUser), `RequestedAt` (DateTime, inherited via `CreationTime`), `Reason` (string, max 500, nullable for cancel), `NewStartTime` (DateTime?, nullable -- reschedule only; NEW slot candidate proposed by requester or picked from a `DoctorAvailability`), `NewDoctorAvailabilityId` (Guid?, nullable), `ProcessedByUserId` (Guid?, FK IdentityUser, nullable), `ProcessedAt` (DateTime?, nullable), `ProcessingNotes` (string, max 500, nullable). Implements `FullAuditedAggregateRoot<Guid>, IMultiTenant`. Tenant FK inherited via parent; still implement `IMultiTenant` so ABP's filter applies.

Domain service `AppointmentChangeRequestManager : DomainService` exposes `CreateAsync(appointmentId, requestType, reason, newStartTime?, newDoctorAvailabilityId?)`, `ApproveAsync(id, processingNotes?, concurrencyStamp)`, `RejectAsync(id, processingNotes, concurrencyStamp)`. Each method (a) loads the parent `Appointment`, (b) calls into G2-01's state-machine primitive `IAppointmentStateMachine.TransitionToAsync(...)` to flip the parent's status atomically, (c) writes the ChangeRequest row, (d) invokes G2-02's booking cascade for approved reschedules (release old slot, book new slot). Each method wraps work in `[UnitOfWork(isTransactional: true)]` so a partial failure rolls back both sides.

AppService `AppointmentChangeRequestAppService : CaseEvaluationAppService, IAppointmentChangeRequestAppService` -- declared `[RemoteService(IsEnabled = false)]` per ADR-002 -- surfaces the CRUD + workflow endpoints and delegates to the Manager. Controller at `src/.../HttpApi/Controllers/AppointmentChangeRequests/AppointmentChangeRequestController.cs` with `[Route("api/app/appointment-change-requests")]`.

Mapperly wiring: three partial classes in `CaseEvaluationApplicationMappers.cs` following the `Appointment*` pattern.

EF Core: `DbSet<AppointmentChangeRequest>` + `OnModelCreating` configuration in both `CaseEvaluationDbContext` and `CaseEvaluationTenantDbContext` (per ADR-003) outside `IsHostDatabase()`. Two migrations generated via `dotnet ef migrations add AddAppointmentChangeRequest -c CaseEvaluationDbContext` and the same with `-c CaseEvaluationTenantDbContext`.

Permissions: new nested static class `AppointmentChangeRequests` in `CaseEvaluationPermissions.cs` (Default / Create / Edit / Delete) plus new permissions `Approve` and `Reject` as siblings (the OLD equivalent `accessItem` was coarser; NEW's finer permissions give the Permission Management UI per-action grant control). Register in `CaseEvaluationPermissionDefinitionProvider.cs` matching the `Appointments` block layout.

Angular: new module `angular/src/app/appointment-change-requests/appointment-change-request/` following the `appointments/appointment/` layout (per CLAUDE.md reference pattern). Routes in `appointment-change-request-routes.ts`:

1. `''` -- reschedule queue (filters to `RequestType=Reschedule, Status=Pending`). Menu label "Reschedule Requests". Menu permission `CaseEvaluation.AppointmentChangeRequests` via `permissionGuard`. Path: `/appointment-rescheduled-requests` (root-mounted in `app.routes.ts` for URL parity with OLD UI-04).
2. Shared list component parameterized by `requestType`; mount a second route `/appointment-cancel-requests` pointing at the same component with `data: { requestType: 'Cancel' }`. Matches UI-05.
3. `'pending-request/:id'` -- detail view showing the parent Appointment summary plus the change-request decision panel (approve/reject buttons, notes field). Route-data `requiredPolicy: 'CaseEvaluation.AppointmentChangeRequests'`. Matches UI-06 and R-10.

Auto-generated proxy at `angular/src/app/proxy/appointment-change-requests/` produced via `abp generate-proxy` once the backend lands. Shared Angular menu metadata in `providers/appointment-change-request-base.routes.ts` registers the two queue routes under an "Appointments" menu group matching OLD's dropdown structure (see UI-04/UI-05 nav inventory in `09-ui-screens.md`).

Permission seeding for internal roles: once [internal-role-seeds](internal-role-seeds.md) lands, grant `Approve` + `Reject` to StaffSupervisor/ClinicStaff (per track-5 heuristic) and `Create` to all 4 external roles (external users can request reschedule/cancel for their own appointments). Exact matrix deferred to Q22 (seeded grants vs runtime admin-assigned) -- see "Open sub-questions" below.

## Why this solution beats the alternatives

- Normalised entity (A1) is LINQ-queryable for the admin queue UI (UI-04, UI-05) whereas `ExtraProperties` (A2) is not. The queue UI is the primary surface for this capability; it must be fast and paginatable.
- G2-01 + G2-02 ownership stays clean. The Manager delegates state transitions to the state machine rather than duplicating transition logic, so when G2-01 hardens its guards the change-request path inherits them for free.
- Mirrors proven NEW pattern (`ApplicantAttorney`, `AppointmentEmployerDetail` are comparable child aggregates), so implementation risk is low and review patterns are already established. New-developer onboarding cost is zero.
- Keeps ABP audit log (A3) for "what happened" and a dedicated entity for "what is pending". The two responsibilities remain separated; HIPAA audit continuity is preserved via the planned [appointment-change-log-audit](appointment-change-log-audit.md).

  - [appointment-change-log-audit](appointment-change-log-audit.md) (G2-13) -- audit trail for change-request state transitions depends on this entity existing.
- **Blocked by:**
  - [appointment-state-machine](appointment-state-machine.md) (G2-01) -- CreateAsync/ApproveAsync/RejectAsync must call into the state-machine primitive. Hard dependency; do not ship this capability before G2-01 exposes a `TransitionToAsync(...)` method or equivalent. Without G2-01, the Manager would duplicate state-transition logic and drift from the (future) canonical state machine.
  - [appointment-booking-cascade](appointment-booking-cascade.md) (G2-02) -- approved reschedules must (a) release the old `DoctorAvailability` slot and (b) book the new slot atomically. G2-02 owns the slot-release path; this capability calls into it.
  - [lookup-data-seeds](lookup-data-seeds.md) (DB-15) -- if the admin UI's reason dropdown is ever seeded from a lookup table (OLD has free-text; NEW could optionally wire `AppointmentStatus` reason labels), DB-15 must seed those rows first. Soft dependency; with free-text Reason (matches OLD), there is no hard block.
- **Blocked by open question:** **none directly blocks implementation.** The UX decision on UI-06 pending-detail scope (show parent appointment summary or just the change-request details) can be decided during Angular implementation. Q22 (seeded role permission grants) is needed to finalize the permission matrix but not to build the permissions and grant them at runtime.

## Risk and rollback

- **Blast radius:** large. An incorrect approval flow can write an Appointment into an invalid state (e.g., `RescheduledNoBill` without actually swapping slots), which affects the whole booking pipeline -- slot inventory goes wrong, patient/doctor see two bookings or none. HIPAA impact: the Reason field carries potentially-PHI text; a logging or audit-redaction miss leaks that text into diagnostic channels. Permission miss-grant can let external attorneys approve their own requests.
- **Rollback:**
  - Migration-level: `dotnet ef migrations remove -c CaseEvaluationDbContext` and same for `-c CaseEvaluationTenantDbContext`. This drops the `AppointmentChangeRequests` table cleanly (no dependent data since approval writes to the parent Appointment via the state machine; rejecting the migration re-orphans nothing).
  - Code-level: branch revert. The entity is net-new; removing it does not regress any existing behaviour. Shipping-in-progress parent appointments that were transitioned to `RescheduleRequested (12)` during the failed roll-forward stay in that state; an admin can manually set them back to their prior status via direct DB edit or an admin override path (tie-in with [appointment-full-field-snapshot](appointment-full-field-snapshot.md) if that capability ships a manual-override status change).
  - Feature flag (optional): gate the Angular menu entries and the controller routes behind an ABP `FeatureManagement` feature flag (`CaseEvaluation.AppointmentChangeRequests.Enabled`). Set `false` in production until all dependencies land. Flip to `true` per tenant once end-to-end smoke passes.

## Open sub-questions surfaced by research

1. **UI-06 pending-detail scope.** OLD's `/appointment-pending-request/:id` route shows the full parent Appointment with a child change-request panel. NEW could either (a) match OLD and show the parent summary + request details, or (b) ship a request-focused view that links back to `/appointments/view/:id`. Recommend (b) for simpler component ownership -- the parent-appointment view already exists. Confirm during Angular implementation. Not a scope-blocker; the backend is identical either way.
2. **`AdminReScheduleReason` column parity.** OLD's entity has a separate `AdminReScheduleReason` admin-facing free-text field. Recommend collapsing into `ProcessingNotes` (single admin-authored free-text on the decision). Confirm with Adrian; the collapsed field reduces schema surface.
3. **Attached documents for a ChangeRequest.** OLD has `AppointmentChangeRequestDocuments` collection (documents uploaded alongside a reschedule/cancel request to justify it). Recommend handling this entirely via [appointment-documents](appointment-documents.md) (G2-08) by letting document uploads reference a `changeRequestId` in addition to `appointmentId`. Tracked as a follow-up within the documents capability, not here.
4. **`IsBeyodLimit` (sic) flag.** OLD carries a boolean that distinguishes "within-policy reschedule" from "outside-policy reschedule" (ties into G2-03 lead-time limits). Recommend computing this dynamically at query time from the parent Appointment date and `SystemParameter.AppointmentLeadTime` rather than snapshotting on the entity, to avoid stale data when the lead-time policy changes. Soft dependency on G2-03 -- ship without the flag if G2-03 is late; admins see the raw dates and can judge. Also: parse the typo correctly -- NEW would name this `IsBeyondLimit`.
5. **External-user submission UX.** Can an external user (Patient / Attorney) submit a ChangeRequest from `/home` or only from an appointment-detail view? Recommend both -- a "Request Reschedule" CTA on `/home` and on `/appointments/view/:id`. UX-level, not architecture-level. Defer to [external-user-home](external-user-home.md) capability for the /home CTA placement.
6. **Concurrency model.** Two admins approve the same pending request simultaneously. Recommend standard ABP optimistic concurrency via the inherited `ConcurrencyStamp` on `FullAuditedAggregateRoot`, with a user-friendly "This request was already processed by another user" message on `AbpDbConcurrencyException`. No novel concurrency work required; matches existing pattern on every other aggregate in the codebase.
