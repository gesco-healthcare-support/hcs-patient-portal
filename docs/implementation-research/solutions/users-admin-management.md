## Source gap IDs

- [5-G09](../../gap-analysis/05-auth-authorization.md) -- Permission group: Users management. Source line 108 of README says "covered by ABP Identity". Source line 197 of track 05 cites `access-permission.service.ts:74` (OLD) vs "Only ABP `AbpIdentity.Users.*`" in NEW. Effort: Low.
- [A8-02](../../gap-analysis/08-angular-proxy-services-models.md) -- Users admin CRUD Angular proxy. Source line 202 cites OLD `user\users\users.service.ts:1-68` vs NEW `user-extended.service.ts` "ABP Identity extension only". Effort: S.

All three gap IDs target the same capability: user admin CRUD. NEW delegates to ABP Identity (both API and Angular UI) and extends it via `UserExtendedAppService` to keep the `Doctor` aggregate in sync when admin edits a user. This brief verifies that delegation against live services and documents the (zero-code) recommendation.

## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict. `Volo.Abp.Identity.Pro.*` modules ship the complete user admin surface; the project already `[DependsOn]`s `AbpIdentityApplicationModule` (Application/Module.cs:24). Cannot substitute a non-ABP Identity without unwinding OAuth/OIDC (would regress ADR-004 and the OIDC swap documented in `05-auth-authorization.md`).
- Row-level `IMultiTenant` (ADR-004), doctor-per-tenant. ABP Identity is tenant-aware out of the box (`IIdentityUserRepository` respects `ICurrentTenant`; host admins see the host-scope user list, tenant admins see only users in their tenant). No additional tenant-awareness wiring is needed; cannot be bypassed without breaking doctor-tenant isolation.
- Mapperly (ADR-001), manual controllers (ADR-002), dual DbContext (ADR-003), no ng serve (ADR-005). None of these apply. The Identity Pro module ships its own controllers, auto-generates its own Angular proxies, and owns its own DbContext projection inside `AbpIdentityDbContext`; no HCS project code participates in mapping or routing.
- HIPAA applicability: user admin pages display PII (usernames, emails, lockout status) but NOT PHI. Audit coverage for user changes is handled by ABP's `AbpAuditLogs` and `AbpSecurityLogs` tables (track 05 lists `/api/identity/security-logs` + `/api/identity/sessions` in the surface). Any bespoke admin UI would need to replicate this audit trail -- ABP's native UI already integrates with it.
- Capability-specific: the Doctor-sync override in `UserExtendedAppService.UpdateAsync` is the only bespoke behaviour. Unless an additional OLD-only field (e.g. OLD's retired `IsAccessor` flag) is reintroduced, no extension beyond Doctor name/email sync is needed. The `UserExtendedAppService` already provides the extension seam; a future field add would append an override here rather than forking a new controller.

## Alternatives considered

1. **A1. Adopt ABP Identity + `UserExtendedAppService` as-is (CHOSEN)** -- keep the current wiring. Confirmation of wiring is the deliverable. No code changes. Any project-specific field on the User entity that needs to sync to another aggregate is added as an override in `UserExtendedAppService` (already proven by the Doctor sync). ABP owns all user admin UX, audit, lockout, 2FA, password policy. **Tag: chosen**. Matches the intentional architectural swap at gap-analysis `05-auth-authorization.md:216-227` and the gap README line 108 verdict "covered by ABP Identity". Effort: S (~1 day), consumed by screenshot verification rather than implementation.
2. **A2. Build a custom `/users` admin screen + `UsersAppService` + custom permissions** -- port OLD's `users.service.ts:1-68` + the `/users` route. **Tag: rejected**. Duplicates `AbpIdentity.Users.*`, loses 2FA / lockout / security-log integration, multiplies maintenance surface. No business requirement for a custom UX per README Q21/Q22 phrasing (both are about role seeding and baseline grants, not UI). Would also re-introduce the OLD `IsAccessor` pattern the rewrite intentionally deleted.
3. **A3. Delete `UserExtendedAppService` and rely on pure ABP defaults** -- skip the Doctor-sync override; admins would have to manually update the Doctor aggregate after editing a user's name/email. **Tag: rejected**. Introduces a real product defect (stale Doctor record when admin renames a user via the Identity UI). Track 04 documents `UserExtendedAppService` as an intentional extras column in NEW ("Extras in NEW" table at `gap-analysis/README.md:216-223`). Keeping it is cheaper than removing it and patching the resulting data drift.
4. **A4. Expose the custom extension via a new top-level `/users-admin` route + AppService instead of extending `IdentityUserAppService`** -- wrap calls to `IdentityUserAppService` from a new thin facade. **Tag: rejected**. Adds a round trip, a mapping layer, and a separate proxy to maintain. The ABP-sanctioned pattern is class inheritance (per research source 3 above); switching to composition loses the automatic inheritance of 30+ methods already tested and maintained upstream. Also violates "ABP Conventions" in root `CLAUDE.md` ("Extend `CaseEvaluationAppService` or an ABP base class; do not wrap").
5. **A5. Seed additional internal roles (ItAdmin / StaffSupervisor / ClinicStaff) and grant them `AbpIdentity.Users.*`** -- extends the capability scope to include role-specific user admin. **Tag: conditional -- belongs to `internal-role-seeds` capability, not this one**. Once `internal-role-seeds` resolves Q21 (admin-only vs 3-tier split), whichever non-admin internal tiers exist will need `AbpIdentity.Users.*` grants assigned via the Permission Management UI. This capability inherits that as a dependency (see "Blocks" edge below), not as scope.

## Recommended solution for this MVP

Adopt `Volo.Abp.Identity.Pro` + `UserExtendedAppService` as-is. Zero code changes in this capability. Deliverables:

1. **Verification artifacts** (this brief + the probe log) document that `/api/identity/users/*` (30 endpoints), `/api/app/user-extended/*` (26 endpoints), and the Angular `/identity/**` route are all registered and reachable.
2. **Operator walkthrough** (post-approval, Phase 6): Angular screenshot run that captures `/identity/users` list, user edit drawer, role assignment dialog, lockout toggle, and security-log tail, saved to `docs/implementation-research/screenshots/users-admin/` -- using synthetic seed users only (the built-in `admin` account is acceptable; any additional capture data must be disposable test accounts created inside a disposable tenant).
3. **Edge-case behaviour to preserve** on any future `UserExtendedAppService` override: always call `base.<Method>Async(...)` first, then perform the domain sync. `UpdateAsync` is the only current override. If Q22 (external roles baseline grants) lands with grants that include `AbpIdentity.Users.ManagePermissions`, verify the existing override still works for non-admin callers with that grant.

Touch-points only if an extension lands (not in this capability's scope):
- `src/HealthcareSupport.CaseEvaluation.Application/Users/UserExtendedAppService.cs` -- add a new override (e.g. `DeleteAsync` that soft-deletes the Doctor, `CreateAsync` that auto-creates a Doctor entity for certain role assignments).
- `src/HealthcareSupport.CaseEvaluation.Domain/Doctors/IDoctorRepository.cs` -- add any new repository method the override needs.
- Regenerate Angular proxy via `abp generate-proxy` to pick up the override signature changes.

No entity, no controller, no permission group, no migration, no new Angular component, no new route is required.

## Why this solution beats the alternatives

- **Upgrade-safe**: staying on stock `Volo.Abp.Identity.Pro.*` and `@volo/abp.ng.identity` means every Volo 10.x patch release (security fixes, lockout refinements, 2FA UX tweaks) applies automatically. Any bespoke UI (A2) would miss those without active porting.
- **Audit and security coverage baked in**: `/api/identity/security-logs` and `/api/identity/sessions` (observed in the 58 identity-path swagger count) give HIPAA-relevant "who changed which user, when" coverage without us writing an audit table. Meeting HIPAA's access-control and audit requirements via an off-the-shelf, vendor-maintained surface is strictly lower risk than rolling our own.
- **Preserves the intentional architectural swap**: the rewrite removed the OLD `User` + `IsAccessor` pattern deliberately (gap-analysis "Extras in NEW"). Reverting (A2/A3) re-introduces exactly what the migration deleted; the single Doctor-sync override (`UserExtendedAppService`) is the minimal surface for the one extension that survived.
- **Zero Q-blocking path**: this capability depends on no open questions today and does not block any capability that is itself blocked. `internal-role-seeds` (Q21/Q22) is the only downstream touchpoint, and even there the dependency is "verify seeded roles appear in the `/identity/users` role-assign dropdown", which is pure operator verification.

## Risk and rollback

- **Blast radius:** the single file in scope is `UserExtendedAppService.cs`. A defect in the Doctor-sync override would desynchronise the Doctor's name/email from the IdentityUser's name/email but would NOT affect the Identity user record itself (the override calls `base.UpdateAsync` first, then the Doctor sync is best-effort). The `/api/identity/users/*` surface is owned by ABP and not reachable via this override. Angular `/identity/**` route failure would break admin user management only; other routes use their own guards and are isolated.
- **Rollback:** zero-change capability, so rollback is a no-op for the verification work. If a future override on `UserExtendedAppService` breaks something, revert the single file on the feature branch. No migration to revert. No data to unroll. The fallback "disable `UserExtendedAppService` entirely" works: remove the class, and requests for `PUT /api/app/user-extended/{id}` stop resolving; callers fall back to `PUT /api/identity/users/{id}` (base ABP route) which is still registered. The `user-extended` Angular proxy would 404; regenerate the proxy to remove it.

## Open sub-questions surfaced by research

- **Should `UserExtendedAppService.DeleteAsync` soft-delete the linked Doctor entity?** Today `DeleteAsync` is inherited unchanged from the base class. Deleting an IdentityUser that owns a Doctor row leaves an orphaned Doctor with a dangling `IdentityUserId`. Open sub-question to raise during `new-sec-03-transactional-tenant-provisioning` or a separate Doctor-lifecycle capability, not blocking this brief.
- **Should `UserExtendedAppService.CreateAsync` auto-create a Doctor row when the created user is assigned the `Doctor` role?** Currently no. `DoctorTenantAppService.CreateAsync` provisions Doctor + admin user atomically for the tenant-provisioning flow, but ad-hoc "admin edits a user and adds the Doctor role" does not create the Doctor entity. Open sub-question; same caveat -- belongs to a Doctor-lifecycle brief.
- **Should the Angular `/identity/users` list be customised to show the Doctor-linkage column (e.g. "is doctor", "tenant")?** ABP's `@volo/abp.ng.identity` supports UI customisation via component extenders. Out of MVP scope per 09-ui-screens.md line 148 ("delegate to ABP"); revisit post-MVP if operators ask for it.
- **Confirm Q21 (internal role structure) resolution:** once Adrian resolves Q21, this capability's post-resolution runbook (grant `AbpIdentity.Users.*` to non-admin internal tiers) becomes concrete. Pure follow-on, not a blocker.
