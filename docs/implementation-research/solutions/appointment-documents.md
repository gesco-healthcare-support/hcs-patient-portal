## Constraints that narrow the solution space

- ABP Commercial 10.0.2, .NET 10, Angular 20, OpenIddict, SQL Server LocalDB (dev), row-level `IMultiTenant` via ABP auto-filter. All per repo root `CLAUDE.md`.
- **ADR-001 Mapperly** -- all DTO <-> entity mappings for `AppointmentDocument`, `AppointmentDocumentCreateDto`, `AppointmentDocumentUpdateDto`, `AppointmentDocumentDto`, `AppointmentDocumentWithNavigationPropertiesDto`, `GetAppointmentDocumentsInput` live in `src/HealthcareSupport.CaseEvaluation.Application/Mappers/CaseEvaluationApplicationMappers.cs` as `[Mapper(RequiredMappingStrategy = RequiredMappingStrategy.Target)]` partial classes extending `MapperBase<TSource, TDest>`.
- **ADR-002 manual controllers** -- `AppointmentDocumentsAppService` (and `AppointmentDocumentTypesAppService`) must carry `[RemoteService(IsEnabled = false)]`; hand-written controller classes live in `src/HealthcareSupport.CaseEvaluation.HttpApi/Controllers/` and extend `AbpController, I{Entity}AppService`, delegating per method.
- **ADR-003 dual DbContext** -- `AppointmentDocument` and `AppointmentNewDocument` are tenant-scoped (via `IMultiTenant` on the entity). Register OnModelCreating in `CaseEvaluationDbContext` without the host guard; the ABP auto-filter handles tenant isolation. `AppointmentDocumentType` is host-scoped; its configuration must be inside `if (builder.IsHostDatabase())`.
- **ADR-004 doctor-per-tenant** -- document rows always resolve to a tenant; uploads through the authenticated tenant path flow through the ABP data filter. The anonymous-magic-link path (sibling brief `anonymous-document-upload`) is the only route that must explicitly `using (_currentTenant.Change(tenantId)) { ... }` around `IBlobContainer.SaveAsync` after resolving appointment -> tenant.
- **ADR-005 no `ng serve`** -- Angular work uses `npx ng build --configuration development` + `npx serve -s dist/CaseEvaluation/browser -p 4200`. The appointment-document Angular feature must not add `yarn start` / `ng serve` scripts.
- **HIPAA** -- IME documents carry PHI (medical records, claim files, patient IDs). Constraints: (1) no `PublicReadWrite` ACL replication from OLD (blob-storage-provider brief alternative A uses DB BLOB + SQL Server TDE, which keeps the HIPAA surface tight for MVP); (2) encryption at rest mandatory -- default provider relies on SQL Server TDE or storage-layer encryption; (3) retention policy 7 years per workers-comp standard, which means soft-delete via `ISoftDelete` + retention job downstream (retention job is out-of-scope for this brief but must be plannable); (4) audit-log each upload/approve/reject/download via `[Audited]` attributes on the AppService methods + `AbpAuditLog` rows; (5) restrict download routes by permission + optional anonymous verification-code path for external stakeholders (out-of-scope here; lives in `anonymous-document-upload`).
- **Per-appointment scoping + role visibility** -- documents belong to a single `AppointmentId`. Visibility: admin / clinic-staff (all in tenant), patient (only their appointments), applicant-attorney (only linked appointments via `AppointmentAccessor` or `AppointmentApplicantAttorney`), defense-attorney (TBD per Q1). Filter applied in `GetListAsync` via join to `AppointmentAccessor` when caller is external (mirrors the pattern in `DoctorsAppService` with `IDataFilter.Disable<IMultiTenant>` for host-scoped cross-tenant queries -- NOT applicable here; tenant filter stays on).
- **Verification code gating** -- OLD uses a Guid `VerificationCode` per `AppointmentDocument` row to permit anonymous download. Keep it as a nullable Guid column; only populate for external-visible rows. The magic-link flow itself is the `anonymous-document-upload` brief.
- **Blob backend** -- Q17 open (blob provider: DB BLOB vs S3). Recommendation per `blob-storage-provider.md` is DB BLOB for MVP via `IBlobContainer<AppointmentDocumentsContainer>`. Consumers here use the marker class; post-MVP S3 swap is config-only.
- **Lookup seed** -- `AppointmentDocumentType` seed rows required for dropdowns (MedicalRecords, ClaimFile, XRay, MRI, etc.). Adds a row to the `lookup-data-seeds` capability (DB-15, Q23 open) or a dedicated `IDataSeedContributor` that writes to the host DB.
- **Permission group** -- new nested static class `CaseEvaluationPermissions.AppointmentDocuments` with children `Default`, `Create`, `Edit`, `Delete`, plus `Approve` (OLD has explicit Approve action; mirrors UI workflow). Register in `CaseEvaluationPermissionDefinitionProvider.cs`. This resolves 5-G05.
- **File-size + antivirus** -- not addressed in OLD. For MVP: enforce single-file upload size cap in the AppService (suggest 25 MB matching clinical PDF norms); AV scan is flagged as a downstream hook (out-of-scope for this brief; tracked in Open sub-questions).

## Alternatives considered

- **A. Full domain aggregate: `AppointmentDocument` + `AppointmentNewDocument` + `AppointmentDocumentType` with `IBlobContainer<AppointmentDocumentsContainer>`, full AppService + manual controller + Angular feature module + permission group.** Chosen. Matches OLD feature parity, respects ADRs (manual controller, Mapperly, dual-DbContext), uses already-wired ABP blob storage, adds per-appointment scoping + verification code + role visibility. The separate `AppointmentNewDocument` sibling entity (matching OLD) preserves the pre-approval-vs-post-approval distinction that drives different due-date / notification workflow.
- **B. Single `AppointmentDocument` entity with `IsPostApproval` bool flag instead of separate `AppointmentNewDocument`.** Conditional. Simpler schema (one table), one AppService, UI differentiates via the flag. Downside: OLD consumer code branched on the entity type -- queries like "pending pre-approval docs" vs "post-approval uploads" become conditional filters instead of table scans; the due-date gate logic lives in a branch rather than a separate workflow. Reason not chosen primary: requires an extra decision from Adrian (is the pre-vs-post distinction meaningful in NEW?) and would make the DB search/approval UI less clean. Retained as a possible simplification if Q17 resolves toward "collapse document surface".
- **C. Use ABP FileManagement Module folders directly instead of a purpose-built aggregate.** Rejected. FileManagement is already mounted (its 12 `/api/file-management/**` paths are live per blob-storage-provider Probe 1), but it exposes a generic hierarchical file tree with no business workflow (no approval status, no rejection reason, no verification code, no `DueDate` gate, no `UserType` tracking). Using it would require rebuilding the approval workflow in a separate parallel entity tree, producing a clumsier surface than alternative A. FileManagement remains appropriate for ad-hoc admin uploads that are not appointment-scoped.
- **D. Store blobs as S3 paths in string columns (OLD parity literal port).** Rejected. Replicates OLD's leakage risk (`PublicReadWrite` ACL, plaintext AWS keys in `server-settings.json`); requires Q17 to be answered in the S3 direction AND a Gesco AWS + BAA + KMS setup before any appointment-document work can start. Alternative A uses the same `IBlobContainer<T>` shape so a post-MVP migration to S3 is a config-only swap with no app-code changes.
- **E. Use OpenAPI file-descriptor upload + separate `AppointmentDocumentMetadata` entity that holds only metadata and a `FileDescriptorId` FK.** Conditional post-MVP. Splits blob storage from appointment-document metadata, allowing files to live in FileManagement with metadata in the appointment aggregate. Downside: two systems of record for one logical entity; joint queries become cross-module. Reason not chosen now: no immediate need to decouple, and alternative A keeps the aggregate clean.

## Recommended solution for this MVP

Build a full `AppointmentDocuments` aggregate under `src/HealthcareSupport.CaseEvaluation.Domain/AppointmentDocuments/` containing three entities -- `AppointmentDocument` (pre-approval, `FullAuditedAggregateRoot<Guid>` + `IMultiTenant`, fields: Id, AppointmentId, DocumentTypeId nullable + OtherDocumentTypeName string, BlobName string, BlobContainerName derived, FileName string, FileSize long, ContentType string, Status enum `DocumentStatus { Pending=1, Uploaded=2, Accepted=3, Rejected=4, Deleted=5 }`, UploadedByUserId Guid, ResponsibleUserId Guid, DueDate DateTime nullable, VerificationCode Guid nullable, RejectionNotes string nullable, RejectedByUserId Guid nullable, IsJointDeclaration bool, ConcurrencyStamp); `AppointmentNewDocument` (post-approval sibling, same shape minus IsJointDeclaration, VerificationCode, DueDate -- since post-approval uploads have no gate -- but including a `SourceDocumentId` Guid nullable to link back to the pre-approval original when a re-upload replaces a rejected document); `AppointmentDocumentType` (host-scoped lookup, `FullAuditedAggregateRoot<Guid>`, DocumentTypeName, DisplayOrder, IsActive). Add three `*Manager` domain services (`AppointmentDocumentManager`, `AppointmentNewDocumentManager`, `AppointmentDocumentTypeManager`) following the slim-manager convention of the existing 11 managers -- null + length checks only; the upload-side orchestration (blob write, status initialisation) lives in the AppService. Add the three corresponding `*AppService` classes extending `CaseEvaluationAppService`, all carrying `[RemoteService(IsEnabled = false)]` (ADR-002). The AppServices inject `IBlobContainer<AppointmentDocumentsContainer>` (marker class added in the blob-storage-provider capability) and orchestrate: upload (save stream + insert entity with `DocumentStatus.Pending` or `Uploaded` based on caller role); approve (flip to `Accepted`, set ResponsibleUserId, raise email event via `IDistributedEventBus` so `email-sender-consumer` can fan-out); reject (flip to `Rejected`, capture RejectionNotes, email event); delete (soft-delete flip to `DocumentStatus.Deleted` + `ISoftDelete.IsDeleted`); verification-code lookup for external read. Add manual controllers under `src/HealthcareSupport.CaseEvaluation.HttpApi/Controllers/AppointmentDocuments/` with flat route `[Route("api/app/appointment-documents")]` for approval search / admin ops plus nested route `[Route("api/app/appointments/{appointmentId}/documents")]` for per-appointment list + upload (OLD parity). Upload actions use `[HttpPost("upload")] [Consumes("multipart/form-data")]` accepting `IFormFile` bound via `[FromForm]`. Download actions use `[HttpGet("{id}/download")]` returning `FileStreamResult` from `IBlobContainer.GetAsync`. Add DTOs (`AppointmentDocumentCreateDto`, `UpdateDto`, `Dto`, `WithNavigationPropertiesDto`, `GetAppointmentDocumentsInput` plus `ApproveAppointmentDocumentDto`, `RejectAppointmentDocumentDto`) in `src/HealthcareSupport.CaseEvaluation.Application.Contracts/AppointmentDocuments/`. Add Mapperly partial classes (`[Mapper(RequiredMappingStrategy = RequiredMappingStrategy.Target)]`) in the existing `CaseEvaluationApplicationMappers.cs`. Add permission group `CaseEvaluationPermissions.AppointmentDocuments` with children `Default`, `Create`, `Edit`, `Delete`, `Approve`; wire in `CaseEvaluationPermissionDefinitionProvider.cs` (fixes 5-G05). Add `[Authorize(Permissions.Create/Edit/Delete/Approve)]` method-level checks on all mutating AppService methods (prevents NEW-SEC-02 class of defect). Add one EF migration `Added_AppointmentDocuments_<timestamp>` via `dotnet ef migrations add` creating the three tables with tenant-scoped indexes on `(AppointmentId, Status)` and `(VerificationCode)`; DocumentType in host DB via `builder.IsHostDatabase()` guard. Add one `AppointmentDocumentTypeDataSeedContributor` writing seed rows for the common IME document types (MedicalRecords, ClaimFile, MRI, XRay, PQMEReport, AMEReport, OtherReport) -- feeds into the `lookup-data-seeds` capability. Angular side: generate proxy via `abp generate-proxy` in a separate step (NOT hand-edited per repo `CLAUDE.md`); build feature module at `angular/src/app/appointment-documents/` with three components (`appointment-document-list.component`, `appointment-document-upload.component`, `appointment-document-search.component`) and a sibling `appointment-new-documents/` module (two components: list + upload) plus `appointment-document-types/` admin module (list + add + edit forms for the host-scoped lookup). Register routes under the `PageAccess`-style permission guard already present in NEW. Verification-code anonymous download is OUT-OF-SCOPE here (separate `anonymous-document-upload` brief). Email send on status change is OUT-OF-SCOPE here (raise domain event; `email-sender-consumer` handles fan-out). Document-package compound support is OUT-OF-SCOPE here (`document-packages` brief, DB-13, Q9 open).

## Why this solution beats the alternatives

- Alternative A keeps the OLD pre-approval / post-approval workflow distinction by mirroring the two entities, preserving the due-date gate + verification-code pattern without leaking OLD's plaintext-S3 defects. Alternative B (single flag) forces branching logic into every query and complicates the approval UI; the extra table in A costs ~20 lines of DbContext mapping but buys workflow clarity.
- Alternative A composes with existing ADR-wired infrastructure (BlobStoringDatabaseDomainModule, FileManagementDomainModule, `IMultiTenant` auto-filter, `FullAuditedAggregateRoot`, Mapperly mapper conventions) with no new cross-cutting wiring. Alternative C (FileManagement direct) would replicate the approval workflow in a parallel tree; alternative D (S3 literal port) reopens the Q17 blocker and the HIPAA defects.
- Alternative A emits domain events (`AppointmentDocumentStatusChanged`) on approve/reject so the scheduler-notifications / email-sender-consumer briefs can subscribe without requiring this capability to know about notification templates; this keeps the blast radius of each brief contained.
- Alternative A writes the permission group once (5-G05) and uses method-level `[Authorize]` on mutations, which structurally fixes the NEW-SEC-02 class of defect across the document surface without depending on the NEW-SEC-02 brief landing first. Other alternatives do not change the permission shape.

## Risk and rollback

- **Blast radius**: a bug in `AppointmentDocumentsAppService.UploadAsync` could write blob content without inserting the entity row (orphaned blob), or insert the row with no blob (dangling metadata). Mitigation: wrap upload + insert in a single ABP `[UnitOfWork(true)]`-scoped operation; the `IBlobContainer.SaveAsync` call happens inside the UoW transaction (ABP DB blob provider uses the same DbContext). A bug in the permission group registration could open unauthorised access -- mitigated by method-level `[Authorize]` on every mutation plus the permission-enforcement pattern in manual controllers. A bug in the tenant-filter scoping could leak PHI across tenants -- mitigated by `IMultiTenant` on both document entities and automatic filter application; validated by the `new-qual-01-critical-path-test-coverage` brief which includes a multi-tenancy filter test.
- **Rollback**: revert the feature branch (all code is new, no existing code modified); run `dotnet ef database update <previous-migration>` to drop the three new tables and the host-DB `AppointmentDocumentTypes` table. No ABP module wiring changes (we only consume already-wired `BlobStoringDatabaseDomainModule`), so rollback requires no module-graph edits. Feature-flag rollback: wrap the permission group registration in `if (AppSettings.FeatureAppointmentDocumentsEnabled)` and toggle off -- the AppService routes still exist but `[Authorize]` denies all callers, effectively soft-disabling the surface.

## Open sub-questions surfaced by research

- Does the NEW schema need `DueDate` on `AppointmentDocument`? OLD uses `Appointment.DueDate` as the cutoff; the document itself does not carry a per-document due date. Recommendation: omit per-document DueDate; the AppService reads `Appointment.DueDate` at upload time (if the Appointment entity lacks `DueDate`, this becomes a cross-cutting gap tracked under the appointment-full-field-snapshot brief G2-10). The appointment-full-field-snapshot brief is the natural owner of `DueDate` on Appointment -- flag as coordination risk.
- AV scan hook -- should `UploadAsync` block until a ClamAV / Defender-for-Cloud scan returns clean, or upload-then-async-scan? Recommendation: upload-then-scan with a separate `DocumentStatus.Quarantined` transitional state that the UI surfaces as "scanning" until scan completes. AV integration is out-of-scope for this brief; flag as follow-up capability `document-av-scan` if Adrian confirms the requirement.
- Retention policy (7-year workers-comp standard) -- where does the retention job live? Recommendation: separate `document-retention-job` capability after background-jobs-infrastructure lands; not needed for MVP walkability but must be on the backlog before production.
- File-size cap for uploads -- confirm with Adrian. Recommendation: 25 MB per file (covers typical IME PDF bundles; matches ABP's default `FormOptions.MultipartBodyLengthLimit` precedent).
- Should `AppointmentNewDocument` really be a separate entity, or would the unified-flag alternative B serve better given the smaller NEW codebase? Recommendation: keep separate per OLD parity, UNLESS Adrian decides to simplify -- this is a tactical call, not an architectural one.
- Should uploads accept `IRemoteStreamContent` (ABP's multi-platform stream abstraction) instead of `IFormFile`? Recommendation: use `IFormFile` at the controller boundary and convert internally; `IRemoteStreamContent` adds complexity with no MVP benefit because we only have one client (Angular).
- Does the external-visible verification-code column need a per-tenant or per-appointment expiration policy (e.g. "code invalidates after 30 days")? Recommendation: defer to `anonymous-document-upload` brief; this capability stores the column but does not enforce expiration.
