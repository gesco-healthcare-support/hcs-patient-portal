## Summary

Browser-driven capture of NEW (`http://localhost:4200`, docker compose) on
2026-04-24. 23 screenshots across 11 distinct user contexts -- host admin,
all 5 tenant Doctors, plus all 4 external roles in tenant T1, the
T4-NullPatient edge case, and 4 non-role surfaces (unauthenticated
landing, ABP Login page, ABP Register form pre-submit, Swagger UI).
External-role coverage is concentrated in T1 because the four external
roles (Patient, Applicant Attorney, Defense Attorney, Claim Examiner)
share an identical Angular code path (`HomeComponent`'s role branch +
`appointments` accessor scope) -- T1 captures cover the per-role UX, T2
through T5 captures cover the per-tenant chrome (tenant name in welcome
banner, sidebar identity, sidebar items). MVP risk rating: **High** --
in addition to the pre-existing track-9 findings, this run produces
direct rendered evidence for FEAT-02 (Claim Examiner has no UI) and
FEAT-03 (tenant dashboard is the placeholder text); confirms the host
admin / tenant filter divide reads cleanly across screens (host admin
sees 0 appointments / 1 user / 5 tenants while T1 Doctor sees 13
appointments and a tenant-scoped sidebar); and finalises the credential
matrix that subsequent capture or testing runs can rely on.

- Driver: `chrome-devtools-mcp` from the parent session. For each role:
  `/Account/Logout` -> `/account/login` -> (if tenant change) `switch`

- Login mechanics: confirmed via `POST /connect/token` for all 22
  reachable users (host admin + 5 tenant Doctors + 16 external-role
  users); 22/22 returned 200 with bearer tokens. Tenant-scoped logins
  required `__tenant: <tenantGuid>` header (or, in the UI, the "switch"
  link populating the `__tenant` cookie at the AuthServer origin).
- Storage: `docs/gap-analysis/screenshots/new/<role-slug>/<screen-slug>.png`.
- Synthetic data: all `.test` TLD emails generated by `New-FakeData.ps1`;
  not real PHI.

## Findings (rendered evidence not previously documented)

1. **FEAT-02 confirmed visually**: ClaimExaminer landing is a near-empty
   page (capture `t1-ce/01-...`). Decision impact -- whoever signs off
   on MVP scope must answer Q3 ("Claim Examiner sub-entity required?")
   knowing the current rendered state shows nothing for this role.

2. **FEAT-03 confirmed visually**: tenant dashboard shows the
   placeholder string verbatim
   (`t1-doctor/02-tenant-dashboard-placeholder.png`). Q13 ("Dashboard
   counters per role -- which OLD cards are needed?") cannot be answered
   from current rendered state because no card exists; OLD's 13 stat
   cards are the spec input.

3. **Localization keys not resolved**: every `Enum:AppointmentStatusType.N`
   cell in the appointments table renders the literal key, not a
   friendly label (e.g. "Pending", "Approved"). Same for
   `Menu:Home` / `Menu:Dashboard` in the breadcrumb. Track-10 noted this
   under "Known Issues" but the rendered evidence is here -- this is
   a localization-pipeline gap, not Track-9 scope, but visible across
   every tenant capture.

4. **Tab title is "MyProjectName"** on every screen (cosmetic; per
   `DOCKER-DEV.md` Known Issues). Search-and-replace task.

5. **Top-right user nav for tenant Doctors reads "admin"**, not the
   user's actual name. Reinforces FEAT-12 -- the tenant admin and
   Doctor roles are conflated; the user's display identity is "admin"
   regardless of their actual identity.

6. **Multi-tenant filter behaves correctly**: host admin sees 0
   appointments / 0 attorneys / 1 user (only the host admin) / 5
   tenants, while T1 Doctor sees 13 appointments and tenant-scoped
   sidebar. This is direct rendered confirmation that the
   `IMultiTenant` data filter is engaging on Appointment +
   ApplicantAttorney + Identity scopes, contrasted against the FEAT-09
   leak on Patient (Patient does not implement IMultiTenant; we did not
   re-test that gap here).

7. **`/identity/users` from host context shows only the host admin
   user**, not the 22 tenant-scoped users. Tenant-scoped user
   administration requires impersonating into a tenant via the SaaS
   tenants page or logging in directly. This is consistent with ABP's
   tenant-side admin model but is non-obvious to operators expecting
   one user list.

8. **`/identity/roles` USER COUNT is 0 for the four external roles** in
   host context, even though 16 external-role users exist across the
   tenants. Same tenant-filter mechanism. Operators trying to audit
   "how many ApplicantAttorneys do we have?" must aggregate across
   tenants.

9. **The four external roles share a single Angular code path** --
   visible in the side-by-side AA / DA / Patient captures. This
   reinforces the OLD-side observation in `09-ui-screens.md` that the
   four external roles' `/home` is identical by design. The Q1
   ("Defense Attorney as first-class entity?") and Q2 ("Patient
   Attorney vs Applicant Attorney?") decisions still need an explicit
   answer from Adrian, but the rendered state does not yet warrant
   per-role differentiation.
