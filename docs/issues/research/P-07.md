# P7: Nature and Circumstances of Handover -- Research

**Risk tier**: Context (misreading "done" vs "abandoned")
**Status**: Open (verified 2026-04-17)
**Nature**: Not code-resolvable. Lives in the prior developer's memory and Gesco leadership's recollection.

---

## Indirect evidence found in repo (2026-04-17)

- **Single initial commit** on 2026-04-08 by `AdrianG <arajeev@gesco.com>` with message "Initial commit: Patient Portal (workers' comp IME scheduling)". All prior git history was scrubbed at import.
- **Starter docs are good**: comprehensive `README.md`, `docs/INDEX.md`, feature-level CLAUDE.md files, decision records, 29 catalogued issues. Suggests the import was not hostile -- reasonable transitional investment was made.
- **Incompleteness signals**: 29 tracked issues (SEC/DAT/BUG/FEAT/ARC), 22 technical open questions, orphan fields and stubbed features (see [Q-04](Q-04.md), [Q-05](Q-05.md)) -- Adrian has been reconstructing context for the past 10 days.

- No handover document, no KT notes, no walkthrough video under `docs/`. If any exist, they live outside the repo.

## Likely answer

**MEDIUM confidence**: Handover was *incomplete but not hostile*. Evidence:
- Starter docs exist and are structured -- reasonable effort.
- 29 open issues and 20+ open questions show Adrian had to reconstruct context; suggests KT was partial.
- Squashed-history import has two benign explanations: (a) clean start on Gesco's preferred SCM; (b) prior repo in a private vendor account that couldn't be wholesale transferred.
- Loss of git archaeology (blame, PR context, release notes) costs real onboarding time but is not automatically a red flag.

**Assumptions that would flip this:**
- If the prior developer's original repo still exists with full history.
- If contracts (if any) include a KT clause still within its window.
- If a handover document was written but stored outside the repo.

## Recommended approach to obtain a definitive answer

- Ask Gesco business owner: "Was the handover planned or abrupt? Are there features the prior dev considers 'done' that weren't tested? Any known issues they chose not to raise?"
- Check whether any contractor/employment contract exists with a KT clause; if yes, is the window still open?
- Request a **one-hour structured Q&A call** with the prior developer. Per Holland & Knight, NDA almost certainly does not bar architectural explanations (general knowledge vs trade secrets).
- Ask the prior developer: "Why did you squash the history at import? Does the original full-history repo still exist? Can we get a read-only clone?"

## Default assumption if unanswered

Treat the handover as incomplete-but-not-hostile. Assume the codebase represents genuine effort but has significant unfinished work. Plan for a continuing reconstruction period (~3 months from 2026-04-08). Treat any "done" claim with suspicion until independently verified.

## Risk if assumption is wrong

**If handover was abrupt / hostile:**
- Features labelled "complete" may be production-broken.
- Orphan code may be actively malicious (though unlikely given internal-project context).
- Security vulnerabilities may have been known and not disclosed (see [P-10](P-10.md)).

**If handover was cleaner than it looks:**
- The scrubbed history may have been deliberate for business-confidentiality reasons that Gesco already knows about; time spent trying to recover it is wasted.
