## Context

`GeneratePreviewAsync` at `src/HealthcareSupport.CaseEvaluation.Application/DoctorAvailabilities/DoctorAvailabilitiesAppService.cs:160` computes preview time slots for doctor availability bulk-creation. ~150 lines across:

- 5 up-front guard clauses (null/empty input, LocationId, duration, date range, time range)
- Min/Max date aggregation + existing-availability query
- Nested `foreach` over inputs x `while` over dates x `while` over time slots â€” slot-generation
- Grouping + month-level preview construction
- 3-level nested conflict-detection loop (preview list x existing availabilities x `FirstOrDefault` with overlap check + 2 `if` branches)
- Final validation with conditional assignment

SonarCloud reports cognitive complexity **41** (target <= 15).

## Why this was deferred

Phase B-5c-3 planned a small refactor (extract 1-2 helpers) to bring complexity under threshold. On closer read (2026-04-20) the method's logic is genuinely complex and **no tests cover it today**. Refactoring without tests violates Chesterton's Fence â€” the current shape enforces an implicit contract we cannot see because no spec documents it.

This method underpins the slot-booking calendar used by healthcare staff to place patients into IME slots. Breaking it silently is a regression into the core scheduling loop.
