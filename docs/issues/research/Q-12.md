# Q12: Hardcoded Default Password for Auto-Created Patients -- Research

**Severity**: Critical (every auto-created patient across every tenant gets the same password)
**Status**: Open (verified 2026-04-17)
**Source files**:
- `src/HealthcareSupport.CaseEvaluation.Application/Patients/PatientsAppService.cs` line 123
- `src/HealthcareSupport.CaseEvaluation.Domain/CaseEvaluationConsts.cs` line 10
- `src/HealthcareSupport.CaseEvaluation.Domain/Identity/ChangeIdentityPasswordPolicySettingDefinitionProvider.cs` (SEC-05 relaxed policy)

---

## Current state (verified 2026-04-17)

- `PatientsAppService.GetOrCreatePatientForAppointmentBookingAsync` line 123 uses `CaseEvaluationConsts.AdminPasswordDefaultValue`, which resolves to `IdentityDataSeedContributor.AdminPasswordDefaultValue` -- the standard ABP default admin password (typically `1q2w3E*`).
- Every auto-created patient across every tenant gets this same password.
- No force-password-change on first login.
- No email verification (FEAT-05 email system is not wired).
- No temporary-password notification.
- Combined with SEC-05 (all four complexity flags off), any patient account is trivially compromised.

## Official documentation

- [NIST SP 800-63B Rev 4 (July 2025 final)](https://pages.nist.gov/800-63-4/sp800-63b.html) / [NIST PDF](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-63B-4.pdf) -- Modern standard. Passwords must be user-chosen OR randomly generated by the CSP. Complexity rules removed; length floor raised (15 chars password-only). Forced rotation only on evidence of compromise. Enrollment secrets must be short-lived and typically one-time.
- [NIST SP 800-63 FAQ](https://pages.nist.gov/800-63-FAQ/) -- Clarifies enrollment-secret lifecycle: single-use, time-bound, out-of-band delivery.
- HIPAA Security Rule § 164.308(a)(5)(ii)(D) -- Password Management (addressable); shared defaults do not meet the "safeguarding" prong. Summaries: [HIPAA Journal](https://www.hipaajournal.com/hipaa-password-requirements/) and [HIPAA Guide](https://www.hipaaguide.net/hipaa-password-requirements/).
- [OWASP Authentication Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html) -- Require force-change on first login for any system-issued credential; do not email passwords; invalidate sessions on reset.
- [OWASP Forgot Password Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html) -- Token via out-of-band channel, short TTL, single use, consistent response, rate-limited.
- [ABP Identity module](https://abp.io/docs/latest/modules/identity) -- `IdentityUserManager` derives from Microsoft `UserManager<T>`; supports `GeneratePasswordResetTokenAsync`, `ResetPasswordAsync`, `CreateAsync` without password; `IAccountEmailer` sends reset/confirmation emails; `IdentityUser.ShouldChangePasswordOnNextLogin` flag is built in.

## Community findings

- [ABP support #4915 -- Create user without password](https://abp.io/support/questions/4915/Create-user-without-password) -- Documented pattern: create user inactive, generate reset token, email link, activate on token use.
- [ABP support #2181 -- Password reset flow](https://abp.io/support/questions/2181/Password-Reset) -- `GeneratePasswordResetTokenAsync` + `IAccountEmailer.SendPasswordResetLinkAsync` recipe.
- [Netwrix -- NIST 800-63B Rev 4 summary](https://netwrix.com/en/resources/blog/nist-password-guidelines/) -- Plain-English Rev 4 recap; confirms removal of complexity + rotation mandates.
- [Enzoic -- NIST Rev 4 changes](https://www.enzoic.com/blog/nist-sp-800-63b-rev4/) -- Breached-password screening now explicitly required.
- [Keeper -- HIPAA password requirements](https://www.keepersecurity.com/blog/2023/01/05/how-to-stay-compliant-with-hipaa-password-requirements/) -- HIPAA defers to NIST in practice; OCR investigations have cited shared/default passwords as a § 164.308 failure.

## Recommended approach

- **Short-term (before any pilot) (HIGH)**:
  - Replace `CaseEvaluationConsts.AdminPasswordDefaultValue` with a per-user cryptographically random password (e.g., `RandomNumberGenerator.GetBytes(16)` -> base64url, post-processed to meet policy).
  - Set `IdentityUser.ShouldChangePasswordOnNextLogin = true`.
  - Do NOT log or return the generated password; discard after hashing.
  - This addresses the "same password across tenants" defect even before email is wired.
- **Longer-term (after FEAT-05 ships) (HIGH)**:
  - Switch to invite-token flow: `IdentityUserManager.CreateAsync(user)` with no password arg, user set to non-active; `GeneratePasswordResetTokenAsync`; send link via `IAccountEmailer`. Token TTL short (24 hours), single use. Matches OWASP Forgot Password and NIST 800-63B-4 enrollment-secret guidance.
  - Optional: passwordless magic-link via OpenIddict extensions (MEDIUM).
- **Email dependency**: invite-token path is blocked until FEAT-05 email wiring ships. Short-term random-password + force-change fix is NOT blocked; ship immediately.
- **Fix SEC-05 in the same PR** (HIGH): turn password complexity back on per NIST Rev 4 (length 15+ password-only, or 8+ with MFA; screen against breached-password lists). The two defects compound.
- **Make force-change-on-first-login a per-tenant ABP setting** (MEDIUM) so carrier tenants with SSO can opt out if they provision via SCIM. Default ON.

## Gotchas / blockers

- Auto-created patients have no verified email (FEAT-05 gap), so there is no secure out-of-band channel for a reset link yet. SMS OTP may be a faster path (HHS OCR permits; NIST downgraded SMS but did not prohibit).
- `ShouldChangePasswordOnNextLogin` interacts with OpenIddict's grant flow -- confirm Angular login UX surfaces the password-change screen before issuing a real access token.
- `CaseEvaluationConsts.AdminPasswordDefaultValue` is also used for the seeded admin; keep that path untouched, only fix the patient-creation path.
- OCR has historically treated shared credentials as a reportable breach precursor; an audit would flag this immediately.

## Open questions

- Do patients log in at all today, or is "auto-create patient" a booking-convenience-only path where the account is never meant to be used? If never used, the simpler fix is to create the user locked/inactive with no usable password hash.
- Is SMS acceptable as the out-of-band channel (HHS yes with consent; NIST discourages but allows)?
- Will the invite flow be reusable for Claim Examiner (Q-04) and Attorney roles?
