## Current state (verified 2026-04-17)

- 13 status values in the enum: Pending, Approved, Rejected, Confirmed, CheckedIn, CheckedOut, NoShow, Billed, CancelledNoBill, CancelledLate, RescheduleRequested, CancellationRequested, Rescheduled.
- Status is set at creation and never mutable via any supported code path. `AppointmentManager.UpdateAsync` explicitly excludes it; `AppointmentUpdateDto` does not expose it.
- No state-machine library, no transition guards, no role-to-status matrix, no audit log for status changes.
- Angular UI treats status as display-only: no dropdowns, buttons, or actions trigger transitions.

## Recommended approach

- **Library**: `Stateless` (MIT, mature, actively maintained). Elsa is overkill for a single-aggregate lifecycle with no branching/parallel work. Implement the machine inside `AppointmentManager` (the existing domain service); read/write `Appointment.AppointmentStatus` via constructor lambdas so the enum column stays the source of truth.
- **Transition location**: Add intent-named methods (`ApproveAsync`, `CheckInAsync`, `CancelAsync(reason)`, `RescheduleAsync`, `BillAsync`, etc.) to `AppointmentManager`. Each method instantiates the Stateless machine, calls `Fire(trigger)`, persists. AppService orchestrates; `AppointmentUpdateDto` stays status-free.
- **Permission model**: Add child permissions under `CaseEvaluationPermissions.Appointments` (`Approve`, `CheckIn`, `CheckOut`, `Bill`, `Cancel`, `Reschedule`). Map transitions 1:1 to permissions; enforce with `[Authorize(...)]` on each AppService method.
- **Audit strategy (two layers)**:
  - Domain-specific `AppointmentStatusHistory` child entity (who, from, to, reason, timestamp) written by the `OnTransitioned` hook -- richer than ABP's generic entity log.
  - Keep `[Audited]` on `Appointment` so the ABP Commercial audit UI still shows diffs. HIPAA 164.312(b) expects application-level action logs, not just row diffs.
- **Default answers to sub-questions (confirm with business before wiring):**
  - Q1a Approve: Admin role (MEDIUM confidence -- standard IME flow; doctor accepts slot at creation, admin approves the booking).
  - Q1b CheckIn / CheckOut: Front-desk / clinic staff role (HIGH -- Epic/Cerner convention + DWC workflow assumes clinic staff track arrival).
  - Q1c CancellationRequested vs CancelledNoBill vs CancelledLate: *Request* is patient/attorney-initiated and pending admin action; `CancelledLate` applies when cancel lands inside the 6-business-day window (DWC section 34 triggers missed-appointment fee); `CancelledNoBill` is a good-cause cancel with no charge (MEDIUM -- matches DWC terminology).
  - Q1d Billed reopen: No. Terminal state. Corrections require a compensating adjustment, not a status rewind (HIGH -- HIPAA audit + accounting integrity).
  - Q1e RescheduleRequested: Either party per 8 CCR section 34, but operationally almost always patient/attorney -> admin approves -> lands on `Rescheduled` (MEDIUM).

## Gotchas / blockers

- `AppointmentStatus` is an `int` column with no CHECK constraint; invalid values can slip in via raw SQL or future migrations. Consider `HasConversion<string>()` so the DB stores readable names.
- `[Audited]` on entities does not capture *who fired a transition* unless `IAuditingManager.Current.Log` is called explicitly. Default logs attribute to HTTP request user, which works when every transition goes through an authenticated AppService.
- `Rejected` / `Rescheduled` are terminal in most IME systems but the enum does not distinguish terminal from active -- the state machine must enforce this in `Permit` config.
- Multi-tenancy: `Appointment` is `IMultiTenant`; transitions must not cross tenant boundaries when loading availability slots, attorneys, or accessors.

## Open questions

- Does the business want a `reason` string required on every cancel/reject, or only on some? DWC section 34 requires written reason on cancellation.
- Should `NoShow` auto-fire after a time threshold via a background worker, or be manually set by staff?
- Should `RescheduleRequested` hold the existing slot or release it immediately?
- Retention policy: HIPAA is 6 years minimum, California medical records 7 years after last visit -- which drives the audit-log retention policy?
