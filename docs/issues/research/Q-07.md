## Current state (verified 2026-04-17)

```csharp
// DoctorConsts.cs line 14
public const int EmailMaxLength = 49;

// PatientConsts.cs line 15
public const int EmailMaxLength = 50;
```

Both are far below:
- RFC-5321 practical maximum of 254.
- ASP.NET Core Identity's default 256.
- Legitimate corporate addresses, `+tag` aliases, long domain names.

The `49` is almost certainly a typo off the already-undersized 50 -- or a legacy downstream-system constraint.

---

## Official documentation

- [RFC 5321 section 4.5.3.1.3](https://datatracker.ietf.org/doc/html/rfc5321#section-4.5.3.1.3) -- authoritative SMTP path maximum is 256 octets for full `<Path>` including angle brackets -> 254-character practical maximum for bare address.
- [RFC 5322 sections 3.2.3, 3.4.1](https://datatracker.ietf.org/doc/html/rfc5322) -- defines `addr-spec`; with RFC 5321, local-part capped at 64, domain at 255 octets, but *transportable* max is 254.
- [Microsoft -- Identity model customization](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/customize-identity-model?view=aspnetcore-9.0) -- ASP.NET Core Identity's `AspNetUsers.Email` is `nvarchar(256)` default. ABP's `AbpUsers` follows.
- [Microsoft -- EF Core Entity Properties (MaxLength/HasMaxLength)](https://learn.microsoft.com/en-us/ef/core/modeling/entity-properties) -- widening via `HasMaxLength(254)` generates `AlterColumn` migration automatically on next `dotnet ef migrations add`.

## Community findings

- [Wikipedia -- Email address](https://en.wikipedia.org/wiki/Email_address) -- cross-references RFC 5321/5322 and the 254-character practical limit.
- [Max Data Types in ASP.NET Core Identity Schema](https://joshthecoder.com/2020/04/28/max-data-types-in-aspnet-core-identity-schema.html) -- rationale for Identity's 256: MX lookups, Unicode normalisation, display formatting overhead.
- [dotnet/efcore #28655 -- Override MaxLength convention](https://github.com/dotnet/efcore/issues/28655) -- `[MaxLength]`/`HasMaxLength` is authoritative over convention defaults, propagates to migrations.
- [ASP.NET Zero Support #5701 -- Changing size of database column](https://support.aspnetzero.com/QA/Questions/5701/Changing-size-of-database-column) -- ABP-ecosystem-specific `AlterColumn` walk-through for widening string properties.

## Recommended approach

1. **Standardise on 256** to match ASP.NET Core Identity and ABP's `AbpUsers.Email`. Using 254 is technically RFC-correct but creates a one-off value future maintainers will question. 256 is the dominant convention in .NET and matches the adjacent `AbpUsers.Email` column.
2. Fix BOTH `DoctorConsts` and `PatientConsts` in the SAME migration -- same problem, inconsistency is its own bug.
3. INFERENCE: audit other `*Consts.cs` files for similar short caps. `CityMaxLength = 50`, `ZipCodeMaxLength = 15`, `AddressMaxLength = 100` look suspicious against US postal standards. Not blocking but worth a sweep while the migration is open.

## Gotchas / blockers

- Widening `nvarchar(49)` to `nvarchar(256)` is non-destructive -- no existing data at risk. Indexes on the column may need rebuild; SQL Server handles this automatically on `ALTER COLUMN` unless the index is filtered.
- INFERENCE: if `EmailMaxLength` appears in Angular validators (reactive form `Validators.maxLength(49)`), update in lockstep or client will reject valid input after backend accepts. Grep `angular/` folder for `maxLength` and the number 49.
- INFERENCE: `49` could originate from inherited downstream system (CSV export width, legacy EDI field). Unlikely given adjacent 50 in PatientConsts, but confirm no report/export is hard-coded before widening.

## Open questions

- Why does Patient use 50 and Doctor use 49? Typo is most likely; worth confirming with `git blame` -- if either was recently changed, there may be a downstream reason.
- Are these consts referenced anywhere besides EF `HasMaxLength` configuration (DTO validation attributes, Angular validators)? Sweep needed before migration ships.
