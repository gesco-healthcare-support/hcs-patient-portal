## Recommended approach

Decision depends on 6-12 month roadmap. Ask Adrian:

- **Remove** if "attorney docs / injury photos" is not on the 2026 roadmap: unused modules add DB tables, migration surface, bundle weight, exposed `/file-management` Angular route that confuses users.
- **Keep** if attachments are a foreseeable feature: removal-then-readd wastes ~1 day each way and risks migration ordering issues in multi-tenant DB.

**If removing** (recommended default given current zero usage):
1. Single PR. Remove `[DependsOn(FileManagementDomainModule)]`.
2. Remove `BlobStoringDatabaseDomainModule` ONLY if no other module uses it (inferred: ships with FileManagement but also used by Chat, CMS Kit, etc. -- grep to confirm).
3. Remove `app.routes.ts` entry.
4. Remove `@volo/abp.ng.file-management` from `package.json`.
5. Create migration: `dotnet ef migrations add RemoveFileManagement` -- emits `DropTable` for `FmFileDescriptors` + `FmDirectoryDescriptors`.
6. Regenerate Angular proxies.

**If keeping**: hide `/file-management` route behind a feature flag or comment out so not reachable until needed.

## Gotchas / blockers

- `FmFileDescriptors` may have FK references from other tables (chat attachments if that module is wired). Grep before dropping.
- Blob-storing DB tables (`AbpBlobContainers`, `AbpBlobs` -- inferred) are a separate removal concern; may serve other ABP modules. Leaving them is safe.
- Removing `@volo/abp.ng.file-management` requires Angular workspace rebuild and possibly proxy regeneration if stale types leak.
- Commercial NuGet packages cached locally may require re-auth to ABP commercial feed if re-adding later.
