## Current state (verified 2026-04-17)

Three fields on `Appointment.cs`:
- Line 28: `public virtual bool IsPatientAlreadyExist { get; set; }`
- Line 36: `public virtual string? InternalUserComments { get; set; }` (max 250)
- Line 38: `public virtual DateTime? AppointmentApproveDate { get; set; }`

All three:

- Never written by any code path (constructor doesn't set them, `AppointmentManager.UpdateAsync` explicitly excludes them, no AppService writes them).

- Read-only from creation (default values) through entire lifecycle.

## Official documentation

- [Microsoft -- MigrationBuilder.DropColumn](https://learn.microsoft.com/en-us/dotnet/api/microsoft.entityframeworkcore.migrations.migrationbuilder.dropcolumn?view=efcore-8.0) -- `DropColumn(name, table, schema)`. Reversible via `AddColumn` in `Down()`.
- [Microsoft -- Managing Migrations](https://learn.microsoft.com/en-us/ef/core/managing-schemas/migrations/managing) -- removing property from entity + running `dotnet ef migrations add` generates the correct `DropColumn` automatically. Applied migrations must not be edited; new migration required.
- [Refactoring catalog -- Remove Dead Code](https://refactoring.com/catalog/removeDeadCode.html) -- Fowler: dead code is a comprehension tax; remove it, rely on version control.

## Community findings

- [dotnet/efcore #27165 -- Easier property deletion in 0-downtime scenarios](https://github.com/dotnet/efcore/issues/27165) -- standard two-phase pattern for production databases: (1) remove property from entity, ship; (2) after all instances on new code, ship migration that drops column. Avoids runtime errors where old instances insert a row missing the now-dropped column.
- [dotnet/efcore #27017 -- Snapshot out of sync after DropColumn](https://github.com/dotnet/efcore/issues/27017) -- known quirk: model snapshot can retain dropped column; regenerate via `dotnet ef migrations add`.
- [Learn EF Core -- Drop Column from SQLite](https://www.learnentityframeworkcore5.com/whats-new-in-ef-core-5/drop-column-from-sqlite-database) -- SQLite's `DROP COLUMN` support is newer (EF Core 5+). Your test project uses SQLite in-memory, so test migrations still apply cleanly.

## Recommended approach

**Decide intent per field (not in bulk):**

1. **`AppointmentApproveDate` -- KEEP.** [FEAT-01](FEAT-01.md) (status state machine) is the documented consumer; when that feature lands, the `Approved` transition should stamp this field. Until then, it is a stub for planned feature, not dead code.

2. **`InternalUserComments` -- CLARIFY.** Classic admin-notes field with 250-char max; INFERENCE: intentional UX hook never wired to the edit form. Confirm with business owner before deleting. If kept, add to `AppointmentUpdateDto` behind the same permission check.

3. **`IsPatientAlreadyExist` -- DELETE CANDIDATE.** Boolean suggesting a historical duplicate-detection workflow. Domain already handles patient get-or-create in `PatientsAppService`. INFERENCE: vestigial. Delete unless business describes a reporting use case.

Document each decision in `docs/features/appointments/overview.md` (keep) or in a single migration message (delete).

## Gotchas / blockers

- **Data loss risk on DropColumn**: for rows where field is non-default, values are lost. VERIFIED:
  - `IsPatientAlreadyExist` has no writer -- every value is `false` (default). Safe.
  - `AppointmentApproveDate` is nullable, never set -- every value is NULL. Safe.
  - `InternalUserComments` is nullable, never set -- every value is NULL. Safe.
  - Still run pre-migration `SELECT COUNT(*) WHERE ... IS NOT NULL` in case DBAs wrote directly.
- Multi-tenant schemas: tenant DBs apply migrations independently via `DbMigrator`.
- INFERENCE: if any reports, BI exports, or downstream consumers read these columns, dropping breaks them silently. Grep `docs/` and talk to business before removing.
