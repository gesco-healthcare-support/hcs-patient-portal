# P9: California DWC / Insurance Carrier Integration Discussions -- Research

**Risk tier**: Product (self-contained vs connected architecture)
**Status**: Open (verified 2026-04-17)
**Nature**: Not code-resolvable. Lives in Gesco roadmap discussions with carriers and the prior developer's notes.

---

## Indirect evidence found in repo (2026-04-17)

- No EDI / X12 / HL7 / FHIR code or configuration anywhere in `src/` or `angular/`.
- No references to Jopari, EAMS, JET File, EDEX, or carrier-specific APIs in any file.
- No SFT / SFTP / AS2 transport packages in `.csproj` or `package.json`.
- No integration-test fixtures for outbound messages.
- Data model contains IME ecosystem entities (WcabOffice, ApplicantAttorney, AppointmentAccessor) but no outbound-message or integration-state tracking.
- Localhost-only pre-production stance is incompatible with active carrier-integration work.

## Industry / regulatory context

- [DWC -- EAMS Home](https://www.dir.ca.gov/dwc/eams/eams.htm) -- State case-management system for all CA workers' comp adjudication. Filing methods: JET File, e-forms, OCR paper.
- [DWC -- JET File](https://www.dir.ca.gov/dwc/eams/JetFiling/EAMS_JetFile.html) -- API-like secure SFT transmission channel for 6 form types; closest thing to a real API.
- [DWC -- EAMS E-Team](https://www.dir.ca.gov/dwc/EAMS/JetFiling/EAMS_eTeam.html) -- Onboarding portal for developers building JET File integrations.
- [DWC -- EDEX](https://www.dir.ca.gov/dwc/edex.html) -- Electronic Data Exchange; subscriber-based WCAB case summary data.
- [DWC -- E-Billing Vendor Intro](https://www.dir.ca.gov/dwc/ebilling/EBilling_vendors.html) -- Lists approved CA e-billing clearinghouses including Jopari.
- [DWC -- 2025 EAMS Rules Update](https://www.dir.ca.gov/DIRNews/2025/2025-108.html) -- Regulatory flux; integration targets are moving.
- [Jopari Solutions](https://www.jopari.com/) -- Dominant workers'-comp clearinghouse for ANSI X12 837 (medical bills) and 275 (attachments).
- [Jopari eBill](https://www.jopari.com/jopari-ebill/) -- Confirms workers'-comp EDI is 837/275; 278 auth sparingly used in this space.
- [DOL OWCP EDI Clearinghouse](https://owcpmed.dol.gov/portal/node/517) -- Federal parallel; useful reference.
- [daisyBill -- Clearinghouse / Payer IDs](https://blog.daisybill.com/how-to-e-bill-clearinghouse-payer-ids) -- Payer-ID routing is the practical pain point; few carriers accept direct API.
- [DataCare Workers' Comp Software](https://www.datacare.com/workers-comp-software/) -- SOC 2-certified CA-based WC software example.

## Likely answer

**LOW-MEDIUM confidence**: No carrier / DWC integrations were in active scope for v1. Industry norm for pre-production IME scheduling apps is standalone with CSV/PDF exports; EDI/EAMS is added later if scope expands. Repo contains no integration plumbing.

**Assumptions that would flip this:**
- An abandoned integration branch in the prior developer's original repo.
- Sandbox credentials or test accounts provisioned with Jopari / EAMS that Gesco finance records reveal.
- A client-specific verbal promise to push appointment outcomes to a carrier adjuster.

## Recommended approach to obtain a definitive answer

- `git log --all --grep -iE "edi|eams|jet|jopari|x12|fhir|carrier"` -- scan for abandoned branches.
- Grep source tree for `837`, `275`, `278`, `HL7`, `FHIR`, `Jopari` -- expect zero hits.
- Ask Gesco principals: "Is the product intended as a standalone scheduler, or will it need to push appointment outcomes to carrier adjusters or the DWC? Are we planning to become a JET File submitter or an X12 sender?"
- Ask prior developer: "Were any external system integrations discussed with the client or investigated, even if never started?"

## Default assumption if unanswered

No EDI / carrier / EAMS integrations in scope for v1. Future roadmap likely includes:
- JET File for WCAB filings (if Gesco acts as medical-legal provider).
- X12 837/275 via Jopari for billing (if scope expands beyond scheduling).

## Risk if assumption is wrong

If a carrier/TPA was promised an API integration, discovering it 6+ months into development forces a major architectural retrofit: EDI clearinghouse onboarding, X12 schema tooling, SFT transport, BAA with Jopari, PHI data-residency decisions. Early signal matters -- ask now.
