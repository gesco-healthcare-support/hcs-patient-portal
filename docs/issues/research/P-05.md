# P5: External Service Accounts to Take Ownership Of -- Research

**Risk tier**: Operational (services expire / fail silently if owner leaves)
**Status**: Open (verified 2026-04-17)
**Nature**: Partly code-resolvable (ABP licence + GitHub org). The rest is business-record territory.

---

## Indirect evidence found in repo (2026-04-17)

- `NuGet.Config` contains a live per-user ABP Commercial API key (`afccfa61-0b0f-410d-ad9a-f062655770f0`) -- a live third-party credential of unknown account ownership.
- `angular/package.json` references `@volo/abp.ng.*` commercial packages requiring authenticated `npm.abp.io` access.
- Repository lives in GitHub org `gesco-healthcare-support` -- owner identity unknown.
- No SMTP / SendGrid / Sentry / Azure / AWS / DigitalOcean configuration in any `appsettings*.json`, env var, or secret file.
- FEAT-05 (email not wired) confirms no email provider is provisioned yet.
- README mentions SonarCloud and Codecov as "pending" -- badges are placeholders, not live integrations.

## Industry / regulatory context

- [HHS FAQ -- Cloud Services and BAA Requirement](https://www.hhs.gov/hipaa/for-professionals/faq/2075/may-a-hipaa-covered-entity-or-business-associate-use-cloud-service-to-store-or-process-ephi/index.html) -- Any CSP handling ePHI is a BA requiring a signed BAA.
- [HHS -- Sample BAA Provisions](https://www.hhs.gov/hipaa/for-professionals/covered-entities/sample-business-associate-agreement-provisions/index.html) -- Model BAA language.
- [ICANN Transfer Policy](https://www.icann.org/en/contracted-parties/accredited-registrars/transfer-policy-01-06-2016-en) -- 60-day lock after registration / transfer / registrant change; AuthInfo code required.
- [ICANN Registrant FAQ -- Transfers](https://www.icann.org/resources/pages/name-holder-faqs-2017-10-10-en) -- Registrar must provide AuthInfo within 5 calendar days.
- [GitHub Docs -- Transferring Organization Ownership](https://docs.github.com/en/organizations/managing-organization-settings/transferring-organization-ownership) -- Requires an existing owner; without one, GitHub Support ticket is the only path.
- [GitHub Docs -- Maintaining Ownership Continuity](https://docs.github.com/en/organizations/managing-peoples-access-to-your-organization-with-roles/maintaining-ownership-continuity-for-your-organization) -- Recommends ≥2 owners per org; sole-owner orgs can become stranded.
- [ABP Email Sending Docs](https://abp.io/docs/latest/framework/infrastructure/emailing) -- Native support is SMTP only; password encrypted via `IStringEncryptionService`.
- [ABP Support #3296 -- SendGrid Integration](https://abp.io/support/questions/3296/Integration-with-SendGrid) -- SendGrid/Mailgun/SES require a custom `EmailSenderBase` subclass.
- [ABP Commercial FAQ](https://commercial.abp.io/faq) -- Licence file delivered by email to purchaser; tied to Volosoft account.
- [Volo.Abp NuGet Package](https://www.nuget.org/packages/Volo.Abp/) -- Commercial feed uses per-user token in `NuGet.Config`.

## Likely answer

**HIGH confidence** on runtime services: the app currently has **zero** live third-party runtime integrations (no SMTP, APM, analytics, cloud, feature flags).

Admin / dev-time accounts that DO exist:

1. **ABP Commercial / Volosoft account** -- licence file + NuGet token in repo. Overlaps with [Q-11](Q-11.md). Live credential.
2. **GitHub org `gesco-healthcare-support`** -- owner unknown; check with Gesco principals. Risk of stranded ownership if sole-owner.
3. **Gesco-side domain registrar** for any current/future domain (e.g., `gesco.com`).

Future-required accounts (not yet provisioned but pre-deploy essential):
- SMTP relay (SendGrid / Mailgun / AWS SES / Azure Communication Services) with BAA.
- Error monitoring (Sentry / Application Insights / Datadog) with BAA.
- Cloud host (Azure App Service / Container Apps; or AWS ECS/Fargate) with BAA.
- DNS provider; SSL via Let's Encrypt (no "transfer" -- reissue only).
- Optional: feature flags, CDN, analytics.

**Assumptions that would flip this:**
- Discovery of live SendGrid / Sentry / Azure keys in a force-pushed commit.
- Gesco finance records showing paid subscriptions to IME-stack services.
- A `gesco.com` admin email with an inbox full of third-party activation notices.

## Recommended approach to obtain a definitive answer

- Log in to `abp.io` with every plausible Gesco email and check for active subscriptions tied to the NuGet token.
- `gh api orgs/gesco-healthcare-support/members --paginate | jq '.[] | select(.role=="admin")'` to list org admins.
- Audit `NuGet.Config` and rotate the ABP token under a Gesco-owned account.
- Before any deployment: provision new accounts under a shared Gesco email (e.g., `devops@gesco.com`), not a personal email.
- Ask Gesco finance: list of all paid third-party software subscriptions for the last 24 months.

## Default assumption if unanswered

The app has zero live third-party runtime integrations. ABP token + GitHub org are the only real ownership questions today. All future accounts should be provisioned under a shared Gesco email from day one.

## Risk if assumption is wrong

If the prior developer owns the Volosoft licence / GitHub org and becomes unreachable, build/deploy is blocked. ABP tokens in committed `NuGet.Config` are a credential leakage regardless of ownership. Any undisclosed production account could be an unmonitored attack surface.
