# Q3: Confirmation Number Uniqueness -- Global vs Per-Tenant -- Research

**Severity**: Medium (blocks DAT-07 unique-constraint design)
**Status**: Open (verified 2026-04-17)
**Source files**:
- `src/HealthcareSupport.CaseEvaluation.Application/Appointments/AppointmentsAppService.cs` lines 254-282 (`GenerateNextRequestConfirmationNumberAsync`)
- `src/HealthcareSupport.CaseEvaluation.Domain/Appointments/Appointment.cs` (`RequestConfirmationNumber` property)
- `src/HealthcareSupport.CaseEvaluation.EntityFrameworkCore/EntityFrameworkCore/CaseEvaluationDbContext.cs` (no unique index)

---

## Current state (verified 2026-04-17)

- `GenerateNextRequestConfirmationNumberAsync` queries `_appointmentRepository.GetQueryableAsync()` with no explicit tenant filter.
- `Appointment` implements `IMultiTenant`, so ABP's automatic data filter makes the query effectively tenant-scoped.
- Each tenant gets its own `A00001..A99999` sequence.
- No unique constraint on `(TenantId, RequestConfirmationNumber)` -- a race condition can produce duplicates within a tenant.
- Format is hardcoded: prefix `A` + 5 zero-padded digits. Hard cap at 99,999 per tenant.

## Official documentation

- [ABP Data Filtering](https://abp.io/docs/latest/framework/infrastructure/data-filtering) -- `IDataFilter.Disable<IMultiTenant>()` bypasses the tenant filter; must be used inside `using`.
- [ABP Multi-Tenancy](https://abp.io/docs/latest/framework/architecture/multi-tenancy) -- Automatic tenant filter applies to every query when the entity implements `IMultiTenant`.
- [EF Core Sequences](https://learn.microsoft.com/en-us/ef/core/modeling/sequences) -- SQL Server `CREATE SEQUENCE` surfaces as `modelBuilder.HasSequence<int>(...)` with start/increment/min/max/cyclic.
- [EF Core `UseHiLo`](https://learn.microsoft.com/en-us/dotnet/api/microsoft.entityframeworkcore.sqlserverpropertybuilderextensions.usehilo) -- EF caches a block of sequence values; gaps are unavoidable on restart or scale-out.
- [SQL Server filtered indexes](https://learn.microsoft.com/en-us/sql/relational-databases/indexes/create-filtered-indexes) -- Composite uniqueness with `WHERE` clause to ignore NULLs; AND predicates only (no OR).

## Community findings

- [Stack Overflow -- Filtered unique index with multiple conditions](https://stackoverflow.com/questions/46277584/unique-filtered-index-with-multiple-conditions-in-sql-server) -- `CREATE UNIQUE INDEX ... (TenantId, RequestConfirmationNumber) WHERE RequestConfirmationNumber IS NOT NULL` is the canonical per-tenant pattern.
- [Stack Overflow -- Unique constraint in a multi-tenant database](https://stackoverflow.com/questions/40836035/how-to-set-a-unique-constraint-in-a-multi-tenant-database) -- Composite `(tenant_id, X)` unique index is the accepted pattern; leading `TenantId` also aids tenant-scoped scans.
- [Stack Overflow -- Generate unique user number per tenant](https://stackoverflow.com/questions/37419447/how-to-best-generate-a-unique-user-number-without-duplicates-in-a-multi-tenant-d) -- Recommends table-based counter with row lock OR per-tenant SEQUENCE OR unique-index + retry loop.
- [DBA StackExchange -- Sequential numbers per group under concurrent inserts](https://dba.stackexchange.com/questions/349798/how-can-i-generate-sequential-numbers-per-group-safely-under-concurrent-inserts) -- Gap-free requires SERIALIZABLE or row-locked counter table; throughput cost is real.
- [DBA StackExchange -- Prevent gaps while using IDENTITY/SEQUENCE](https://dba.stackexchange.com/questions/77891/prevent-gaps-while-using-identity-sequence-object-in-sql-server) -- True gap-free is inherently serialising.
- [dotnet/efcore #21067 -- HiLo gaps in scale-out](https://github.com/dotnet/efcore/issues/21067) -- HiLo is NOT gap-free when multiple app instances or restarts occur.
- [SQL Workbench blog -- Gapless sequence generation](https://blog.sql-workbench.eu/post/gapless-sequence/) -- Walks through why true gapless is inherently serialising.

## Recommended approach

- **Keep per-tenant scope** (HIGH confidence). Current behaviour is correct for the domain: confirmation numbers are operator-facing within a tenant; cross-tenant collisions are irrelevant. Do NOT switch to global.
- **Add a DB-level unique constraint** (HIGH). `CREATE UNIQUE INDEX IX_Appointment_Tenant_ConfirmationNumber ON Appointment (TenantId, RequestConfirmationNumber) WHERE RequestConfirmationNumber IS NOT NULL` -- turns race conditions into retriable `DbUpdateException` rather than silent duplicates.
- **Concurrency strategy: unique-index + bounded optimistic retry** (HIGH). The current `MAX + 1` pattern under Read Committed can race. Wrap generate-and-insert in a 3-5-attempt retry on unique-constraint violation. Avoid SERIALIZABLE (throughput hit). Avoid HiLo (gap-risk here is fine but HiLo doesn't fit the A##### format).
- **Plan for A99999 overflow** (HIGH). Mitigations (in order of preference): widen to 6-7 digits (`A000001`), add year/tenant-prefix rollover (`A26-00001`), or alarm at 90k. Pick one before any real tenant approaches 10k appointments.
- **Do NOT disable the tenant filter here**. The silent per-tenant behaviour is what you want; document it and add the constraint.

## Gotchas / blockers

- SQL Server filtered unique indexes support only AND predicates; for `IS NULL`/`IS NOT NULL` use AND, which is fine.
- `GenerateNextRequestConfirmationNumberAsync` parses the string with `Substring(1)` + `int.Parse`. If format changes (overflow fix), the parser must change too.
- `[RemoteService(IsEnabled = false)]` on the AppService means any controller override must be kept in sync -- watch for duplicate routes.
- ABP automatic tenant filter can be bypassed by raw SQL or by disabling the filter; the DB constraint is the last line of defence.

## Open questions

- Is the A##### format user-facing (printed on paper) or purely internal? Drives overflow mitigation choice.
- Does the business want gap-free numbering (audit optics) or is gap-tolerant acceptable? Current behaviour is gap-tolerant.
- What should happen at A99999 -- hard fail, auto-rollover, or widen?
