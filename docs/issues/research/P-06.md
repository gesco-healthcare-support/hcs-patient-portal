# P6: Verbal Commitments to Client Not Reflected in Code -- Research

**Risk tier**: Product (building the wrong thing)
**Status**: Open (verified 2026-04-17)
**Nature**: Not code-resolvable. Lives in emails, Slack, meeting notes, and the prior developer's memory.

---

## Likely answer

**LOW confidence on specifics**: what was verbally promised is unknowable from the repo. **HIGH confidence on the recommendation**: run a scope-validation workshop.

The orphan fields and unused role are strong indirect evidence that *something* was discussed and partially built. Whether that reflects verbal commitments to an external client or internal Gesco stakeholders cannot be determined from code.

**Assumptions that would flip this:**
- Surfacing a PRD, design deck, or recorded demo.
- A client-facing email trail with explicit promises.
- Any ticket / issue in an external tracker (Jira, Linear, Asana) referencing the orphan fields or CE role.

## Recommended approach to obtain a definitive answer

- Ask Gesco principals: any email threads, meeting notes, Slack/Teams channels, Figma files, or design docs that contain client-facing product discussions?
- Ask prior developer: "Were any features demoed that aren't yet implemented? Any specific behaviours promised that differ from standard IME practice? Anything explicitly ruled out?"
- Run a **2-4 hour JAD-style scope-validation workshop** (PMBOK Validate Scope) with business owner + 1-2 end users to reconfirm what the product must do.
- Use the orphan fields and unused role as discussion prompts -- each likely has a story behind it.

## Default assumption if unanswered

Assume the codebase represents Gesco's internal understanding of scope. Treat orphan features as unintentional debt (not verbal commitments). Before building anything new, hold a scope workshop so new features are Gesco-sanctioned and documented in `docs/`.

## Risk if assumption is wrong

Building (or failing to build) a feature the client was promised is the classic "building the wrong thing" failure. In a healthcare context, an undocumented verbal commitment about data retention, access control, or audit trails could be legally significant (HIPAA-adjacent obligations documented only in email).
