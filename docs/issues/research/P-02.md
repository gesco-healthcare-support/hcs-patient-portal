# P2: Prior Production/Staging Deployment with Real PHI -- Research

**Risk tier**: Legal (HIPAA Breach Notification Rule)
**Status**: Open (verified 2026-04-17)
**Nature**: Not code-resolvable. Direct answer lives in Gesco business records + cloud invoices + prior-developer memory.

---

## Indirect evidence found in repo (2026-04-17)

- `README.md`: "Deployed environments: None", "Pre-production, localhost / Docker only", "Maintained by Gesco".
- `angular/src/environments/environment.prod.ts`: still points to `http://localhost:4200` and `https://localhost:44327`.
- `appsettings.json`: only localhost / LocalDB connection strings.
- No SMTP / SendGrid / APM / Application Insights configuration in source code (FEAT-05 email not wired).
- GitHub org: `gesco-healthcare-support`.
- Seeded doctor GUIDs are hardcoded and look demo-style; no PII patterns suggest real names in any seed file.
- Initial commit 2026-04-08 (single "Initial commit" message) -- any deployment history prior to the import is not in git.

## Industry / regulatory context

- [eCFR 45 CFR § 164.402 -- Breach Definitions](https://www.ecfr.gov/current/title-45/subtitle-A/subchapter-C/part-164/subpart-D/section-164.402) -- Breach is presumed unless a risk assessment shows low probability that PHI was compromised. Four-factor assessment applies even in non-production environments.
- [eCFR 45 CFR § 164.404 -- Notification to Individuals](https://www.ecfr.gov/current/title-45/subtitle-A/subchapter-C/part-164/subpart-D/section-164.404) -- 60-day clock from *discovery* of breach (not from deployment date).
- [HHS -- Breach Notification Rule overview](https://www.hhs.gov/hipaa/for-professionals/breach-notification/index.html) -- "Unsecured PHI" does not distinguish dev vs prod; any real PHI loaded anywhere unsecured counts.
- [HHS -- Cloud Computing Guidance](https://www.hhs.gov/hipaa/for-professionals/special-topics/health-information-technology/cloud-computing/index.html) -- Even encrypted ePHI stored by a CSP without a BAA is a violation.
- [Cal. Civ. Code § 56.35](https://leginfo.legislature.ca.gov/faces/codes_displaySection.xhtml?lawCode=CIV&sectionNum=56.35) -- CMIA compensatory + up to $3,000 punitive + $1,000 attorney fees per patient. Applies regardless of environment label.
- [Cal. Civ. Code § 56.36](https://leginfo.legislature.ca.gov/faces/codes_displaySection.xhtml?sectionNum=56.36.&lawCode=CIV) -- Additional administrative penalties and private right of action for negligent release.
- [TruffleHog](https://github.com/trufflesecurity/trufflehog) -- `trufflehog git file://.` scans full git history for 800+ credential patterns including connection strings, AWS/Azure keys that would reveal undocumented deployments.
- [TruffleHog -- Finding Deleted & Private Commits](https://trufflesecurity.com/blog/trufflehog-now-finds-all-deleted-and-private-commits-on-github) -- Detects secrets in force-pushed commits.
- [Thoughtworks -- Lessons from inheriting a codebase](https://www.thoughtworks.com/insights/blog/lessons-inheriting-another-team-s-codebase) -- Advises independent reproduction of deployments to catch undocumented infrastructure.
- [canro91 -- 8 Proven Steps to Take Over a Legacy Codebase](https://canro91.github.io/2025/11/13/LegacyTakeOver/) -- Check CI/CD history, deployment logs, DNS records.
- [Claranet -- Cloud Services Enumeration](https://www.claranet.com/us/blog/2020-10-12-cloud-services-enumeration-aws-azure-and-gcp) -- Tools (ScoutSuite, Pacu, ROADtools) to enumerate what discovered cloud credentials reach.

## Likely answer

**HIGH confidence**: App has never been deployed outside localhost with real PHI. README explicitly says so; env files still localhost; no deployment credentials in source; initial-commit timing matches a clean handover import.

**Assumptions that would flip this:**
- Discovery of an Azure / AWS / DigitalOcean / Heroku account paid to the prior developer's email that shows workloads in 2025 or earlier.
- Access logs to a non-Gesco domain that point at any server.
- Any email from Gesco staff mentioning "the staging instance" or similar.

## Recommended approach to obtain a definitive answer

- Run `trufflehog git file://.` on the full repo including all branches; look for deployment-hinting secrets.
- `git log --all --source --pretty=format:"%H %an %ae %s" | grep -iE "prod|azure|aws|heroku|deploy"`.
- Ask Gesco finance: any past cloud invoices paid to the prior developer's email (`arajeev@gesco.com`) or to Gesco directly in the 18 months before 2026-04-08.
- DNS-check any domains Gesco owns; check WHOIS for admin contact; look for A-records to unfamiliar IPs.
- Ask prior developer directly: "Was this ever deployed outside localhost? Were any real patients, doctors, or case records ever entered?"

## Default assumption if unanswered

No prior deployment with real PHI. Treat repo as greenfield for breach-notification purposes.

## Risk if assumption is wrong

Undisclosed past breach of unsecured PHI triggers HIPAA 60-day notification obligation from the moment Gesco *discovers* it (not when it happened). CMIA exposure per CA patient record. Best-case remediation: document discovery, run risk assessment, possibly self-report to HHS OCR via the breach portal.
