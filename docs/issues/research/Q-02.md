# Q2: AppointmentStatus Lookup Table vs Enum -- Research

**Severity**: Medium (blocks DB cleanup and FEAT-01 design)
**Status**: Open (verified 2026-04-17)
**Source files**:
- `src/HealthcareSupport.CaseEvaluation.Domain.Shared/Enums/AppointmentStatusType.cs` (13-value enum)
- `src/HealthcareSupport.CaseEvaluation.Domain/AppointmentStatuses/AppointmentStatus.cs` (entity with Name field)
- `src/HealthcareSupport.CaseEvaluation.Application/AppointmentStatuses/AppointmentStatusesAppService.cs` (full CRUD)
- `src/HealthcareSupport.CaseEvaluation.EntityFrameworkCore/EntityFrameworkCore/CaseEvaluationDbContext.cs` (host-scoped configuration)

---

## Current state (verified 2026-04-17)

- Two parallel representations.
  - `AppointmentStatusType` enum: 13 values (1-13), used by `Appointment.AppointmentStatus` column.
  - `AppointmentStatus` entity: `Name` string, host-scoped, full CRUD.
- The enum and the table are not linked in code or migration. No FK binds Appointment to the table; no seeding keeps them in sync.
- The DbContext configures the table under `if (builder.IsHostDatabase())` -- it lives in host DB only.
- The CRUD AppService is proxy-generated into Angular but no component consumes it.

## Official documentation

- [ABP Entities -- best practices](https://abp.io/docs/9.0/framework/architecture/best-practices/entities) -- Enums for bounded, code-coupled type sets; lookup tables for user-editable reference data.
- [ABP Localization -- Enum convention](https://github.com/abpframework/abp/blob/dev/docs/en/framework/fundamentals/localization.md) -- `Enum:{EnumName}.{Value}` keys auto-localize enum display across Razor/Angular/MVC without a lookup table.
- [ABP Angular Localization pipe](https://abp.io/docs/latest/framework/ui/angular/localization) -- `| abpLocalization` resolves `Enum:AppointmentStatusType.Pending` identically to table-backed names.
- [EF Core #12248 -- Enum as lookup table](https://github.com/dotnet/efcore/issues/12248) -- EF team's stance: hybrid pattern (enum column + seeded lookup table for FK integrity) is the recommended middle ground when a DB-level CHECK is desired.
- [Stack Overflow -- EF Core enum + seed data](https://stackoverflow.com/questions/50375357/how-to-create-a-table-corresponding-to-enum-in-ef-core-code-first) -- Canonical hybrid: enum in C#, seeded table in DB, `HasData` keeps them in sync.
- [Ardalis SmartEnum](https://github.com/ardalis/SmartEnum) -- Strongly-typed enum alternative with behaviour; a middle option between raw enum and full lookup entity.

## Community findings

- [Software Engineering SE -- DB enumerations vs app enumerations vs lookup tables](https://softwareengineering.stackexchange.com/questions/378321/db-enumerations-vs-app-enumerations-vs-lookup-tables) -- Consensus: enum for closed, code-coupled sets; lookup when end users need to edit.
- [Software Engineering SE -- Enforce correspondence between enum and lookup](https://softwareengineering.stackexchange.com/questions/435403/enforce-correspondence-between-enum-and-lookup-table) -- Seed the lookup from the enum; FK still works because IDs match.
- [Stack Overflow -- Code-First enumerations into lookup tables](https://stackoverflow.com/questions/31372549/code-first-enumerations-put-into-lookup-tables) -- Migration patterns for maintaining enum-table parity.
- [Reddit r/dotnet -- Type/Status tables in EF Core](https://www.reddit.com/r/dotnet/comments/15tlfv8/how_do_you_typically_approach_sql_tables_like/) -- Practitioner consensus: status values driven by code logic belong in enums.
- [Stack Overflow -- Auto-localized enums + datatables in ABP](https://stackoverflow.com/questions/71616659/how-do-i-get-both-auto-localized-select-list-enums-and-datatables-in-abp-framewo) -- ABP community solution uses enum + `Enum:` localization keys for dropdowns and grids.
- [Stack Overflow -- DDD enum-like entities](https://stackoverflow.com/questions/2237420/ddd-enum-like-entities) -- If the "enum" participates in business rules, promote to SmartEnum/value object; if pure label, keep as enum.
- [timabell/ef-enum-to-lookup](https://github.com/timabell/ef-enum-to-lookup) -- Auto-generates lookup tables from enum references.

## Recommended approach

- **Source of truth: the enum** (HIGH confidence). Status participates in state-machine transitions, permission checks, and reporting; business staff should not be able to add "SuperPending" via CRUD and break guards. The 13 values are domain-coupled, not reference data.
- **Drop the `AppointmentStatus` entity**. It is orphaned -- no FK references it, no UI consumes it, no seed keeps it in sync. Delete entity + table + AppService + Angular proxy. Cleaner than keeping a dead branch with live CRUD endpoints.
- **If kept, demote to read-only label catalog only** -- seed rows with `Id = (int)enum` and `Name = "Enum:AppointmentStatusType.Pending"` (the localization key, not a display string), and remove Create/Update/Delete from the AppService. But ABP's enum-localization convention already does this without the table, so keeping adds sync burden for zero benefit.
- **Deprecate CRUD endpoints first** -- if a future consumer integrates against the AppService before you delete it, the deletion becomes a breaking change.
- **Recommendation confidence: HIGH**. Remove the table; rely on the enum + `Enum:` localization keys.

## Gotchas / blockers

- Angular proxy generator has already produced `angular/src/app/proxy/appointment-statuses/`; grep that folder before deletion to find any consumer.
- Host-scoped table means all tenants share labels. If any tenant ever needs custom status names, the enum-only path blocks it -- that's a deliberate tradeoff worth recording as an ADR.
- If a future SQL report joins `Appointment.AppointmentStatus` to the lookup table on `(int)status == AppointmentStatus.Id`, deleting the table silently breaks the join. Check BI queries before removing.
- DbContext config, AppService, DTOs, permissions, and migration must all be removed together or the build breaks.
- Removal is a migration that drops a table -- plan for the `Down()` path in case of rollback.

## Open questions

- Was the `AppointmentStatus` entity created as part of an abandoned feature (intended to replace the enum) or as shadow metadata (labels/colours/ordering)? Git blame on the entity + migration would reveal intent.
- Do any stakeholders (reporting, BI, external integration) depend on the SQL table existing today?
- Does the business want per-tenant status customisation in future? If yes, the enum-only path blocks it but that may still be the right call today.
- Do `AppointmentType` and `AppointmentLanguage` follow the same zombie-lookup pattern? If yes, clean them up in the same sweep.
