**Severity**: Medium (orphan tech question)
**Status**: Open (verified 2026-04-17)

## Current state (verified 2026-04-17)

Google, Microsoft, Twitter all registered via `AddDynamicExternalLoginProviderOptions<...>()` in both modules:
```csharp
context.Services
    .AddDynamicExternalLoginProviderOptions<GoogleOptions>(GoogleDefaults.AuthenticationScheme, ...)
    .AddDynamicExternalLoginProviderOptions<MicrosoftAccountOptions>(MicrosoftAccountDefaults.AuthenticationScheme, ...)
    .AddDynamicExternalLoginProviderOptions<TwitterOptions>(TwitterDefaults.AuthenticationScheme, ...);
```

No provider is activated at runtime -- client IDs/secrets live in gitignored `appsettings.secrets.json`. This is ABP's default Account-module scaffolding, untouched. The question is: finish wiring (add credentials, test, enable) or remove entirely given HIPAA concerns.

## Official documentation

- [ABP Account Module (Pro)](https://abp.io/docs/latest/modules/account-pro) -- "The application startup template comes with Twitter, Google and Microsoft logins pre-installed." Credentials configured via Admin UI Settings; multi-tenant can override per tenant.
- [ABP Account Module -- Commercial](https://docs.abp.io/en/commercial/latest/modules/account) -- `WithDynamicOptions<>()` extension lets admin-UI-stored settings populate `FacebookOptions`/`GoogleOptions`/etc at request time rather than startup.
- [Microsoft -- Google external login](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/social/google-logins?view=aspnetcore-9.0) -- OAuth 2.0 consent screen setup, Client ID/Secret via Google Cloud Console.
- [Microsoft -- Microsoft Account external login](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/social/microsoft-logins?view=aspnetcore-9.0) -- Azure AD/Entra ID app registration for OAuth 2.0.
- [Twitter Developer -- OAuth 2.0](https://developer.x.com/en/docs/authentication/oauth-2-0) -- X (Twitter) still supports OAuth 2.0 PKCE; programmatic API now requires paid plan, but OAuth identity sign-in remains under Free tier (as of research date).
- [Google Cloud -- Identity Platform HIPAA Implementation Guide](https://cloud.google.com/security/compliance/hipaa/identity-platform) -- **CRITICAL**: "Any synchronization of PHI from external identity systems to Identity Platform is not recommended or supported." Google warns against sending PHI through identity products.
- [HHS -- Business Associates FAQ](https://www.hhs.gov/hipaa/for-professionals/faq/business-associates/index.html) -- BAAs required for any vendor creating, receiving, maintaining, or transmitting PHI on behalf of covered entity.

- [ABP Support #9034 -- Configure external providers on tenant level](https://abp.io/support/questions/9034/Configure-external-providers-on-tenant-level) -- per-tenant admin-UI configuration is the intended path, not `appsettings.secrets.json`.

- [MojoAuth -- Social login on high-security applications](https://mojoauth.com/ciam-qna/social-login-high-security-applications) -- "HIPAA regulations often demand more control over authentication and data security than social login can easily provide." Industry consensus.
- [Accountable HQ -- Patient Portal Authentication MFA/SSO HIPAA](https://www.accountablehq.com/post/patient-portal-authentication-mfa-sso-and-hipaa-compliant-best-practices) -- dedicated auth providers with BAAs (Okta, Auth0 with BAA, AWS Cognito HIPAA) are the defensible path.
- [Superface -- Twitter API Changes FAQ](https://dev.to/superface/twitter-api-changes-the-missing-faq-for-free-basic-access-2lil) -- post-2023 rebrand paid-tier changes impact data API not identity; but X's developer trust is a separate business risk.

## Recommended approach

**Short-term**: do nothing with credentials. Current state (registered, no secrets) means buttons don't appear on login page -- harmless. Document as intentional pending HIPAA review.

**Medium-term**: decide between two paths based on product direction:

1. **Remove** all three providers from `ConfigureExternalProviders` if patient-facing login is primary use case and social login is HIPAA liability. Staff use OpenIddict local accounts with enforced MFA.
2. **Keep Microsoft only** (not Google, not Twitter) if internal staff log in via corporate Entra ID. Entra ID is lowest-friction HIPAA-safe option because Microsoft offers BAAs for Entra ID + Microsoft 365 under Microsoft OPDA. Google and Twitter do not offer BAAs for general consumer OAuth.

**If keeping any provider**:
- Disclaimer on login page ("Social login shares your email and profile with [Provider]. Do not use for protected health information.").
- Audit OpenIddict claim mapping to ensure no PHI ends up in ID tokens.
- Execute any required BAAs.

## Gotchas / blockers

- Google, Microsoft (consumer), Twitter don't sign BAAs for consumer OAuth. Microsoft **Entra ID** (enterprise) CAN be BAA-covered under Microsoft OPDA. INFERENCE: authentication-only flows typically don't transmit PHI -- provider sees only email + display name -- but HHS OCR treats any third-party touching authentication as potentially in-scope.
- Twitter's OAuth 2.0 still works but X developer trust and product stability lower than pre-2023. Long-term reliability risk.
- INFERENCE: gitignored `appsettings.secrets.json` fine for local dev but not prod -- Azure Key Vault / AWS Secrets Manager / ABP Settings-stored-in-DB are prod options. Confirm whichever is already used for OpenIddict signing certificate.
- If providers stay registered but unconfigured, any admin entering partial credentials in Settings UI can inadvertently enable a broken login flow. INFERENCE: remove `AddDynamicExternalLoginProviderOptions<>` calls entirely rather than leaving dormant.

## Open questions

- Is this portal patient-facing, staff-facing, or both? Determines plausible identity providers. Workers' comp IME scheduling suggests mixed use.
- Has Gesco executed a BAA with any identity vendor? If yes, use that one; if no, safest path is local accounts + MFA.
- Does ABP Account Module UI expose social-login settings to tenant admins today? INFERENCE: yes by default -- removing the registrations is the durable fix.
