# ASP.NET / ABP Conventions -- Appointment Portal

- .NET 10 with ABP Commercial 10.0.2
- EF Core for data access; Riok.Mapperly for object mapping (NOT AutoMapper)
- Follow ABP module architecture: Domain.Shared -> Domain -> Application.Contracts -> Application -> EntityFrameworkCore -> HttpApi
- Never edit proxy/ (auto-generated by ABP)
- Adrian is new to ASP.NET/C# -- explain by comparing to Node.js/Express equivalents
- Always cite ABP docs or Microsoft docs when introducing new patterns

## AppService wiring

IMPORTANT: Every new AppService MUST carry `[RemoteService(IsEnabled = false)]` and extend
`CaseEvaluationAppService`. Omitting the attribute lets ABP auto-expose duplicate routes
alongside the manual controller, causing 500s on ambiguous route resolution.

## Manual controllers

Every AppService needs a paired controller:
- Extends `AbpController` and implements `I<Entity>AppService`
- Route: `[Route("api/app/<entity-plural>")]` (kebab-case plural)
- The controller is thin -- it delegates every method to the AppService; no logic lives here

## Mapperly mappers

IMPORTANT: Use Riok.Mapperly only. Never `ObjectMapper.Map<>` or AutoMapper.
- Mappers are `partial class` annotated `[Mapper(RequiredMappingStrategy = RequiredMappingStrategy.Target)]`
- Extend `MapperBase<TSource, TDest>` and register in `CaseEvaluationApplicationMappers.cs`
- Mapper source-gen runs at build time; missing member mappings are compile errors, not runtime errors

## Two DbContexts

- `CaseEvaluationDbContext` -- host database; wrap host-only entity configs with
  `if (builder.IsHostDatabase())` or they will run against tenant DBs and fail
- `CaseEvaluationTenantDbContext` -- per-tenant database; do not add host-only tables here
- `Patient` is NOT `IMultiTenant`; guard any cross-tenant query attempts at the service boundary
